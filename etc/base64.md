# Base64

## 한 줄 요약

> **바이너리 데이터를 어디서나 깨지지 않고 통과할 수 있는 안전한 텍스트(ASCII) 형태로 변환하는 인코딩 방식**

---

## 사용하는 이유

시스템마다 데이터를 처리하는 방식이 다르기 때문에, 모든 환경에서 통용되는 **"최소 공통 분모"** 형태가 필요합니다.

### 1. 텍스트만 허용하는 환경에서 바이너리 전송
- 이메일(SMTP), JSON, XML, URL 등은 원래 텍스트 기반 프로토콜
- 이미지, 파일 같은 바이너리를 그대로 넣으면 깨지거나 거부됨
- Base64로 인코딩하면 `A-Z`, `a-z`, `0-9`, `+`, `/` 64자만 사용하므로 어디서나 안전

### 2. 제어 문자/특수 문자 회피
- 바이너리에는 줄바꿈(`0x0A`), NULL(`0x00`) 같은 제어 문자가 섞여 있어 파서가 오작동할 수 있음
- Base64는 출력 가능한 ASCII만 사용해서 문제 없음

### 3. 문자 인코딩 차이로부터 안전
- UTF-8, EUC-KR 등 시스템별 인코딩 차이에 영향받지 않음

---

## 비유

**"국제 표준 컨테이너"** 같은 개념입니다.
안에 뭐가 들었든(이미지, 음악, 실행파일 등) 같은 규격 박스에 담으면 어떤 배, 트럭, 기차에 실어도 문제없이 운반되는 것과 같은 원리입니다.

---

## 흔한 사용처

| 용도 | 예시 |
|------|------|
| 이메일 첨부파일 | MIME 인코딩 |
| 웹 이미지 인라인 | `data:image/png;base64,iVBOR...` |
| API 토큰 | JWT의 헤더/페이로드 |
| HTTP 인증 | `Authorization: Basic dXNlcjpwYXNz` |
| 인증서/키 파일 | PEM 형식 |

---

## 동작 원리 (간단히)

- 바이너리 데이터를 **6비트 단위**로 쪼갬
- 각 6비트(0~63)를 64개 문자 중 하나로 매핑
- 3바이트(24비트) → 4문자(24비트)로 변환
- 길이가 3의 배수가 아니면 `=`로 패딩

```
입력: "Man" (3바이트)
ASCII: 77, 97, 110
2진수: 01001101 01100001 01101110
6비트 단위로 재배열: 010011 010110 000101 101110
10진수: 19, 22, 5, 46
Base64: T,  W,  F,  u
결과: "TWFu"
```

---

## 주의할 점

- **암호화가 아닙니다.** 누구나 디코딩 가능
  - → 비밀번호 같은 민감 정보를 "Base64로 숨긴다"는 건 보안이 아님
- **크기가 약 33% 증가** (3바이트 → 4문자)
- 성능이 중요하면 가능한 한 바이너리로 직접 전송하는 게 좋음
- Base64는 **"텍스트 채널을 통과해야만 할 때"** 쓰는 게 원칙

---

## 변종

- **Base64 URL-safe**: `+`, `/` 대신 `-`, `_` 사용 (URL에서 안전)
- **Base32**, **Base16(Hex)**: 사용 문자 집합이 더 제한적인 변종

---

## ❓ 남은 질문

1. 인코딩 후 크기가 왜 정확히 약 33% 늘어나는가?

   → **답:** 3바이트(24비트)를 4문자로 표현하므로 4/3배가 되어 약 33% 증가한다. 각 Base64 문자는 6비트만 담지만 1바이트(8비트)로 저장되어 2비트가 낭비되는 셈이다.

2. JWT는 왜 암호화가 아닌 Base64URL 인코딩을 쓰는가?

   → **답:** JWT의 헤더·페이로드는 기밀이 아니라 누구나 읽어도 되는 정보이고, 위변조 방지는 서명(signature)이 담당하기 때문이다. Base64URL은 URL·헤더에서 안전하게 전달하기 위한 인코딩일 뿐 기밀성을 제공하지 않는다.

3. URL-safe 변종이 `+`, `/`를 `-`, `_`로 바꾸는 이유는?

   → **답:** 표준 Base64의 `+`, `/`, `=`는 URL/쿼리스트링에서 각각 공백·경로구분자·예약문자로 오해되거나 퍼센트 인코딩이 필요해지기 때문이다. `-`, `_`로 치환하고 패딩 `=`를 생략하면 추가 이스케이프 없이 URL에 넣을 수 있다.
