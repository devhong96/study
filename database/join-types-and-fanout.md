# SQL JOIN — 논리적 종류와 fan-out (곱 → 필터로 이해하기)

> **한 줄 정의:** 모든 JOIN은 개념적으로 *"두 테이블의 **카티션 곱(CROSS)**을 만든 뒤 조건으로 **필터**한 것"*이다. 이 "곱 → 필터" 모델 하나로 INNER·OUTER·fan-out·ON vs WHERE가 전부 설명된다.
>
> *(1~5장 = 논리 층, 6장 = 물리 층. 슬로우 쿼리 진단·실행계획 읽기 심화는 별도 노트 예정.)*

> 관련 문서:
>
> - [db-index.md](db-index.md) — 조인 조건 컬럼에 인덱스가 있어야 조인이 빠르다 (물리 알고리즘과 직결)
> - [transaction-and-isolation.md](transaction-and-isolation.md) — 함께 DB 면접 핵심
> - [../jpa/persistence-context-and-n-plus-one.md](../jpa/persistence-context-and-n-plus-one.md) — fan-out은 JPA 컬렉션 fetch join의 `MultipleBagFetchException`·중복 엔티티 문제의 뿌리

## 1. 뿌리 — CROSS JOIN은 "곱"이다 (합집합 아님)

두 테이블 `member(3행)`, `orders(3행)`를 CROSS JOIN(`FROM member, orders`)하면 **3 × 3 = 9행**. 한쪽의 **각 행**을 다른 쪽 **모든 행**과 짝짓는 **카티션 곱**이다.

| 연산                      | 하는 일                                         | 결과                                  |
| ------------------------- | ----------------------------------------------- | ------------------------------------- |
| **UNION (합집합)**  | 두 결과를**세로로 쌓음** (같은 컬럼 구조) | 행이 더해짐 (3+3=6)                   |
| **CROSS JOIN (곱)** | 모든 행 × 모든 행                              | 옆으로 컬럼 붙고 행이 곱해짐 (3×3=9) |

> ⚠️ **흔한 오개념:** "CROSS = 합집합"은 틀림. 합집합은 `{a,b} ∪ {b,c}`처럼 **더하기**, 카티션 곱은 `{a,b} × {c,d}`처럼 **곱하기**. 그래서 6이 아니라 9.

---

## 2. INNER JOIN = 곱 → 필터

`INNER JOIN ... ON member.id = orders.member_id`는 위 9행 카티션 곱에서 **ON 조건을 만족하는 행만 남긴 것**.

```
CROSS(9행)  --  ON m.id = o.member_id 필터  -->  INNER(3행)
```

> **정의:** INNER JOIN = 두 테이블의 카티션 곱에서 ON 조건을 만족하는 행만 남긴 것.

> ⚠️ **"곱 → 필터"는 명세이지 실행 방식이 아니다** (아주 중요)
>
> | 층                    | 질문                          | "곱 → 필터"인가                    |
> | --------------------- | ----------------------------- | ---------------------------------- |
> | **논리(semantics)** | 결과가 **무엇이어야** 하나 | **그렇다** — 표준의 정의     |
> | **물리(execution)** | 실제로 **어떻게** 계산하나 | **아니다** — 절대 안 그럼    |
>
> `member` 100만 × `orders` 1000만이면 곱은 10조 행. DB가 이걸 만들 리 없다. 옵티마이저는 **결과만 같으면 계산 방법은 자유**라서 NL·Hash 같은 알고리즘(6장)으로 곱을 **회피**한다.
>
> - 정확한 표현: *"조인은 카티션 곱을 ON으로 필터한 것과 **같은 결과**를 낸다. 단, 그렇게 **계산하지는** 않는다."*
> - 같은 구조가 5.1의 논리적 실행 순서에도 적용된다 — 그것도 **의미론적 순서**이지 실제 처리 순서가 아니다(→ predicate pushdown).
> - 비유: 컴파일러가 `x * 2`를 `x << 1`로 바꾸는 것. **명세는 곱셈, 실행은 시프트.**

- "교집합" 비유는 벤다이어그램용 직관일 뿐 **반쪽짜리**다. 결과 컬럼은 양쪽을 **가로로** 붙인 것이고, 1:N이면 결과 행이 **어느 원본 테이블보다 많아질 수** 있다(순수 교집합은 원소가 늘 수 없음). → 이게 곱→필터의 흔적.

## 3. fan-out — 1:N 조인의 행 증폭과 집계 오염 ⭐

### 3.1 정의

> **fan-out** = 조인에서 **왼쪽 한 행이 오른쪽 여러 행과 매칭돼 결과 행이 증식하는 현상**.
> 조인 결과의 카디널리티가 원본 테이블보다 커지는 것 자체를 가리킨다.

**형식화 — 조인 결과 행 수는 "덧셈"이 아니라 "각 행의 매칭 수 총합"이다.**

```
INNER JOIN 결과 행 수 = Σ (왼쪽 각 행이 매칭한 오른쪽 행 수)
                       r∈L
```

- 회원 1명 · 주문 100건 → 1행이 100번 복제 → 결과 **100행**
- 회원 3명이 각각 2/0/5건 → 결과 = 2+0+5 = **7행** (회원 수 3도, 주문 수 7도 아닌 "매칭 수 총합" — 여기선 우연히 주문 수와 같지만, 주문에 매칭 안 되는 회원이 있으면 INNER에선 0으로 빠진다)

**관계 방향에 따라 fan-out 여부가 갈린다:**

| 조인 방향                                   | 왼쪽 1행당 매칭 | 결과                        |
| ------------------------------------------- | --------------- | --------------------------- |
| **N:1** (`orders` → `member`)         | 정확히 1행      | 행 수 **유지** (fan-out 없음) |
| **1:N** (`member` → `orders`)         | 0~N행           | 행 수 **증식** ⭐ fan-out    |
| **1:N × 1:N** (member → orders, addresses) | N행 × M행      | **곱**으로 폭발 (3.2)       |

> N:1이 안전한 건 **오른쪽 조인 컬럼이 UNIQUE/PK**여서 매칭이 최대 1행이기 때문이다. FK가 아무 컬럼이나 가리키면 N:1처럼 보여도 fan-out이 난다. — *"1:N이라 위험"이 아니라 **"오른쪽이 유일하지 않아서" 위험**하다*가 더 정확한 표현.

**중요 — fan-out은 버그가 아니라 조인의 정상 동작이다.** "곱 → 필터"(2장)의 필연적 결과다.
문제가 되는 건 **그 증식된 행을 그대로 집계하거나(→ 3.2) 개수로 착각할 때**뿐이다.
`member JOIN orders`로 주문 목록을 뽑는 건 fan-out이 일어나도 **정상**이고 원하는 결과다.

<details>
<summary>용어 메모</summary>

- **fan-out은 SQL 표준 용어가 아니다** *(거의 확실)*. ANSI SQL 스펙에 정의된 절이 아니라 실무·옵티마이저 문헌에서 쓰는 표현이다. 같은 현상을 **row multiplication / join amplification / 카디널리티 증가**로도 부른다.
- 원래는 디지털 회로에서 **게이트 출력 하나가 구동하는 입력 개수**를 뜻한다 *(확실)*. 하나가 여럿으로 갈라져 나가는 그림이 같아 차용된 것으로 보인다 *(차용 경로 자체는 확인 안 됨)*.

</details>

### 3.2 왜 위험 — 집계가 뻥튀기된다

**서로 관계없는 1:N 가지 두 개**를 한 쿼리에서 조인하면 두 자식 집합이 서로 곱해진다.

```
        member(김철수)
        /            \
   orders(2건)    addresses(3건)   ← 둘은 서로 무관 (각각 member에만 매달림)
   5000, 3000     집/회사/부모님댁
```

```sql
SELECT SUM(o.amount)
FROM member m
JOIN orders    o ON m.id = o.member_id
JOIN addresses a ON m.id = a.member_id   -- 주문·배송지 둘 다 member에 1:N
WHERE m.id = 1;
```

주문 2건이 배송지 3건과 무차별로 곱해져 **6행** → 각 주문 금액이 3번씩 더해짐:

`SUM = 5000×3 + 3000×3 = 24000` (진짜 합 8000의 **×3**)

> 📌 **규칙:** 1:N 가지 두 개를 한 쿼리에서 집계하면, 한쪽 집계값은 **반대쪽 가지의 행 수만큼** 곱해진다. (배송지 5건이면 ×5 = 40000)

> ⚠️ **오해 주의 — "8000인데 왜 40000?"**
> 배송지가 5건이면 각 주문 행이 5번 복제돼 `SUM(o.amount) = 8000 × 5 = 40000`. 진짜 합 8000으로 **착각하기 딱 좋다**.
> - **배송지엔 amount가 없다.** 배송지는 금액에 기여하는 게 아니라 **주문 행을 복제하는 배수**로만 작용한다.
> - **뻥튀기 배수는 값 분포와 무관.** 주문이 `5000+3000`이든 `1+7999`든 합이 8000이면 결과는 똑같이 40000 (`1×5 + 7999×5 = 8000×5`). 개별 금액이 아니라 **(진짜 합) × (반대 가지 행 수)**만 결정한다.
> - 그래서 더 위험하다 — 정확히 N배로 **균일하게** 틀려 "그럴듯한 큰 수"로 보이고, 로그로는 눈치채기 어렵다.
> - **8000을 정확히 얻으려면** → 3.3의 pre-aggregation (스칼라 서브쿼리 / 파생 테이블).

### 3.3 어떻게 막나 — 조인/필터가 아니라 "집계 시점"

문제는 **곱해진 뒤에 집계**해서 생긴다. → **곱하기 전에 각 가지를 회원당 1행으로 미리 접는다(pre-aggregation).**

```sql
-- ① 스칼라 서브쿼리 (독립 실행 → 배송지와 안 곱해짐)
SELECT m.name,
       (SELECT SUM(amount) FROM orders WHERE member_id = m.id) AS total
FROM member m;

-- ② 파생 테이블 (미리 GROUP BY로 1행 접기 → 1×1 조인)
SELECT m.name, o.total, a.cnt
FROM member m
JOIN (SELECT member_id, SUM(amount) total FROM orders    GROUP BY member_id) o ON o.member_id = m.id
JOIN (SELECT member_id, COUNT(*)   cnt   FROM addresses GROUP BY member_id) a ON a.member_id = m.id;
```

> ❌ **"조인 방식/필터로 막는다"는 오답.** 배송지를 어떻게 조인·필터하든 1건이라도 남으면 주문 집계는 그만큼 여전히 오염된다. 조인 방식이 아니라 **집계하는 시점**의 문제.

## 4. OUTER JOIN — 안 맞는 행을 NULL로 살린다

INNER는 매칭 안 된 행을 **버린다**(주문 없는 박민수 탈락). OUTER는 살린다.

```
member LEFT JOIN orders ON m.id = o.member_id
| m.name | o.id | amount |
| 김철수 | 10   | 5000   |
| 김철수 | 11   | 3000   |
| 이영희 | 12   | 8000   |
| 박민수 | NULL | NULL   |  ← 매칭 없어도 왼쪽은 전부 남고, 오른쪽은 NULL
```

> **정의:** LEFT JOIN = INNER 결과 + **왼쪽 테이블에서 매칭 안 된 행 전부**(오른쪽은 NULL). "**왼쪽은 무조건 전부 나온다**"가 보장.
>
> - **핵심 주의:** 왼쪽은 조건으로 필터되지 않는다. ON은 "오른쪽에서 뭘 붙일지"만 결정.
> - RIGHT = 방향만 반대, FULL = 양쪽 다 보존. (CROSS는 조건 없는 전체 곱.)

## 5. 논리적 실행 순서 & ON vs WHERE

### 5.1 작성 순서 ≠ 처리 순서

SQL은 `SELECT`부터 쓰지만 **처리는 `FROM`부터**. (이건 옵티마이저가 실제 물리 실행을 재배치하기 전의 **의미론적 순서**다.)

| 순서 | 절                              | 하는 일                                                           |
| ---- | ------------------------------- | ----------------------------------------------------------------- |
| 1    | **FROM / JOIN**           | 테이블 결합(곱)                                                   |
| 2    | **ON**                    | 조인 조건 평가 →**매칭 + outer join의 NULL 채움이 여기서** |
| 3    | **WHERE**                 | 완성된 행 필터                                                    |
| 4    | GROUP BY                        | 그룹핑                                                            |
| 5    | HAVING                          | 그룹 필터                                                         |
| 6    | **SELECT**                | 컬럼·별칭 생성                                                   |
| 7    | DISTINCT / 8 ORDER BY / 9 LIMIT |                                                                   |

- 부산물: **WHERE에서 SELECT 별칭 못 씀**(3 < 6), **ORDER BY에선 됨**(8 > 6).

### 5.2 ON vs WHERE — 정의

둘 다 "조건을 쓰는 자리"지만 **역할 자체가 다르다.**

> - **ON = 조인 술어(join predicate).** *"두 테이블의 **어떤 행끼리 짝지을지**"*를 결정한다. **조인 연산의 일부**이며, 짝짓기(2단계)와 outer join의 NULL 채움이 여기서 일어난다.
> - **WHERE = 필터 술어(filter predicate).** *"**이미 만들어진 결과 행** 중 무엇을 남길지"*를 결정한다. 조인이 **끝난 뒤**(3단계) 집합을 걸러낸다.

한 줄 요약: **ON은 "짝을 만드는 규칙", WHERE는 "만들어진 행을 거르는 체".**

부속 사실:
- `CROSS JOIN`엔 ON이 없다 — 짝짓기 규칙이 "전부"라서.
- ON은 **그 시점까지 조인된 테이블만** 참조 가능, WHERE는 FROM 절 전체를 참조 가능.
- **INNER JOIN에선 둘이 사실상 같다** (결과 동일 — 옵티마이저가 어차피 재배치). 차이는 **NULL 행이 생기는 OUTER JOIN에서만** 드러난다. ↓

```sql
-- WHERE 버전: 조인 후 필터 → 박민수의 NULL 행 제거 → LEFT가 INNER로 변질
LEFT JOIN orders o ON m.id = o.member_id
WHERE o.amount >= 3000        -- NULL >= 3000 = UNKNOWN → 컷

-- ON 버전: 매칭 단계에서 필터 → 왼쪽(박민수) 보존, 안 맞으면 NULL
LEFT JOIN orders o ON m.id = o.member_id AND o.amount >= 3000
```

|                    | ON                        | WHERE                 |
| ------------------ | ------------------------- | --------------------- |
| 시점               | 조인 매칭 중(2)           | 조인 후(3)            |
| 역할               | 어떤 오른쪽 행을 붙일지   | 완성된 결과 필터      |
| outer join 왼쪽 행 | **보존**(NULL 가능) | NULL 행**제거** |

> 실무 감각: outer join에서 **오른쪽 테이블 조건은 ON에**, **왼쪽(보존할) 테이블 조건은 WHERE에**. (INNER에선 ON/WHERE가 사실상 동일 — 차이는 NULL 행이 있는 OUTER에서만 의미.)
> `NULL` 비교가 `UNKNOWN`이 되는 건 SQL **3값 논리**(TRUE/FALSE/UNKNOWN) 때문. WHERE는 TRUE만 통과시킴.

## 6. 물리 조인 알고리즘 — 실제로는 이렇게 계산한다

논리 조인 하나를 수행하는 방법은 **세 가지**. 옵티마이저가 비용을 보고 고른다.

### 6.1 Nested Loop Join (NL) — "하나씩 찾아본다"

```
for outer 각 행:
    inner에서 매칭 행 찾기
```

승부는 **"inner에서 찾기"의 비용**에서 갈린다.

| inner 조인 컬럼 인덱스 | 안쪽 1회 비용 | 총 비용                       |
| ---------------------- | ------------- | ----------------------------- |
| **없음**         | 풀스캔 (N행)  | outer × inner ← **재앙** |
| **있음**         | 인덱스 탐색   | outer × log(inner) ← 실용    |

> 📌 **면접·실무 핵심 한 줄:** **NL 조인은 inner 테이블의 조인 컬럼에 인덱스가 없으면 죽는다.**
> "조인이 느려요"의 압도적 다수가 이것. → [db-index.md](db-index.md)와 여기서 만난다.

- **강점:** outer가 작을 때 최강. **비등치 조인(`<`, `BETWEEN`)이 가능한 유일한 알고리즘.**
- **약점:** 양쪽 다 크면 폭사. → 옵티마이저는 **작은 쪽을 outer(구동 테이블)로** 두려 한다.
- 중간 결과(곱)를 **저장하지 않는다** — 짝 하나씩 검사하고 버림. 그래서 **문제는 메모리가 아니라 시간**(비교 횟수).

### 6.2 Hash Join — "해시테이블 만들고 던진다"

```
① Build : 작은 쪽을 읽어 조인키로 해시테이블 생성 (메모리)
② Probe : 큰 쪽을 흘리며 해시테이블 조회 → O(1)
```

각 테이블을 **한 번씩만** 읽는다. **인덱스가 없어도 빠르다.**

- **강점:** 대용량 × 대용량, 인덱스 없는 조인. 배치·집계성 쿼리의 기본기.
- **약점:** ① **등치 조인(`=`)만 가능** — 해시는 "같다"만 판별할 수 있어 `<`는 불가. ② 해시테이블이 메모리를 넘으면 디스크로 흘러(spill) 급격히 느려짐.
- MySQL은 **8.0.18부터** 지원 *(거의 확실)*. 그전엔 없어서 NL로만 버텼고, 그게 "MySQL은 조인이 약하다"는 말의 배경.

### 6.3 Sort-Merge Join — "정렬해서 나란히 훑는다"

양쪽을 조인키로 정렬한 뒤 두 포인터로 전진하며 매칭.

- **강점:** 인덱스 덕에 **이미 정렬돼 있으면** 정렬 비용 0. 초대용량에 안정적.
- **약점:** 정렬 비용.
- **MySQL엔 없다** *(확실)*. Oracle·PostgreSQL엔 있음 → **MySQL만 쓴다면 6.1·6.2만 알면 된다.**

### 6.4 정리표

|                | 언제 선택되나                      | 인덱스 의존       | 비등치 조인 |
| -------------- | ---------------------------------- | ----------------- | ----------- |
| **NL**   | outer가 작음 + inner에 인덱스      | **절대적** | ✅          |
| **Hash** | 양쪽 크고 인덱스 없음, 등치 조건   | 없어도 됨         | ❌          |
| **Merge**| 양쪽 크고 이미 정렬됨 (MySQL 없음) | 정렬에 도움       | ❌          |

**옵티마이저가 보는 것:** 통계정보로 추정한 **행 수(카디널리티)**, **인덱스 유무**, **가용 메모리**.
→ 통계가 낡으면 엉뚱한 걸 고른다 (`ANALYZE TABLE`).

### 6.5 실행계획에서 무엇을 보나

**PostgreSQL**은 이름을 그대로 찍는다: `Nested Loop` / `Hash Join` / `Merge Join`.
**MySQL** 전통 `EXPLAIN`은 알고리즘 이름을 안 찍는다. 대신:

| 볼 곳          | 위험 신호                                                                          |
| -------------- | ---------------------------------------------------------------------------------- |
| `type`       | **`ALL`** = 풀스캔. 조인 대상 테이블에 뜨면 적신호                     |
| `key`        | **`NULL`** = 인덱스 안 씀                                              |
| `Extra`      | `Using join buffer (Block Nested Loop / hash join)` = **인덱스 못 써서 우회** |
| `rows`       | 추정 행 수 — 실제와 크게 다르면 통계 문제                                         |

> 알고리즘 이름을 직접 보려면 `EXPLAIN FORMAT=TREE` *(8.0.16 무렵부터 — 정확한 버전 확인 안 함)* 또는 `EXPLAIN ANALYZE`.

> 🗣️ **면접 답변 템플릿:** "조인이 느리면 실행계획부터 봅니다. MySQL이면 `type`이 `ALL`인지, `Extra`에 join buffer가 떴는지 확인해요. NL 조인은 inner 테이블 조인 컬럼에 인덱스가 없으면 outer 행마다 풀스캔이라 급격히 느려지거든요. 그래서 보통 **조인 컬럼(FK)에 인덱스**를 걸어 해결합니다."

---

> ## 📌 핵심 요약
>
> 모든 조인은 **의미론적으로** **곱(CROSS) → 필터(ON)** — 단 실제로 곱을 만들지는 않는다. INNER는 매칭만 남기고, OUTER는 안 맞는 쪽을 **NULL로 살린다**. 1:N 가지 두 개를 한 쿼리에서 집계하면 **fan-out**으로 집계값이 반대 가지 행 수만큼 뻥튀기 → **조인 전에 미리 집계(pre-aggregation)**로 막는다. ON은 조인 시점(2)·WHERE는 조인 후(3)라, outer join에서 오른쪽 조건을 WHERE에 두면 NULL 행이 제거돼 INNER로 변질된다. 물리적으로는 **NL / Hash / Merge**로 실행되며, **NL은 inner 조인 컬럼 인덱스가 생명**이다.
