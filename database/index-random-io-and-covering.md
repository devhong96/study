# 인덱스의 랜덤 I/O · 2번 탐색 · 커버링 · MRR

> **한 줄 정의:** 보조 인덱스로 행을 찾으면 *"인덱스 정렬 순서(컬럼순)와 테이블 저장 순서(PK순)가 어긋나"* 행을 가지러 갈 때 디스크 여기저기로 점프하는 **random I/O**가 생긴다. 이게 인덱스의 진짜 비용이고, **커버링 인덱스**로 테이블 밖에 안 나가면 사라진다.

## 1. 디스크 읽기: 순차 vs 랜덤

디스크(와 그걸 캐싱하는 버퍼 풀)는 데이터를 **페이지(블록, 보통 4~16KB)** 단위로 통째 읽는다. 그래서 읽기에 두 종류가 있다.

| 종류 | 동작 | 비용 |
|------|------|------|
| **순차 읽기 (sequential I/O)** | 디스크에서 **붙어 있는** 페이지를 연달아 읽음 (0→1→2번) | 쌈 |
| **랜덤 읽기 (random I/O)** | 매번 **흩어진** 페이지로 점프 (87→3→512번) | 비쌈 |

- 왜 랜덤이 느린가: 다음 읽을 위치를 매번 새로 찾아가야 함(HDD는 헤드 이동, SSD/메모리도 prefetch·캐시 지역성을 못 살림).
- 옵티마이저는 보통 **random ≈ sequential × 4** 정도로 비용을 매긴다. *(엔진·버전·스토리지에 따라 다른 추정치 — 확실치 않은 구체 수치)*

## 2. 왜 보조 인덱스 조회가 random I/O인가 (핵심)

전제 — InnoDB 기준:
- **클러스터형 인덱스(= PK 인덱스)**: 테이블 데이터 자체가 **PK 순서로 정렬·저장**됨. leaf에 **행 전체**가 들어있다.
  ```
  페이지1: id 1~100   페이지2: id 101~200   페이지3: id 201~300  ...
  ```
- **보조 인덱스**: PK 아닌 컬럼 기준으로 정렬된 **별도 B+tree**. leaf에는 **`(컬럼값 + PK)` 둘만** 있고 행 전체는 없다.
  ```
  abe@... → id 8921    amy@... → id 12    ann@... → id 543   ...   (email 알파벳순)
  ```

`WHERE email LIKE 'a%'` 실행:
1. email 인덱스 leaf를 따라 `abe, amy, ann...`을 **순차로** 읽음 → PK `8921, 12, 543...` 획득. *(여기까진 leaf가 linked list라 sequential)*
2. 이 PK들로 **행 전체를 가지러 테이블(클러스터형)로** 감 = **2번 탐색(back-to-table)**.
3. 그런데 PK가 **email 순서**로 나왔는데 테이블은 **PK 순서**로 놓여 있음 → `8921`(페이지90) → `12`(페이지1) → `543`(페이지6)... **여기저기 점프 = random I/O.**

> **두 정렬 순서의 어긋남**이 random I/O의 근본 원인. "논리적으로는 PK 리스트를 차례차례 처리"하지만, "물리적으로는 매번 흩어진 페이지로 점프" → 차례대로 방문 ≠ sequential I/O.

> **"그럼 인덱스를 아예 PK순으로 정렬해두면 안 되나?"** → 그러면 email 검색 때 정렬이 깨져 **다시 풀스캔**이 된다 = 인덱스의 존재 이유(email 빠른 검색) 소멸. 핵심 원리: **한 묶음의 데이터는 "한 가지 순서로만" 정렬할 수 있다.** email순으로 빨리 찾는 대가로 테이블(PK순)과 어긋나 random이 생기는 것 — 피할 수 없는 트레이드오프다. (인덱스 *전체*는 못 바꿔도, **이번 쿼리 결과 PK만** PK순 재정렬하는 우회가 → 3장 MRR.)
>
> 참고로 2번 탐색의 두 단계 중 **느린 건 ②**다: ① email로 인덱스에서 PK 찾기 = 인덱스가 email순 정렬이라 B+tree 탐색 + leaf 순차 → **빠름**. ② 그 PK로 테이블에서 행 찾기 = PK가 email순으로 나와 테이블을 점프 → **느림(random)**.

## 3. MRR (Multi-Range Read) — random을 sequential에 가깝게

인덱스가 준 PK 리스트(`8921, 12, 543...`)를 **테이블 가기 전에 PK 순서로 재정렬**한 뒤 가지러 간다.

```
인덱스가 준 순서(email순):  8921, 12, 543, 300, ...
        ↓ 결과 PK를 PK순으로 재정렬   (★ 인덱스 자체가 아니라 "이번 쿼리 결과"를 정렬)
테이블 갈 순서(PK순):       12, 300, 543, 8921, ...
```

- 효과: PK 오름차순 → 디스크를 **한 방향으로 훑음** → random이 sequential에 가까워짐. 가까운 PK들이 같은 페이지에 모여 **페이지 읽기 횟수 자체도 감소**.
- 한계: **"테이블 밖으로 나가는 행위 자체"는 못 없앤다.** 덜 아프게 할 뿐. → 교차점(4장)을 **뒤로 미뤄줄 뿐 근본 해결은 아님**.

## 4. 풀스캔 역전 — 선택도(행 기준!)

가져올 행이 많으면 옵티마이저는 인덱스를 버리고 **풀스캔(테이블 통째 순차 읽기)**을 택한다.

- **교차점 ≈ 선택도 20~25%** (매칭 행 ÷ 전체 행). *대략값 — 데이터·엔진에 따라 다름.*
- 왜: ① random 단가가 sequential의 몇 배라, **25%를 랜덤하게 읽는 비용 ≈ 100%를 순차로 읽는 비용**. ② 매칭 행이 많으면 그 행들이 **거의 모든 페이지에 퍼져** 있어 어차피 테이블 대부분을 만짐 → "어차피 다 만질 거면 인덱스 트리·재정렬·점프 오버헤드 없이 처음부터 쭉 읽자".

> ⚠️ **자주 헷갈리는 두 축 (서로 무관!)**
>
> | 축 | 무엇 | 정해지는 시점 |
> |----|------|--------------|
> | **A. 행 (선택도, "30%")** | 이 쿼리가 **몇 개의 행**을 가져오나 (WHERE 조건에 맞는 행 ÷ 전체) | 쿼리 실행 시 |
> | **B. 열 (커버링, 폭)** | 가져올 **컬럼**이 인덱스 안에 다 있나 | 인덱스 설계(DDL) 시 |
>
> "30% 가져올 바엔 풀스캔"은 **축 A(행)** 얘기. 쿼리는 조건에 맞는 행만 가져온다 — **다 가져오는 게 아니다.**

## 5. 인덱스 = "행 수는 같고 컬럼만 추린 날씬한 사본"

흔한 오해: "인덱스는 행이 몇 개 없다" → **틀림.**

- 보조 인덱스 엔트리 수 = **테이블 행 수와 동일**(행 하나당 인덱스 칸 하나, 일반 비-유니크 기준). 100만 행이면 인덱스도 100만 엔트리.
- 인덱스가 "작다"는 건 **개수가 적은 게 아니라 폭이 좁다**는 뜻.
  ```
  테이블 행 1개:   id|email|name|age|address|phone|...   (뚱뚱)
  인덱스 엔트리 1개: email|id                            (날씬)
  ```
- 인덱스가 빠른 이유 = 적어서가 아니라 **날씬 → 한 페이지에 많이 들어감 → fanout↑ → 트리 얕음 → 디스크 읽기 횟수↓.**

## 6. 커버링 인덱스 — random read 자체를 없애기

문제의 뿌리가 "인덱스에 원하는 데이터가 다 없어서 **테이블 밖으로 나가야 함**"이라면, **필요한 컬럼이 전부 인덱스 안에 있으면** 나갈 필요가 없다 → 2번 탐색·random I/O **소멸**(index-only scan).

**커버링 여부는 데이터를 안 뒤져도 안다 — 순전히 구조 비교:**
```
쿼리가 필요한 컬럼 (SELECT + WHERE + ORDER BY)  ⊆  인덱스가 담은 컬럼 (인덱스 컬럼 + PK)
```
- "이 인덱스가 어떤 **컬럼**을 담나" = 스키마(CREATE INDEX 시 기록) → **미리 안다(구조)**.
- "특정 행의 **값**이 조건에 맞나" = 실행하며 안다(데이터).
- → 옵티마이저는 **실행 전 플래닝 단계**에서 커버링 여부를 판단.

### 커버링이면 4장의 "선택도 역전(~25%)"이 깨진다

20~25% 역전점은 **random I/O가 만든 숫자**(4장 이유 ①)다. 커버링은 그 random이 **없으니** 역전 논리가 사실상 무력화 → **선택도와 거의 무관하게 인덱스(index-only scan)가 유리.**

- 극단으로 **100%를 읽어도**: `인덱스 전체 순차 읽기(index full scan)` vs `테이블 전체 순차 읽기(full table scan)` — **둘 다 random 없는 순차**라 승부는 이제 **크기(읽을 페이지 수)**로 갈린다. 인덱스가 **날씬**(5장)해 페이지를 적게 읽으므로 인덱스가 이긴다.
- ⚠️ **절대불변은 아니다** — 커버링 인덱스가 컬럼을 너무 많이 품어 테이블만큼 뚱뚱해지면 이 이점이 줄어든다. 최종 판단은 옵티마이저의 비용 비교(통계 기반).

> 정리: **비-커버링** → 선택도 ~25%↑면 풀스캔 역전 / **커버링** → 역전점이 사실상 사라져 거의 항상 인덱스.

## 7. WHERE 컬럼 vs SELECT 컬럼 — 인덱스에서 역할이 다르다

| 인덱스가 하는 일 | 관여 컬럼 | 없으면 |
|---|---|---|
| **걸러내기 (어떤 행을 고를까)** | **WHERE / JOIN / ORDER BY** 조건 컬럼 | 인덱스로 못 거름 → **풀스캔** |
| **테이블 밖 안 나가기 (커버링)** | **SELECT** 등 가져올 컬럼 | 테이블로 나감 → 2번 탐색(random I/O) |

세 경우로 정리 (`country`에 인덱스 있음 가정):

```
① SELECT country WHERE country='KR'
   WHERE(country) 인덱스 O → 거름 가능 / SELECT(country)도 인덱스 안 → 밖에 안 나감
   ✅ 커버링, 테이블 접근 0, 최速

② SELECT name WHERE country='KR'
   WHERE 인덱스 O → 거름 가능 / SELECT(name)은 인덱스 밖 → name 가지러 테이블로 (2번 탐색, random)
   매칭 적으면 굿, ~25%↑면 풀스캔에 역전

③ SELECT * WHERE age=30   (age엔 인덱스 없음)
   WHERE(age) 인덱스 X → 인덱스로 못 거름
   ⚠️ 풀스캔 (모든 행 읽어 age 확인)
```

## 📌 핵심 요약
보조 인덱스 leaf는 `(컬럼값+PK)`를 **컬럼순**으로, 테이블 행은 **PK순**으로 저장 → 두 순서가 어긋나 **2번 탐색 시 random I/O**가 발생한다. **MRR**은 결과 PK를 PK순 재정렬해 random을 sequential에 가깝게 하지만 "테이블로 나가는 것" 자체는 못 없앤다. 매칭 행이 많으면(선택도 ~20~25%↑) 어차피 거의 모든 페이지를 만지고 random 단가도 비싸 **풀스캔이 역전**한다. **WHERE 컬럼**은 걸러내기용(없으면 풀스캔), **SELECT 컬럼**은 커버링 여부를 결정 — 필요 컬럼이 전부 인덱스에 있으면(**커버링**) random read 자체가 사라진다.

## 🌱 심화 키워드
- **back-to-table / bookmark lookup** — 보조 인덱스 → 클러스터형 재탐색(2번 탐색)
- **MRR (Multi-Range Read)** — 결과 PK를 물리 순서로 정렬해 random→sequential
- **clustering factor** — 인덱스 순서와 물리 저장 순서의 일치도(낮을수록 random 심함)
- **covering index / index-only scan** — 테이블 접근 제거
- **selectivity / cardinality** — 행 기준 필터링 비율, 풀스캔 역전 판단 근거
- **ICP (Index Condition Pushdown)** — 테이블 가기 전 인덱스 단계에서 추가 조건 거르기

## ❓ 남은 질문
1. clustering factor가 나쁜(인덱스순≠물리순) 인덱스를 실무에서 어떻게 진단·개선하나? (PK 설계와의 관계)
   → **답:** 인덱스 순서와 테이블 물리(PK) 순서가 어긋날수록 나빠져 랜덤 I/O가 는다. Oracle은 `CLUSTERING_FACTOR` 지표로 진단하고(테이블 행수에 가까울수록 나쁨), 개선은 자주 범위조회하는 컬럼을 PK 순서에 맞추거나 커버링 인덱스로 테이블 재방문 자체를 없애는 것.
2. `EXPLAIN`에서 ①②③(커버링/2번탐색/풀스캔)을 각각 어떤 신호로 구분하나? (`type`, `Extra`의 `Using index` vs `Using where`)
   → **답:** `Extra`가 `Using index`면 커버링(테이블 재방문 없음), 인덱스 `ref/range`인데 `Using index`가 없으면 2번 탐색(인덱스→테이블), `type=ALL`이면 풀 테이블 스캔. `Using where`만 있고 인덱스 표시가 없으면 테이블에서 필터링한다는 신호.
3. UUID처럼 긴 PK를 쓰면 모든 보조 인덱스 leaf가 그 PK를 들고 있어 같이 뚱뚱해진다 — 얼마나 손해고, 정렬되지 않은 UUID가 페이지 분할에 주는 영향은? (→ [db-index.md](db-index.md) 남은 질문 1번과 연결)
   → **답:** 모든 보조 인덱스가 16B PK를 복제해 인덱스 크기·메모리·I/O가 커지고, 랜덤한 UUIDv4는 삽입 위치가 사방이라 B-tree **페이지 분할·단편화**를 유발한다. 그래서 시간순 정렬되는 UUIDv7/ULID나 auto-increment PK가 삽입에 유리하다.
