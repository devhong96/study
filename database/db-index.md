# DB 인덱스 (B+tree · 선택도 · 복합 인덱스)

> **한 줄 정의:** 인덱스는 *"정렬된 B+tree로 검색 시 **읽어야 할 디스크 페이지 수**를 O(log n)으로 줄이는"* 조회 가속 자료구조다. 대신 쓰기 비용·공간을 치르므로 **선별적으로** 건다.

## 1. 왜 빠른가 — 핵심은 "디스크 접근 횟수"

정렬 안 된 1000만 건을 다 훑으면 **O(n)**. 인덱스(B+tree)는 정렬+트리라 **O(log n)** → 약 23번 비교면 도달.

**진짜 포인트는 디스크 I/O다:**
- DB는 디스크에 있고 디스크 읽기가 비싸다. B+tree는 **한 노드(페이지)에 수백 개 키**가 들어가 넓고 얕다 → 1000만 건이어도 **트리 높이 3~4단계**.
- → 원하는 행에 **디스크 페이지 3~4번 읽기**로 도달. 풀스캔은 수만~수십만 페이지를 읽어야 함.

> 인덱스가 빠른 이유 = 비교 횟수(O(log n))보다 **"읽어야 할 디스크 페이지 수"가 압도적으로 줄어서.**

## 2. B-tree vs B+tree (DB 인덱스 표준은 B+tree)

> **트리 노드 용어**: **root**(맨 위) · **internal 노드(내부 노드)**=자식을 거느린 비-leaf, B+tree에선 **데이터 없이 길 안내(이정표)만** · **leaf 노드(잎)**=맨 아래, B+tree에선 **실제 데이터(행 포인터)**가 여기.

공통 토대: 둘 다 **균형 트리(balanced)**. BST는 한쪽으로 치우치면 O(n)으로 퇴화하지만, B/B+tree는 **모든 leaf가 같은 깊이**가 되도록 균형을 유지 → **항상 O(log n) 보장**. 한 노드가 자식을 수백 개(m-way) 가져 넓고 얕다.

**B-tree** — 데이터가 모든 노드에 흩어짐:
```
            [ 30•data | 60•data ]        ← internal 노드에도 데이터 있음
           /          |          \
    [10•d|20•d]  [40•d|50•d]  [70•d|80•d] ← leaf끼리 연결 없음
```
- 장점: `30` 찾으면 root에서 바로 데이터 발견(운 좋으면 일찍 끝남).
- 단점: `30~70 범위`는 트리를 위아래로 헤집어야(중위순회) → 랜덤 I/O 많음.

**B+tree** — 데이터는 leaf에만, internal은 이정표(키)만:
```
              [ 30 | 60 ]                 ← internal: 키(이정표)만, 데이터 X
             /     |     \
    [10|20]→[30|40|50]→[60|70|80]         ← leaf: 실제 키+데이터, 그리고
       └────────→─────────→┘                leaf끼리 linked list로 연결
```
- `30`은 internal(이정표)·leaf(실제)에 **둘 다** 등장 — internal은 길 안내용.
- 모든 검색이 **leaf까지** 감(경로 길이 균일).

| | **B-tree** | **B+tree** |
|--|--|--|
| 데이터(행 포인터) 위치 | **모든 노드**(internal+leaf)에 분산 | **leaf 노드에만** (internal은 키=이정표만) |
| leaf 연결 | 없음 | **leaf끼리 linked list로 연결** |
| 범위 검색 | 약함(트리 헤집음) | **강함**(leaf 따라 순차로 쭉) |

**DB가 B+tree를 쓰는 이유 (구조에서 직접 나옴):**
1. **트리가 더 얕다** — internal이 데이터 없이 키만 담아 fanout↑ → 높이↓ → **디스크 I/O 더 적음**.
2. **범위 검색 최적** — `BETWEEN`·`>`·`ORDER BY`·`LIKE '홍%'`에서 leaf **linked list를 순차로** 읽으면 됨.
3. **조회 성능 균일** — 모든 검색이 leaf까지 가므로 일관.

→ MySQL(InnoDB)·PostgreSQL 등 거의 모든 RDBMS 인덱스가 **B+tree**. (B-tree의 "일찍 끝남" 장점보다 **얕은 트리+범위검색**이 DB엔 훨씬 중요.)

### 왜 "데이터를 leaf에만" 두면 빠른가 — fanout과 트리 높이 (★)

오해 주의: **"leaf가 internal보다 빨라서"가 아니다.** 단일 키 1개만 찾으면 B-tree가 오히려 빠를 수도 있다(이정표에서 일찍 발견). 진짜 이유는 **트리 높이**다.

DB는 디스크를 **고정 크기 페이지**(InnoDB 16KB) 단위로 읽고, 한 노드 = 한 페이지. "한 페이지에 키를 몇 개 담느냐(fanout)"가 트리 높이를 좌우한다.

| | 한 칸 크기 (예시) | 16KB 페이지당 키 수(fanout) |
|--|--|--|
| **B-tree internal** | 키8 + 포인터6 + **데이터200** ≈ 214B | **약 76개** |
| **B+tree internal** | 키8 + 포인터6 ≈ 14B | **약 1170개** |

1000만 건이면 — B+tree: 1170³≈16억 → **높이 ~3**(디스크 3번 읽기), B-tree: 76⁴≈3300만 → **높이 ~4~5**(4~5번 읽기).

- **인과**: internal에 데이터가 박힘 → 노드가 뚱뚱 → 페이지당 키 적음 → **트리 높이↑ → 디스크 읽기 횟수↑**. 느림의 원인은 *데이터 처리 시간*이 아니라 **읽어야 할 페이지 수(트리 높이)**.
- **비유**: 책 **목차**에 페이지 번호(키)만 적으면 목차가 얇아 금방 도달. 칸마다 내용 전문(데이터)까지 적으면 목차가 두꺼워져 **여러 장 넘겨야** 함. B-tree가 그 "두꺼운 목차".
- **구조 재확인**: internal=**키만**(이정표) / leaf=**키+데이터**(둘 다). 키는 양쪽에 있고 **데이터만 leaf로 모은** 것.
- **보너스**: internal이 작아 **버퍼 풀(메모리)에 상주** → 윗단계는 RAM, 마지막 leaf만 디스크. 캐시 효율까지 좋다.

## 3. 선택도(selectivity) — 인덱스가 있어도 안 타는 경우

가져올 행이 전체의 **약 20~25%를 넘으면** 옵티마이저가 **인덱스를 버리고 풀스캔**을 택한다.

- **이유**: 인덱스로 행을 찾으면 실제 데이터를 가지러 테이블로 가는 **랜덤 I/O**가 붙는다. 가져올 행이 많아지면 이 랜덤 I/O가 쌓여, 차라리 **테이블을 순차로 읽는(sequential I/O) 풀스캔이 더 싸지는** 지점이 온다.
- 그래서 **카디널리티(값의 다양성)가 낮은 컬럼**(성별, 상태값 등)은 인덱스 효과가 적다. 카디널리티 높은 컬럼(email, 주문번호)이 인덱스에 적합.

## 4. 비용 — 왜 모든 컬럼에 안 거나

1. **쓰기 비용** — INSERT/UPDATE/DELETE마다 인덱스도 갱신. B+tree는 **정렬 유지**라 값 위치에 끼워넣고, 노드가 꽉 차면 **페이지 분할(split)**까지 일어나 쓰기가 느려짐.
2. **저장 공간** — 별도 자료구조라 디스크를 추가로 먹음(인덱스가 테이블보다 클 수도).
3. **안 쓰는 인덱스 = 순수 손해** — 조회엔 안 쓰이는데 쓰기·공간 비용만.

→ **읽기를 빠르게 하는 대가로 쓰기를 느리게 + 공간.** 조회 조건에 자주 등장하는 컬럼에만 선별적으로.

## 5. 복합 인덱스 — 선두 컬럼(leftmost prefix) 규칙 ★

`(name, age)` 복합 인덱스 = **name으로 먼저 정렬, 같은 name 안에서 age로 정렬**된 하나의 인덱스. (전화번호부: 성→이름순)

| 쿼리 | 인덱스 | 이유 |
|------|--------|------|
| `name='홍' AND age=30` | ✅ 완전 활용 | name으로 좁히고 그 안에서 age로 또 좁힘 |
| `name='홍'` | ✅ 탄다 | name이 **선두 컬럼**이라 정렬됨 (age 조건 없어도 OK) |
| `age=30` | ❌ 못 탄다 | age는 두 번째 컬럼 — **선두(name)를 건너뛰면** 정렬이 흩어져 못 씀 |

- **선두 컬럼 규칙**: 복합 인덱스는 **왼쪽 컬럼부터 순서대로** 써야 탄다. 선두를 건너뛴 조건은 못 탐.
- → **컬럼 순서가 설계의 핵심.** 자주 단독 검색하는 컬럼을 **앞**에 둔다. (또 카디널리티 높은 컬럼을 앞에 두는 게 일반적으로 유리.)
- **커버링 인덱스**: 조회하는 컬럼이 전부 인덱스 안에 있으면 테이블 접근(랜덤 I/O) 없이 **인덱스만으로** 응답 → 매우 빠름. (→ 6장)

## 6. 클러스터형 인덱스 vs 보조 인덱스 — "2번 탐색" (InnoDB ★)

- **클러스터형 인덱스(clustered)** = **PK 인덱스**. leaf 노드에 **실제 행(row) 전체**가 들어있다. 테이블 데이터 자체가 PK 순서로 정렬·저장됨 → PK로 찾으면 **한 번에 row 도달**.
- **보조 인덱스(secondary)** = PK 아닌 일반 인덱스. leaf에 **(인덱스 컬럼 값 + PK)**만 있고 **row는 없다**.

그래서 보조 인덱스 조회는 트리를 **2번** 탄다:
```
SELECT * FROM member WHERE email = 'hong@..';   -- email = 보조 인덱스
① email 보조 인덱스 탐색 → leaf에서 PK(id=42) 획득
② 그 PK로 클러스터형(PK) 인덱스 재탐색 → 실제 row 획득
   = "테이블 백(back to table)"
```

**커버링 인덱스 = 2번 탐색 제거:**
```sql
-- 보조 인덱스 (email, name) 가 있을 때
SELECT name FROM member WHERE email = 'hong@..';   -- 필요한 컬럼이 인덱스 안에 다 있음
```
조회 컬럼(`name`)이 인덱스 안에(인덱스 컬럼+PK) 다 있으면 클러스터형으로 안 가도 됨 → **① 탐색만으로 끝**(인덱스 온리 스캔). "필요한 컬럼을 인덱스에 포함"이 튜닝 기법.

## 7. 인덱스 무력화 패턴 — 정렬 전제가 깨질 때

인덱스는 **컬럼 값이 "그대로" 정렬돼 있다**는 전제로 동작. 전제를 깨면 못 탄다.

**(가) 앞 와일드카드 LIKE:**
```sql
WHERE name LIKE '홍%'    -- ✅ 탄다  (앞 고정 → '홍'으로 시작하는 범위로 좁힘)
WHERE name LIKE '%홍'    -- ❌ 못 탄다 (뒤로 끝나는 건 앞에서부터 정렬이라 못 좁힘)
WHERE name LIKE '%홍%'   -- ❌ 못 탄다
```
사전순(앞부터) 정렬이라 **앞이 고정돼야** 범위를 좁힌다. 뒤만 알면 전부 뒤져야 함.

**(나) 컬럼에 함수/연산:**
```sql
WHERE YEAR(created) = 2026                                  -- ❌ 못 탄다
WHERE created >= '2026-01-01' AND created < '2027-01-01'    -- ✅ 탄다 (컬럼 그대로 + 범위)
```
인덱스는 `created` **원본 값**으로 정렬돼 있는데 `YEAR(created)`로 **가공하면** 그 정렬을 못 쓴다. → **컬럼은 가공하지 말고 조건 쪽을 바꿔라.** (`col+1=10`, `SUBSTRING(col)` 다 같은 함정.) 정 필요하면 **함수 기반 인덱스**.

> ## 📌 핵심 요약
> 인덱스(B+tree)가 빠른 이유는 **디스크 페이지 읽기 횟수**가 O(log n)·트리가 얕아 3~4번으로 줄기 때문. DB는 **범위 검색·얕은 트리** 때문에 B-tree가 아니라 **B+tree**(데이터는 leaf에만, leaf는 linked list)를 쓴다. 많은 행을 가져올 땐 랜덤 I/O 탓에 풀스캔이 역전(**선택도 20~25%**). 비용은 **쓰기 느려짐·공간·페이지 분할**이라 선별적으로. 복합 인덱스는 **선두 컬럼부터** 써야 타는 **leftmost prefix 규칙**이 핵심 → 컬럼 순서가 설계의 전부.
