# JPA 2차 캐시 (Second-Level Cache)

> **한 줄 정의:** 2차 캐시는 *"1차 캐시가 트랜잭션을 못 넘는 한계를 메우려, 앱(SessionFactory) 레벨에 엔티티의 **상태 스냅샷**을 두고 요청을 가로질러 재사용하는 옵트인 캐시"*다. 불변·읽기 많은 데이터에만 쓰는 게 정석.

> 관련 문서:
> - [persistence-context-and-n-plus-one.md](persistence-context-and-n-plus-one.md) — 1차 캐시·N+1·조회 최적화(2차 캐시는 그 다음 카드)
> - [../database/transaction-and-isolation.md](../database/transaction-and-isolation.md) — stale read·일관성 트레이드오프

## 1. 캐시 3층 구조 — 먼저 지도부터

JPA에는 캐시가 한 종류가 아니라 **세 층**이 있다. 이걸 구분 못 하면 2차 캐시 얘기가 전부 헷갈린다.

| 층 | 범위(scope) | 키 | 수명 | 누가 켜나 |
|----|------------|-----|------|----------|
| **1차 캐시** | **트랜잭션(영속성 컨텍스트)** | 엔티티 `@Id` | TX 끝나면 소멸 | 항상 켜짐(끌 수 없음) |
| **2차 캐시** | **애플리케이션 전체(SessionFactory)** | 엔티티 `@Id` | 명시적 evict 전까지 | 옵트인(`@Cacheable`) |
| **쿼리 캐시** | 애플리케이션 전체 | **쿼리문 + 파라미터** | evict 전까지 | 별도 옵트인 |

- **1차 캐시**: "같은 TX 안에서 같은 id 재조회 시 DB 안 감 + `==` 동일성 보장". TX가 끝나면 통째로 버려진다 → *요청 간(between requests)* 캐싱 효과 0. 두 번째 HTTP 요청은 새 영속성 컨텍스트라 또 DB를 친다.
- **2차 캐시**: 그 한계를 넘으려고 **앱 전체 레벨**에 엔티티를 보관. TX A가 캐싱하면 TX B도 재사용 → DB 안 침.

```
요청1 ─┐
요청2 ─┼─→ [2차 캐시: 앱 전체 공유] ─→ DB
요청3 ─┘        ↑ 여기서 막으면 DB 트래픽 ↓

각 요청 안: [1차 캐시] (TX 끝나면 버려짐)
```

> **존재 이유는 단 하나** — 1차 캐시가 TX를 못 넘기 때문. "요청을 가로질러" 재사용하려고 앱 레벨로 올린 것.

---

## 2. 2차 캐시는 "무엇을" 저장하나 — 결정적 포인트

가장 많이 오해하는 지점. **2차 캐시는 엔티티 객체를 통째로 저장하지 않는다. "분해된 상태(hydrated/dehydrated state)"를 저장한다.**

```
2차 캐시 엔트리 (개념)
key:   Member#42
value: [ "홍길동", 30, team_id=7 ]   ← 식별자별 컬럼값 배열, 연관은 FK(id)만
```

**왜?** 엔티티 인스턴스를 그대로 공유하면 여러 스레드가 같은 객체를 동시에 만지는 재앙이 난다. 그래서 캐시엔 **불변 상태 스냅샷**만 두고, 각 영속성 컨텍스트가 그걸 읽어 **자기만의 엔티티 인스턴스로 재조립(hydrate)**한다.

두 가지 중요한 귀결:

1. **연관관계는 id만 저장된다.** `Member`를 캐시에서 꺼내도 `member.getTeam()`은 또 다른 조회(1차/2차/DB)다. 연관 엔티티도 캐싱하려면 그쪽도 `@Cacheable`이거나 `@Cache`로 컬렉션을 따로 캐싱해야 한다.
2. **캐시 히트해도 새 인스턴스다.** 1차 캐시 같은 `==` 동일성 보장은 2차 캐시엔 없다(같은 TX 안에선 1차가 잡아줌).

---

## 3. 동시성 전략 (CacheConcurrencyStrategy)

2차 캐시를 "쓴다"고 끝이 아니다. **여러 트랜잭션이 같은 엔트리를 동시에 읽고 쓸 때 어떻게 일관성을 지킬지** 골라야 한다 → `@Cache(usage = ...)`.

| 전략 | 동작 | 언제 |
|------|------|------|
| **READ_ONLY** | 한 번 캐싱하면 수정 불가. 변경 시 예외 | 코드성/참조 데이터 등 **불변** — 가장 빠르고 안전 |
| **NONSTRICT_READ_WRITE** | 수정 시 캐시를 **무효화(evict)만**. 락 없음 | 가끔 바뀌고 **stale 잠깐 허용** 가능 |
| **READ_WRITE** | **soft lock**으로 일관성 유지. commit 시점 갱신 | 자주 읽고 가끔 쓰며 stale 곤란 |
| **TRANSACTIONAL** | JTA 분산 트랜잭션과 묶임 | JTA 환경(드묾) |

직관: **위 → 아래로 갈수록 안전 ↑, 성능 ↓**. 일관성을 지키려면 락/조율 비용이 드니까. **READ_ONLY가 압도적 권장** — 안 바뀌는 데이터엔 락이 필요 없어 캐시 이점을 100% 누린다.

`NONSTRICT` vs `READ_WRITE` (핵심 질문거리):
- **NONSTRICT**: DB 갱신 후 캐시를 *지우기만*. 갱신과 evict 사이 짧은 틈에 다른 TX가 **옛 값을 읽을 수 있다**(stale).
- **READ_WRITE**: 그 틈을 **soft lock**으로 막음. 대신 분산 캐시(여러 노드)에선 노드 간 락 조율이 안 돼 보장이 약해진다.

## 4. 쿼리 캐시 — 함정 덩어리라 따로 본다

엔티티 2차 캐시가 "id로 엔티티 1건"이라면, **쿼리 캐시는 "이 JPQL + 이 파라미터의 결과"**를 캐싱한다. 악명 높은 함정:

> **쿼리 캐시는 결과 엔티티를 통째로 저장하지 않는다. id 리스트만 저장한다.**

```
쿼리캐시[ "select m from Member where age>?", [30] ] = [ id 1, 5, 42, 88 ]
                                                          ↓ 각 id를 2차 캐시/DB에서 다시 조회
```

- **쿼리 캐시만 켜고 엔티티 2차 캐시를 안 켜면** → id 리스트는 캐시에서 나오지만 각 엔티티는 다시 DB로 → **오히려 N+1**. → 쿼리 캐시는 반드시 엔티티 2차 캐시와 **세트로**.
- 쿼리 캐시는 **대상 테이블이 한 번이라도 write되면 통째 무효화**(타임스탬프 비교). write 잦은 테이블엔 거의 무용지물.

---

## 5. 2차 캐시를 쓰면 안 되는 경우 (Common Pitfalls)

"캐시 = 빠름"이 아니다. 실무에서 제일 중요한 부분.

1. **자주 변하는 데이터** → evict가 끊임없어 유지비 > 이득. 오히려 느려짐.
2. **분산 환경 + 로컬 캐시(Ehcache 등)** → 노드마다 캐시가 따로 논다. 노드 A 갱신해도 B는 stale. → 분산 캐시(Hazelcast/Infinispan/Redis)로 가야 하는데, 그럼 네트워크 왕복이 생겨 "캐시인데 안 빠른" 상황도.
3. **쓰기 많은 트랜잭션 데이터(주문·결제)** → 일관성 리스크 > 이득.
4. **메모리 압박** → 2차 캐시는 힙(또는 오프힙)을 먹는다. 무제한 캐싱 = GC 압박/OOM. → TTL·최대 엔트리 수 필수.

**좋은 후보**: 코드 테이블, 카테고리, 설정값 — 거의 안 바뀌고 자주 읽는 참조 데이터. 보통 READ_ONLY.

> **한 줄 통찰**: 대부분의 "조회가 느려요"는 2차 캐시가 아니라 **인덱스·fetch join·DTO projection**으로 풀린다. 2차 캐시는 그 *다음* 카드다. (그래서 실무에선 적용 영역을 좁게 잡는 게 정석.)

---

> ## 📌 핵심 요약
> 2차 캐시는 1차 캐시가 TX를 못 넘는 한계를 메우려 앱 레벨에 **상태 스냅샷(연관은 id만)**을 두는 옵트인 캐시. 동시성 전략(READ_ONLY→READ_WRITE)으로 안전·성능을 저울질하며 **불변·읽기 많음 데이터에만** 정석. 쿼리 캐시는 id만 저장해 반드시 엔티티 캐시와 세트.

> ## 🌱 심화 키워드
> - **hydration / dehydration** — 캐시 상태 ↔ 엔티티 인스턴스 변환
> - **CacheConcurrencyStrategy** — READ_ONLY/NONSTRICT/READ_WRITE/TRANSACTIONAL의 락 모델
> - **soft lock** — READ_WRITE가 일관성 지키는 방식
> - **RegionFactory / Ehcache · Infinispan · Hazelcast** — 2차 캐시 provider 구현체

> - **shared-cache-mode (ENABLE_SELECTIVE 등)** — 캐시 적용 범위 설정

> ## ❓ 남은 질문
> 1. 1차 캐시는 `==` 동일성을 보장하는데 2차 캐시는 왜 못 할까? (저장하는 게 인스턴스가 아니라 상태라는 것과 연결)
>    → **답:** 1차 캐시는 영속성 컨텍스트가 **엔티티 인스턴스 자체**를 들고 있어 같은 조회가 같은 객체를 반환(== 보장). 2차 캐시는 인스턴스가 아니라 **분해된 상태(값)**를 저장해 조회 때마다 새 인스턴스로 재조립하므로 ==는 깨지고 내용만 같다.
> 2. 분산 서버 3대 + `READ_WRITE` 로컬 캐시 → 어떤 일관성 문제가 생기고 왜 분산 캐시 provider가 필요할까?
>    → **답:** 각 서버가 자기 JVM에 로컬 2차 캐시를 따로 들면, A서버가 갱신해도 B·C서버 캐시는 옛 값이라 **노드 간 stale**이 생긴다. 그래서 무효화를 전파하거나 캐시를 공유하는 분산 provider(Hazelcast·Infinispan·Redis)가 필요하다.
> 3. 쿼리 캐시가 "id 리스트만" 저장하는 설계의 이점은? (메모리·일관성 관점)
>    → **답:** 결과로 엔티티 전체가 아니라 **PK 리스트만** 저장하면 실제 엔티티는 엔티티 캐시에서 가져와 중복 저장을 피하고 메모리를 아낀다. 엔티티가 갱신돼도 엔티티 캐시만 무효화하면 돼 일관성 관리도 단순해진다.
