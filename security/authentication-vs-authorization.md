# 인증(Authentication) vs 인가(Authorization)

둘 다 보안의 진입 단계지만 **묻는 질문이 다르다.** 인증은 "너 누구냐", 인가는 "너 이거 해도 되냐"이다. 항상 **인증 먼저, 인가 나중** 순서로 일어난다.

## 한 줄 비교

| | 인증 (Authentication, AuthN) | 인가 (Authorization, AuthZ) |
|---|---|---|
| 묻는 것 | **누구인가** (신원 확인) | **무엇을 할 수 있는가** (권한 확인) |
| 시점 | 먼저 | 인증 후 |
| 실패 시 | `401 Unauthorized` | `403 Forbidden` |
| 근거 | 비밀번호, 토큰, 생체, OTP 등 | 역할(Role), 권한(Permission), 정책 |
| 비유 | 건물 입구에서 신분증 확인 | 출입증으로 들어갈 수 있는 층 제한 |

> 이름이 헷갈리는 함정: HTTP `401`의 이름이 "Unauthorized"지만 실제로는 **인증 실패**를 뜻한다. **인가 실패는 `403 Forbidden`**.

## 인증 (Authentication) — "너 누구냐"

사용자가 주장하는 신원이 진짜인지 검증.

- **검증 수단(factor)**
  - 지식 기반: 비밀번호, PIN
  - 소유 기반: OTP 기기, 스마트폰, 보안 키
  - 속성 기반: 지문, 얼굴 등 생체
- **MFA(다중 인증)**: 위 factor를 2개 이상 조합 (예: 비밀번호 + OTP)
- **세션 vs 토큰**
  - 세션 방식: 서버가 세션 저장, 클라이언트는 세션 ID 쿠키 보유 (stateful)
  - 토큰 방식(JWT): 토큰 자체에 정보 담아 서명, 서버 저장 불필요 (stateless)

## 인가 (Authorization) — "너 이거 해도 되냐"

인증된 사용자가 특정 리소스/작업에 접근할 권한이 있는지 결정.

- **RBAC (Role-Based)**: 역할에 권한을 묶고, 사용자에게 역할 부여 (예: `ADMIN`, `USER`)
- **ABAC (Attribute-Based)**: 사용자·리소스·환경 속성으로 정책 판단 (예: "본인이 작성한 글만 수정 가능")
- **ACL (Access Control List)**: 리소스마다 누가 무엇을 할 수 있는지 목록 관리
- **Scope (OAuth)**: 토큰에 허용된 작업 범위 명시 (예: `read:profile`)

## 동작 순서

```
요청 도착
  ↓
[인증] 토큰/세션 검증 → 실패 시 401
  ↓ (신원 확인됨)
[인가] 이 사용자가 이 리소스에 접근 가능? → 실패 시 403
  ↓ (권한 있음)
실제 비즈니스 로직 실행
```

## 자주 쓰는 프로토콜/표준

- **OAuth 2.0**: **인가** 프레임워크. "제3자 앱에 내 리소스 접근 권한을 위임"이 핵심 (인증이 아님!)
- **OpenID Connect (OIDC)**: OAuth 2.0 위에 **인증** 계층을 얹은 것. 로그인(소셜 로그인)에 사용
- **JWT (JSON Web Token)**: 인증·인가 정보를 담아 서명하는 토큰 포맷 (프로토콜 아님)
- **SAML**: 기업 SSO에서 많이 쓰는 XML 기반 인증/인가 표준

## 흔한 오해 정리

- "OAuth로 로그인한다" → 엄밀히는 OAuth는 인가용. 로그인(인증)은 그 위의 **OIDC**가 담당
- "401 = 권한 없음" → ✗. 401은 **인증 실패**, 권한 없음은 **403**
- "JWT가 곧 인증" → ✗. JWT는 정보를 담는 **포맷**일 뿐, 인증·인가 양쪽에 쓰임

## ❓ 남은 질문

1. Stateless한 JWT는 서버에 저장이 없는데, 탈취되거나 조기 무효화해야 할 때 어떻게 하나?

   → **답:** JWT는 만료 전까지 자체 유효하므로 즉시 폐기가 어렵다. 실무에서는 액세스 토큰 수명을 짧게 두고 리프레시 토큰으로 재발급하며, 강제 무효화가 필요하면 블랙리스트(무효화 목록)를 서버에 두는데, 이는 stateless 장점을 일부 포기하는 절충이다.

2. 인증은 게이트웨이에서 한 번 했는데, 인가(권한 판단)는 왜 각 서비스에서 또 하게 되나?

   → **답:** 게이트웨이는 신원 확인과 거친 수준의 접근 제어까지만 하고, 리소스 소유권 같은 세밀한 인가(예: "본인 글만 수정")는 해당 데이터를 아는 서비스만 판단할 수 있기 때문이다. 인증은 공통화하되 도메인 인가는 서비스에 두는 게 일반적이다.

3. RBAC와 ABAC는 언제 갈리나?

   → **답:** 역할 수가 적고 권한이 역할에 깔끔히 묶이면 RBAC가 단순해 좋고, "본인 소유·근무시간·부서 일치" 같은 문맥/속성 조건이 필요하면 ABAC가 적합하다. 실무에선 RBAC를 기본으로 하고 세밀 조건만 속성 기반으로 보강하는 혼합이 흔하다.
