# 컨텍스트 스위칭 (Context Switching)

CPU가 실행 중인 스레드를 잠깐 내리고 다른 스레드를 올리는 교대 행위. **"일이 다른 프로세스로 전이"되는 게 아니라, 유한한 CPU 자리를 시간으로 쪼개(시분할) 돌려쓰는 것.**

## 1. 왜 존재하나 — 자리 vs 시간

- 물리 CPU **코어 1개**는 한 순간에 스레드 **1개**만 실행한다. (SMT/하이퍼스레딩이면 논리 코어 2개 — 레지스터 세트를 2벌 두어 2개를 번갈아 태움. 단 실행 유닛·캐시는 공유라 성능은 ~1.2~1.3배지 2배가 아님.)
- 그런데 실행하려는 스레드는 코어 수보다 **훨씬 많다**(코어 8×2=16인데 스레드 수천 개).
- **자리는 유한, 손님은 초과** → 부족한 자리를 **'시간'으로 쪼개(시분할, time-slicing)** 각 스레드에 짧은 조각(타임 슬라이스, 보통 수 ms)을 나눠준다.
- 시간이 다 되면 잠깐 내려오고 다음 스레드가 올라온다. **이 교대의 순간 = 컨텍스트 스위칭.** 빠르게 돌려서 "다 같이 도는 것 같은 착시(동시성)"를 만든다.

> ⚠️ **오개념 1**: "일이 다른 프로세스로 넘어가서 CS가 발생한다" ❌
> → 일이 전이되는 게 아니라 **실행 주체가 무대(CPU)에서 내려오고 다른 주체가 올라오는 것.** "다른 프로세스로 넘어가는 경우"는 있지만, 그건 *스케줄러가 다음에 고른 스레드가 마침 다른 프로세스 소속*이라는 뜻이지 작업 전이가 아니다.

## 2. 언제 일어나나 — 트리거 2갈래

| 갈래 | 뜻 | 예시 | 이후 상태 |
|------|-----|------|-----------|
| **비자발적 (선점, preemption)** | OS가 **강제로** 끌어내림 | 타이머 인터럽트(타임 슬라이스 만료), 더 높은 우선순위 스레드 등장 | `Running → Ready` |
| **자발적 (voluntary)** | 스레드가 **스스로** 자리를 비켜줌 | I/O 응답 대기(블로킹), 락 대기, sleep, yield | `Running → Blocked` |

> "시분할 인터럽트"가 아니라 **타이머 인터럽트**가 정확한 이름. "I/O가 발생해서"가 아니라 **I/O 응답을 기다리며 블로킹에 들어가서**.

## 3. 내려간 스레드는 어떻게 되나 — 상태 머신 + 저장 위치

### 상태 머신 (3-state 뼈대)

```
        타임 슬라이스 만료 / 선점
   ┌───────────────────────────────┐
   ▼                               │
[Ready] ──스케줄러가 CPU 배정──▶ [Running] ──I/O·락 대기──▶ [Blocked]
   ▲                                                          │
   └──────────── 이벤트(I/O 완료 등) 도착 ◀────────────────────┘
```

- **타이머로 밀린 스레드**: 아직 실행하고 싶음(자리만 뺏김) → **Ready(=Runnable)**. 레디 큐에서 차례를 기다림.
- **I/O 걸린 스레드**: 지금은 실행조차 못 함(응답이 와야 함) → **Blocked(=Waiting)**. 이벤트가 오면 그제서야 `Blocked → Ready`.
- 둘을 같은 대기 줄에 세우면 안 되는 이유가 이것 — "언제든 부르면 뛸 수 있음" vs "뭐가 와야 뛸 수 있음".

### 저장 대상과 저장 위치 (자주 틀리는 곳)

> ⚠️ **오개념 2**: "PC(Program Counter)에 저장된다" ❌
> **PC는 저장되는 '대상'이지 저장되는 '장소'가 아니다.**

- **저장 대상**: PC 하나가 아니라 **레지스터 세트 전체** — PC + SP(스택 포인터) + 범용 레지스터 + 상태 플래그.
- **저장 장소**: 레지스터가 아니라 **메모리 안의 커널 자료구조** — **TCB(Thread Control Block) / PCB(Process Control Block)**.
- **왜 메모리로 대피?** 곧 올라올 다음 스레드가 그 레지스터를 덮어쓰기 때문. 지우기 전에 스냅샷을 떠서 메모리로 빼둬야 나중에 정확히 이어서 실행할 수 있다.

## 4. 컨텍스트 스위칭의 비용 — 4겹

| 겹 | 무슨 일 | 스레드 전환 | 프로세스 전환 |
|----|---------|:----------:|:------------:|
| **① 레지스터 저장·복원** | 내려가는 스레드 레지스터를 TCB에 복사 + 올라오는 스레드 것을 레지스터로 복원 | ✅ (작음) | ✅ (작음) |
| **② 커널 진입 + 스케줄러** | 유저→커널 모드 전환, "다음 누구?" 결정 | ✅ | ✅ |
| **③ 주소공간 교체 + TLB flush** | 페이지 테이블 베이스 레지스터(x86 `CR3`) 교체 → TLB 무효화 | ❌ **없음** | ✅ **있음** |
| **④ 캐시 오염 (숨은 최대 비용)** | 새 스레드 데이터는 L1/L2/L3에 없음 → 콜드 캐시 → 캐시 미스 폭탄 | 조금 | 심함 |

- ①②는 **양쪽 공통·상대적으로 작음**. 진짜 차이는 **③·④**에서 갈린다.
- **직접 비용(①②)**보다 **간접 비용(④ 캐시 오염)**이 종종 더 크다 — 눈에 안 보여서 놓치기 쉬움.

## 5. 스레드 전환 vs 프로세스 전환

> 💡 **TLB(Translation Lookaside Buffer)란?**
> **가상 주소 → 물리 주소 변환 결과를 캐싱하는, CPU 내부의 작고 빠른 캐시.**
> - 프로그램은 **가상 주소**로 메모리에 접근하지만, 실제 **물리 주소**를 알려면 메모리에 있는 **페이지 테이블**을 뒤져야 한다(page table walk, ~100ns). 메모리 접근 **매번** 이걸 하면 너무 느림.
> - 그래서 MMU가 한 번 구한 "가상→물리" 변환을 **TLB에 캐싱**해둔다. → **TLB hit이면 즉시(~1ns)** 물리 주소 확보, **miss면 페이지 테이블 walk** 후 채움.
> - TLB 엔트리는 **특정 프로세스의 페이지 테이블** 기준이다. 그래서 **프로세스가 바뀌면(주소 공간 교체)** 이전 엔트리는 무효 → **flush(비움)** → 이후 miss가 쌓이며 walk 재발생 = 비용. (스레드 전환은 같은 페이지 테이블이라 TLB 유효 → flush 불필요.)
> - 더 자세히는 → [process-thread-memory.md](./process-thread-memory.md#tlb-translation-lookaside-buffer)

- **스레드 전환(같은 프로세스)**: 주소 공간 그대로 → **TLB flush 없음**, 캐시도 어느 정도 재활용 → **쌈** ✅
- **프로세스 전환**: 주소 공간이 통째로 바뀜 → **③ 주소공간 교체 + TLB flush + ④ 캐시 싹 무효** → **비쌈** ⚠️

즉 "프로세스 전환이 비싼 진짜 이유"는 레지스터 저장이 아니라 **TLB flush + 캐시 오염** 두 가지다.

> 🔬 **더 깊게 — PCID / ASID**: 요즘 CPU는 TLB 엔트리에 "이건 어느 프로세스 것"이라는 꼬리표(x86 **PCID**, ARM **ASID**)를 달 수 있다. 그러면 프로세스 전환 때 TLB를 **전부 flush하지 않고** 남겨둘 수 있어 ③의 비용이 완화된다. ("프로세스 전환 = 무조건 전체 TLB flush"는 옛 모델.) *(거의 확실)*

## 6. 모드 전환(system call) vs 컨텍스트 스위칭

> ⚠️ **자주 헷갈림**: 시스템콜은 커널로 들어가지만 **컨텍스트 스위칭이 아니다.**

- **모드 전환(mode switch)** = **같은 스레드**가 **권한 레벨**을 바꾸는 것 (user mode ↔ kernel mode)
- **컨텍스트 스위칭** = **실행 스레드 자체**가 바뀌는 것 (누가 CPU를 쓰냐)
- → 바뀌는 대상이 다르다: 하나는 **"모드"**, 하나는 **"주인(스레드)"**.

### 왜 시스템콜은 CS가 아닌가

`read()`·`write()` 같은 시스템콜은 커널 권한이 필요해 CPU가 user→kernel로 올라갔다 내려온다. 그런데:

| | 스레드 바뀌나? | 주소 공간 바뀌나? | 스케줄러 도나? | TCB 교체? |
|--|:---:|:---:|:---:|:---:|
| **모드 전환 (syscall)** | ❌ 그대로 | ❌ 그대로 | ❌ | ❌ |
| **컨텍스트 스위칭** | ✅ A→B | (프로세스면) ✅ | ✅ | ✅ |

시스템콜은 **호출한 그 스레드가 계속** 커널 코드를 "자기 대신" 실행할 뿐 — 실행 주체가 안 바뀌니 CS가 아니다. 권한 레벨만 잠깐 올라갔다 온다.

> 💡 **단, 시스템콜이 CS를 *유발*할 수는 있다.** `read()`가 디스크를 기다려야 하면 → 그 스레드가 **Blocked** → 그제서야 다른 스레드로 **컨텍스트 스위칭**. 즉 **"모드 전환(항상) + CS(블로킹일 때만)"**. 데이터가 이미 준비돼 있으면 모드 전환만 하고 CS 없이 리턴.

### 비용 차이

| | 하는 일 | TLB flush | 캐시 | 대략 규모 |
|--|--------|:---:|:---:|--------|
| **모드 전환** | 최소 상태 저장 + 권한 플립 + 커널 핸들러 + 복귀 | ❌ | 대부분 warm | **~수십~수백 ns** |
| **스레드 CS** | 레지스터 세트 전체 + 스케줄러 | ❌ | 일부 오염 | **~1μs + α** |
| **프로세스 CS** | 위 + 주소공간 교체 | ✅ | 싹 오염 | **가장 비쌈** |

**모드 전환이 CS보다 대략 한 자릿수(order of magnitude) 이상 싸다.** 주소 공간이 그대로라 TLB·캐시 후폭풍이 없다는 게 결정적. *(숫자는 하드웨어·OS 의존 대략치)*

> 🔬 **더 깊게 — KPTI**: Meltdown 취약점 대응으로 도입된 **KPTI(커널 페이지 테이블 격리)** 이후엔 시스템콜 때 유저/커널 페이지 테이블을 분리·교체해 **모드 전환이 예전보다 비싸졌다**(일부 TLB flush 발생). "syscall이 요즘 왜 느려졌나"의 답. *(거의 확실)*

## 7. 비동기와 컨텍스트 스위칭 — 작업 전환 ≠ 스레드 전환

> ⚠️ **오개념 3**: "비동기는 한 스레드로 여러 작업을 하니까 CS가 더 많다" ❌
> 숨은 가정 = **"작업(task)을 바꾸는 것 = 컨텍스트 스위칭"**. 아니다. (오개념 1의 재발)

| | 무엇 | OS가 끼나? | 비용 |
|--|------|:---------:|------|
| **작업(task) 전환** | 같은 스레드가 콜백 A 끝내고 콜백 B 실행 | ❌ | 그냥 **함수 호출** (레지스터 세트 교체·TLB·커널 진입 전부 없음) |
| **스레드 전환 (= 컨텍스트 스위칭)** | OS가 스레드 A를 내리고 B를 올림 | ✅ | 4겹 비용 전부 |

- **블로킹 모델**: 커넥션 1만 개 → 스레드 ~1만 개, 각자 I/O에서 Blocked됐다 깨어남 → **OS 컨텍스트 스위칭 폭발** (+ 스택 1만 개 = 수 GB).
- **비동기 모델**: 스레드 1개(또는 코어 수만큼)가 안 멈추고 여러 I/O를 굴림. 작업 사이 전환은 전부 **유저 공간 함수 호출** → **OS 컨텍스트 스위칭 극소**. OS 눈엔 여전히 스레드 1개. → **비동기의 존재 이유.**
- **단, 0은 아니다**: 이벤트 루프 스레드도 OS 스레드라 타이머로 선점당하고, 할 일 없으면 `epoll_wait`에서 스스로 Blocked(이것도 CS), 워커 풀 있으면 그 사이 전환도 있음. → **"줄인다"지 "없앤다"가 아님.**
- **가상 스레드(Loom)**: 같은 트릭의 다른 얼굴 — 블로킹처럼 코드를 짜도 JVM이 유저 공간에서 park/unmount해서 **OS 컨텍스트 스위칭을 안 늘림.**

## 8. 곁가지 — 스핀락 vs 블로킹 락 (폴링 vs 알림)

컨텍스트 스위칭에 비용이 있으니, 그걸 **일부러 피하는** 전략도 있다.

| | 하는 짓 | CPU/자원 | 컨텍스트 스위칭 |
|--|---------|----------|:---------------:|
| **스핀락 (busy-wait)** | "락 풀렸어?" 루프 돌며 계속 확인. CPU를 **안 놓음**(Running 유지) | 낭비함(도는 동안) | **안 함** |
| **블로킹 락 / Redisson pub/sub** | 확인 대신 **구독해두고 잠듦**(Blocked). 풀리면 **메시지로 깨워줌** | 안 낭비 | **함** |

- 축은 **폴링(계속 되물음) vs 알림(오면 깨워줌)**.
- **스핀락은 대기가 아주 짧을 때 유리**: 잠들었다 깨는 것(= CS 2번)보다 몇 사이클 도는 게 더 싸니까. (반대로 대기가 길면 CPU 낭비라 손해.)
- **Redisson 분산락**: 락 못 잡으면 Redis 키를 계속 폴링(스핀)하지 않고 pub/sub 채널을 **구독**해뒀다가 unlock 메시지를 받고 깨어남 → 불필요한 되물음(부하) 제거. *(거의 확실 — Redisson 구현 기준)*

---

> ## 📌 핵심 요약
> 컨텍스트 스위칭은 **유한한 CPU 자리를 시분할로 나눠쓰는 교대**다(일의 전이 ❌, 자리의 교대 ⭕). 트리거는 비자발적(타이머 선점)·자발적(I/O·락 대기) 2갈래. 내려간 스레드는 왜 내려왔냐에 따라 Ready/Blocked가 되고, **레지스터 세트 전체를 메모리 TCB/PCB에 저장**(PC는 저장 '대상'이지 '장소' 아님). 비용은 4겹(레지스터·커널·주소공간+TLB·캐시)이고, **프로세스 전환이 비싼 건 TLB flush + 캐시 오염** 때문. 비동기가 CS를 줄이는 핵심은 **작업 전환(유저 공간 함수 호출) ≠ 스레드 전환(OS 개입)**.

> ## 🌱 심화 키워드
> - **Dispatcher / dispatch latency** — 스케줄러가 고른 뒤 실제 전환까지의 지연
> - **PCID(x86) / ASID(ARM)** — 태그드 TLB로 프로세스 전환 시 전체 flush 회피
> - **Cache pollution / working set** — 전환 후 콜드 캐시가 성능에 주는 실질 타격
> - **Voluntary vs involuntary context switch** — `pidstat -w`의 `cswch/s` vs `nvcswch/s`
> - **Mode switch vs context switch** — 시스템콜(모드 전환)은 CS가 아님 (같은 스레드)

> ## ❓ 남은 질문
> 1. `vmstat`/`pidstat -w`로 실제 서버의 CS 수를 어떻게 읽고, 어느 수치부터 "과하다"고 판단하나? (voluntary vs involuntary 비율의 의미)
>    → **답:** `cswch/s`(자발적=I/O·락 대기)와 `nvcswch/s`(비자발적=타임슬라이스 소진·선점)를 본다. 절대 임계치는 없고, nvcswch가 높으면 CPU 경합(스레드 과다)·cswch가 높으면 잦은 블로킹이 원인 — 처리량이 떨어지는데 CS가 급증하면 과하다고 본다.
> 2. (모드 전환 vs CS는 6장에서 정리됨) → 후속: **KPTI**로 syscall이 비싸진 만큼, 실무에서 **syscall 횟수를 줄이는** 기법(배치 I/O, `io_uring`, `mmap`)은 각각 어떻게 비용을 아끼나?
>    → **답:** 배치 I/O(`readv/writev`·다건 묶기)는 호출 횟수 자체를 줄이고, `io_uring`은 커널과 공유하는 링버퍼로 제출·완료를 모아 매 연산의 syscall을 없애며, `mmap`은 파일을 메모리에 매핑해 read/write syscall 대신 페이지폴트로 접근한다 — 모두 유저↔커널 전환 비용을 절감.
> 3. 스레드 풀 크기를 코어 수보다 훨씬 크게 잡으면 CS가 늘어 오히려 느려진다 — 최적 크기는 어떻게 추정하나? (CPU-bound vs I/O-bound, Little's Law) (→ [thread-pool.md](./thread-pool.md))
>    → **답:** CPU-bound는 코어 수(±1)면 충분하고, I/O-bound는 대기가 길수록 더 크게 잡는다. `N = 코어수 × (1 + 대기시간/서비스시간)`이나 Little's Law(동시성=도착률×응답시간)로 추정하고 부하 테스트로 보정한다.
