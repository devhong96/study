# 프로세스와 스레드의 기초 개념

이 문서는 프로세스와 스레드의 핵심 개념, 메모리 영역(Heap, Stack)의 분배 방식, 그리고 많은 학습자들이 혼동하는 **"작업의 단위"**와 **"실행 흐름의 단위"**의 명확한 차이점을 직관적인 비유와 함께 정리합니다.

---

## 1. 프로세스와 스레드의 메모리 구조 (Heap & Stack)

프로그램이 실행될 때 메모리는 다음과 같이 관리됩니다.

### 힙 (Heap)
* **개념**: 동적으로 메모리를 할당하는 공동 공간입니다.
* **소속**: **프로세스 단위**로 하나씩 주어집니다.
* **특징**: 같은 프로세스에 속한 모든 스레드가 이 힙 영역을 **공동으로 공유**합니다. 이를 통해 스레드 간 데이터 교환이 쉽지만, 동시에 접근할 때 동기화 문제(Race Condition)가 발생할 수 있습니다.

### 스택 (Stack)
* **개념**: 함수 호출 정보, 지역 변수, 매개 변수 등이 임시 저장되는 독립 공간입니다.
* **소속**: **스레드 단위**로 하나씩 개별 지급됩니다.
* **특징**: 프로세스에 스택 영역이 하나만 존재하는 것이 아닙니다. **새로운 스레드가 생성될 때마다 해당 스레드만을 위한 독점적인 스택이 하나씩 생성**됩니다. 따라서 스레드끼리는 서로의 스택 영역을 침범할 수 없으며 안전하게 자신만의 함수 호출 흐름을 가집니다.

```
[ 프로세스 (Process) 가상 메모리 공간 ]
┌────────────────────────────────────────────────────────┐
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │   Text (코드)   │   Data / BSS   │   Heap (힙)    │  │ ◀── 모든 스레드가 공유하는 영역
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐  │
│  │  스택 (Stack)  │  │  스택 (Stack)  │  │  스택 (Stack)  │  │ ◀── 스레드마다 독립적인 영역
│  │   - 스레드 1    │  │   - 스레드 2    │  │   - 스레드 3    │  │
│  └───────────────┘  └───────────────┘  └───────────────┘  │
└────────────────────────────────────────────────────────┘
```

## 2. "작업의 단위" vs "실행 흐름의 단위"

프로세스(작업 단위)와 스레드(실행 흐름 단위)의 차이는 **OS가 자원을 분배하는 기준**과 **CPU가 실제로 일하는 기준**의 차이입니다.

### ① 작업의 단위 (Process) = 자원 관리의 주체
* 운영체제(OS)가 메모리(힙 공간), 파일 디스크립터, CPU 시간 등 **컴퓨터 자원을 나누어 주는 기준**입니다.
* OS는 독립된 자원을 안전하게 관리하기 위해 **프로세스 단위로 메모리 공간을 격리**합니다.
* 프로세스 자체는 자원을 소유한 껍데기(혹은 울타리) 역할을 하며, 스스로 움직여 실행되지는 않습니다.

### ② 실행 흐름의 단위 (Thread) = 실행의 주체
* 프로세스가 나누어 받은 자원 안에서 **실제 명령어 코드를 한 줄씩 읽어가며 동작하는 주체**입니다.
* CPU 스케줄러가 실제로 CPU를 할당하여 **일을 시키는 대상**은 프로세스가 아니라 스레드입니다.
* 하나의 프로세스 안에서 여러 실행 흐름(멀티 스레드)이 동시에 돌아가며 일을 처리할 수 있습니다.

## 3. 직관적인 비유: "백종원의 식당 주방" 🏠

이해를 돕기 위해 **새로운 식당 주방을 차리는 과정**에 비유해 보겠습니다.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  [식당 프로젝트]                                                         │
│                                                                          │
│  ● 주방 (프로세스) = "작업의 단위"                                       │
│    - 백종원(OS)으로부터 할당받은 공간(가상 메모리)                       │
│    - 냉장고와 조리대(힙 영역)를 갖추고 있음                             │
│                                                                          │
│  ● 요리사 (스레드) = "실행 흐름의 단위"                                  │
│    - 주방에서 실제로 요리책 레시피(코드)를 보고 요리를 만드는 일꾼       │
│    - 요리사 각자는 본인의 앞치마 주머니(스택 영역)에 개인 도구를 가짐    │
└──────────────────────────────────────────────────────────────────────────┘
```

* **프로세스 (주방 개설)**: 
  백종원(OS)이 새로운 요리 사업(작업)을 위해 임대 공간을 마련하고 설비(자원)를 들여놓습니다. 주방 자체가 혼자서 요리를 하지는 않으므로, 이는 자원을 다루는 **작업(Task)의 준비 단계**입니다.
* **스레드 (요리사 채용)**: 
  요리를 하려면 주방에 요리사(스레드)가 들어가야 합니다. 
  * **공유 자원(Heap)**: 요리사들은 주방 안의 **공동 냉장고와 공동 조리대(힙)**를 같이 씁니다. 한 요리사가 요리 재료를 썰어서 조리대에 놓으면, 다른 요리사가 이를 바로 가져다 쓸 수 있습니다. (빠른 소통 가능)
  * **독립 자원(Stack)**: 하지만 각 요리사는 자기만의 **앞치마 주머니(스택)**를 가지고 있어, 본인이 수시로 적는 메모나 개인 소도구들은 서로 침범하지 않습니다.
  * **안정성 (Crash)**: 만약 한 요리사가 크게 실수하여 주방 가스를 터뜨리면(심각한 에러 발생), 그 주방에 있던 **모든 요리사와 주방 전체가 폐쇄(프로세스 강제 종료)**됩니다. 옆집 다른 주방(다른 프로세스)은 아무런 타격을 받지 않습니다.

## 4. 최종 요약 비교표

| 구분 | 프로세스 (Process) | 스레드 (Thread) |
| :--- | :--- | :--- |
| **정의** | OS가 독립적인 자원을 할당하는 **작업의 단위** | 프로세스 내에서 실제 코드를 실행하는 **흐름의 단위** |
| **메모리 할당** | 독립된 가상 주소 공간 (Text, Data, Heap, Stack) | 프로세스의 주소 공간 공유 + 자신만의 **Stack** 지급 |
| **자원 공유** | 불가능 (IPC 등 특수 기법 필요) | 같은 프로세스 내의 다른 스레드와 **Heap 공유** |
| **생성/스위칭 비용**| **매우 높음** (메모리 격리 및 전환 시 무거움) | **낮음** (공유 자원이 많아 가볍고 빠름) |
| **영향도 (오류)** | 독립적이어서 문제 발생 시 해당 프로세스만 죽음 | 한 스레드가 죽으면 **프로세스 전체가 죽을 수 있음** |
