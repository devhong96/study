# 260710 : 원자성과 멱등성은 어떻게 다른가요?

**오늘의 백엔드 질문**

원자성과 멱등성은 어떻게 다른가요?

이 질문은 트랜잭션, 재시도, 중복 요청 방어를 제대로 이해하고 있는지 확인하기 좋은 질문입니다. 특히 결제, 주문, 쿠폰 발급, 메시지 처리 같은 상황에서 자주 이어집니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- 원자성은 하나의 작업 단위가 모두 성공하거나 모두 실패해야 한다는 개념입니다.
- 예를 들어 주문 생성과 결제 기록 저장이 하나의 트랜잭션이라면, 둘 중 하나만 성공한 상태로 남으면 안 됩니다.
- 멱등성은 같은 요청이 여러 번 실행되어도 결과가 한 번 실행한 것과 같아야 한다는 개념입니다.
- 예를 들어 결제 승인 요청이 네트워크 문제로 재시도되더라도 실제 결제는 한 번만 처리되어야 합니다.
- 즉, 원자성은 “작업 묶음의 성공/실패 경계”에 가깝고, 멱등성은 “중복 실행에 대한 결과 보장”에 가깝습니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- 트랜잭션을 사용하면 멱등성도 자동으로 보장되나요?
- 멱등성 key는 어떤 기준으로 설계하나요?
- 같은 요청이 동시에 두 번 들어오면 어떻게 방어하나요?
- DB unique key와 Redis idempotency key는 각각 어떤 차이가 있나요?
- 메시지 큐에서 at-least-once 처리 시 멱등성이 왜 중요할까요?

---

## 답변

> **한 줄 핵심**: 원자성은 "한 번의 실행 **안**"의 성질(부분 성공 방지, 도구는 트랜잭션), 멱등성은 "여러 번의 실행 **사이**"의 성질(중복 반영 방지, 도구는 멱등키·unique 제약) — 서로 다른 실패 모델을 방어하며, 둘은 직교한다.

### 면접 답변 (구술용)

두 개념은 서로 다른 실패 모델을 방어합니다. 원자성은 한 번의 실행 내부에 대한 성질입니다 — 작업 묶음이 전부 성공하거나 전부 실패해야 하고 부분 성공 상태가 남으면 안 됩니다. 방어 대상은 "실행 도중의 중단"이고 도구는 DB 트랜잭션입니다. 멱등성은 여러 번의 실행에 걸친 성질입니다 — 같은 요청이 N번 실행돼도 결과가 1번과 같아야 합니다. 방어 대상은 "실행의 반복" — 타임아웃 재시도, 더블클릭, 메시지 재전달이고, 도구는 멱등성 키와 unique 제약입니다. 둘이 직교한다는 걸 예시로 보이면 — "잔액 100원 차감 + 거래내역 기록"을 트랜잭션으로 묶으면 완벽히 원자적이지만, 이 요청이 재시도되면 두 번째 실행도 각자 완벽히 원자적으로 커밋되어 200원이 깎입니다. 원자성은 두 번 다 지켜졌는데 결과는 틀린 거죠. 전형적 시나리오가 결제 타임아웃입니다 — 클라이언트는 실패로 알지만 서버는 커밋을 마쳤고, 재시도가 들어오면 중복 결제가 됩니다. 여기서 문제는 트랜잭션이 아니라 "같은 논리적 요청의 두 번째 실행을 식별해서 차단할 장치"의 부재이고, 그게 멱등성 설계입니다. 정리하면 — 원자성은 하나의 시도 안에서 반쪽 상태를 막고, 멱등성은 여러 시도에 걸쳐 중복 반영을 막습니다. 신뢰성 있는 시스템에는 둘 다 필요합니다.

### 원리 이해 (왜 그런가)

**비교표:**

| | 원자성 (Atomicity) | 멱등성 (Idempotency) |
|---|---|---|
| 적용 범위 | 한 번의 실행 **내부** | 여러 번의 실행 **사이** |
| 방어하는 실패 | 실행 도중의 중단 (부분 성공) | 실행의 반복 (재시도·더블클릭·재전달) |
| 깨졌을 때 증상 | "재고만 줄고 주문이 없다" | "결제가 두 번 됐다" |
| 도구 | DB 트랜잭션 (commit/rollback) | 멱등성 키, unique 제약, 조건부 연산 |
| 출처 개념 | ACID의 A | 수학의 f(f(x)) = f(x) |

**직교성 증명 (면접 득점 포인트):**

```
트랜잭션 { 잔액 -= 100; 내역 INSERT; }   ← 완벽히 원자적
같은 요청 2회 실행 → 각각 원자적으로 커밋 → 잔액 -200   ← 멱등하지 않음
∴ 원자성 ⊅ 멱등성 — 트랜잭션은 "이번 실행"만 알고, "이전에 같은 요청이 있었는지"는 모른다
```

**멱등 처리의 표준 흐름:**

```
요청(멱등키 포함) → 멱등키 INSERT 시도 (unique 제약)
  ├ 성공: 최초 요청 → 비즈니스 처리 + 결과 저장 (가능하면 같은 트랜잭션)
  └ 실패(중복): 기존 처리 결과를 조회해 그대로 반환 (에러가 아니라 "같은 응답")
```

### 꼬리질문 Q&A

**Q. 트랜잭션을 사용하면 멱등성도 자동으로 보장되나요?**
→ **아니오 — 트랜잭션은 실행 1회의 원자성만 보장하고, 재실행하면 또 한 번 원자적으로 반영된다.**
위 직교성 예시가 근거입니다. 멱등성은 "이 논리적 작업이 이미 수행됐는가"를 기록하고 확인하는 별도 장치가 있어야 합니다.

**Q. 멱등성 key는 어떤 기준으로 설계하나요?**
→ **"논리적 작업 1건 = key 1개" — 재시도는 같은 key를 가져야 하므로, key는 시도 시점이 아니라 작업 생성 시점에 만들어져야 한다.**
클라이언트가 결제 버튼을 누른 순간 키를 생성해 재시도에도 동일하게 보내는 방식(HTTP Idempotency-Key 헤더 패턴)이나 자연 비즈니스 키(주문ID + 작업유형)를 씁니다. 반대로 서버가 요청 수신마다 새 키를 발급하면 재시도가 서로 다른 작업으로 보여 멱등성이 무의미해집니다. 보관 기간(TTL — 재시도 가능 기간보다 길게)과 저장 위치도 함께 정합니다.

**Q. 같은 요청이 동시에 두 번 들어오면 어떻게 방어하나요?**
→ **"조회해서 없으면 처리"(check-then-act)는 레이스가 있어 안 되고, 원자적 연산에 맡겨야 한다.**
멱등키를 unique 제약이 있는 테이블에 INSERT하게 하면 DB가 두 요청을 직렬화해 하나만 성공시키고, 실패한 쪽은 "이미 처리됨/처리 중"으로 응답합니다. Redis SET NX로 선점 마킹하는 것도 같은 원리(원자적 check-and-set)입니다. 분산락으로 구간을 직렬화할 수도 있지만 락은 경계 상황에서 뚫릴 수 있으므로(260706), 최종 방어선은 DB 제약이어야 합니다.

**Q. DB unique key와 Redis idempotency key는 각각 어떤 차이가 있나요?**
→ **DB unique는 데이터와 같은 트랜잭션에서 원자적으로 처리되는 최종 방어선, Redis key는 빠른 선차단·다중 서비스 방어용 — 실무는 조합이다.**
DB의 결정적 장점은 "키 기록됐는데 데이터는 롤백" 같은 어긋남이 구조적으로 불가능하다는 것(같은 커밋)입니다. Redis는 빠르고 TTL 관리가 쉬워 DB 도달 전 차단(부하 절감)과 여러 저장소에 걸친 방어에 유리하지만, 데이터와 별도 시스템이라 "Redis엔 처리됨인데 DB는 롤백" 틈이 생길 수 있어 기록 순서와 실패 시 삭제/만료 설계가 따로 필요합니다.

**Q. 메시지 큐에서 at-least-once 처리 시 멱등성이 왜 중요할까요?**
→ **at-least-once에서 재전달은 버그가 아니라 설계된 정상 동작이기 때문이다.**
consumer가 처리 후 ack/commit 전에 죽으면 브로커는 "처리 안 됨"으로 보고 다시 보냅니다 — 유실을 막기 위한 선택의 필연적 대가입니다. 즉 같은 메시지가 두 번 오는 것은 언젠가 반드시 일어나는 이벤트이고, consumer가 멱등하지 않으면 중복 반영이 일상적으로 발생합니다. "유실은 브로커+커밋 전략이 막고, 중복은 consumer 멱등성이 막는다"는 역할 분담이 메시징 신뢰성 설계의 기본 구도입니다(→ 260709, 260618).

### 🌱 심화 키워드
- **Idempotency-Key 헤더** — 결제 API 업계 표준 패턴 (Stripe가 대중화)
- **check-then-act 레이스** — 동시성 버그의 근본 패턴 (락·원자 연산으로만 해결)
- **UPSERT / 조건부 UPDATE** — 연산 자체를 멱등하게 만드는 SQL 수단
- **exactly-once processing effect** — "전달은 at-least-once, 효과는 1회"라는 실무 목표
- **자연 멱등 연산** — 상태 덮어쓰기(SET), 집합 추가처럼 반복해도 같은 결과인 연산 설계

### ❓ 더 파볼 질문
- **HTTP 메서드별 멱등성 규약은 어떻게 되나?**
  ↳ RFC 9110 기준 GET/HEAD/PUT/DELETE는 멱등, POST는 비멱등이다. PUT이 멱등인 근거는 "전체 교체" 의미라 반복해도 같은 상태가 되기 때문이고, 그래서 재시도 로직·프록시는 멱등 메서드만 자동 재시도해도 안전하다고 간주한다. POST에 멱등성이 필요하면 Idempotency-Key를 별도로 얹는 것이다.
- **중복 요청에 "같은 응답"을 돌려주려면 무엇을 저장해야 하나?**
  ↳ 멱등키와 함께 처리 상태(IN_PROGRESS/DONE)와 응답 본문까지 저장한다(response replay). 그래야 재시도가 "이미 처리됨 에러"가 아니라 원래 성공 응답을 그대로 받아, 클라이언트 입장에서 첫 시도와 구분이 없어진다 — Stripe 방식이 이 형태다. IN_PROGRESS 상태를 두는 이유는 동시 중복 요청이 처리 중간에 들어왔을 때 "기다리거나 거절"할 분기점이 필요해서다.
- **수학적 멱등(f(f(x))=f(x))과 API 멱등성은 어떻게 연결되나?**
  ↳ "잔액 -= 100"은 f(x)=x-100이라 f(f(x))≠f(x), 비멱등이다. "잔액 = 500으로 설정"은 상수 함수라 멱등이다. 즉 증분 연산을 절대값 연산(또는 조건부 연산)으로 바꾸는 것이 연산 수준에서 멱등성을 얻는 일반 기법이고, UPSERT·상태 덮어쓰기가 그 실무 형태다.
