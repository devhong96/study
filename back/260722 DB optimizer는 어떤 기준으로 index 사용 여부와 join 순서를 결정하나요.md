# 260722 : DB optimizer는 어떤 기준으로 index 사용 여부와 join 순서를 결정하나요?

DB optimizer는 어떤 기준으로 index 사용 여부와 join 순서를 결정하나요?

이 질문은 인덱스를 "걸면 빨라진다" 수준이 아니라, DB가 실제로 어떤 실행 계획을 선택하는지 이해하고 있는지 확인하기 좋은 질문입니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- DB optimizer는 여러 실행 계획 후보 중 비용이 가장 낮다고 판단되는 계획을 선택한다.
- 이때 테이블 통계, 인덱스 통계, 조건의 선택도, cardinality, 예상 row 수, join 방식, 정렬 비용 등을 고려한다.
- 인덱스가 있어도 조건이 많은 데이터를 읽어야 한다면 full scan이 더 낫다고 판단할 수 있다.
- Join 순서도 SQL에 작성된 순서대로 정해지는 것이 아니라, 중간 결과를 줄일 수 있는 방향으로 결정된다.
- 따라서 쿼리 성능을 분석할 때는 `EXPLAIN`으로 실제 실행 계획을 보고, 어떤 인덱스를 탔는지, 예상 row 수와 실제 처리량이 맞는지 확인해야 한다.

> 관련 문서: [인덱스의 랜덤 I/O와 커버링](../database/index-random-io-and-covering.md)

> **용어 약속(혼동 주의)**: 이 노트에서 **선택도**는 "조건이 **매칭시키는 row의 비율**"을 뜻한다(비율이 **높을수록** = 많이 걸림 → 인덱스 불리·풀스캔 유리). **cardinality**는 "컬럼/인덱스의 **구별되는 값의 수**"다(높을수록 값이 잘게 갈려 매칭 비율이 낮아짐 → 인덱스 유리). 즉 cardinality↑ ⇒ 매칭 비율↓ ⇒ 인덱스 유리. (주의: 업계·MySQL 문맥에선 "index selectivity = cardinality/rows(높을수록 좋은 인덱스)"라는 **정반대 용법**도 흔하다 — 면접에선 어느 정의로 쓰는지 먼저 밝히자.)

## 답변

> **한 줄 핵심**: 옵티마이저는 SQL을 적힌 대로 실행하지 않고, **통계로 각 후보 계획의 예상 비용(I/O+CPU)을 계산해 최저 비용 하나를 고르는** 비용 기반(cost-based) 엔진이다. index 사용 여부는 "그 조건이 몇 row를 고르나(매칭 비율)"로, join 순서는 "중간 결과를 가장 빨리 줄이는 순서"로 정해진다 — 둘 다 **통계(cardinality·예상 row 수) 추정치의 정확성**에 목숨이 걸려 있다.

### 1문 1답

**Q. DB optimizer는 어떤 기준으로 index 사용 여부와 join 순서를 결정하나요?**

**A.** 옵티마이저는 하나의 SQL에 대해 여러 실행 계획 후보(어떤 인덱스를 쓸지, 어떤 순서로 조인할지)를 만들고, 각 후보의 **예상 비용(디스크 I/O + CPU 연산량)을 통계로 계산해 가장 싼 계획을 고른다**. 그래서 "적힌 순서대로, 인덱스 있으면 무조건 탄다"가 아니다. index는 그 조건이 **매칭시키는 row 비율**이 낮을 때(=적게 걸러질 때) 유리한데, 매칭 비율이 크면 보조인덱스로 row를 찾은 뒤 테이블을 랜덤하게 재방문하는 비용이 풀스캔의 순차 읽기보다 오히려 비싸져 풀스캔을 택한다. join 순서는 SQL 텍스트 순서가 아니라, **먼저 조인해서 중간 결과 row를 최대한 줄이는 방향**으로 재배치된다. 이 모든 판단의 입력이 통계라, **통계가 낡으면 추정이 틀어져 엉뚱한 계획**을 고른다 — 그래서 실무에선 `EXPLAIN`으로 예상 row와 실제를 대조하는 게 핵심이다.

**Q. 인덱스가 있는데도 왜 풀스캔이 더 싸다고 판단하나요?**

**A.** 보조(secondary) 인덱스는 조건에 맞는 키를 찾은 뒤, 실제 컬럼을 읽으려면 **PK로 테이블을 다시 찾아가는 랜덤 I/O**가 매 row마다 붙는다(→ 관련 노트: 인덱스 랜덤 I/O). 매칭 비율이 낮으면 이 재방문이 몇 건뿐이라 이득이지만, 비율이 커지면 랜덤 재방문이 수없이 쌓여 **디스크를 순차로 쭉 읽는 풀스캔보다 비싸진다**. 경험칙으로 대략 **전체의 20~25%를 넘게 매칭**하면 풀스캔이 유리해지는 지점이 온다(정확한 임계는 랜덤/순차 I/O 비용비·row 크기에 따라 달라지는 **경험칙**이다). 단 인덱스만으로 조건과 SELECT 컬럼이 다 해결되는 **커버링 인덱스**면 테이블 재방문 자체가 없어 이 임계가 크게 완화된다.

**Q. join 순서는 무엇을 최소화하려고 정해지나요?**

**A.** 기본이 되는 nested loop join에서 비용은 대략 `(바깥 테이블 row 수) × (안쪽 테이블 1건 탐색 비용)`이다. 그래서 옵티마이저는 **row 수가 적거나 조건이 강하게 걸리는 테이블을 driving(바깥) 테이블로 앞에 두어**, 안쪽을 탐색(probe)하는 횟수 자체를 줄이려 한다. 안쪽 테이블은 조인 키에 인덱스가 있으면 1건 탐색이 싸지므로 함께 고려된다. 테이블이 많아지면 순서 조합이 폭발(n!)하므로, MySQL은 **그리디/휴리스틱 탐색**(`optimizer_search_depth`로 탐색 깊이 제한)으로 현실적인 시간 안에 좋은 순서를 찾는다 — 항상 전역 최적을 보장하진 않는다.

**Q. 이 판단에 쓰이는 통계는 무엇이고 왜 중요한가요?**

**A.** 대표적으로 **테이블의 대략적 row 수, 인덱스의 cardinality(구별되는 값의 수), 값 분포**다. 옵티마이저는 이걸로 "이 조건이면 몇 row 나오겠다"를 추정하는데, 추정 row 수가 실제와 어긋나면 인덱스 선택·조인 순서가 통째로 틀어진다. 예컨대 통계가 낡아 실제론 100만 건 매칭인데 100건으로 추정하면, 랜덤 재방문이 폭발할 인덱스 계획을 "싸다"고 골라버린다. 그래서 `EXPLAIN`의 예상 `rows`와 실제 처리량이 크게 벌어지면 **통계 갱신(`ANALYZE TABLE`)**을 먼저 의심한다.

### 면접 답변 (구술용)

DB 옵티마이저는 SQL을 적힌 순서대로 실행하는 게 아니라, 가능한 실행 계획 후보들을 만들어 각각의 예상 비용을 통계로 계산한 다음, 비용이 가장 낮은 계획을 고르는 비용 기반 엔진입니다. 인덱스를 탈지 말지는 그 조건이 전체에서 몇 퍼센트의 row를 걸러내느냐로 갈립니다. 보조 인덱스는 매 row마다 테이블을 랜덤하게 재방문하는 비용이 있어서, 매칭되는 비율이 크면 오히려 풀스캔으로 순차 읽는 게 싸다고 판단합니다. 경험적으로 대략 20에서 25퍼센트를 넘게 매칭하면 풀스캔이 유리해지는 구간이 옵니다. 조인 순서도 SQL에 적은 순서가 아니라, 중간 결과 row를 가장 빨리 줄일 수 있는 테이블을 먼저 조인하는 방향으로 재배치됩니다. 보통 row가 적거나 조건이 강한 테이블을 driving 테이블로 앞에 둬서 안쪽 탐색 횟수를 줄이는 식입니다. 이 모든 판단이 통계에 의존하기 때문에, 통계가 오래되면 추정 row 수가 실제와 어긋나 엉뚱한 인덱스나 조인 순서를 고르게 됩니다. 그래서 성능 분석할 때는 EXPLAIN으로 어떤 인덱스를 탔는지, 예상 rows와 실제 처리량이 맞는지를 보고, 벌어지면 통계 갱신부터 의심합니다.

### 원리 이해 (왜 그런가)

**옵티마이저의 판단 파이프라인:**

| 단계 | 하는 일 | 입력(통계) | 틀어지면 |
|------|---------|-----------|----------|
| ① 후보 생성 | 인덱스 조합·조인 순서·조인 방식 후보 열거 | 스키마·인덱스 정의 | (탐색 폭발 시 그리디로 제한) |
| ② 비용 추정 | 각 후보의 예상 I/O+CPU 계산 | 테이블 row 수, cardinality, 값 분포 | 통계 낡으면 추정 row 오차 |
| ③ 선택 | 최저 예상비용 계획 채택 | ②의 비용값 | 오차가 그대로 잘못된 선택으로 |

**인덱스 vs 풀스캔 — 비용이 역전되는 이유:**

```
매칭 row 비율 낮음                        매칭 row 비율 높음(≳20~25%, 경험칙)
─────────────────────                    ─────────────────────
보조인덱스 → 소수만 랜덤 재방문 → 싸다      보조인덱스 → 대량 랜덤 재방문 → 비싸다
                                          풀스캔 → 순차 읽기 → 더 싸다  ✅
```
- 핵심은 **랜덤 I/O 단가 > 순차 I/O 단가**. 재방문 건수가 많아지면 랜덤 비용 총합이 풀스캔을 추월한다.
- **커버링 인덱스**면 재방문이 없어 이 역전 지점이 훨씬 뒤로 밀린다.

**join 순서 — nested loop 비용 모델:**
- `비용 ≈ 바깥 row 수 × 안쪽 1건 탐색 비용`
- → 바깥은 **작게(선택도 강한 테이블)**, 안쪽은 **조인 키 인덱스로 탐색 싸게**. 이 둘을 만족하는 순서를 탐색.
- MySQL 조인 방식: 기본 **nested loop**. 인덱스를 못 쓰는 equi-join은 **hash join**(8.0.18+ 도입, 8.0.20에서 **block nested loop를 대체·제거**). 큰 테이블끼리 인덱스 없는 조인에서 hash join이 유리.

### 꼬리질문 Q&A

**Q. Index가 있는데도 full scan이 발생하는 이유는 무엇인가요?**

**A.** **인덱스가 "있다"와 "쓸 수 있다·쓰는 게 싸다"는 다르다.** 대표 원인:
- **매칭 비율이 큼**(낮은 선택도) → 위의 비용 역전으로 풀스캔이 쌈.
- **통계가 낡음** → 추정 row가 틀려 풀스캔을 잘못 선택.
- **non-sargable 조건**: 인덱스 컬럼에 함수·연산(`WHERE DATE(col)=...`, `col+1=...`)을 씌우거나 **암묵적 타입 변환**(문자열 컬럼에 숫자 비교)이 일어나면 인덱스를 못 탄다.
- **복합인덱스 선두 컬럼 미사용**(leftmost prefix 위반), `LIKE '%x'`(선두 와일드카드), `OR`로 인덱스 못 묶임.
- **테이블이 아주 작음** → 그냥 풀스캔이 더 쌈.

**Q. Cardinality가 낮은 컬럼에 인덱스를 걸면 어떤 문제가 생길 수 있나요?**

**A.** **한 값이 대량 row에 매칭돼 매칭 비율이 커지므로 옵티마이저가 대개 그 인덱스를 안 탄다.** 그러면:
- 조회 이득은 없는데 **쓰기마다 인덱스 갱신 비용 + 저장 공간**만 늘어난다(순손해).
- 예외: 값 분포가 **한쪽으로 크게 치우쳐** 있고(예: `status='ERROR'`가 0.1%) 그 희소 값만 자주 조회하면, 그 값에 한해선 유효할 수 있다 — 통계/히스토그램이 이 편향을 알아야 한다.

**Q. 멀티 컬럼 인덱스에서 컬럼 순서는 어떤 기준으로 정하나요?**

**A.** **① 등호(`=`) 조건 컬럼을 앞에, range(`>`, `<`, `BETWEEN`) 조건 컬럼을 뒤에.** range가 걸린 컬럼 **다음** 컬럼들은 인덱스로 범위를 더 좁히지 못하기 때문이다.
- ② 그 다음 **cardinality가 높은(잘 갈리는) 컬럼**을 앞쪽에 둬 빨리 좁힌다.
- ③ **leftmost prefix**: `(a,b,c)` 인덱스는 `a`, `a,b`, `a,b,c`엔 쓰이지만 `b`만·`c`만으론 못 쓴다.
- ④ `ORDER BY`/커버링까지 고려하면 순서가 달라질 수 있다(정렬 컬럼을 뒤에 붙여 filesort 회피).

**Q. `EXPLAIN`에서 어떤 지표를 보고 문제를 판단하나요?**

**A.** **`type`, `key`, `rows`, `filtered`, `Extra`를 본다.**
- `type`: `const`<`eq_ref`<`ref`<`range`<`index`<`ALL` 순으로 나빠짐. **`ALL`(풀스캔)·`index`(인덱스 풀스캔)** 이 큰 테이블에 뜨면 경고.
- `key`: 실제 선택된 인덱스(NULL이면 인덱스 안 탐).
- `rows`×`filtered`: 이 단계에서 다음으로 넘길 예상 row 수. **`rows`가 실제와 크게 다르면 통계 문제.**
- `Extra`: `Using index`(커버링, 좋음), **`Using filesort`·`Using temporary`**(정렬/임시테이블, 자주 튜닝 대상).
- 확정 진단은 **`EXPLAIN ANALYZE`**(8.0.18+)로 **예상 vs 실제 row·시간**을 대조.

**Q. 통계 정보가 오래되면 optimizer 판단에 어떤 영향이 생기나요?**

**A.** **추정 row 수가 실제와 어긋나 인덱스 선택·조인 순서가 통째로 틀어진다.**
- 대량 INSERT/DELETE 후 cardinality가 옛날 값이면, 실제론 대량 매칭인데 소량으로 착각해 랜덤 재방문 폭발하는 인덱스 계획을 고르는 식.
- 대응: **`ANALYZE TABLE`**로 수동 갱신. InnoDB는 **`innodb_stats_auto_recalc`**로 테이블의 **약 10%** row가 바뀌면 자동 재계산하고, **`innodb_stats_persistent`**로 통계를 디스크에 영속화한다(재시작에도 유지).

### 🌱 심화 키워드
- **Cost-Based Optimizer(CBO)** — 통계 기반 비용 추정으로 계획을 고르는 방식. 반대는 규칙 기반(RBO)
- **Cardinality / Selectivity** — 구별되는 값 수 / 조건이 매칭시키는 row 비율. 인덱스 유효성의 핵심 추정치
- **Index dive** — MySQL이 range 조건의 매칭 row 수를 인덱스를 실제로 훑어 추정하는 기법(`eq_range_index_dive_limit`)
- **Histogram(컬럼 통계)** — MySQL 8.0+ 값 분포를 담아 편향된 데이터의 선택도 추정을 개선(`ANALYZE TABLE ... UPDATE HISTOGRAM`)
- **Nested Loop / Hash Join** — 조인 실행 알고리즘. 인덱스 유무·테이블 크기로 선택
- **STRAIGHT_JOIN / 옵티마이저 힌트** — 옵티마이저 판단을 강제로 덮어쓸 때(최후 수단)
- **Sargable** — 인덱스를 탈 수 있는 형태의 조건(Search ARGument ABLE)

### ❓ 더 파볼 질문

**Q. 옵티마이저가 명백히 틀린 계획을 고를 때, 힌트로 강제하는 것과 통계/인덱스를 고치는 것 중 무엇이 우선인가?**

**A.** 원칙은 **통계 갱신·인덱스 설계로 옵티마이저가 스스로 옳게 고르게** 하는 것. 힌트(`STRAIGHT_JOIN`, `USE INDEX`)는 데이터 분포가 바뀌면 되레 독이 되므로, 근본 원인(낡은 통계·나쁜 인덱스)을 못 고칠 때의 **최후 수단**으로 본다.

**Q. 히스토그램은 인덱스와 무엇이 다르고 언제 쓰나?**

**A.** 인덱스는 조회 경로 + 통계를 겸하지만 쓰기 비용이 크다. 히스토그램은 **조회 경로 없이 값 분포만** 저장해, cardinality는 낮지만 분포가 크게 치우친 컬럼(인덱스 걸긴 아깝지만 선택도 추정은 정확히 하고 싶은 컬럼)의 추정을 개선한다.

**Q. 분산/샤딩 환경에선 이 옵티마이저 판단이 어떻게 달라지나?**

**A.** 단일 노드 비용 모델에 **네트워크 왕복·샤드 간 데이터 이동(shuffle)** 비용이 추가돼, 로컬에선 싼 조인이 분산에선 비쌀 수 있다. "조인을 어디서 수행하나(push down)"가 새 변수로 등장한다.
