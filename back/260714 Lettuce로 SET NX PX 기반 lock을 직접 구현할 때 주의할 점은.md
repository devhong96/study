# 260714 : Lettuce로 SET NX PX 기반 lock을 직접 구현할 때 주의할 점은 무엇인가요?

Lettuce로 `SET NX PX` 기반 lock을 직접 구현할 때 주의할 점은 무엇인가요?

이 질문은 Redis 분산락을 라이브러리 없이 직접 구현할 때, 락 획득과 해제의 안전성을 이해하고 있는지 확인하는 질문입니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- `SET key value NX PX ttl`은 key가 없을 때만 값을 저장하고, 동시에 만료 시간을 설정합니다.
- `NX`는 이미 락이 잡혀 있으면 획득하지 못하게 하고, `PX`는 애플리케이션 장애 시 락이 영원히 남지 않도록 TTL을 둡니다.
- 락 value에는 UUID 같은 고유 값을 넣어야 합니다. 그래야 내가 잡은 락인지 확인할 수 있습니다.
- 락 해제 시에는 단순히 `DEL key`를 호출하면 안 됩니다. 내 락이 만료된 뒤 다른 요청이 새로 잡은 락을 지워버릴 수 있기 때문입니다.
- 그래서 해제는 Lua script로 “value가 내가 저장한 값과 같을 때만 delete”하도록 원자적으로 처리하는 것이 안전합니다.
- TTL은 실제 작업 시간보다 충분히 길게 잡아야 하지만, 너무 길면 장애 시 대기 시간이 길어지므로 작업 시간, 재시도 정책, 장애 복구 시간을 함께 고려해야 합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- `SETNX` 후 `EXPIRE`를 따로 호출하면 어떤 문제가 생길 수 있나요?
- Lock value를 UUID로 두는 이유는 무엇인가요?
- Lua script 없이 `GET` 후 `DEL`을 따로 호출하면 어떤 race condition이 생기나요?
- TTL이 작업 시간보다 먼저 만료되면 어떤 문제가 발생하나요?
- 직접 구현한 lock과 Redisson lock의 차이는 무엇인가요?

---

## 답변

> **한 줄 핵심**: 직접 구현의 모든 주의점은 한 원칙에서 나온다 — **"확인과 변경 사이에 틈이 생기는 지점을 전부 원자적으로 만들어라"**: 획득은 SET NX PX 한 방, 소유자는 value의 UUID, 해제는 Lua의 compare-and-delete.

### 1문 1답

**Q. Lettuce로 SET NX PX 기반 lock을 직접 구현할 때 주의할 점은 무엇인가요?**

**A.** 모든 주의점은 "확인과 변경 사이에 틈이 생기는 지점을 전부 원자적으로 만들어라"는 한 원칙에서 나오고, 핵심은 세 군데의 원자성이다. 첫째, 획득은 SET key value NX PX ttl 한 번으로 해야 한다 — SETNX 후 EXPIRE를 따로 부르면 두 명령 사이에 죽었을 때 TTL 없는 락이 영구히 남아 모든 후속 요청이 막힌다. 둘째, value에 UUID 같은 소유자 고유값을 넣어야 한다 — 락 key는 모두가 같은 이름을 쓰므로 해제 시 "이 락이 내 것인지"를 확인할 근거가 value뿐이다. 셋째, 해제는 Lua로 "값 비교 후 삭제"를 원자적으로 해야 하는데, GET 확인과 DEL 사이에 내 락이 만료되고 다른 요청이 새로 잡는 순간이 끼면 남의 락을 지워 세 번째 요청까지 진입하기 때문이다. 그 외에 TTL은 작업 p99에 GC·네트워크 여유를 더해 잡되 무한정 길지 않게 하고, Lettuce에는 대기 메커니즘이 없어 획득 실패 시 백오프 재시도를 직접 짜야 하며 재진입·TTL 연장도 없다. 또 TTL 만료 후 늦은 쓰기는 락만으로 못 막으니 조건부 UPDATE 같은 데이터 레벨 방어가 필요하다 — 획득은 한 줄이지만 안전한 운영은 한 줄이 아니라는 게 요지다.

**Q. SET key value NX PX ttl에서 NX와 PX는 각각 어떤 역할을 하나요?**

**A.** NX는 key가 없을 때만 값을 저장하도록 해서, 이미 락이 잡혀 있으면 획득하지 못하게 하는 경쟁 판정 역할을 합니다. PX는 애플리케이션 장애가 나도 락이 영원히 남지 않도록 만료 시간(TTL)을 거는 역할입니다. 핵심은 이 둘을 SET 한 명령으로 묶는다는 점인데, 그래야 "락이 생기는 순간 만료도 함께 생긴다"가 원자적으로 보장됩니다. 만약 이를 SETNX와 EXPIRE로 나누면 두 명령 사이에 crash가 났을 때 TTL 없는 락이 영구히 남는 틈이 생깁니다. 그래서 획득은 반드시 SET key value NX PX ttl 한 방으로 처리해야 합니다.

**Q. 락 value에 UUID 같은 고유 값을 넣어야 하는 이유는 무엇인가요?**

**A.** 락 key는 모든 요청이 같은 이름을 공유하므로, 해제나 연장 시점에 "이 락이 내 것인지"를 확인할 수 있는 유일한 근거가 value뿐이기 때문입니다. 시나리오로 보면, 내 작업이 길어져 TTL이 만료되고 B가 같은 key로 락을 잡은 뒤, 내가 뒤늦게 작업을 마치고 삭제를 호출하는 상황이 생길 수 있습니다. 이때 value 검증이 없으면 B의 락을 지워 버리고, 그 틈에 C까지 진입해 상호배제가 연쇄적으로 무너집니다. UUID 같은 소유자 고유값이 있으면 "value가 내 것일 때만 삭제"가 가능해져 이 연쇄를 끊습니다. 즉 UUID는 소유자 검증의 핵심 재료입니다.

**Q. 락 해제 시 DEL key만 호출하거나 GET 후 DEL을 따로 하면 왜 안 되고, 왜 Lua script로 해야 하나요?**

**A.** 단순히 DEL만 호출하면 내 락이 이미 만료된 뒤 다른 요청이 새로 잡은 락을 지워버릴 수 있습니다. 이를 피하려고 GET으로 내 값을 확인한 뒤 DEL하는 방식도, 확인과 삭제 사이에 틈이 있어 위험합니다 — GET으로 통과했는데 그 직후 내 TTL이 만료되고 B가 락을 잡으면, 뒤이은 DEL이 B의 락을 지우고 C까지 진입해 상호배제가 붕괴됩니다. 이것이 전형적인 check-then-act 레이스, 즉 "확인 시점과 행동 시점 사이에 세계가 바뀔 수 있다"는 동시성의 기본 문제입니다. Redis는 Lua 스크립트를 단일 원자 단위로 실행하므로, "value가 내 값일 때만 삭제"하는 compare-and-delete를 스크립트 하나로 만들면 그 사이에 만료나 타인 획득이 끼어들 틈 자체가 사라집니다. 그래서 안전한 해제는 Lua compare-and-delete가 표준입니다.

**Q. TTL은 무엇을 기준으로 정해야 하나요?**

**A.** TTL은 실제 작업 시간보다 충분히 길게 잡되, 무한정 길지 않게 하는 균형이 필요합니다. 구체적으로는 작업 p99에 GC pause와 네트워크 지연 여유를 더한 값을 기준으로 잡는 것이 안전합니다. 너무 짧으면 작업 도중 락이 풀려 동시 진입과 lost update가 생기고, 너무 길면 홀더 장애 시 대기 시간이 그만큼 길어지기 때문입니다. 그래서 작업 시간, 재시도 정책, 장애 복구 시간을 함께 고려해서 정해야 합니다. 다만 직접 구현에서는 TTL 연장(watchdog) 기능이 없으므로, 애초에 보수적으로 잡는 것이 특히 중요합니다.

**Q. SETNX 후 EXPIRE를 따로 호출하면 어떤 문제가 생기나요?**

**A.** 두 명령 사이에 원자성이 없다는 것이 문제의 핵심입니다. SETNX가 성공한 직후, EXPIRE를 부르기 전에 애플리케이션이 crash하면 TTL이 없는 락이 영구히 남습니다. 그러면 이후 모든 요청이 그 락에 막히고, 누군가 수동으로 key를 지울 때까지 해당 기능 전체가 정지합니다. 이는 락이라는 안전장치가 오히려 전면 장애의 원인이 되는 상황입니다. SET의 NX·PX 옵션 결합은 "락이 생기는 순간 만료도 함께 생긴다"를 명령 하나로 보장해 바로 이 틈을 없앱니다.

**Q. TTL이 작업 시간보다 먼저 만료되면 어떤 문제가 발생하나요?**

**A.** 락이 풀린 줄 모르는 A와 새로 잡은 B가 동시에 실행되어 상호배제가 깨지고, A의 뒤늦은 커밋이 B의 결과를 덮는 lost update가 발생할 수 있습니다. 대응은 층위별로 나눕니다. 예방 층위에서는 보수적 TTL과 주기적 연장(watchdog 자작)으로 만료 확률 자체를 낮춥니다. 검출·차단 층위에서는 최종 쓰기 단계에서 fencing token, 버전 비교(낙관적 락), 조건부 UPDATE로 "내가 읽은 상태 그대로일 때만 쓰기"를 강제합니다. 이 두 번째 층위가 있어야 예방이 뚫려도 데이터가 지켜지므로, 락만으로 정합성을 보장하려 하지 않는 것이 핵심입니다.

**Q. 직접 구현한 lock과 Redisson lock의 차이는 무엇인가요?**

**A.** 직접 구현은 원자적 획득/해제, 대기(재시도 루프와 백오프), TTL 연장, 재진입을 각자 직접 책임져야 합니다. 반면 Redisson은 그 각각이 소유자 검증까지 포함해 검증된 구현으로 내장돼 있고, 공정 락·ReadWriteLock·Semaphore 같은 부가 도구도 제공합니다. 예를 들어 대기의 경우 직접 구현은 재시도 루프에 백오프를 안 넣으면 스핀 부하가 생기지만, Redisson은 pub/sub 대기로 처리합니다. 그래서 실무 기준은 "동작 원리 학습이나 아주 단순한 케이스는 직접 구현도 가능하지만, 운영 락은 라이브러리"입니다. 단 Redisson도 TTL 경계의 늦은 쓰기나 failover 시 비동기 복제로 인한 락 유실 같은 분산 시스템의 본질적 한계는 없애 주지 못한다는 점은 동일합니다.

### 면접 답변 (구술용)

핵심 주의점은 세 군데의 원자성입니다. 첫째, 획득은 SET key value NX PX ttl 한 번으로 해야 합니다 — SETNX 후 EXPIRE를 따로 부르면 두 명령 사이에 애플리케이션이 죽었을 때 TTL 없는 락이 영구히 남아 모든 후속 요청이 막힙니다. 둘째, value에 UUID 같은 소유자 고유값을 넣어야 합니다 — 락 key는 모두가 같은 이름을 쓰므로, 해제할 때 "이 락이 내 것인지"를 확인할 근거가 value뿐입니다. 셋째, 해제는 Lua 스크립트로 "값 비교 후 삭제"를 원자적으로 해야 합니다 — GET으로 확인하고 DEL하는 두 단계 사이에 내 락이 만료되고 다른 요청이 새로 잡는 순간이 끼어들 수 있고, 그러면 남의 락을 지워 세 번째 요청까지 진입합니다. 그 외에 TTL은 작업 p99에 GC pause와 네트워크 지연 여유를 더해 잡되 무한정 길지 않게 하고, Lettuce에는 대기 메커니즘이 없으니 획득 실패 시의 재시도 루프도 백오프를 넣어 직접 짜야 하며, 재진입과 TTL 연장도 지원되지 않는다는 점, TTL 만료 후 늦은 쓰기는 락만으로 못 막아서 조건부 UPDATE 같은 데이터 레벨 방어가 필요하다는 점까지 챙겨야 합니다. 획득은 한 줄이지만 안전한 운영은 한 줄이 아니라는 게 이 질문의 요지입니다.

### 원리 이해 (왜 그런가)

**락의 생애와 원자성이 필요한 3지점:**

```
[획득]  SET lock:stock {uuid} NX PX 10000     ← NX(경쟁 판정) + PX(만료)를 한 명령으로
[보호]  value = 내 UUID                        ← "내 락"의 유일한 증거
[해제]  Lua: if GET(key) == my_uuid then DEL   ← 비교와 삭제 사이의 틈 제거
```

**해제 Lua 스크립트 (compare-and-delete):**

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```
Redis는 스크립트를 단일 원자 단위로 실행하므로(단일 스레드 순차 실행 — 260701), 비교와 삭제 사이에 만료·타인 획득이 끼어들 수 없습니다.

**GET 후 DEL의 race 타임라인 (왜 Lua여야 하나):**

```
t1: A가 GET → 내 값 확인 (통과)
t2: A의 TTL 만료
t3: B가 SET NX 성공 (락 획득)
t4: A가 DEL 실행 → B의 락 삭제됨
t5: C가 SET NX 성공 → B와 C 동시 실행 (상호배제 붕괴)
```
전형적인 check-then-act 레이스 — "확인 시점과 행동 시점 사이에 세계가 바뀔 수 있다"는 동시성의 기본 문제입니다.

**직접 구현이 추가로 감당해야 하는 목록:**

| 항목 | Lettuce 직접 구현 | Redisson |
|------|-------------------|----------|
| 원자적 획득/해제 | SET NX PX + Lua 직접 작성 | 내장 (소유자 검증 포함) |
| 대기 | 재시도 루프 직접 (백오프 없으면 스핀 부하) | pub/sub 대기 (→ 260717) |
| TTL 연장 | 없음 (직접 구현해야) | watchdog (→ 260716) |
| 재진입 | 불가 | 지원 (스레드별 카운트) |
| 부가 도구 | — | 공정 락, ReadWriteLock, Semaphore 등 |

### 꼬리질문 Q&A

**Q. SETNX 후 EXPIRE를 따로 호출하면 어떤 문제가 생기나요?**

**A.** **두 명령 사이에 원자성이 없어서, SETNX 성공 직후 crash하면 TTL 없는 락이 영구 잔존한다.**
이후 모든 요청이 그 락에 막히고 수동으로 key를 지울 때까지 해당 기능 전체가 정지합니다. SET의 NX·PX 옵션 결합은 "락이 생기는 순간 만료도 함께 생긴다"를 명령 하나로 보장해 이 틈을 없앱니다.

**Q. Lock value를 UUID로 두는 이유는 무엇인가요?**

**A.** **해제·연장 시점에 소유자를 검증할 유일한 근거이기 때문이다.**
시나리오로 보면 — 내 작업이 길어져 TTL 만료 → B가 같은 key로 락 획득 → 내가 작업을 마치고 DEL 호출. value 검증이 없으면 여기서 B의 락이 지워지고 C까지 진입합니다. UUID가 있으면 "value가 내 것일 때만 삭제"가 가능해져 이 연쇄를 끊습니다.

**Q. Lua 없이 GET 후 DEL을 따로 호출하면 어떤 race condition이 생기나요?**

**A.** **위 타임라인 그대로 — 확인(GET)과 삭제(DEL) 사이에 내 락 만료 + 타인 획득이 끼어들어 남의 락을 지우게 된다.**
Lua 스크립트는 Redis에서 단일 원자 단위로 실행되어 그 사이에 다른 명령이 끼어들 수 없으므로, compare-and-delete를 스크립트 하나로 만들면 틈 자체가 사라집니다.

**Q. TTL이 작업 시간보다 먼저 만료되면 어떤 문제가 발생하나요?**

**A.** **락이 풀린 줄 모르는 A와 새로 잡은 B가 동시 실행되고(상호배제 실패), A의 뒤늦은 커밋이 B의 결과를 덮을 수 있다(lost update).**
대응은 층위별로: ① 예방 — 보수적 TTL + 주기적 연장(watchdog 자작), ② 검출·차단 — 최종 쓰기 단계에서 fencing token, 버전 비교(낙관적 락), 조건부 UPDATE로 "내가 읽은 상태 그대로일 때만 쓰기"를 강제. ②가 있어야 ①이 뚫려도 데이터가 지켜집니다(→ 260706).

**Q. 직접 구현한 lock과 Redisson lock의 차이는 무엇인가요?**

**A.** **직접 구현은 위 표의 전 항목을 각자 책임지는 것이고, Redisson은 그것들이 검증된 구현으로 내장돼 있다는 것이다.**
그래서 실무 기준은 "동작 원리 학습이나 아주 단순한 케이스는 직접 구현도 가능하지만, 운영 락은 라이브러리"입니다. 단 Redisson도 분산 시스템의 본질적 한계 — TTL 경계의 늦은 쓰기, failover 시 비동기 복제로 인한 락 유실 — 는 없애 주지 못한다는 점은 동일합니다(→ 260715).

### ❓ 더 파볼 질문
**Q. Redlock 알고리즘은 무엇을 더 보장하려 했고, 왜 논쟁이 됐나?**

**A.** 독립된 Redis 노드 N개(보통 5) 중 과반에서 시간 안에 락을 획득하면 유효한 락으로 보는 방식 — 단일 노드 장애·failover 유실을 견디려는 설계다. Kleppmann은 "클라이언트 GC pause에는 여전히 무력하고(fencing token 부재), 시계 가정도 위험하다"고 비판했고 antirez가 반론했다. 논쟁 자체가 분산락의 한계를 이해하는 최고의 교재다.

**Q. WAIT 명령으로 복제 확인을 하면 failover 유실을 막을 수 있나?**

**A.** WAIT는 지정 수의 replica에 쓰기가 전파될 때까지 기다려 유실 확률을 줄여주지만, Redis 복제는 여전히 비동기 기반이라 완전 보장은 아니고 지연 비용이 추가된다. "확률을 낮추는 장치"이지 합의(consensus)가 아니라는 구분이 중요하다.

**Q. 락 구현에서 시계(clock)는 어디에 숨어 있나?**

**A.** TTL 판정은 Redis 서버의 시계가, "내 작업이 TTL 안에 끝났는지" 판단은 클라이언트의 시계가 한다. 클라이언트 쪽은 벽시계(currentTimeMillis)가 아니라 모노토닉 클록(nanoTime)으로 경과 시간을 재야 NTP 보정에 흔들리지 않는다 — 분산 시스템에서 시계를 믿지 말라는 원칙의 축소판이다.
