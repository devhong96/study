# 260617 DB Connection Pool이 고갈되는 원인은 무엇일까요?

면접에서 “Connection Pool을 왜 쓰나요?”는 꽤 자주 나오는 질문인데요.
조금 더 실무적으로 들어가면 이런 식으로 물어볼 수 있습니다.

“Connection Pool이 고갈된다면 어떤 상황을 의심해야 하나요?”
“Slow Query 때문인가요, Lock Wait 때문인가요, 아니면 외부 API 호출 때문인가요?”

이 질문은 단순히 커넥션 풀 개념을 아는지보다,
장애 상황에서 원인을 레이어별로 나눠서 볼 수 있는지를 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- 쿼리가 오래 실행되어 커넥션을 오래 점유하는 경우
- DB 락 대기로 트랜잭션이 끝나지 않는 경우
- 트랜잭션 안에서 외부 API를 호출해 커넥션 반환이 늦어지는 경우
- 커넥션 풀 크기보다 동시에 DB를 사용하는 요청이 많은 경우
- 커넥션 누수로 반납되지 않는 경우

---

## 답변

> **한 줄 핵심**: 고갈 = (커넥션 점유 시간 × 동시 요청 수)가 풀 용량을 초과하는 것 — 원인은 "점유가 길어졌다"(쿼리·락·외부호출·누수) 아니면 "수요가 늘었다" 둘 중 하나다.

### 1문 1답

**Q. DB Connection Pool이 고갈되는 원인은 무엇일까요?**

**A.** 고갈은 결국 (커넥션 점유 시간 × 동시 요청 수)가 풀 용량을 넘는 것으로, "점유가 길어졌다" 아니면 "수요가 늘었다" 둘 중 하나다. 레이어별로 나눠 의심하면, DB 레이어에서는 Slow Query로 쿼리 자체가 오래 걸리거나 락 대기로 트랜잭션이 안 끝나는 경우가 있다. 애플리케이션 레이어에서는 트랜잭션 안에서 외부 API를 호출해 응답 지연만큼 커넥션을 쥐고 있는 경우가 실무에서 가장 흔하고, 커넥션 누수도 있다. 용량 레이어에서는 동시 요청이 풀 크기를 넘는 경우인데, 이때 풀만 늘리면 병목이 DB 내부 경합으로 옮겨갈 수 있어 근본 원인부터 확인해야 한다. 진단은 HikariCP 메트릭으로 풀 상태를 보고, DB에서 실행 중 쿼리·락을 확인하고, 그래도 없으면 스레드 덤프로 커넥션을 쥔 채 외부를 기다리는 스레드를 찾는 순서로 한다.

**Q. 쿼리가 오래 실행되어 커넥션을 오래 점유하는 경우는 어떻게 확인하나요?**

**A.** Slow Query는 쿼리 실행 시간이 곧 커넥션 점유 시간이 되기 때문에, 느린 쿼리 한 건이 커넥션을 오래 붙잡아 전체 동시 처리량을 갉아먹는다. 이때 DB는 실제로 바쁜 상태로 나타나므로, slow query log와 실행 계획으로 어떤 쿼리가 오래 걸리는지 확인한다. 진단 순서로 보면 먼저 HikariCP 메트릭으로 정말 고갈인지 본 뒤, active가 만석이면 DB에서 SHOW PROCESSLIST로 실행 중인 쿼리를 살핀다. 주의할 점은 느린 쿼리가 원인일 때 풀 크기부터 키우면 증상만 가리게 된다는 것이다. 근본적으로는 쿼리 자체(인덱스·실행 계획)를 손봐 점유 시간을 줄여야 한다.

**Q. DB 락 대기로 트랜잭션이 끝나지 않는 경우는 어떻게 알아채나요?**

**A.** 락 경합에는 두 종류의 세션을 나눠 봐야 한다. 락을 능동적으로 기다리는 **대기자**는 InnoDB row lock 기준으로는 SHOW PROCESSLIST에서 쿼리가 실행 중인 것처럼 보이며 Time만 계속 증가하고, 대기가 끝나 쿼리가 완료되면 그 대기 시간이 Query_time에 포함되어 slow query log에도 잡힐 수 있다 — 즉 대기자는 오히려 흔적이 남는 편이다. (참고: 'Waiting for ... lock' 같은 상태 문자열은 InnoDB row lock이 아니라 메타데이터 락·테이블 락에서 나타난다.) 진짜 함정은 락을 쥔 **보유자** 쪽인데, 쿼리는 이미 끝냈지만 커밋하지 않은 채 트랜잭션을 열어두고 idle 상태로 락을 쥐고 있는(idle-in-transaction) 세션이다. 이건 실행 중인 쿼리가 없어서 SHOW PROCESSLIST의 실행 쿼리로도, slow query log로도 드러나지 않는다. 그래서 보유자는 `performance_schema.data_locks`·`performance_schema.data_lock_waits`(또는 `sys.innodb_lock_waits` 뷰, `SHOW ENGINE INNODB STATUS`)로 "누가 무슨 락을 쥐고 누가 기다리는지"의 관계를 봐야 찾을 수 있다. 진단할 때는 실행 중 쿼리뿐 아니라 이 락 보유·대기 관계를 함께 조회해야 한다.

**Q. 트랜잭션 안에서 외부 API를 호출해 커넥션 반환이 늦어지는 경우는 왜 위험한가요?**

**A.** 외부 API의 응답 시간이 그대로 커넥션 점유 시간이 되기 때문에, 외부 서비스가 느려지거나 장애가 나면 그 지연만큼 커넥션이 묶여 풀이 고갈된다. 이는 외부 장애가 우리 시스템의 고갈로 번지는 전형적인 장애 전파 경로이며, 실무에서 가장 흔한 원인 중 하나다. 증상으로는 외부 장애와 거의 동시에 풀이 고갈되고, DB는 한가한데 커넥션이 안 돌아오는 패턴으로 나타난다. 이때는 스레드 덤프를 떠서 커넥션을 쥔 채 외부 응답을 기다리는 스레드를 찾는다. 근본 대책은 외부 호출을 트랜잭션 밖으로 빼거나, bulkhead로 외부 API용 자원을 격리하는 것이다.

**Q. 커넥션 풀 크기보다 동시에 DB를 사용하는 요청이 많은 경우는 어떻게 대응하나요?**

**A.** 동시 요청이 풀 크기(HikariCP 기본 10)를 넘으면 초과분이 커넥션을 얻지 못하고 pending으로 쌓이며, 이 증가는 트래픽 피크와 동기화되어 나타난다. 언뜻 풀을 키우면 될 것 같지만 그게 항상 답은 아니다. DB가 동시에 실제로 처리할 수 있는 작업 수는 CPU·디스크에 묶여 있어서, 풀만 늘리면 대기 위치가 애플리케이션(풀 대기)에서 DB 내부 경합으로 옮겨갈 뿐이고 오히려 전체 처리량이 떨어질 수 있다. 그래서 HikariCP 위키도 풀을 작게 유지하라고 권한다. 크기를 늘리기 전에 점유 시간을 줄일 수 있는지부터 확인하고, 최종 크기는 부하 테스트로 정하는 게 맞다.

**Q. 커넥션 누수로 반납되지 않는 경우는 어떻게 잡나요?**

**A.** 예외 경로에서 close가 누락되는 등 커넥션이 반납되지 않으면, 트래픽과 무관하게 active가 서서히 우상향하다 결국 고갈된다. 트래픽 피크와 상관없이 천천히 차오르는 패턴이 누수의 대표적 신호다. 이걸 잡는 가장 빠른 방법은 HikariCP의 `leakDetectionThreshold`를 설정하는 것인데, 지정한 시간(N초) 이상 반납되지 않은 커넥션의 획득 지점 스택트레이스를 로그로 남겨준다. 그 스택을 따라가면 반납이 빠진 코드 위치를 특정할 수 있다. 진단 플로우상 "서서히 고갈되는 패턴"이 보이면 누수를 의심하고 이 설정을 켜는 순서로 접근한다.

### 면접 답변 (구술용)

커넥션 풀 고갈은 결국 "빌린 커넥션을 오래 반납하지 않거나, 동시에 빌리려는 요청이 풀 크기를 넘는" 상황이고, 레이어별로 나눠 의심합니다. DB 레이어에서는 Slow Query로 쿼리 자체가 오래 걸리는 경우와, 락 대기로 트랜잭션이 안 끝나는 경우가 있습니다. 애플리케이션 레이어에서는 트랜잭션 안에서 외부 API를 호출해 응답 지연만큼 커넥션을 쥐고 있는 경우 — 실무에서 가장 흔한 원인 중 하나입니다 — 와 커넥션 누수가 있습니다. 용량 레이어에서는 동시 요청이 풀 크기를 넘는 경우인데, 이때 풀만 늘리면 병목이 DB 내부 경합으로 옮겨갈 수 있어서 근본 원인부터 확인합니다. 진단은 HikariCP 메트릭으로 풀 상태를 보고, DB에서 실행 중 쿼리와 락을 확인하고, 그래도 없으면 스레드 덤프로 커넥션을 쥔 채 외부를 기다리는 스레드를 찾는 순서로 합니다.

### 원리 이해 (왜 그런가)

**원인 분류표 (레이어별 의심 순서):**

| 레이어 | 원인 | 메커니즘 | 특징적 증상 / 확인 |
|--------|------|----------|--------------------|
| DB | Slow Query | 쿼리 실행 시간 = 점유 시간 | slow query log, 실행 계획 |
| DB | Lock 대기 | 대기자는 남의 락을 기다리고, 보유자는 커밋 없이 락을 쥔 채 idle → 트랜잭션 미종료 | 대기자(row lock)는 실행 중처럼 Time만 증가(완료 시 Query_time 반영·slow log 흔적) / 보유자(idle-in-transaction)는 실행 쿼리가 없어 `data_locks`·`data_lock_waits`(또는 `sys.innodb_lock_waits`)로만 찾음 |
| 앱 | 트랜잭션 안 외부 API | 외부 응답 시간 = 커넥션 점유 시간 | 외부 장애와 동시에 풀 고갈 (장애 전파 경로) |
| 앱 | 커넥션 누수 | 반납 누락 (예외 경로 close 누락 등) | 트래픽과 무관하게 active 우상향, 서서히 고갈 |
| 용량 | 동시 요청 > 풀 크기 | 수요 초과 (HikariCP 기본 10) | 트래픽 피크와 동기화된 pending 증가 |

**"풀 늘리기"가 항상 답이 아닌 근거**: DB가 동시에 실제 처리할 수 있는 작업 수는 CPU·디스크에 묶여 있다. 풀만 늘리면 대기 위치가 애플리케이션(풀 대기)에서 DB(내부 경합)로 이동할 뿐이며 오히려 전체 처리량이 떨어질 수 있다(HikariCP 위키 "About Pool Sizing"의 논지). 느린 쿼리가 원인일 때 풀 확대는 증상 은폐다.

**진단 플로우:**

```
① HikariCP 메트릭 (active / idle / pending) — 정말 고갈인가?
② active 만석 → DB에서 SHOW PROCESSLIST + lock wait 조회 (느린 쿼리? 락?)
③ DB는 한가 → 스레드 덤프 (커넥션 쥐고 외부 API 대기 중인 스레드?)
④ 서서히 고갈 패턴 → 누수 의심 → leakDetectionThreshold 설정
   (N초 이상 미반납 커넥션의 획득 스택트레이스를 로그로 → 누수 위치 특정)
```

### 꼬리질문 Q&A (체크포인트 기준)

**Q. Slow Query 때문인지, Lock Wait 때문인지, 외부 API 때문인지 어떻게 구분하나?**
→ **DB가 바쁘면 쿼리, DB에 대기 세션이 보이면 락, DB가 한가한데 커넥션이 안 돌아오면 앱(외부 호출·누수).**
- 함정은 락을 쥔 채 커밋 없이 idle한 **보유자**(idle-in-transaction) — 실행 쿼리가 없어 SHOW PROCESSLIST·slow log에 안 드러나고 `data_locks`·`data_lock_waits`(또는 `sys.innodb_lock_waits`)로만 보인다. (락을 기다리는 대기자는 실행 중처럼 Time만 늘며 완료 시 slow log에 흔적이 남아 오히려 잡기 쉽다.)

**Q. 커넥션 누수는 어떻게 잡나?**
→ **HikariCP `leakDetectionThreshold`로 미반납 커넥션의 획득 지점 스택을 확보하는 게 가장 빠르다.**

**Q. 풀 크기는 어떻게 정하나?**
→ **무작정 크게가 아니라 DB의 실동시성 한계에 맞춰 작게 — 부족하면 늘리기 전에 점유 시간부터 줄인다.**
- 커넥션 수를 늘려 해결되는 문제인지, 점유 시간을 줄여야 하는 문제인지 구분이 실력.

### 🌱 심화 키워드
- **HikariCP maximumPoolSize / connectionTimeout / maxLifetime** — 풀 동작을 결정하는 3대 설정과 DB wait_timeout의 관계
- **leakDetectionThreshold** — 누수 추적의 표준 도구
- **OSIV(Open Session In View)** — 스프링부트 기본 true; 뷰 렌더링까지 영속성 컨텍스트가 살아남는 구조가 커넥션 점유에 미치는 영향
- **bulkhead** — 기능별 풀 분리로 장애 전파 차단 (외부 API용 커넥션/스레드 격리)
- **backpressure / admission control** — 풀 크기 = 시스템이 받아들일 동시 작업량의 상한이라는 관점

### ❓ 더 파볼 질문
**Q. OSIV가 켜져 있으면 커넥션은 정확히 언제 반납되나?**

**A.** 트랜잭션이 끝나면 그 커넥션은 반납되지만, 영속성 컨텍스트(EntityManager)는 요청 끝까지 살아 있어서 뷰 렌더링 중 lazy loading이 일어나면 그 시점에 커넥션을 다시 얻어 쓴다. 즉 "컨트롤러/뷰 계층에서도 DB 접근이 생길 수 있다"는 게 문제의 본질 — 커넥션 사용 시점이 서비스 계층 밖으로 새어 나가 점유 예측이 어려워지고, 외부 API를 뷰 데이터 구성 중에 부르는 코드와 겹치면 고갈을 가속한다.

**Q. HikariCP 권장 공식 `(core_count × 2) + effective_spindle_count`의 근거는? SSD 시대에도 유효한가?**

**A.** "CPU가 코어당 하나를 실행하는 동안 디스크 대기 중인 커넥션이 하나 더 있을 수 있다 + 회전 디스크 수만큼 병렬 I/O가 가능하다"는 HDD 시대 모델에서 나온 경험식이다. SSD·클라우드 스토리지에서는 spindle 개념이 무의미해져 문자 그대로는 안 맞지만, "풀은 생각보다 훨씬 작아야 한다(수십이 아니라 십 단위)"는 공식의 메시지는 여전히 유효하다 — 최종값은 부하 테스트로.

**Q. maxLifetime을 DB의 wait_timeout보다 짧게 잡아야 하는 이유는?**

**A.** DB가 유휴 커넥션을 먼저 끊으면 풀은 그 사실을 모른 채 죽은 커넥션을 들고 있다가, 애플리케이션이 꺼내 쓰는 순간 오류가 터진다("connection is closed"류의 간헐 장애). 풀이 먼저 은퇴시키면(maxLifetime < wait_timeout, 여유 포함) 항상 살아있는 커넥션만 대여되므로 이 간헐 오류가 사라진다 — "끊는 주도권을 풀이 갖게 하라"는 원칙이다.
