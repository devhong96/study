# 260716 : Redisson lock watchdog은 어떤 문제를 해결하고 어떤 부작용을 만들 수 있나요?

**오늘의 백엔드 질문**

Redisson lock watchdog은 어떤 문제를 해결하고 어떤 부작용을 만들 수 있나요?

이 질문은 Redisson을 단순히 “분산락 라이브러리”로만 알고 있는지, 실제 운영에서 락 유지 시간과 장애 상황을 어떻게 이해하는지 확인하기 좋습니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- Redisson watchdog은 락을 잡은 스레드가 정상적으로 작업 중일 때 lock TTL을 자동으로 연장해주는 기능입니다.
- Lease time을 명시하지 않고 lock을 획득하면, Redisson이 내부적으로 주기적으로 TTL을 갱신해서 작업 도중 락이 만료되지 않게 합니다.
- 이 기능은 작업 시간이 예상보다 길어져도 락이 먼저 풀려 다른 요청이 동시에 들어오는 문제를 줄여줍니다.
- 하지만 애플리케이션이 멈추거나 네트워크 문제가 생기면 TTL 갱신이 중단되고, 일정 시간이 지난 뒤 락이 풀립니다.
- 부작용으로는 작업이 비정상적으로 오래 걸릴 때 락도 계속 연장되어 다른 요청이 오래 대기할 수 있고, 실제로는 장애인데 살아 있는 것처럼 락이 유지될 수 있습니다.
- 그래서 watchdog을 쓰더라도 timeout, fallback, 모니터링, 보상 처리 설계를 함께 해야 합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- Lease time을 직접 지정하면 watchdog은 어떻게 동작하나요?
- Watchdog이 TTL을 연장하는 주기는 어떻게 정해지나요?
- Watchdog이 있어도 deadlock이나 장기 대기를 완전히 막을 수 있을까요?
- 작업이 오래 걸리는 로직에서 lease time 지정과 watchdog 중 무엇을 선택하겠나요?
- Lock이 오래 유지되는 상황을 운영에서 어떻게 감지하고 대응하나요?

---

## 답변

> **한 줄 핵심**: watchdog은 "짧은 TTL + 살아있는 동안 주기 갱신"으로 **lease 추정 딜레마(짧으면 조기 해제, 길면 장기 잠금)를 제거**한다 — 대신 그 대가로 **최악 점유 시간의 상한이 사라진다**(hang인데 살아있으면 무한 연장). 얻은 것과 잃은 것이 정확히 같은 지점이다.

### 면접 답변 (구술용)

해결하는 문제는 lease time을 미리 못 박아야 한다는 딜레마입니다. 고정 TTL은 작업보다 짧으면 도중에 락이 풀려 동시 진입이 생기고, 넉넉히 길게 잡으면 홀더가 죽었을 때 그 시간 내내 모든 요청이 막힙니다. 문제의 뿌리는 작업 시간은 변동하는데 TTL은 획득 시점에 고정해야 한다는 정보 비대칭입니다. watchdog은 이걸 "짧은 TTL + 살아있는 동안 계속 갱신"으로 풉니다 — lease를 지정하지 않고 락을 잡으면 기본 TTL 30초가 걸리고, 락을 보유한 스레드가 살아있는 동안 주기적으로 TTL을 재설정합니다. 그래서 작업이 얼마나 길어지든 도중 만료가 없고, JVM이 죽으면 갱신이 멈춰 최대 30초 안에 자동 해제됩니다 — "정상이면 안 풀리고, 죽으면 풀린다"의 자동화입니다. 부작용은 얻은 것의 정확한 반대편에 있습니다. 첫째, 최악 점유 시간의 상한이 사라집니다 — 스레드가 죽진 않았지만 진전이 없는 hang 상태(내부 교착, 응답 없는 외부 API 대기)면 watchdog은 그걸 살아있음으로 보고 계속 연장해서 다른 요청들이 무기한 대기할 수 있습니다. 둘째, 락이 계속 갱신되니 겉보기엔 정상인데 실제로는 멈춰 있는, 장애처럼 보이지 않는 장애가 가능해집니다. 셋째, 갱신도 네트워크를 타므로 갱신 실패가 겹치면 작업 도중 만료가 여전히 가능합니다 — 줄어들 뿐 사라지지 않습니다. 그래서 watchdog을 쓰더라도 작업 자체의 타임아웃, 락 점유 시간 모니터링과 알람, 뚫렸을 때의 데이터 방어를 함께 설계해야 합니다.

### 원리 이해 (왜 그런가)

**동작 메커니즘:**

```
lease 미지정 lock()/tryLock(wait, unit)
  → TTL = lockWatchdogTimeout (기본 30초) 설정
  → 백그라운드 타이머가 기본 10초 간격(30초의 1/3 — Redisson 문서 기준, 거의 확실)으로 TTL을 30초로 재설정
  → unlock 시 타이머 취소
  → JVM crash 시: 갱신 중단 → 남은 TTL(최대 30초) 후 자동 해제

갱신 주기를 TTL의 1/3로 두는 근거: 갱신 한두 번이 일시 장애로 실패해도
다음 기회가 TTL 안에 남아 있도록 여유를 둔 것 (갱신 주기 < TTL/2 설계)
```

**얻은 것과 잃은 것 (같은 동전의 양면):**

| | 고정 lease | watchdog |
|---|---|---|
| 작업이 길어지면 | 도중 만료 → 동시 진입 위험 | 만료 없음 ✅ |
| JVM crash 시 | lease 만료까지 대기 | 최대 30초 내 해제 ✅ |
| hang(살아있지만 진전 없음) | **lease가 상한이 되어 결국 풀림** ✅ | **무한 연장 — 상한 소멸** ❌ |
| 운영 예측 가능성 | "최악 N초"가 보장됨 | 보장 없음 → 모니터링으로 복원해야 |

watchdog이 구분하는 것은 "JVM의 생사"이지 "작업의 건강함"이 아니다 — 이 한 줄이 부작용 전체의 근거입니다(liveness는 보지만 progress는 못 봄).

### 꼬리질문 Q&A

**Q. Lease time을 직접 지정하면 watchdog은 어떻게 동작하나요?**
→ **동작하지 않는다 — lease 명시는 "상한을 내가 책임진다"는 선언으로 취급되어, 지정 TTL로 고정되고 작업이 안 끝났어도 만료된다.**
`tryLock(wait, lease, unit)`처럼 lease를 준 코드에는 watchdog 보호가 없습니다. "Redisson 쓰니까 자동 연장되겠지"라고 생각하고 lease를 지정하는 것이 흔한 착각이자 사고 지점입니다.

**Q. Watchdog이 TTL을 연장하는 주기는 어떻게 정해지나요?**
→ **lockWatchdogTimeout(기본 30초)의 1/3 간격 — 기본 설정으로 10초마다 TTL을 30초로 재설정한다 (거의 확실 — Redisson 공식 문서 기준).**
1/3로 잡은 근거는 여유입니다 — 갱신이 한두 번 실패해도 TTL 안에 다음 기회가 남아 있어야 일시적 네트워크 순단이 곧바로 락 만료로 이어지지 않습니다.

**Q. Watchdog이 있어도 deadlock이나 장기 대기를 완전히 막을 수 있을까요?**
→ **막지 못한다 — 오히려 hang 상황에서는 watchdog이 락을 계속 살려둬 대기를 연장하는 쪽으로 작용한다.**
watchdog이 보는 것은 JVM 생사뿐이라 "살아있지만 아무것도 못 하는" 상태를 장애로 인식하지 못합니다. 장기 대기 방지는 wait time 상한, 작업 레벨 타임아웃(외부 호출 타임아웃), 점유 시간 알람이 담당해야 할 몫입니다.

**Q. 작업이 오래 걸리는 로직에서 lease time 지정과 watchdog 중 무엇을 선택하겠나요?**
→ **판단 축은 "조기 해제 위험 vs 무기한 점유 위험 중 어느 쪽이 더 치명적인가"다.**
작업 시간이 예측 가능하면(정해진 배치 등) 명시적 lease로 최악 점유 시간의 상한을 보장하는 쪽 — 운영 예측 가능성이 생깁니다. 작업 시간 편차가 크고 도중 만료(동시 진입)가 더 위험하면 watchdog — 단 이 경우 반드시 작업 자체의 타임아웃과 점유 시간 알람을 함께 둬서, watchdog이 없앤 상한을 모니터링으로 복원해야 합니다.

**Q. Lock이 오래 유지되는 상황을 운영에서 어떻게 감지하고 대응하나요?**
→ **감지: 락 획득/해제 시각 로깅 → 점유 시간 메트릭화 + 임계 초과 알람, tryLock 실패율 추이, Redis에서 락 key TTL·존재 직접 확인, APM 트레이스에 락 구간 표시.**
대응은 원인 규명이 먼저입니다 — 해당 인스턴스의 스레드 덤프로 홀더가 어디서 멈췄는지 확보하고, 외부 API hang이면 그 호출의 타임아웃을 고치고, 필요 시 작업 재기동과 함께 이미 부분 수행된 작업을 멱등 재시도나 보상으로 정리합니다. 락 key 수동 삭제는 최후 수단입니다 — 홀더가 실제로는 진행 중이면 동시 실행을 사람 손으로 만드는 셈이기 때문입니다.

### 🌱 심화 키워드
- **lockWatchdogTimeout** — watchdog의 기준 TTL이자 "죽었을 때 풀리기까지의 최대 시간"
- **liveness vs progress** — "살아있음"과 "진전"의 구분: watchdog 부작용을 한 단어로 설명하는 프레임
- **lease** — 시간 제한부 소유권이라는 분산 시스템 일반 개념 (watchdog = lease 자동 갱신)
- **작업 레벨 타임아웃** — 락이 아니라 작업 자체에 상한을 두는 보완 장치
- **split-brain 유사 상황** — 갱신 실패로 락은 만료됐는데 작업은 계속 도는 구간

### ❓ 더 파볼 질문
- **watchdog의 TTL 갱신이 실패하고 있다는 건 어떻게 알 수 있나?**
  ↳ 갱신 실패는 Redisson 내부 로그/예외로 남고, 근본적으로는 Redis 연결 문제의 신호다. 락 key의 TTL을 주기적으로 관측해서 "TTL이 갱신 주기보다 오래 줄어들기만 한다"면 갱신이 멎었다는 뜻 — 이걸 알람으로 걸어두면 "만료 후 늦은 쓰기" 사고를 사전에 잡을 수 있다.
- **liveness와 progress의 구분은 시스템 설계 어디에 또 나타나나?**
  ↳ k8s의 liveness probe가 프로세스 생존만 확인하면 hang을 못 잡는 것과 같은 구조다 — 그래서 readiness probe나 "실제 일이 진행되는지"를 보는 커스텀 헬스체크가 따로 필요하다. "살아있음 확인과 일하고 있음 확인은 다른 검사다"라는 원칙은 락, 헬스체크, 하트비트 설계에 공통으로 적용된다.
- **watchdog 방식과 fencing token은 어떤 관계인가?**
  ↳ 상호 보완이다 — watchdog은 "만료가 일어날 확률"을 줄이는 예방 장치이고, fencing token(또는 조건부 UPDATE)은 "그래도 만료가 일어났을 때" 늦은 쓰기를 데이터 레벨에서 거부하는 최종 방어다. 예방과 최종 방어를 같은 층으로 착각하지 않는 것이 설계의 핵심이다(→ 260706).
