# 260713 : Wait time과 lease time은 각각 어떤 의미인가요?

Wait time과 lease time은 각각 어떤 의미인가요?

이 질문은 Redis 분산락이나 Redisson 기반 락을 실제 운영에서 어떻게 다루는지 확인하기 좋은 질문입니다. 단순히 “락을 건다”가 아니라, 락 대기 시간과 락 점유 시간을 어떻게 제한할지 설명할 수 있어야 합니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- Wait time은 락을 얻기 위해 최대 얼마나 기다릴지를 의미합니다.
- Lease time은 락을 얻은 뒤 최대 얼마 동안 락을 유지할지를 의미합니다.
- Wait time이 너무 길면 요청이 오래 대기하면서 응답 지연이 커질 수 있습니다.
- Lease time이 너무 짧으면 작업이 끝나기 전에 락이 풀려 다른 요청이 동시에 들어올 수 있습니다.
- Lease time이 너무 길면 락을 잡은 애플리케이션에 문제가 생겼을 때 다른 요청이 오래 막힐 수 있습니다.
- 그래서 실제 처리 시간, 장애 복구 방식, 재시도 정책, 사용자 응답 허용 시간을 기준으로 값을 정해야 합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- Wait time을 짧게 잡으면 어떤 장단점이 있나요?
- Lease time이 작업 시간보다 짧으면 어떤 문제가 생기나요?
- Redisson watchdog은 lease time 문제를 어떻게 보완하나요?
- 결제성 로직에서는 lock timeout을 어떻게 설계해야 할까요?
- Lock 획득 실패 시 바로 실패 처리할지 재시도할지는 어떻게 결정하나요?

---

## 답변

> **한 줄 핵심**: wait time = 락을 얻기 위해 **기다리는** 최대 시간(획득 전), lease time = 얻은 뒤 **점유하는** 최대 시간(획득 후, TTL) — 둘 다 "장애가 무한히 번지지 않게 하는 상한선"이며, 값은 감이 아니라 작업 시간 분포·장애 허용 시간·사용자 응답 예산·실패 정책에서 도출한다.

### 1문 1답

**Q. Wait time과 lease time은 각각 무엇을 의미하나요?**

**A.** Redisson의 tryLock(waitTime, leaseTime, unit)를 기준으로 보면, wait time은 락이 풀리기를 기다리는 최대 시간으로 획득 전 구간을 제한하고(기본 비공정 락이라 FIFO 대기열이 아니라 알림 후 재경쟁 방식), lease time은 락을 얻은 뒤 임계 구역 안에서 점유할 수 있는 최대 시간, 즉 TTL로 획득 후 구간을 제한합니다. 하나의 락 생애에서 서로 다른 두 구간을 제한하는 값인 셈입니다. 각각이 왜 필요한지 보면, wait가 없으면 경합 시 요청 스레드가 무한정 묶여 스레드 풀 고갈로 번지고, lease가 없으면 락을 쥔 서버가 죽었을 때 락이 영원히 안 풀리는 데드락이 됩니다. 그래서 둘 다 "장애가 무한히 번지지 않게 하는 상한선"이라는 공통 목적을 가지며, 다만 제한하는 구간이 다를 뿐입니다. 값은 감이 아니라 작업 시간 분포·장애 허용 시간·사용자 응답 예산·실패 정책에서 도출해야 합니다.

**Q. Wait time이 너무 길면 어떤 문제가 생기나요?**

**A.** wait가 과대하면 경합 구간에서 요청이 줄줄이 대기하면서 응답이 밀립니다. 더 심각한 것은 대기하는 요청마다 스레드가 묶여 스레드 풀이 고갈되는 것으로, "락 하나가 서버 전체 장애로 증폭되는 경로"가 된다는 점입니다. 즉 단순한 지연을 넘어 자원 고갈로 확산될 수 있다는 것이 핵심입니다. 사용자 대면 상황에서는 오래 기다리게 하는 것 자체가 이미 나쁜 UX이고, 기다리는 사이 스레드뿐 아니라 커넥션도 함께 잠깁니다. 그래서 wait는 무작정 길게 잡기보다 사용자 응답 예산 안에서 짧게 두고 실패 처리를 설계하는 편이 안전합니다.

**Q. Lease time이 작업 시간보다 짧으면 어떤 문제가 생기나요?**

**A.** lease가 작업보다 짧으면 작업 도중에 락이 만료되어, 락이 풀린 줄 모르는 원래 요청 A와 새로 진입한 B가 동시에 실행됩니다. 이것은 락으로 막으려던 바로 그 상황, 즉 상호배제 붕괴가 재현되는 것입니다. 더 나쁜 것은 A가 뒤늦게 커밋하면서 B의 결과를 덮어쓰는 lost update로, 데이터 정합성까지 깨집니다. 구현 차원의 신호도 있는데, Redisson에서는 A가 작업을 마치고 unlock을 부르는 시점에 이미 자기 락이 아니라서 IllegalMonitorStateException이 발생합니다. 이 예외는 락을 쥐지 않은 스레드가 unlock하는 등 다른 원인으로도 나지만, 이 시나리오에서는 "lease가 작업보다 짧았다"는 신호로 의심해 볼 수 있습니다.

**Q. Lease time이 너무 길면 어떤 문제가 생기나요?**

**A.** lease가 과대하면 락을 쥔 홀더에 장애가 생겼을 때 그 lease가 만료될 때까지 모든 후속 요청이 내내 막힙니다. 홀더가 죽으면 만료까지 전 요청이 블로킹되므로, 복구 시간이 lease 값에 종속됩니다. 즉 실제로는 죽은 홀더 하나 때문에 전체 처리가 lease 시간만큼 멈추는 셈입니다. wait 과대가 "경합 시" 문제를 키운다면, lease 과대는 "장애 시" 복구를 지연시킨다는 점에서 방향이 다릅니다. 그래서 lease는 홀더 장애 시 허용 가능한 잠금 시간을 근거로 상한을 정해야 합니다.

**Q. wait time과 lease time 값은 무엇을 기준으로 정해야 하나요?**

**A.** 두 값 모두 감이 아니라 네 가지 근거에서 도출합니다. 첫째, 실제 작업 시간 분포로, 평균이 아니라 p99에 GC pause와 네트워크 지연 여유를 더해야 합니다. 둘째, 홀더 장애 시 허용 가능한 잠금 시간입니다. 셋째, 사용자 응답 예산으로, 업스트림 타임아웃 안에 wait와 작업이 모두 들어가야 의미가 있습니다(deadline propagation). 넷째, 획득 실패 시 재시도할지 즉시 실패할지의 정책이며, 재시도할 계획이라면 wait를 더 짧게 나눠 쓰는 편이 낫습니다.

**Q. Wait time을 짧게 잡으면 어떤 장단점이 있나요?**

**A.** 장점은 빠른 실패(fail-fast)로 응답성을 유지하고 스레드가 오래 묶이지 않는다는 것입니다. 단점은 경합 시 실패율이 올라가서, 실패했을 때의 UX와 재시도 정책 설계가 필수가 된다는 점입니다. 특히 사용자 대면 API에서는 짧은 wait에 "잠시 후 다시 시도해 주세요" 같은 명확한 실패 처리나 자동 재시도를 붙이는 것이 보통 옳은 방향입니다. 그 이유는 오래 기다리게 하는 것 자체가 이미 나쁜 UX인 데다, 기다리는 사이 스레드와 커넥션도 함께 잠기기 때문입니다. 결국 짧은 wait는 "빨리 실패해 자원과 UX를 지킨다"는 fail-fast 원칙을 적용한 것입니다.

**Q. Redisson watchdog은 lease time 문제를 어떻게 보완하나요?**

**A.** watchdog은 "lease를 얼마로 잡지"라는 추정 문제 자체를 제거하는 장치입니다. lease를 지정하지 않고 락을 잡으면 기본 30초 TTL을 걸고, 락을 보유한 스레드가 살아있는 동안 주기적으로(기본 30초의 1/3인 10초 간격) TTL을 재설정합니다. 그래서 작업이 살아있는 한 도중에 만료되지 않고, JVM이 죽으면 갱신이 멈춰 최대 30초 안에 자동으로 풀립니다. 결과적으로 "정상이면 안 풀리고 죽으면 풀린다"가 자동화되는 셈입니다. 다만 스레드가 살아있지만 진전이 없는 hang 상태에서도 계속 연장된다는 부작용이 있어, 상세는 260716에서 다룹니다.

**Q. 결제성 로직에서는 lock timeout을 어떻게 설계해야 할까요?**

**A.** 원칙은 두 요청의 동시 실행이 절대 없어야 하고, 락 타임아웃 하나에 정합성을 걸지 않는 것입니다. 구체적으로 lease는 결제 작업의 전체 타임아웃 체인(PG 호출 타임아웃 + 재시도 + DB 커밋)보다 길게 잡아, 작업이 락보다 오래 살아남는 순간이 없게 해야 합니다. wait는 짧게 두고 실패 시 "이미 처리 중입니다"로 안내하는데, 결제에서 대기 후 재진입은 중복 시도를 유도할 뿐이기 때문입니다. 그리고 락과 별도로 상태머신(PENDING→APPROVED), 멱등성 키, unique 제약을 함께 둡니다. 락이 경계 상황에서 뚫려도 돈이 두 번 나가지 않는 구조가 진짜 방어선이기 때문입니다.

**Q. Lock 획득 실패 시 바로 실패 처리할지 재시도할지는 어떻게 결정하나요?**

**A.** 기준은 락을 건 "목적"입니다. 목적이 같은 작업의 중복 방지(쿠폰 발급, 캐시 재생성)라면 누군가 이미 하고 있다는 뜻이므로 내가 또 할 이유가 없어, 즉시 실패하고 앞선 요청의 결과를 쓰면 됩니다. 반대로 목적이 순차화(재고 차감처럼 모든 요청이 결국 각자 처리돼야 함)라면 실패시키는 것은 정당한 요청을 버리는 것이므로, 짧은 백오프 재시도나 큐잉이 맞습니다. 여기에 사용자 대기 예산(동기 API인가, 백그라운드 작업인가)을 겹쳐 최종 결정합니다. 요약하면 "중복 억제면 즉시 실패, 순차화면 재시도"가 출발점입니다.

### 면접 답변 (구술용)

Redisson의 tryLock(waitTime, leaseTime, unit) 기준으로, wait time은 락을 얻기 위해 기다리는 최대 시간이고 lease time은 락을 얻은 뒤 점유할 수 있는 최대 시간, 즉 TTL입니다. 락의 생애에서 서로 다른 구간을 제한하는 값입니다 — wait는 획득 전 대기열에서의 시간, lease는 획득 후 임계 구역 안에서의 시간. 각각이 왜 필요한지부터 보면, wait가 없으면 경합 시 요청 스레드가 무한정 묶여 스레드 풀 고갈로 번지고, lease가 없으면 락을 쥔 서버가 죽었을 때 락이 영원히 안 풀리는 데드락이 됩니다. 둘 다 장애 확산을 끊는 상한선입니다. 트레이드오프는 방향이 서로 다릅니다. wait가 너무 길면 경합 구간에서 요청이 줄줄이 대기해 응답이 밀리고 대기 스레드가 쌓입니다. lease가 너무 짧으면 작업 도중 락이 풀려 동시 진입이 생기고 — 락을 쓴 이유가 무효가 되는 데다 늦게 끝난 원래 요청의 쓰기가 새 요청 결과를 덮을 수 있습니다. lease가 너무 길면 홀더 장애 시 그 시간 내내 전부 막힙니다. 그래서 값은 실제 작업 시간 분포(p99에 GC·네트워크 여유), 장애 시 허용 가능한 잠금 시간, 사용자 응답 예산, 그리고 획득 실패 시 어떻게 할지(재시도냐 즉시 실패냐)를 근거로 정해야 합니다.

### 원리 이해 (왜 그런가)

**락의 생애와 두 타임아웃:**

```
tryLock 호출 ──[wait time: 대기 상한]──▶ 획득 ──[lease time: 점유 상한(TTL)]──▶ 해제/만료
              못 얻으면 false 반환                작업 안 끝났어도 만료되면 풀림
```

**세 방향의 트레이드오프:**

| 설정 | 잘못되면 생기는 일 | 메커니즘 |
|------|--------------------|----------|
| wait 과대 | 응답 지연 + 스레드 풀 고갈 | 경합 시 대기 요청마다 스레드가 묶임 — 락 하나가 서버 전체 장애로 증폭되는 경로 |
| lease 과소 | 동시 진입 + lost update | 락이 풀린 줄 모르는 A와 새로 진입한 B가 동시 실행, A의 늦은 커밋이 B를 덮음 |
| lease 과대 | 장애 복구 지연 | 홀더가 죽으면 만료까지 전 요청 블로킹 — 복구 시간이 lease에 종속 |

**값 도출의 근거 4가지**: ① 작업 시간 분포(p99 + GC pause·네트워크 여유 — 평균 기준은 위험), ② 홀더 장애 시 허용 가능한 잠금 시간, ③ 사용자 응답 예산(업스트림 타임아웃 안에 wait + 작업이 들어가야 함), ④ 실패 시 정책(재시도할 거면 wait를 더 짧게 나눠 쓰는 게 낫다).

### 꼬리질문 Q&A

**Q. Wait time을 짧게 잡으면 어떤 장단점이 있나요?**

**A.** **장점: 빠른 실패(fail-fast)로 응답성 유지 + 스레드가 오래 안 묶임. 단점: 경합 시 실패율 상승 → 실패 시 UX·재시도 정책 설계가 필수가 된다.**
사용자 대면 API에서는 짧은 wait + 명확한 실패 처리("잠시 후 다시 시도해 주세요" 또는 자동 재시도)가 보통 옳은 방향입니다 — 오래 기다리게 하는 것 자체가 이미 나쁜 UX이고, 그 사이 스레드·커넥션도 함께 잠기기 때문입니다.

**Q. Lease time이 작업 시간보다 짧으면 어떤 문제가 생기나요?**

**A.** **락이 풀린 줄 모르는 원래 요청 A와 새로 진입한 B가 동시 실행된다 — 락으로 막으려던 바로 그 상황이 재현된다.**
더 나쁜 것은 A가 뒤늦게 커밋하면서 B의 결과를 덮는 lost update입니다. 구현 차원의 신호도 있습니다 — Redisson에서는 A가 작업을 마치고 unlock을 부르는 시점에 이미 자기 락이 아니라서 IllegalMonitorStateException이 발생합니다. 이 예외가 로그에 보이면 "lease가 작업보다 짧았다"는 경고로 읽어야 합니다.

**Q. Redisson watchdog은 lease time 문제를 어떻게 보완하나요?**

**A.** **"lease를 얼마로 잡지"라는 추정 문제 자체를 제거한다 — lease 미지정 시 기본 30초 TTL을 걸고, 스레드가 살아있는 동안 주기적으로(기본 30초의 1/3인 10초 간격 — Redisson 문서 기준, 거의 확실) TTL을 재설정한다.**
작업이 살아있으면 안 풀리고, JVM이 죽으면 갱신이 멈춰 최대 30초 안에 풀립니다. 대신 hang 상태(살아있지만 진전 없음)에서도 계속 연장된다는 부작용이 생깁니다 — 상세는 260716.

**Q. 결제성 로직에서는 lock timeout을 어떻게 설계해야 할까요?**

**A.** **원칙: 두 요청의 동시 실행이 절대 없어야 하고, 락 타임아웃 하나에 정합성을 걸지 않는다.**
구체적으로 — ① lease는 결제 작업의 전체 타임아웃 체인(PG 호출 타임아웃 + 재시도 + DB 커밋)보다 길게: 작업이 락보다 오래 살아남는 순간이 없어야 하기 때문입니다. ② wait는 짧게 + 실패 시 "이미 처리 중입니다" 안내: 결제에서 대기 후 재진입은 중복 시도를 유도할 뿐입니다. ③ 락과 별도로 상태머신(PENDING→APPROVED), 멱등성 키, unique 제약을 함께 둡니다 — 락이 경계 상황에서 뚫려도 돈이 두 번 나가지 않는 구조가 진짜 방어선입니다.

**Q. Lock 획득 실패 시 바로 실패 처리할지 재시도할지는 어떻게 결정하나요?**

**A.** **락을 건 "목적"으로 결정한다 — 중복 억제 목적이면 즉시 실패, 순차화 목적이면 재시도/큐잉.**
목적이 "같은 작업의 중복 방지"(쿠폰 발급, 캐시 재생성)라면 누군가 이미 하고 있다는 뜻이므로 내가 또 할 이유가 없습니다 — 즉시 실패하고 앞선 요청의 결과를 쓰면 됩니다. 목적이 "순차화"(재고 차감처럼 모든 요청이 결국 각자 처리돼야 함)라면 실패시키면 정당한 요청을 버리는 것이므로 짧은 백오프 재시도나 큐잉이 맞습니다. 여기에 사용자 대기 예산(동기 API인가, 백그라운드 작업인가)을 겹쳐 최종 결정합니다.

### 🌱 심화 키워드
- **fail-fast** — 빨리 실패해 자원과 UX를 지키는 원칙 (wait 짧게의 이론적 근거)
- **백오프 + 지터(jitter)** — 재시도 뭉침(thundering herd)을 분산시키는 표준 기법
- **deadline propagation** — 상류 타임아웃 예산을 하류로 배분하는 설계 (wait+작업 ≤ 업스트림 타임아웃)
- **lost update** — lease 과소가 만드는 대표 이상 현상
- **IllegalMonitorStateException** — "lease < 작업 시간"의 런타임 신호

### ❓ 더 파볼 질문
- **공정 락(fair lock)으로 대기 순서를 보장하면 어떤 비용이 생기나?**
  ↳ 대기열 관리(누가 먼저 왔는지 기록·유지) 오버헤드가 추가되고, 항상 "가장 오래 기다린 쪽"에게 넘겨야 하므로 처리량이 떨어진다. 순서가 비즈니스 요구일 때만 쓰고, 아니면 비공정 + 재시도가 일반적으로 낫다.
- **재시도에 지터(jitter)를 섞는 이유는?**
  ↳ 같은 시각에 실패한 대기자들이 같은 백오프로 재시도하면 또 같은 시각에 몰린다(thundering herd). 대기 시간에 무작위성을 섞으면 재시도가 시간축에 퍼져 경합과 스파이크가 줄어든다 — AWS의 "Exponential Backoff and Jitter" 글이 이 효과를 실험으로 보여준 표준 레퍼런스다.
- **업스트림 타임아웃이 3초라면 wait time은 어떻게 정해야 하나?**
  ↳ wait + 실제 작업 시간 + 응답 직렬화까지 3초 안에 들어와야 의미가 있다. 예를 들어 작업 p99가 1초면 wait는 1초 남짓이 상한이고, 그보다 길게 기다리게 하는 것은 "성공해도 클라이언트는 이미 타임아웃"인 낭비를 만든다 — 타임아웃 예산은 항상 위에서 아래로 배분한다(deadline propagation).
