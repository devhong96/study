# 260615 : Redis가 빠른 이유를 설명해 주세요.

Redis가 빠른 이유를 설명해 주세요.
단, “인메모리라서 빠릅니다”, “싱글스레드라서 빠릅니다” 말고 다른 관점에서도 설명할 수 있나요?

Redis 관련 질문은 정말 자주 나오는데요.
많은 분들이 “메모리에 저장해서 빠릅니다” 정도로 답변하고 끝내는 경우가 많습니다.

그런데 면접관이 한 단계 더 들어가면 이런 식으로 물어볼 수 있습니다.

“메모리 기반이면 다 Redis처럼 빠른가요?”
“싱글스레드인데 어떻게 많은 요청을 처리하나요?”
“네트워크 I/O는 어떻게 처리하나요?”

이 질문은 Redis를 단순 캐시 도구로 써봤는지가 아니라,
Redis의 구조와 동작 원리를 어느 정도 이해하고 있는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- Redis는 왜 디스크 기반 DB보다 빠른가?
- 싱글스레드 구조가 오히려 유리한 지점은 무엇인가?
- I/O Multiplexing은 Redis에서 어떤 역할을 하는가?
- Context Switching 비용이 줄어드는 이유는 무엇인가?
- Redis의 자료구조가 성능에 어떤 영향을 주는가?

---

## 답변

> **한 줄 핵심**: Redis의 속도 = 디스크 제거(메모리) + 스레드 비용 제거(이벤트 루프) + 락 제거(단일 스레드 실행) + 연산 비용 최소화(전용 자료구조) — 병목을 층층이 제거한 **조합 설계**다.

### 1문 1답

**Q. Redis는 왜 디스크 기반 DB보다 빠른가?**
→ 데이터가 메모리(RAM)에 상주하기 때문에 디스크 탐색·회전 지연이나 페이지 I/O 같은 저장 계층의 병목이 아예 없다. 메모리 접근은 나노초 단위라 디스크와는 자릿수가 다르게 빠르다. 다만 이건 Redis 속도를 만드는 병목 제거 4층(저장·연결·실행·연산) 중 저장 계층 하나만 해결한 것이다. 그래서 "메모리 기반이면 다 Redis처럼 빠른가?"에 대한 답은 '아니오'인데, 커넥션 모델·락·자료구조의 병목이 그대로 남아 있으면 메모리만으로는 이 속도가 안 나온다. 즉 인메모리는 필요조건이지 충분조건이 아니다.

**Q. 싱글스레드 구조가 오히려 유리한 지점은 무엇인가?**
→ 명령이 하나의 스레드에서 순차적으로 실행되므로 락 획득·경합이나 CPU 캐시 동기화 비용이 0이고, 모든 명령이 자연스럽게 원자적으로 처리된다. 멀티스레드였다면 공유 자료구조를 보호하느라 락이 필요하고 그 경합이 병목이 됐을 것이다. 이것이 "싱글스레드라서 빠르다"의 진짜 의미다. 이 설계는 "명령 하나하나가 마이크로초 단위로 짧다"는 전제 위에 서 있어서, 줄을 세워 처리해도 뒤 명령이 밀리지 않는다. 반대로 `KEYS *`나 긴 Lua 같은 긴 명령이 오면 전제가 깨져 전체가 밀리므로, 강점과 약점이 같은 설계에서 나온다.

**Q. I/O Multiplexing은 Redis에서 어떤 역할을 하는가?**
→ epoll/kqueue 같은 커널 기능을 이용해 스레드 하나가 수만 개의 소켓을 동시에 감시하다가, 데이터가 준비된 소켓만 골라 처리한다. 덕분에 커넥션마다 스레드를 만들 필요가 없어져 스레드 생성 비용과 컨텍스트 스위칭이 아예 발생하지 않는다. 즉 "대기를 감시하는" 동시성 부담을 커널에 위임하는 역할이다. "동시에 많은 요청"에서 실제로 시간을 잡아먹는 건 대부분 네트워크 대기인데, 그 대기를 커널이 대신 기다려 준다. 그래서 싱글스레드임에도 많은 커넥션을 감당할 수 있고, Redis 6+는 이 네트워크 read/write 단계만 io-threads로 병렬화하기도 한다.

**Q. Context Switching 비용이 줄어드는 이유는 무엇인가?**
→ 명령을 실행하는 스레드가 1개뿐이라 스레드 간 전환 자체가 일어나지 않는다. 커넥션 수가 수만 개로 늘어도 스레드 수는 그대로이므로 스케줄링·전환 비용이 커지지 않는다. 커넥션당 스레드를 두는 전통적 모델은 커넥션이 늘수록 스레드가 늘고 그만큼 컨텍스트 스위칭이 폭증하는데, Redis는 이 구조를 아예 피한다. 대기 감시는 epoll가, 처리는 짧은 순차 실행이 담당하는 분업이라 애초에 전환이 필요 없는 것이다.

**Q. Redis의 자료구조가 성능에 어떤 영향을 주는가?**
→ Redis는 목적별로 최적화된 전용 자료구조를 써서 연산 자체의 비용을 낮춘다. 예를 들어 길이를 헤더에 저장하는 SDS는 strlen을 O(1)로 만들고, Sorted Set의 skiplist는 범위 검색을 O(logN)에, 소규모 컬렉션의 listpack은 연속 메모리 압축으로 캐시 지역성을 확보한다. 덕분에 대부분의 명령이 O(1)~O(logN)에 동작한다. 연산 비용이 낮으니 단일 스레드에서도 각 명령이 짧게 끝나 전체 처리량이 유지된다. 반대로 대형 컬렉션에 O(N) 명령을 던지면 이 짧은 명령 전제가 깨져 이벤트 루프 전체가 밀린다.

### 면접 답변 (구술용)

인메모리·싱글스레드 외에 세 가지 관점을 더 들 수 있습니다. 첫째, I/O Multiplexing 기반 이벤트 루프입니다 — epoll 같은 커널 기능으로 스레드 하나가 수만 개의 소켓을 감시하므로, 커넥션마다 스레드를 만들 때 발생하는 생성 비용과 컨텍스트 스위칭이 아예 없습니다. 둘째, 락이 필요 없는 실행 모델입니다 — 명령이 단일 스레드에서 순차 실행되므로 락 획득·경합·캐시 동기화 비용이 0이고, 이것이 "싱글스레드라서 빠르다"의 실제 의미입니다. 셋째, 목적에 최적화된 자료구조입니다 — SDS, skiplist, listpack 덕분에 대부분의 명령이 O(1)~O(logN)에 동작합니다. 그래서 "메모리 기반이면 다 빠른가?"에 대한 답은 '아니오'입니다 — 메모리는 디스크 병목만 없앨 뿐, 나머지 병목(스레드·락·연산)을 없앤 설계가 함께 있어야 Redis급 속도가 나옵니다.

### 원리 이해 (왜 그런가)

**병목 제거 4층 구조:**

| 층 | 병목 | Redis의 제거 방식 | 근거 |
|----|------|------------------|------|
| 저장 | 디스크 I/O | 인메모리 | 메모리 접근은 디스크보다 자릿수가 다르게 빠름 |
| 연결 | 커넥션당 스레드 비용 | 이벤트 루프 + epoll/kqueue | 대기는 커널이 감시, 준비된 소켓만 처리 → 스레드 1개로 수만 커넥션 |
| 실행 | 락 경합·컨텍스트 스위칭 | 단일 스레드 순차 실행 | 명령이 마이크로초 단위로 짧아 줄 세워도 안 밀림 + 모든 명령이 원자적 |
| 연산 | 자료구조 비용 | 전용 자료구조 | SDS: 길이 저장으로 strlen O(1) / Sorted Set: skiplist O(logN) / 소규모 컬렉션: listpack 압축 인코딩 |

**전제와 이면**: 이 설계는 "명령 하나하나가 짧다"는 전제 위에 서 있다. 전제가 깨지는 긴 명령(`KEYS *`, 긴 Lua, 대형 컬렉션 O(N) 명령)이 오면 전체가 밀린다 — 같은 설계에서 강점과 약점이 함께 나온다. (→ 260701)

**Redis 6+ 참고**: 병목이 명령 실행이 아니라 네트워크 read/write·파싱이라는 점에 착안해, 그 부분만 io-threads로 병렬화 가능. 명령 실행은 여전히 단일 스레드라 원자성 유지.

### 꼬리질문 Q&A

**Q. 메모리 기반이면 다 Redis처럼 빠른가요?**
→ **아니오. 메모리는 4층 중 1층만 해결한다.**
- 커넥션 모델이 스레드 기반이면 스위칭 비용, 멀티스레드 접근이면 락 경합, 범용 자료구조면 O(N) 연산이 각각 남는다.

**Q. 싱글스레드인데 어떻게 많은 요청을 처리하나요?**
→ **동시성은 epoll(대기 감시)이, 처리 속도는 짧은 순차 실행이 담당하는 분업 구조.**
- "동시에 많은 요청"에서 시간이 걸리는 건 대부분 네트워크 대기 — 그건 커널이 대신 기다려준다.

**Q. 네트워크 I/O는 어떻게 처리하나요?**
→ **논블로킹 소켓을 이벤트 루프에 등록 → 읽기 이벤트에 요청 파싱·실행 → 응답은 버퍼에 쌓아 쓰기 가능 이벤트에 전송.**
- Redis 6+는 이 read/write 단계만 io-threads로 병렬화할 수 있다.

**Q. Context Switching 비용이 줄어드는 이유는?**
→ **실행 스레드가 1개라 스레드 간 전환 자체가 없고, 커넥션 수가 늘어도 스레드 수가 늘지 않기 때문.**

### 🌱 심화 키워드
- **SDS(Simple Dynamic String)** — 길이·여유 공간을 헤더에 저장하는 문자열 구조, O(1) strlen + 바이너리 안전
- **skiplist** — Sorted Set의 범위 검색을 O(logN)으로 만드는 확률적 계층 리스트 (B-tree 대신 채택된 이유 찾아보기)
- **listpack** — 소규모 컬렉션을 연속 메모리에 압축 저장하는 인코딩(구 ziplist 대체), 캐시 지역성 확보
- **RESP(REdis Serialization Protocol)** — 파싱 비용이 낮은 단순 텍스트 프로토콜
- **io-threads** — Redis 6+의 네트워크 단계 한정 멀티스레딩

### ❓ 더 파볼 질문
- **memcached는 멀티스레드인데 Redis와 처리량 특성이 어떻게 다른가?**
  ↳ memcached는 단순 KV + 멀티스레드라 순수 GET/SET 처리량과 멀티코어 확장에서 유리할 수 있다. Redis는 풍부한 자료구조·영속화·복제·스크립팅이 강점이고 코어 하나 기준 효율이 매우 높다. 즉 "단순 캐시 대량 처리 vs 기능이 필요한 데이터 저장소"라는 용도 차이로 갈리는 것이지 절대 우열이 아니다.
- **RDB fork 시 copy-on-write가 latency 스파이크를 만드는 메커니즘은?**
  ↳ RDB 스냅샷은 fork한 자식 프로세스가 뜨는데, fork 순간 페이지 테이블 복사 비용이 크고(힙이 클수록), 이후 부모가 쓰기를 할 때마다 해당 메모리 페이지가 실제로 복제된다(copy-on-write). 쓰기가 많은 워크로드에서 스냅샷 중 메모리 사용량이 치솟고 fork 자체가 수십~수백 ms의 멈춤을 만들 수 있다 — "메모리 2배 여유"를 권고하는 이유다.
- **대량 키 만료(expire cycle)가 이벤트 루프에 주는 부담은 어떻게 관리되나?**
  ↳ Redis는 접근 시 만료를 확인하는 lazy 방식과, 주기적으로 만료 키를 샘플링해 지우는 active 방식을 혼용하고, active 사이클에 시간 상한을 둬 루프를 오래 막지 않게 한다. 그래도 같은 시각에 대량 키가 몰려 만료되면 부담이 되므로, TTL에 지터(무작위 편차)를 섞어 만료 시각을 분산시키는 것이 실무 요령이다.
