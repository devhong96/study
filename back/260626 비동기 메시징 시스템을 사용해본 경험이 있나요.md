## 답변

> **한 줄 핵심**: 도입 목적(응답 시간·장애 격리·버퍼링)이 요구 보장 수준을 결정하고, Redis로 만든다면 자료구조 선택(List < Pub/Sub < Stream)이 곧 보장 수준 선택이다 — 유실은 구조와 영속화로 막고, 중복은 멱등성으로 막는다.

### 면접 답변 (구술용)

먼저 왜 도입하는지부터 명확히 해야 합니다 — 응답 시간 개선(무거운 후속 작업을 요청 경로에서 분리), 장애 격리(소비자가 죽어도 생산자는 정상), 트래픽 버퍼링(폭증을 큐가 흡수) 중 무엇이냐에 따라 유실 허용 여부, 순서, 재처리 요구가 완전히 달라지기 때문입니다. Redis로 구현한다면 첫 결정은 자료구조입니다. List의 BRPOP은 단순하지만 ack 개념이 없어서 꺼낸 직후 소비자가 죽으면 메시지가 유실됩니다. Pub/Sub은 저장하지 않는 방송이라 구독자가 없으면 그대로 소멸합니다. 그래서 메시지 큐 용도라면 사실상 Stream이 답입니다 — append-only 로그로 저장되고, Consumer Group과 ACK를 지원하며, 처리 중 죽은 메시지는 PEL에 남아 다른 소비자가 회수해 재처리할 수 있습니다. 그다음 고려 순서는 유실 방어(Redis 영속화 설정 — AOF everysec도 최대 1초 유실 가능), 중복 방어(재처리가 있는 한 at-least-once이므로 소비자 멱등성 필수), 재처리와 DLQ, 처리량 순입니다. Kafka와 비교하면 Redis 큐는 인프라 추가 없이 가볍게 시작할 수 있고 지연이 낮지만, 내구성이 약하고 메모리가 곧 용량 한계라서, 유실이 돈으로 직결되는 도메인은 Kafka, 가벼운 작업 큐는 Redis Stream이 맞다고 정리할 수 있습니다.

### 원리 이해 (왜 그런가)

**자료구조 선택 = 보장 수준 선택:**

| | List (LPUSH+BRPOP) | Pub/Sub | **Stream** |
|---|---|---|---|
| 저장 | 큐에 저장, 꺼내면 삭제 | **저장 안 함** (방송) | append-only 로그로 보존 |
| 소비 확인(ACK) | 없음 — **꺼낸 순간 큐에서 사라짐** | 없음 | 있음 — ACK 전까지 PEL에 추적 |
| 소비자 죽으면 | 꺼낸 메시지 유실 | 못 받은 메시지 소멸 | PEL에 남음 → 회수·재처리 가능 |
| 적합 용도 | 유실 허용 단순 작업 큐 | 실시간 알림(유실 허용) | **신뢰성 있는 메시지 큐** |

List의 구조적 한계는 "처리 완료"와 "큐에서 제거"가 분리되지 않는다는 것이고, Stream은 그 둘을 분리(XREADGROUP으로 수령 → XACK로 완료)해서 문제를 풀었습니다.

**설계 체크리스트 (순서대로):**

1. **유실** — Redis는 메모리 기반이라 영속화 설정이 내구성을 결정. 기본(RDB 스냅샷)은 마지막 스냅샷 이후 통째 유실 가능, AOF everysec도 최대 1초분 유실 가능. "브로커가 죽어도 절대 유실 불가"면 Redis 단독으로는 한계를 인정해야 함.
2. **중복** — 재처리가 존재하는 한 at-least-once → 소비자 멱등성(메시지 ID + unique 제약) 필수.
3. **재처리** — XPENDING으로 오래 방치된 pending 확인, XAUTOCLAIM으로 소유권 이전 재처리, 재시도 초과분은 DLQ 스트림으로 격리.
4. **처리량** — Consumer Group 내 소비자 수 확장. 단 Kafka의 파티션 같은 내장 샤딩이 없어 스트림 key 분할은 직접 설계.

### 꼬리질문 Q&A

**Q. Redis Pub/Sub과 Stream의 차이는 무엇인가요?**
→ **저장 여부가 본질이다.**
Pub/Sub은 발행 순간 구독 중인 클라이언트에게만 전달하고 저장하지 않으므로, 그 순간 안 듣고 있었으면 영영 못 받습니다(재소비 불가). Stream은 로그로 저장되므로 나중에 합류한 소비자도 과거부터 읽을 수 있고, Consumer Group·ACK·재처리까지 지원합니다. "방송 vs 기록"으로 기억하면 됩니다.

**Q. Stream에서 Consumer가 처리 중 죽으면 메시지는 어떻게 되나요?**
→ **ACK되지 않았으므로 그 소비자 몫의 PEL(Pending Entries List)에 남는다.**
다른 소비자가 XPENDING으로 "일정 시간 넘게 방치된" 메시지를 찾고 XAUTOCLAIM으로 소유권을 넘겨받아 재처리합니다. 즉 유실되지 않고 중복 처리 가능성으로 전환됩니다 — 이것이 소비자 멱등성이 필수인 이유입니다.

**Q. 메시지 중복 처리는 어떤 key로 방어하나요?**
→ **생산 시점에 부여한 고유 메시지 ID(또는 비즈니스 키 조합)로 방어한다.**
소비자가 처리 이력 테이블(unique 제약)에 ID를 기록하면서 처리하고, 이력 기록과 비즈니스 반영을 같은 DB 트랜잭션으로 묶으면 "기록됐는데 반영 안 됨" 같은 틈까지 제거됩니다. 재전달된 메시지는 unique 위반으로 잡혀 안전하게 스킵됩니다.

**Q. Redis를 큐로 쓸 때 Kafka보다 불리한 점은?**
→ **내구성, 용량, 재소비, 확장 모델, 생태계.**
Kafka는 디스크 로그 + 복제가 기본이고 보존 기간 내 offset을 되감아 리플레이할 수 있으며 파티션·리밸런싱이 내장돼 있습니다. Redis는 메모리가 용량 한계라 백로그가 쌓이면 위험하고, 영속화 설정에 따라 유실 가능하며, 샤딩을 직접 설계해야 합니다. 반대로 운영 단순성과 낮은 지연은 Redis가 유리한 경우가 많아서, "유실 비용"과 "운영 비용" 중 무엇이 큰지가 선택 기준이 됩니다.

**Q. DLQ나 재처리 정책은 어떻게 설계하나요?**
→ **한계 재시도까지는 백오프 재시도, 초과하면 DLQ로 격리하고 원본은 ACK한다.**
근거는 poison message 문제입니다 — 실패하는 메시지를 본 큐에서 무한 재시도하면 뒤 메시지 처리를 막고 장애를 증폭시키므로, "격리 후 별도 처리"가 원칙입니다. Stream이면 XPENDING의 delivery count로 재시도 횟수를 추적할 수 있습니다. DLQ에는 알람을 걸고, 원인 수정 후 수동/배치로 재투입하는 운영 절차까지 정해둬야 설계가 완성됩니다.

### 🌱 심화 키워드
- **PEL(Pending Entries List)** — Stream 신뢰성의 핵심 자료구조, "누가 뭘 처리 중인가"의 장부
- **XAUTOCLAIM / XPENDING** — 죽은 소비자의 메시지를 회수하는 명령 셋
- **at-least-once / at-most-once / exactly-once** — 전달 보장 3단계와 각각의 비용 (→ 260709, 260710)
- **poison message / DLQ** — 실패 메시지 격리 패턴
- **backpressure** — 소비 속도 대비 유입 초과 시의 흐름 제어

### ❓ 더 파볼 질문
- **Kafka의 컨슈머 그룹 리밸런싱과 Stream의 Consumer Group은 뭐가 다른가?**
  ↳ Kafka는 파티션 "소유권"을 그룹 프로토콜이 자동 재분배한다(리밸런스). Stream은 파티션 개념이 없고 소유권이 메시지 단위(PEL)라 자동 리밸런싱이 없다 — 죽은 소비자의 몫은 XAUTOCLAIM으로 다른 소비자가 능동적으로 가져가야 한다. "자동 재분배 vs 수동 회수"의 차이다.
- **Stream이 무한히 커지지 않게 하려면?**
  ↳ XADD의 MAXLEN(~ 근사치 옵션) 또는 XTRIM으로 오래된 엔트리를 절단한다. 절단된 메시지는 재소비가 불가능해지므로, 보존 길이는 "가장 느린 소비자가 따라올 시간"을 근거로 정해야 한다.
- **exactly-once 전달은 왜 원리적으로 불가능하다고 하나?**
  ↳ 네트워크 단절 시 "상대가 받고 ack만 유실됐는지, 아예 못 받았는지"를 송신자가 구분할 수 없기 때문이다(두 장군 문제 계열). 그래서 실무의 답은 전달을 at-least-once로 두고 처리의 효과를 멱등성으로 1회화하는 것 — "exactly-once delivery는 불가, exactly-once processing effect는 설계 가능"으로 정리된다.
