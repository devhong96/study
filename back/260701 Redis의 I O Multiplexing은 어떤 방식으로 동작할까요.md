# 260701 Redis의 I/O Multiplexing은 어떤 방식으로 동작할까요?

Redis의 I/O Multiplexing은 어떤 방식으로 동작할까요?

Redis가 빠른 이유를 물어보면 보통
“메모리 기반이라서 빠릅니다”, “싱글 스레드라서 context switching 비용이 적습니다” 정도로 답하는 경우가 많습니다.

그런데 면접에서 한 단계 더 들어가면 이런 질문이 나올 수 있습니다.

“Redis는 싱글 스레드인데, 여러 클라이언트 요청을 어떻게 동시에 처리하나요?”

이 질문은 Redis를 단순 캐시로 써봤는지보다,
Redis의 이벤트 루프와 네트워크 I/O 처리 방식을 이해하고 있는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- Redis는 여러 클라이언트 소켓을 이벤트 기반으로 감시함
- select, poll, epoll, kqueue 같은 I/O multiplexing 기술을 활용함
- 요청이 들어온 소켓만 이벤트 루프에서 처리해 불필요한 blocking을 줄임
- 명령 실행 자체는 주로 단일 스레드에서 순차적으로 처리됨
- 긴 Lua Script나 무거운 명령은 이벤트 루프를 막아 전체 요청 지연을 만들 수 있음

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- Redis가 싱글 스레드인데도 빠른 이유는 무엇인가요?
- I/O multiplexing과 multi-threading은 어떤 차이가 있나요?
- Redis에서 하나의 명령이 오래 걸리면 다른 요청에는 어떤 영향이 있나요?
- Redis Lua Script가 오래 실행되면 왜 위험한가요?

## 답변

> **한 줄 핵심**: "기다림은 커널에 맡기고(epoll이 준비된 소켓만 알려줌), 실행만 직접 한다(단일 스레드 순차)" — 동시성은 I/O 다중화가, 처리 속도는 짧은 명령 실행이 담당하는 분업 구조다.

### 1문 1답

**Q. Redis는 여러 클라이언트 소켓을 어떻게 감시하나요?**
→ Redis는 접속한 모든 클라이언트 소켓을 논블로킹으로 설정한 뒤 커널의 다중화 기능에 등록해 둡니다. 그러면 이벤트 루프가 매 바퀴마다 커널에게 "지금 읽거나 쓸 준비가 된 소켓이 무엇이냐"를 물어 그 목록을 한 번에 받아옵니다. 준비된 소켓만 골라 순회하며 요청을 읽고 처리하므로, 준비 안 된 소켓을 붙잡고 하나씩 기다리는 낭비가 없습니다. 기다림이라는 가장 시간이 오래 걸리는 부분을 커널에 위임한 셈입니다. 그 덕분에 실행 스레드가 하나뿐인데도 수만 개의 커넥션을 동시에 감시할 수 있습니다.

**Q. select, poll, epoll, kqueue 같은 I/O multiplexing 기술을 어떻게 활용하나요?**
→ Redis는 플랫폼에 따라 리눅스에서는 epoll, macOS/BSD에서는 kqueue 같은 커널 다중화 기술을 사용합니다. 이 기술들은 관심 있는 소켓 목록을 커널이 계속 유지하도록 등록해 두고, 그중 준비된 것만 통지받는 방식입니다. Redis는 이런 OS별 구현 차이를 자체 이벤트 라이브러리인 ae로 추상화해 하나의 코드로 여러 플랫폼을 지원합니다. 참고로 select는 호출할 때마다 전체 fd 집합을 커널에 넘기고 커널이 전부를 선형 스캔하기 때문에 fd가 늘수록 O(n)으로 비싸집니다. 반면 epoll은 대기 비용이 "전체 수"가 아니라 "준비된 수"에 비례하므로, 수만 커넥션 시대에 사실상 표준이 되었습니다.

**Q. 요청이 들어온 소켓만 처리해 blocking을 줄인다는 게 무슨 의미인가요?**
→ 이벤트 루프는 epoll_wait로 준비된 fd 목록만 받아오기 때문에, 실제로 데이터가 도착한 소켓만 처리하고 나머지는 아예 건드리지 않습니다. 이렇게 기다림 자체를 커널에 맡겨 두니 루프 어느 지점에서도 소켓 하나를 붙잡고 블로킹으로 멈추는 일이 없습니다. 예를 들어 동시 접속이 5만이어도 어느 순간 실제로 데이터를 보내온 소켓은 그중 일부일 뿐입니다. 그 일부만 짧게 처리하고 바로 다음 바퀴로 넘어가므로 전체 처리량이 유지됩니다. 요청이 겹쳐 보이는 이유는 병렬 실행이 아니라 각 처리가 마이크로초 단위로 너무 짧아서입니다.

**Q. 명령 실행은 왜 주로 단일 스레드에서 순차적으로 처리되나요?**
→ 소켓에서 읽고 RESP 프로토콜로 파싱해 명령을 실행하고 응답을 버퍼에 쓰는 이 전 과정이 하나의 스레드에서 순차적으로 돕니다. 이렇게 하면 어느 순간에도 동시에 실행되는 명령이 없으므로, 락을 걸지 않아도 경쟁 상태가 원천적으로 생기지 않고 모든 명령이 자연히 원자적이 됩니다. 이 구조가 성립하는 근거는 명령 하나하나가 메모리 연산이라 마이크로초 단위로 짧다는 점입니다. 명령이 짧으니 한 줄로 세워 처리해도 뒤가 밀리지 않는 것입니다. 다만 같은 이유로 KEYS나 긴 Lua처럼 오래 걸리는 작업 하나가 끼면 뒤의 모든 요청이 함께 지연되는 약점도 이 설계에서 함께 나옵니다.

**Q. Redis가 싱글 스레드인데도 빠른 이유는 무엇인가요?**
→ 느려질 이유들을 설계로 하나씩 제거했기 때문입니다. 우선 대기는 epoll이 대신 감시해 주므로 커넥션이 늘어도 스레드를 늘릴 필요가 없습니다. 실행은 단일 스레드에서 락 없이 순차로 이뤄져 락 획득·경합·캐시 동기화 비용이 0입니다. 연산은 메모리 접근과 전용 자료구조 덕분에 대부분 O(1)~O(logN)에 끝납니다. 원래 멀티스레드가 필요한 이유는 느린 I/O 대기와 긴 CPU 연산인데, Redis의 워크로드에는 그 두 가지가 거의 없으므로 멀티스레드가 주는 이점 대신 그 비용(스위칭·락)만 피한 셈입니다.

**Q. I/O multiplexing과 multi-threading은 어떤 차이가 있나요?**
→ 두 방식은 동시성을 확보하기 위해 동원하는 자원이 다릅니다. I/O multiplexing은 커널의 이벤트 통지와 스레드 하나로 수많은 연결의 "대기"를 처리합니다. 스레드가 하나뿐이라 컨텍스트 스위칭이나 공유 데이터 동기화 비용이 없지만, 실행이 한 줄로 이뤄지므로 긴 CPU 작업이 끼면 전체가 막히는 취약점이 있습니다. 반면 multi-threading은 "실행 자체"를 여러 스레드로 병렬화하므로 CPU 작업을 나눌 수 있지만, 스레드 생성·전환 비용과 공유 데이터에 대한 락이 필요합니다. 그래서 "짧은 요청을 대량으로 처리"하는 데는 전자가, "긴 연산을 병렬화"하는 데는 후자가 맞는 도구입니다.

**Q. Redis에서 하나의 명령이 오래 걸리면 다른 요청에는 어떤 영향이 있나요?**
→ 단일 스레드가 순차 처리하는 구조이므로, 명령 하나가 오래 걸리면 큐잉된 나머지 요청은 전부 그 명령이 끝날 때까지 대기해야 합니다. 클라이언트 입장에서는 그 순간 Redis 전체가 멈춘 것과 다름없습니다. 게다가 응답이 늦어지면 클라이언트들의 타임아웃과 재시도가 몰려 부하가 오히려 증폭될 수 있습니다. 그래서 운영에서는 slowlog로 느린 명령을 상시 감시하고, KEYS나 대형 SMEMBERS/HGETALL 같은 O(N) 명령을 코드 리뷰 단계에서 걸러냅니다. 또 대형 키를 지울 때는 DEL 대신 실제 메모리 해제를 백그라운드 스레드로 넘기는 UNLINK를 쓰는 것이 기본기입니다.

**Q. Redis Lua Script가 오래 실행되면 왜 위험한가요?**
→ Lua 스크립트는 실행되는 동안 원자성을 보장하기 위해 다른 모든 명령을 배제하므로, 스크립트 실행 시간만큼 서버 전체가 그 스크립트에 독점됩니다. 기본값인 5초(busy-reply-threshold, 과거 이름 lua-time-limit)를 넘어가면 다른 클라이언트에게 BUSY 에러가 반환되기 시작합니다. 문제는 이때도 이미 쓰기를 수행한 스크립트는 SCRIPT KILL로 죽일 수 없다는 점입니다. 중간에 강제로 끊으면 일부만 반영된 부분 쓰기가 남아 원자성이 깨지기 때문입니다. 결국 남는 선택지가 SHUTDOWN NOSAVE 수준이라 "긴 Lua는 장애가 나면 탈출구가 없다"는 것이 위험의 본질이며, 그래서 스크립트는 짧게 유지하고 루프에는 반드시 상한을 두고 설계해야 합니다.

### 면접 답변 (구술용)

Redis는 모든 클라이언트 소켓을 논블로킹으로 설정하고 커널의 다중화 기능 — 리눅스의 epoll, macOS/BSD의 kqueue — 에 등록해 둡니다. 이벤트 루프는 "읽거나 쓸 준비가 된 소켓 목록"을 커널에서 한 번에 받아오고, 준비된 소켓만 순회하면서 요청을 읽고 파싱하고 실행하고 응답을 버퍼에 씁니다. 준비 안 된 소켓을 붙잡고 기다리는 일이 없으니 스레드 하나로 수만 커넥션을 감당할 수 있습니다. 이 구조가 성립하는 근거는 명령 하나하나가 메모리 연산이라 마이크로초 단위로 짧다는 것입니다 — 동시 접속이 5만이어도 어느 순간 실제로 데이터를 보내온 소켓은 일부이고, 각 처리가 짧으니 한 줄로 세워도 밀리지 않습니다. 부수 효과로 모든 명령이 원자적입니다 — 동시에 도는 명령이 없으니 락 없이도 경쟁 상태가 없습니다. 다만 같은 설계의 이면으로, KEYS나 긴 Lua처럼 오래 걸리는 작업 하나가 끼면 뒤의 모든 요청이 일괄 지연됩니다. 그래서 운영에서는 KEYS 대신 SCAN, 대형 키 삭제는 UNLINK를 쓰는 것이 기본입니다.

### 원리 이해 (왜 그런가)

**이벤트 루프 한 바퀴:**

```
① 등록: 모든 클라이언트 소켓(fd)을 논블로킹 + epoll/kqueue에 등록
        (Redis는 자체 이벤트 라이브러리 ae로 플랫폼별 구현을 추상화)
② 대기: epoll_wait — "준비된 fd 목록"을 커널에서 한 번에 수령
        (여기서 블로킹이 사라짐 — 기다림을 커널에 위임)
③ 처리: 준비된 소켓만 순회 — 읽기 → RESP 파싱 → 명령 실행 → 응답 버퍼 쓰기
        (이 전체가 단일 스레드 순차 → 모든 명령 원자적, 락 불필요)
```

**"동시에 처리한다"의 진짜 의미**: 시간이 걸리는 부분(네트워크 대기)은 커널이 병렬로 감시하고, CPU를 쓰는 부분(명령 실행)은 하나씩 빠르게 처리합니다. 요청이 겹쳐 보이는 이유는 각 실행이 너무 짧아서입니다.

**같은 설계의 강점과 약점이 한 몸:**

| 설계 선택 | 얻는 것 | 잃는 것 |
|-----------|---------|---------|
| 단일 스레드 실행 | 락 제거, 명령 원자성, 스위칭 비용 0 | 긴 작업 하나가 전체를 블로킹 |
| 이벤트 루프 | 스레드 1개로 수만 커넥션 | 루프 위에서 "절대 기다리면 안 됨"이라는 규율 필요 |

### 꼬리질문 Q&A

**Q. Redis가 싱글 스레드인데도 빠른 이유는?**
→ **느려질 이유를 설계로 하나씩 제거했기 때문이다.**
대기는 epoll이 대신하니 스레드를 늘릴 필요가 없고, 실행은 락 없이 순차라 경합 비용이 0이고, 연산은 메모리 + 전용 자료구조로 O(1)~O(logN)입니다. 멀티스레드가 필요한 이유(느린 I/O 대기, 긴 연산)가 Redis 워크로드에는 거의 없으므로, 멀티스레드의 비용(스위칭, 락)만 피한 셈입니다. (260615와 연결)

**Q. I/O multiplexing과 multi-threading의 차이는?**
→ **동시성을 확보하는 자원이 다르다 — 전자는 "커널의 이벤트 통지 + 스레드 1개", 후자는 "스레드 여러 개".**
multiplexing은 많은 연결의 "대기"를 스레드 하나로 처리합니다 — 스위칭·동기화 비용이 없지만 실행이 한 줄이라 긴 CPU 작업에 취약합니다. multi-threading은 "실행 자체"를 병렬화합니다 — CPU 작업을 나눌 수 있지만 스레드 비용과 공유 데이터 락이 필요합니다. 그래서 "짧은 요청 대량 처리"는 전자, "긴 연산 병렬화"는 후자가 맞는 도구입니다.

**Q. 하나의 명령이 오래 걸리면 다른 요청에는 어떤 영향이 있나요?**
→ **큐잉된 모든 요청이 그 명령이 끝날 때까지 대기한다 — 클라이언트 입장에선 Redis 전체가 멈춘 것과 같다.**
이어서 타임아웃·재시도 폭주로 증폭될 수 있습니다. 그래서 slowlog로 느린 명령을 상시 감시하고, O(N) 명령(KEYS, 대형 SMEMBERS/HGETALL)을 코드 리뷰에서 걸러내며, 대형 키 삭제는 DEL 대신 UNLINK(실제 메모리 해제를 백그라운드 스레드로 넘김)를 쓰는 것이 운영 기본기입니다.

**Q. Lua Script가 오래 실행되면 왜 위험한가요?**
→ **스크립트는 원자성 보장을 위해 실행 중 다른 모든 명령을 배제한다 — 실행 시간만큼 서버 전체가 독점된다.**
기본 5초(busy-reply-threshold, 과거 이름 lua-time-limit)를 넘으면 다른 클라이언트에 BUSY 에러가 반환되기 시작하는데, 이때도 이미 쓰기를 수행한 스크립트는 SCRIPT KILL로 죽일 수 없습니다 — 중간에 끊으면 부분 쓰기가 남아 원자성이 깨지기 때문입니다. 남는 선택지가 SHUTDOWN NOSAVE 수준이라 "긴 Lua는 장애 시 탈출구가 없다"는 것이 위험의 본질입니다. 스크립트는 짧게, 루프에는 바운드를 두고 설계해야 합니다.

### ❓ 더 파볼 질문
- **epoll은 왜 select보다 빠른가?**
  ↳ select는 호출할 때마다 전체 fd 집합을 커널에 복사하고 커널이 전부를 선형 스캔한다 — fd가 늘수록 O(n)으로 비싸진다. epoll은 관심 fd 목록을 커널이 계속 유지하고(epoll_ctl로 등록) 준비된 것만 통지하므로, 대기 비용이 "전체 수"가 아니라 "준비된 수"에 비례한다. 수만 커넥션 시대에 epoll이 표준이 된 이유다.
- **Netty의 이벤트 루프와 Redis의 이벤트 루프는 뭐가 다른가?**
  ↳ 패턴은 같은 reactor지만, Netty는 이벤트 루프를 여러 개(보통 코어 수) 두고 채널마다 고정 루프를 배정해 멀티코어를 활용한다. Redis는 명령 원자성을 위해 실행 루프를 하나로 유지하고 네트워크 단계만 io-threads로 보조한다 — "원자성 요구가 루프 개수를 결정한다"는 관점으로 비교하면 깔끔하다.
- **Redis keyspace notification은 어떤 용도이고 한계는?**
  ↳ 키 만료·변경 이벤트를 pub/sub 채널로 발행해 캐시 무효화나 후속 처리에 쓸 수 있다. 단 pub/sub 기반이라 전달 보장이 없고(놓치면 끝), 이벤트 폭증 시 부하가 되므로 "정합성 수단"이 아니라 "최적화 힌트"로만 써야 한다.
