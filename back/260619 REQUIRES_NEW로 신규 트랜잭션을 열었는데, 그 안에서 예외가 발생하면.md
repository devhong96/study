# 260619 REQUIRES_NEW로 신규 트랜잭션을 열었는데, 그 안에서 예외가 발생하면 기존 트랜잭션에는 어떤 영향이 있을까요?

REQUIRES_NEW로 신규 트랜잭션을 열었는데, 그 안에서 예외가 발생하면 기존 트랜잭션에는 어떤 영향이 있을까요?

트랜잭션 전파 옵션은 면접에서 은근 자주 나오는 주제입니다.
특히 이력서에 @Transactional, 결제/주문/정산/로그 저장 같은 내용을 적어두면
“어디까지 하나의 트랜잭션으로 묶었나요?”
“실패하면 어디까지 롤백되나요?”
이런 식으로 꼬리질문이 이어질 수 있습니다.

이 질문은 단순히 REQUIRES_NEW를 외웠는지보다,
트랜잭션 경계와 실패 전파를 실제 서비스 흐름에서 이해하고 있는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- REQUIRED와 REQUIRES_NEW의 차이
- 기존 트랜잭션을 잠시 중단하고 새 트랜잭션을 여는 구조
- 신규 트랜잭션 실패가 기존 트랜잭션에 자동으로 전파되는지 여부
- 예외를 잡아서 처리할 때와 밖으로 던질 때의 차이
- 감사 로그, 이력 저장, 외부 연동 상태 저장 등에 REQUIRES_NEW를 쓰는 이유

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- REQUIRES_NEW는 언제 사용하는 게 적절한가요?
- 신규 트랜잭션이 롤백돼도 기존 트랜잭션은 커밋될 수 있나요?
- 반대로 기존 트랜잭션이 롤백되면 이미 커밋된 신규 트랜잭션은 어떻게 되나요?
- 트랜잭션 안에서 외부 API를 호출하면 어떤 문제가 생길 수 있나요?
- 로그 저장을 REQUIRES_NEW로 분리할 때 장단점은 무엇인가요?

---

## 답변

> **한 줄 핵심**: 신규 트랜잭션은 롤백된다. 기존 트랜잭션의 운명은 "자동 전파"가 아니라 **예외 객체가 기존 경계까지 전파되느냐**로 결정된다 — catch하면 살고, 던지면 같이 롤백된다.

### 1문 1답

**Q. REQUIRED와 REQUIRES_NEW의 차이는 무엇인가요?**
→ REQUIRED(기본)는 진행 중인 트랜잭션이 있으면 거기에 참여해서 하나의 물리 트랜잭션을 공유하고 커넥션도 하나만 쓴다. 그래서 참여한 내부에서 롤백이 마킹되면, 바깥에서 예외를 catch하더라도 커밋 시점에 `UnexpectedRollbackException`이 나며 전체가 롤백된다. 반면 REQUIRES_NEW는 기존 트랜잭션을 잠시 suspend하고 별도 커넥션으로 완전히 독립적인 물리 트랜잭션을 새로 연다. 이 경우 커넥션을 2개 동시에 점유하고, 신규의 커밋/롤백이 기존과 독립적이다. 한마디로 "물리 트랜잭션 1개 공유 vs 2개 독립"이 근본 차이이고, 이후 동작들이 모두 여기서 파생된다.

**Q. 기존 트랜잭션을 잠시 중단하고 새 트랜잭션을 여는 구조는 어떻게 동작하나요?**
→ REQUIRES_NEW를 만나면 트랜잭션 매니저가 기존 트랜잭션(커넥션 A)을 suspend해 스레드에서 떼어 잠시 보관한다. 그리고 새 커넥션 B로 신규 트랜잭션을 시작해 그 안의 작업을 처리한 뒤, 신규를 커밋 또는 롤백하고 다시 기존을 resume한다. 이렇게 커넥션이 물리적으로 A와 B 둘로 나뉘어 있다는 사실이, 뒤이은 모든 성질(독립 커밋, 커넥션 2개 점유, 실패 격리)의 근거가 된다. suspend/resume은 트랜잭션 매니저가 트랜잭션-스레드 바인딩을 잠시 풀었다 되돌리는 메커니즘이다.

**Q. 신규 트랜잭션 실패가 기존 트랜잭션에 자동으로 전파되나요? (신규가 롤백돼도 기존은 커밋될 수 있나요?)**
→ 자동으로 전파되지는 않는다. 신규 트랜잭션 안에서 예외가 나면 신규 자신은 롤백되지만, 기존 트랜잭션의 운명은 "신규가 실패했다"가 아니라 "그 예외 객체가 기존의 경계까지 도달하느냐"로 결정된다. 그래서 호출자가 예외를 catch해서 처리하면 기존은 아무 영향 없이 커밋할 수 있다. 반대로 예외를 잡지 않고 밖으로 던지면 예외가 기존 경계(프록시)까지 올라가 기존도 함께 롤백된다. 즉 신규가 롤백돼도 기존은 얼마든지 커밋될 수 있으며, 그 여부는 예외 처리 방식이 쥐고 있다.

**Q. 예외를 잡아서 처리할 때와 밖으로 던질 때는 어떻게 다른가요?**
→ REQUIRES_NEW에서는 호출자가 신규의 예외를 catch하면 기존은 계속 진행해 커밋할 수 있고, 던지면 예외가 기존 경계까지 전파되어 기존도 롤백된다. 그래서 예외를 잡느냐 던지느냐가 곧 "부가 기능이 실패해도 본 작업을 살릴 것인가"라는 비즈니스 정책의 표현이 된다. 반대로 REQUIRED에서는 같은 물리 트랜잭션을 공유하기 때문에 사정이 다르다. 내부 예외로 rollback-only가 이미 마킹되면, 바깥에서 catch하더라도 커밋 시점에 `UnexpectedRollbackException`이 나며 전체가 롤백되어 "잡아도 못 살린다". 즉 같은 catch라도 전파 옵션에 따라 결과가 정반대다.

**Q. 감사 로그, 이력 저장, 외부 연동 상태 저장 등에 REQUIRES_NEW를 쓰는 이유는 무엇인가요? (언제 쓰는 게 적절한가요?)**
→ 이런 기록들은 본 트랜잭션의 성패와 무관하게 반드시 남아야 한다는 공통 요구를 가진다. REQUIRES_NEW는 독립된 물리 트랜잭션이라, 본 작업이 나중에 롤백되더라도 이미 커밋된 기록은 그대로 살아남는다. 커밋은 DB 차원에서 확정된 행위여서 기존이 뒤에 롤백돼도 소급 취소되지 않기 때문이다. "본 작업이 롤백돼도 이 기록만은 커밋되어야 한다"는 요구는 논리적 분리로는 안 되고 물리 트랜잭션 분리로만 달성할 수 있어서 REQUIRES_NEW를 쓴다. 감사 로그·알림 이력·외부 연동 시도/실패 이력이 대표적인 사용처다.

**Q. 반대로 기존 트랜잭션이 롤백되면 이미 커밋된 신규 트랜잭션은 어떻게 되나요?**
→ 그대로 남는다. 커밋은 DB 차원에서 이미 확정된 행위라, 나중에 기존 트랜잭션이 롤백되더라도 소급해서 취소할 수 없다. 이 성질이 바로 감사 로그처럼 "본 작업이 실패해도 남아야 하는 기록"에 REQUIRES_NEW를 쓰는 근거이기도 하다. 다만 반대로 "둘이 함께 무효여야 한다"는 요구라면 REQUIRES_NEW는 애초에 잘못된 선택이며, 이미 커밋된 신규를 되돌리려면 별도의 보상 로직(취소 기록)을 설계해야 한다.

**Q. 트랜잭션 안에서 외부 API를 호출하면 어떤 문제가 생길 수 있나요?**
→ 커넥션 점유 시간이 외부 API의 응답 시간에 종속되기 때문에, 외부가 느려지면 그만큼 커넥션을 오래 쥐고 있어 풀 고갈로 이어진다. 게다가 외부 API 호출은 DB 롤백에 동참하지 않으므로 "DB만 롤백되고 API는 이미 나간" 정합성 틈이 생긴다. 특히 REQUIRES_NEW와 결합되면 커넥션을 2개나 쥔 채 외부 응답을 기다리는 최악의 조합이 된다. 풀이 부족한 상황에서 이런 요청이 겹치면 서로 커넥션을 못 얻어 자원 데드락까지 갈 수 있다. 그래서 외부 호출은 트랜잭션 경계 밖으로 빼는 것이 원칙이다.

**Q. 로그 저장을 REQUIRES_NEW로 분리할 때 장단점은 무엇인가요?**
→ 장점은 본 트랜잭션이 롤백되더라도 로그가 독립적으로 커밋되어 보존된다는 것이다. 단점은 커넥션을 2개 점유하는 비용이 든다는 점인데, 같은 트래픽에서 풀 고갈이 두 배 빨리 올 수 있다. 또 "로그 저장 자체가 실패하면 본 작업은 어떻게 할 것인가"라는 정책을 명시적으로 정해야 한다. 만약 로그가 크리티컬하지 않다면, `AFTER_COMMIT` 이벤트 리스너로 커밋 후에 비동기로 기록하는 방식이 커넥션을 추가로 물지 않는 좋은 대안이다. 즉 REQUIRES_NEW는 "반드시 독립적으로 남아야 하는 기록"에 한정해 쓰는 게 좋다.

### 면접 답변 (구술용)

REQUIRES_NEW는 기존 트랜잭션을 잠시 중단시키고 별도 커넥션으로 완전히 독립적인 물리 트랜잭션을 새로 엽니다. 그래서 신규 트랜잭션 안에서 예외가 나면 신규는 롤백되지만, 기존 트랜잭션에 대한 영향은 예외가 어디까지 전파되느냐에 달려 있습니다. 호출자가 예외를 catch해서 처리하면 기존 트랜잭션은 영향 없이 커밋할 수 있고, 밖으로 던지면 예외가 기존 경계까지 올라가 기존도 롤백됩니다. 즉 "신규가 실패해서" 롤백되는 게 아니라 "예외 객체가 내 경계에 도달해서" 롤백되는 것입니다. REQUIRED와 대비하면 차이가 선명한데, REQUIRED는 같은 물리 트랜잭션을 공유해서 내부 예외 시 rollback-only가 마킹되므로 catch해도 커밋 시 UnexpectedRollbackException으로 전체가 롤백됩니다. 반대 방향도 독립이라, 이미 커밋된 신규 트랜잭션은 기존이 롤백돼도 남습니다 — 이 성질로 감사 로그처럼 "본 작업이 실패해도 남아야 하는 기록"에 씁니다. 운영상 주의점은 스레드당 커넥션을 2개 점유해서 풀 고갈이 빨라진다는 것입니다.

### 원리 이해 (왜 그런가)

**구조 — "물리적으로 둘"이 모든 답의 근거:**

```
기존 트랜잭션(커넥션 A) ──suspend──▶ 신규 트랜잭션(커넥션 B) 시작
                                     └ 예외 발생 → B 롤백
예외를 catch?  ── Yes → 기존(A) 계속 진행, 커밋 가능
              └─ No  → 예외가 A의 경계(프록시)까지 전파 → A도 롤백
```

**REQUIRED vs REQUIRES_NEW (핵심 비교):**

| | REQUIRED (기본) | REQUIRES_NEW |
|---|---|---|
| 물리 트랜잭션 | 1개 공유 | 2개 독립 (기존은 suspend) |
| 커넥션 | 1개 | **2개 동시 점유** |
| 내부 예외 → catch하면? | 못 살림 — rollback-only 마킹 → 커밋 시 `UnexpectedRollbackException` | **살릴 수 있음** — 기존은 무관하게 커밋 가능 |
| 내부 커밋의 독립성 | 없음 (전체가 함께) | 있음 (기존 롤백돼도 신규 커밋 유지) |

**운영 리스크 (면접 가산점):**
- 커넥션 2개 점유 → 같은 트래픽에서 풀 고갈이 2배 빨리 온다.
- 극단 케이스: 풀이 부족하면 전 스레드가 "A를 쥔 채 B를 대기" → 서로를 막는 자원 데드락 가능.
- 두 트랜잭션이 같은 row를 건드리면 자기 자신과 락 경합/데드락 가능.

### 꼬리질문 Q&A

**Q. REQUIRES_NEW는 언제 적절한가?**
→ **본 트랜잭션의 성패와 무관하게 반드시 남아야 하는 기록** — 감사 로그, 알림 이력, 외부 연동 시도/실패 이력.
- 공통 요구 = "본 작업이 롤백돼도 이 기록은 커밋" → 물리 트랜잭션 분리로만 달성 가능.

**Q. 신규가 롤백돼도 기존은 커밋될 수 있나?**
→ **예 — 호출자가 예외를 잡으면.** 예외 처리 방식이 곧 비즈니스 정책의 표현이다("부가 기능 실패 시 본 작업을 살릴 것인가").

**Q. 기존이 롤백되면 이미 커밋된 신규는?**
→ **그대로 남는다 — 커밋은 DB 차원에서 확정된 행위라 소급 불가.**
- "함께 무효여야 한다"면 REQUIRES_NEW가 잘못된 선택이거나, 보상 로직(취소 기록)이 필요하다.

**Q. 트랜잭션 안 외부 API 호출의 문제는?**
→ **커넥션 점유 시간이 외부 응답에 종속(풀 고갈) + API는 롤백 불가(정합성 틈).**
- REQUIRES_NEW와 결합되면 "커넥션 2개를 쥔 채 외부 대기"라는 최악 조합.

**Q. 로그 저장을 REQUIRES_NEW로 분리할 때 장단점은?**
→ **장점: 본 롤백과 무관하게 로그 보존. 단점: 커넥션 2개 비용 + "로그 실패 시 본 작업은?" 정책을 명시해야 함.**
- 로그가 크리티컬하지 않으면 AFTER_COMMIT 이벤트 + 비동기 기록이 커넥션 비용 없는 대안.

### 🌱 심화 키워드
- **물리 트랜잭션 vs 논리 트랜잭션** — 스프링 전파 의미론을 이해하는 열쇠
- **rollback-only 마킹 / UnexpectedRollbackException** — REQUIRED 참여 트랜잭션의 함정
- **NESTED** — 세이브포인트 기반 부분 롤백; REQUIRES_NEW와 무엇이 다른가
- **suspend/resume** — JTA/트랜잭션 매니저가 기존 트랜잭션을 보류하는 메커니즘
- **TransactionSynchronizationManager** — 트랜잭션-스레드 바인딩의 실체 (→ 260623)

- 토비의 스프링 — 트랜잭션 전파 개념 도입부

### ❓ 더 파볼 질문
- **NESTED(세이브포인트)는 REQUIRES_NEW와 달리 왜 "부모가 롤백되면 함께 롤백"되나?**
  ↳ NESTED는 새 물리 트랜잭션이 아니라 **같은 트랜잭션 안에 세이브포인트를 찍는 것**이기 때문이다. 내부 실패 시 세이브포인트까지만 부분 롤백해 부모를 살릴 수는 있지만, 커밋 자체는 부모의 커밋에 묶여 있어서 부모가 롤백되면 전체가 사라진다. "부분 롤백 가능 + 독립 커밋 불가"가 NESTED, "독립 커밋 가능 + 커넥션 2개"가 REQUIRES_NEW — 정확히 반대 방향의 트레이드오프다.
- **REQUIRES_NEW 안에서 다시 REQUIRES_NEW를 열면 커넥션 점유는 어떻게 되나?**
  ↳ 열 때마다 기존 트랜잭션이 suspend 스택에 쌓이며 각자의 커넥션을 계속 쥔다 — 2중이면 스레드당 3개다. 중첩 깊이만큼 커넥션 점유가 곱해지므로, REQUIRES_NEW의 중첩은 사실상 설계 경고 신호로 봐야 한다.
- **커넥션 풀 크기가 N일 때, 동시 요청 몇 개부터 REQUIRES_NEW 자원 데드락 가능성이 생기나?**
  ↳ 각 요청이 커넥션 2개를 쓰므로 안전한 동시 처리량은 N/2다. 최악 시나리오는 동시 요청 K개가 전부 첫 커넥션을 쥔 상태에서 두 번째를 기다리는 경우 — K ≥ N이면 풀에 남은 커넥션이 0이라 전원이 영원히 대기하는 완전 교착이 된다(connectionTimeout으로 예외가 나며 풀리긴 하지만 그 자체가 장애다). N/2 < K < N 구간도 타임아웃 확률이 급증한다 — "REQUIRES_NEW를 쓰는 경로의 동시성은 풀의 절반까지"라는 어림이 나오는 계산이다.
