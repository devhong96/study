# 260711 : Redis 분산락은 실제로 무엇을 막아 주나요?

Redis 분산락은 실제로 무엇을 막아 주나요?

이 질문은 “Redis로 락을 걸 수 있다” 수준을 넘어서, 락이 보장하는 범위와 보장하지 못하는 범위를 구분할 수 있는지 보는 질문입니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- Redis 분산락은 여러 서버 인스턴스가 동시에 같은 임계 영역에 진입하는 것을 막기 위해 사용합니다.
- 예를 들어 같은 쿠폰을 동시에 발급하거나, 같은 재고를 동시에 차감하거나, 같은 예약 건을 중복 처리하는 상황을 제어할 수 있습니다.
- 하지만 분산락이 비즈니스 작업 전체의 성공이나 rollback까지 자동으로 보장하는 것은 아닙니다.
- 락은 “동시에 실행되지 않게 하는 장치”이고, 데이터 정합성은 DB transaction, unique constraint, optimistic lock, 보상 처리와 함께 설계해야 합니다.
- 따라서 Redis 분산락은 중복 실행 가능성을 줄여주지만, 장애 상황까지 고려하면 멱등성과 최종 정합성 설계가 함께 필요합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- 분산락을 사용하면 transaction rollback도 같이 보장되나요?
- Redis lock을 잡은 애플리케이션이 죽으면 어떻게 복구하나요?
- Lease time과 wait time은 어떤 기준으로 정하나요?
- DB unique key와 Redis 분산락은 각각 어떤 문제를 막아주나요?
- Redis 장애가 발생하면 정확성과 가용성 중 무엇을 우선해야 할까요?

---

## 답변

> 260706과 동일 주제 — 여기서는 이 파일의 꼬리질문 기준으로 정리. TTL 경계·커밋 순서·남의 락 해제 시나리오 상세는 260706 참고.

> **한 줄 핵심**: 분산락이 막는 것은 "여러 인스턴스의 임계 구역 **동시 진입**" 하나뿐 — rollback도, TTL 경계의 늦은 쓰기도, 시간차 중복도 못 막는다. 그래서 "락이 없어도 데이터는 안 깨지는 층(DB 제약·멱등성)" 위에 락을 얹는 것이 올바른 구조다.

### 1문 1답

**Q. Redis 분산락은 실제로 무엇을 막아 주나요?**

**A.** 여러 서버 인스턴스가 같은 임계 영역에 동시에 진입하는 것, 즉 상호배제(mutual exclusion)를 막아 줍니다. 같은 쿠폰의 동시 발급, 같은 재고의 동시 차감, 같은 예약 건의 동시 처리처럼 "동시에 실행되면 깨지는 구간"을 직렬화하는 것입니다. 단일 서버라면 synchronized로 충분했을 것을, 인스턴스가 여러 개라 락 상태를 공유 저장소인 Redis에 두는 것뿐입니다. 중요한 것은 락이 보장하는 것은 딱 이 "동시 진입 차단" 하나이고, rollback이나 TTL 경계의 늦은 쓰기, 시간차 중복은 막지 못한다는 점입니다.

**Q. 분산락을 사용하면 비즈니스 작업의 성공이나 transaction rollback도 보장되나요?**

**A.** 아니오입니다. 락(Redis)과 트랜잭션(DB)은 별개 시스템의 별개 메커니즘이라 서로의 상태를 모르기 때문입니다. 락은 실행 순서를 제어하는 장치일 뿐이고, 실패한 작업의 되돌림은 DB 트랜잭션과 보상 로직의 몫입니다. 오히려 주의할 상호작용은 @Transactional 메서드 안에서 락을 잡고 풀 때인데, 커밋은 프록시(메서드 밖)에서 일어나므로 락 해제가 커밋보다 먼저 가기 쉽습니다. 그래서 락이 풀린 뒤 커밋 전 사이에 다른 요청이 들어올 수 있어, 락 안에서 커밋까지 끝나는 구조로 잡아야 보호가 성립합니다.

**Q. 그렇다면 데이터 정합성은 무엇이 지키나요?**

**A.** 정합성의 최종 방어선은 DB 트랜잭션, unique 제약, 낙관적 락, 멱등성 키 같은 데이터 레벨의 장치들입니다. 락은 실행(process) 레벨의 사전 억제 장치이고, 정합성은 데이터(data) 레벨에서 지켜지므로 층이 다릅니다. 따라서 둘은 대체재가 아니라 보완재이며, 함께 써야 합니다. 예를 들어 쿠폰 발급은 coupon_issue(user_id, coupon_id)에 unique를 걸면 락이 없어도 DB가 직렬화해 하나만 성공시켜 정합성이 완벽해지고, 락은 그 위에서 실패할 요청이 DB까지 내려가는 부하를 앞단에서 줄이는 역할을 합니다.

**Q. 장애 상황까지 고려하면 어떤 설계가 함께 필요한가요?**

**A.** 분산락은 중복 실행 가능성을 줄여줄 뿐, 두 가지를 못 막습니다. 하나는 TTL 만료 경계인데, 작업이 TTL을 넘기면 락이 먼저 풀려 다른 요청이 진입하고 원래 요청의 늦은 쓰기가 겹칠 수 있습니다. 다른 하나는 시간차 중복으로, 재시도나 더블클릭은 순서대로 락을 잡고 각각 실행되므로 락 입장에선 정상 흐름입니다. 그래서 TTL 경계의 늦은 쓰기는 fencing token이나 낙관적 락·조건부 UPDATE로, 시간차 중복은 멱등성 키로 막아야 합니다. 결국 최종 정합성과 멱등성 설계가 락과 함께 있어야 장애 상황까지 안전합니다.

**Q. Redis lock을 잡은 애플리케이션이 죽으면 어떻게 복구하나요?**

**A.** TTL이 자동 복구 장치입니다. 죽은 홀더는 락을 갱신하거나 해제하지 못하므로, TTL이 만료되면 락이 자동으로 풀리고 다음 요청이 진행합니다. TTL 없는 락은 홀더가 죽으면 영구 데드락이 되므로 반드시 TTL을 걸어야 하며, Redisson watchdog을 쓰는 경우도 JVM이 죽으면 갱신이 멈춰 기본 최대 30초 안에 풀립니다. 단 "락 해제"가 곧 "데이터 복구"는 아니라는 점이 중요합니다. 죽기 직전까지의 작업 잔재는, 커밋 전이면 트랜잭션 롤백이 정리하고 커밋 후면 멱등 재시도나 보상이 정리합니다.

**Q. Lease time과 wait time은 어떤 기준으로 정하나요?**

**A.** lease는 "작업 p99 + GC/네트워크 여유"보다는 길게, 홀더 장애 시 허용 가능한 잠금 시간보다는 짧게 잡고, 이 둘이 충돌하면 watchdog(자동 연장)을 검토합니다. lease가 너무 짧으면 정상 작업 중에 락이 풀려 늦은 쓰기 문제가 나고, 너무 길면 홀더가 죽었을 때 다음 요청이 오래 막히기 때문입니다. wait는 사용자 응답 예산 기준으로 짧게 잡고 실패 시 정책을 명확히 해야 합니다. wait를 길게 잡으면 경합 시 대기 스레드가 쌓여 락 하나가 서버 전체 스레드 고갈로 번질 수 있다는 것이 짧게 가져가는 근거입니다.

**Q. DB unique key와 Redis 분산락은 각각 어떤 문제를 막아주나요?**

**A.** unique 제약은 커밋 시점에 중복 "데이터"를 차단하는 최종 방어선으로 경계 상황에서도 뚫리지 않고, 분산락은 실행 진입 시점에 중복 "작업"을 줄이는 사전 억제라 경계 상황에서 뚫릴 수 있습니다. 각자의 한계를 보면 왜 둘 다 필요한지가 드러납니다. unique만으로는 "실행은 됐다가 커밋에서 실패"하는 낭비와 DB 밖 부수효과(외부 API 중복 호출)를 못 막습니다. 락만으로는 TTL 경계 같은 상황을 못 막습니다. 그래서 층을 나눠 함께 써서, 락이 뚫려도 제약이 데이터를 지키는 구조로 설계합니다.

**Q. Redis 장애가 발생하면 정확성과 가용성 중 무엇을 우선해야 할까요?**

**A.** 도메인의 "중복 실행 비용"이 결정합니다. 결제·정산·쿠폰처럼 중복이 치명적인 기능은 fail-closed(락을 못 잡으면 거부, 정확성 우선), 캐시 재생성처럼 중복 비용이 낮은 구간은 fail-open(락 없이 진행, 가용성 우선)이 맞습니다. 단 fail-open을 택하려면 "락 없이도 데이터가 안 깨진다"는 DB 레벨 방어가 이미 있어야 합니다. 반대로 fail-closed를 택하면 Redis가 그 기능의 SPOF가 되므로, Sentinel/Cluster 같은 자체 고가용성과 장애 시 사용자 안내까지 설계에 포함해야 합니다.

### 면접 답변 (구술용)

Redis 분산락이 막아 주는 것은 여러 서버 인스턴스에 걸쳐 같은 임계 영역에 동시에 진입하는 것, 즉 상호배제입니다. 같은 쿠폰의 동시 발급, 같은 재고의 동시 차감, 같은 예약 건의 동시 처리처럼 "동시에 실행되면 깨지는 구간"을 직렬화합니다. 단일 서버라면 synchronized로 충분했을 것을, 인스턴스가 여러 개라 락 상태를 공유 저장소인 Redis에 두는 것뿐입니다. 반대로 막아 주지 못하는 것을 구분하는 게 이 질문의 핵심입니다. 첫째, 비즈니스 작업의 성공이나 rollback은 보장하지 않습니다 — 락은 실행 순서 제어 장치이고, 실패한 작업의 되돌림은 DB 트랜잭션과 보상 로직의 몫입니다. 둘째, TTL 만료 경계는 못 막습니다 — 작업이 TTL을 넘기면 락이 먼저 풀려 다른 요청이 진입하고, 원래 요청의 늦은 쓰기가 겹칠 수 있습니다. 셋째, 시간차 중복 — 재시도나 더블클릭은 순서대로 락을 잡고 각각 실행되므로 락 입장에선 정상 흐름이고, 이건 멱등성의 영역입니다. 그래서 결론은 — 정합성의 최종 방어선은 DB 트랜잭션, unique 제약, 낙관적 락, 멱등성 키에 두고, 분산락은 그 위에서 불필요한 경합과 중복 작업을 줄이는 장치로 쓰는 것입니다.

### 원리 이해 (왜 그런가)

**"막는 것"과 "못 막는 것"의 경계:**

| 위협 | 분산락이 막나? | 실제 담당 |
|------|---------------|-----------|
| 여러 인스턴스의 동시 진입 | ✅ (이것이 락의 존재 이유) | 분산락 |
| 락 안 작업의 실패·부분 반영 | ❌ | DB 트랜잭션(원자성) + 보상 |
| TTL 만료 후 늦은 쓰기 | ❌ (확률만 줄임) | fencing token / 낙관적 락 / 조건부 UPDATE |
| 시간차 중복 실행(재시도·더블클릭·재전달) | ❌ (각각 순서대로 정상 진입) | 멱등성 키 (→ 260710) |
| 중복 "데이터" 생성 | ❌ (실행만 제어) | DB unique 제약 (최종 방어선) |

이 표가 말하는 것: 락은 **실행(process) 레벨**의 사전 억제 장치이고, 정합성은 **데이터(data) 레벨**의 장치들이 지킨다 — 층이 다르므로 대체재가 아니라 보완재입니다.

### 꼬리질문 Q&A

**Q. 분산락을 사용하면 transaction rollback도 같이 보장되나요?**

**A.** **아니오 — 락(Redis)과 트랜잭션(DB)은 별개 시스템의 별개 메커니즘이라 서로의 상태를 모른다.**
락 안에서 예외가 나면 DB 롤백은 @Transactional이 처리하고 락은 finally에서 해제될 뿐이며, 이미 커밋된 뒤의 실패는 보상 로직이 필요합니다. 오히려 주의할 상호작용은 "커밋 전에 락부터 풀리는" 순서 문제입니다 — @Transactional 메서드 안에서 락을 잡고 풀면 커밋은 메서드 밖(프록시)에서 일어나므로, 락 해제가 커밋보다 먼저 가기 쉽습니다. 락 안에서 커밋까지 끝나는 구조로 잡아야 보호가 성립합니다.

**Q. Redis lock을 잡은 애플리케이션이 죽으면 어떻게 복구하나요?**

**A.** **TTL이 자동 복구 장치다 — 죽은 홀더는 갱신·해제를 못 하므로 TTL 만료로 풀리고 다음 요청이 진행한다.**
TTL 없는 락은 영구 데드락이 되므로 반드시 TTL을 겁니다. Redisson watchdog을 쓰는 경우도 JVM이 죽으면 갱신이 멈춰 기본 최대 30초 안에 풀립니다. 단 "락 해제"가 곧 "데이터 복구"는 아닙니다 — 죽기 직전까지의 작업 잔재는 커밋 전이면 트랜잭션 롤백이, 커밋 후면 멱등 재시도나 보상이 정리합니다.

**Q. Lease time과 wait time은 어떤 기준으로 정하나요?**

**A.** **lease는 "작업 p99 + GC/네트워크 여유"보다 길게, 홀더 장애 시 허용 가능한 잠금 시간보다 짧게 — 충돌하면 watchdog 검토. wait는 사용자 응답 예산 기준으로 짧게 + 실패 정책 명확히.**
wait를 길게 잡으면 경합 시 대기 스레드가 쌓여 락 하나가 서버 전체 스레드 고갈로 번질 수 있다는 게 짧게 가져가는 근거입니다. 상세 트레이드오프는 260713 참고.

**Q. DB unique key와 Redis 분산락은 각각 어떤 문제를 막아주나요?**

**A.** **unique 제약 = 커밋 시점에 중복 "데이터"를 차단하는 최종 방어선(뚫리지 않음), 분산락 = 실행 진입 시점에 중복 "작업"을 줄이는 사전 억제(경계 상황에서 뚫릴 수 있음).**
unique만으로는 "실행은 됐다가 커밋에서 실패"하는 낭비와 DB 밖 부수효과(외부 API 중복 호출)를 못 막고, 락만으로는 경계 상황을 못 막습니다. 그래서 층을 나눠 함께 씁니다 — 락이 뚫려도 제약이 데이터를 지키는 구조.

**Q. Redis 장애가 발생하면 정확성과 가용성 중 무엇을 우선해야 할까요?**

**A.** **도메인의 "중복 실행 비용"이 결정한다 — 결제·정산·쿠폰은 fail-closed(락 못 잡으면 거부, 정확성 우선), 캐시 재생성 같은 구간은 fail-open(락 없이 진행, 가용성 우선).**
단서가 둘 있습니다. fail-open을 택하려면 "락 없이도 데이터가 안 깨진다"는 DB 레벨 방어가 이미 있어야 하고, fail-closed를 택하면 Redis가 그 기능의 SPOF가 되므로 Redis 자체의 고가용성(Sentinel/Cluster)과 장애 시 사용자 안내까지 설계에 포함해야 합니다.

### 🌱 심화 키워드
- **fail-open / fail-closed** — 의존 시스템 장애 시의 두 가지 태도 (보안·락·인증 공통 프레임)
- **상호배제(mutual exclusion) / critical section** — 락이 보장하는 유일한 것의 정확한 이름
- **fencing token / 낙관적 락** — 경계 상황의 늦은 쓰기를 데이터 레벨에서 차단 (→ 260706)
- **Sentinel / Redis Cluster** — 락 저장소 자체의 가용성 수단과 failover 시 락 유실 한계
- **idempotency key** — 시간차 중복의 담당자 (→ 260710)

### ❓ 더 파볼 질문

**Q. Sentinel failover가 일어나는 동안 락은 어떤 상태가 되나?**

**A.** master가 죽고 replica가 승격되는 사이 락 쓰기가 실패하며(수초), 승격 전에 복제가 안 된 락 key는 새 master에 존재하지 않는다 — 즉 failover는 "락 유실 + 일시적 락 기능 불능"을 동시에 만들 수 있다. 이 구간의 행동(fail-open/closed)을 미리 정해두는 것이 운영 설계다.

**Q. 쿠폰 발급을 락 없이 unique 제약만으로 설계하면 어떻게 되나?**

**A.** coupon_issue(user_id, coupon_id)에 unique를 걸고 전 요청이 INSERT를 시도하게 하면 DB가 직렬화해 하나만 성공한다 — 정합성은 완벽하다. 락을 추가하는 이유는 실패할 요청들이 DB까지 내려가 부하를 만드는 것을 앞단에서 줄이려는 것뿐이다. "락 = 정합성 장치가 아니라 부하 완화 장치"임이 이 사례에서 선명해진다.

**Q. CAP 관점에서 Redis 분산락은 어떤 위치인가?**

**A.** 락은 본질적으로 일관성(C)을 요구하는 기능인데, 단일 master + 비동기 복제 구조의 Redis는 파티션(P) 상황에서 C를 희생할 수 있는 설계다 — 그래서 "정확성이 생명인 락"의 저장소로는 본질적 한계가 있고, 그 지점을 합의 기반 저장소(ZooKeeper/etcd)가 채운다.
