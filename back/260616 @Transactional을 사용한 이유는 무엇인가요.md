# 260616 : @Transactional을 사용한 이유는 무엇인가요?

@Transactional을 사용한 이유는 무엇인가요?

주니어 이력서를 보다 보면
“트랜잭션을 적용했습니다”, “@Transactional로 처리했습니다”라는 문장을 정말 자주 보는데요.

그런데 막상 면접에서
“왜 붙였나요?”
“어디까지 하나의 트랜잭션으로 묶어야 하나요?”
“외부 API 호출도 트랜잭션 안에 있어도 되나요?”
처럼 물어보면 대답이 흐려지는 경우가 꽤 많습니다.

이 질문은 단순히 @Transactional 어노테이션을 아는지보다,
데이터 정합성을 어떤 단위로 보장하려고 했는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- 어떤 작업들을 하나의 원자적 단위로 묶어야 했는가?
- 실패 시 어떤 데이터가 롤백되어야 하는가?
- 트랜잭션 범위가 너무 길어지면 어떤 문제가 생기는가?
- 외부 API 호출, 메시지 발행, 파일 업로드 같은 작업도 트랜잭션 안에 넣어도 되는가?
- 기본 전파 옵션과 롤백 조건을 알고 있는가?

---

## 답변

> **한 줄 핵심**: "비즈니스적으로 하나의 사실"에 해당하는 여러 DB 변경을 all-or-nothing으로 묶기 위해 — 그리고 그 커밋/롤백 코드를 AOP 프록시에 위임(선언적 트랜잭션)하기 위해 사용한다.

### 1문 1답

**Q. 어떤 작업들을 하나의 원자적 단위로 묶어야 했는가?**
→ "함께 성공하지 않으면 거짓말이 되는 것들", 즉 하나의 비즈니스 사실을 구성하는 DB 변경들을 묶어야 한다. 예컨대 주문 생성이 주문 저장·재고 차감·포인트 차감의 세 변경으로 이뤄진다면, 이건 비즈니스적으로 '주문'이라는 하나의 사실이므로 포인트 차감만 실패한 부분 성공 상태가 남으면 안 된다. 판단 기준은 기술이 아니라 도메인인데, "이 중 하나만 반영된 상태를 사용자나 정산이 용납하는가"를 물으면 된다. 용납할 수 없다면 하나의 트랜잭션으로 묶고, 그렇지 않다면 굳이 묶지 않아도 된다. 이렇게 경계를 도메인 기준으로 정하는 것이 핵심이다.

**Q. 실패 시 어떤 데이터가 롤백되어야 하는가?**
→ 트랜잭션 경계 안에서 일어난 모든 변경이 롤백 대상이다. 그래서 어디까지를 한 트랜잭션으로 묶느냐, 즉 경계 설정이 곧 롤백 범위 설계가 된다. 경계를 넓게 잡으면 더 많은 변경이 함께 되돌아가지만 그만큼 커넥션·락을 오래 쥐게 되고, 좁게 잡으면 정합성 보장 범위가 줄어든다. 롤백 대상 예외가 발생하면 그 경계 안의 변경들이 all-or-nothing으로 함께 취소된다. 결국 "무엇을 되돌릴 것인가"를 먼저 정하고 그에 맞춰 경계를 그어야 한다.

**Q. 트랜잭션 범위가 너무 길어지면 어떤 문제가 생기는가?**
→ 트랜잭션은 시작부터 커밋까지 커넥션을 점유하고, 그 사이 변경한 row의 락도 계속 쥐고 있다. 따라서 범위가 길어질수록 커넥션이 오래 묶여 풀 고갈(260617) 위험이 커진다. 동시에 락 유지 시간이 길어져 다른 트랜잭션의 대기와 데드락 확률도 올라가고, undo 같은 DB 자원 사용도 늘어난다. 그래서 트랜잭션은 꼭 필요한 DB 변경만 감싸 짧게 유지하는 것이 원칙이다. 무거운 연산이나 외부 호출은 경계 밖으로 빼야 한다.

**Q. 외부 API 호출, 메시지 발행, 파일 업로드 같은 작업도 트랜잭션 안에 넣어도 되는가?**
→ 넣지 않는 게 좋다. 근거는 두 가지인데, 첫째로 이런 외부 작업들은 DB 롤백에 동참하지 않아서 "DB만 롤백되고 API 호출은 이미 나가버린" 정합성 틈이 생긴다. 둘째로 외부의 응답 지연이 그대로 커넥션 점유 시간이 되어, 외부 장애가 커넥션 풀 고갈이라는 형태로 DB 쪽에 전파된다. 트랜잭션 안에 넣는다고 정합성이 생기는 것도 아니어서, "DB는 롤백됐는데 API는 나간" 상태의 복구(재시도·보상·Outbox)를 어차피 따로 설계해야 한다. 커밋 이후에만 실행돼야 하는 후속 작업이라면 `@TransactionalEventListener(AFTER_COMMIT)`나 Outbox 패턴(260618)을 쓴다.

**Q. 기본 전파 옵션과 롤백 조건을 알고 있는가?**
→ 기본 전파 옵션은 REQUIRED로, 진행 중인 트랜잭션이 있으면 참여하고 없으면 새로 시작한다. 여기서 '참여'는 같은 물리 트랜잭션을 공유한다는 뜻이라, 내부에서 롤백이 마킹되면 바깥까지 전체가 롤백된다(260619). 기본 롤백 대상은 unchecked 예외(RuntimeException·Error)뿐이고, checked 예외는 오히려 커밋되는 것이 함정이다. 그래서 checked 예외에도 롤백이 필요하면 `rollbackFor = Exception.class`를 명시해야 한다. 격리 수준은 별도 지정이 없으면 DB 기본값을 따르는데, MySQL(InnoDB)은 REPEATABLE READ다.

### 면접 답변 (구술용)

여러 DB 변경을 하나의 원자적 단위로 묶어 중간 실패 시 전부 되돌리기 위해 사용했습니다. 예를 들어 주문 생성이 "주문 저장, 재고 차감, 포인트 차감" 세 변경이라면 이건 비즈니스적으로 '주문'이라는 하나의 사실이므로, 포인트 차감만 실패한 부분 성공 상태가 남으면 안 됩니다. @Transactional이라는 수단을 고른 이유는 선언적 트랜잭션이기 때문입니다 — 커넥션 획득, 커밋, 롤백, 반납 코드를 비즈니스 로직에서 분리하고 AOP 프록시가 대신 수행합니다. 다만 범위 설계가 중요합니다. 트랜잭션이 길어지면 커넥션과 락을 오래 쥐어 풀 고갈과 데드락 위험이 커지고, 외부 API 호출은 롤백에 동참하지 않는 데다 응답 지연이 그대로 커넥션 점유가 되므로 트랜잭션 밖으로 빼야 합니다. 커밋 이후에만 실행돼야 하는 후속 작업은 AFTER_COMMIT 이벤트나 Outbox 패턴을 씁니다.

### 원리 이해 (왜 그런가)

**동작 메커니즘 (프록시 흐름):**

```
호출 → [프록시] 트랜잭션 시작 (커넥션 획득, autocommit off, ThreadLocal 바인딩)
     → 실제 메서드 실행
     → 정상 종료: 커밋 / 롤백 대상 예외: 롤백
     → 커넥션 반납
```

**기본값 3종 (외울 것):**

| 항목 | 기본값 | 함정 |
|------|--------|------|
| 전파(propagation) | REQUIRED — 있으면 참여, 없으면 새로 시작 | "참여" = 같은 물리 트랜잭션 공유 → 내부 롤백 마킹이 전체를 롤백시킴 (→ 260619) |
| 롤백 대상 | unchecked(RuntimeException, Error)만 | checked 예외는 **커밋됨** → `rollbackFor = Exception.class` 명시 필요 |
| 격리 수준 | DB 기본값 따름 | MySQL(InnoDB)은 REPEATABLE READ |

**범위 설계 원칙과 근거:**
- **짧게** — 트랜잭션은 시작~커밋까지 커넥션 점유 + 변경 row 락 유지. 길수록 풀 고갈(260617)·락 대기·데드락 확률 증가.
- **외부 I/O 배제** — 근거 2가지: ① 외부 API·메시지 발행·파일 업로드는 DB 롤백에 동참 안 함(정합성 틈), ② 외부 지연 = 커넥션 점유 시간(장애 전파 경로).
- **커밋 후 실행 보장이 필요하면** — `@TransactionalEventListener(phase = AFTER_COMMIT)` 또는 Outbox 패턴(260618).

### 꼬리질문(체크포인트) Q&A

**Q. 어떤 작업들을 하나의 원자적 단위로 묶어야 하나?**
→ **"함께 성공하지 않으면 거짓말이 되는 것들" — 하나의 비즈니스 사실을 구성하는 DB 변경들.**
- 판단 기준은 기술이 아니라 도메인: "이 중 하나만 반영된 상태를 사용자/정산이 용납하는가?"

**Q. 실패 시 어떤 데이터가 롤백되어야 하나?**
→ **트랜잭션 경계 안의 모든 변경 — 그래서 경계 설정이 곧 롤백 범위 설계다.**

**Q. 트랜잭션 범위가 길어지면 생기는 문제는?**
→ **커넥션 점유↑(풀 고갈), 락 유지↑(대기·데드락), DB 자원(undo) 사용↑.**

**Q. 외부 API 호출도 트랜잭션 안에 있어도 되나?**
→ **피해야 한다 — 롤백 불가 + 커넥션 점유 + 외부 장애의 DB 전파.**
- 넣는다고 정합성이 생기는 게 아니라, "DB만 롤백되고 API는 나간" 상태의 복구(재시도·보상·Outbox)를 별도로 설계해야 한다.

### 🌱 심화 키워드
- **선언적 vs 프로그래밍적 트랜잭션** — @Transactional vs TransactionTemplate, 세밀 제어가 필요한 경우
- **JDK Dynamic Proxy / CGLIB** — 프록시 생성 방식과 self-invocation 무효의 근거 (→ 260623)
- **readOnly = true** — flush 생략·최적화 힌트, 조회 전용 메서드의 기본기
- **rollbackFor / noRollbackFor** — 롤백 규칙 커스터마이징
- **@TransactionalEventListener** — 트랜잭션 단계(BEFORE/AFTER_COMMIT)에 후속 작업을 거는 표준 수단

### ❓ 더 파볼 질문
- **readOnly=true는 내부적으로 정확히 무엇을 바꾸나?**
  ↳ Hibernate의 flush mode를 MANUAL로 바꿔 자동 flush와 스냅샷 기반 dirty checking을 생략하게 한다 — 조회만 하는 트랜잭션에서 메모리·CPU를 실제로 아낀다. 추가로 JDBC 드라이버에 읽기 전용 힌트를 전달하고, 라우팅 데이터소스 구성에서는 리플리카(읽기 DB)로 분기하는 근거로도 쓰인다. "성능 최적화 + 의도 선언 + 라우팅 신호"의 3중 역할이다.
- **트랜잭션 안에서 이벤트를 발행하면 리스너는 언제 실행되나?**
  ↳ 일반 @EventListener는 발행 즉시(같은 트랜잭션 안에서 동기) 실행된다 — 리스너 실패가 본 트랜잭션을 롤백시킨다. @TransactionalEventListener는 단계 선택이 가능한데, BEFORE_COMMIT은 커밋 직전이라 실패 시 전체가 롤백되고, AFTER_COMMIT은 이미 커밋된 뒤라 실패해도 본 작업에 영향이 없다 — 대신 그 실패는 아무도 롤백해주지 않으므로 재시도/이력 설계가 따로 필요하다(→ 260623).
- **프록시 기반 AOP 대신 AspectJ 위빙을 쓰면 self-invocation 문제가 왜 사라지나?**
  ↳ 프록시 방식은 "객체 바깥에서 들어오는 호출"만 가로챌 수 있어서 this 호출이 새는 것인데, AspectJ는 컴파일/로드 시점에 대상 클래스의 바이트코드 자체에 어드바이스를 짜 넣는다. 호출 경로가 아니라 메서드 본체가 바뀌므로 내부 호출에도 적용된다 — 대신 빌드 복잡도가 올라 실무에서는 보통 "구조를 바꿔 프록시를 태우는" 쪽을 택한다.
