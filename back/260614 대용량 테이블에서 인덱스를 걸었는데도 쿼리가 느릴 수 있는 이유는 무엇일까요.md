# 260614 : 대용량 테이블에서 인덱스를 걸었는데도 쿼리가 느릴 수 있는 이유는 무엇일까요?

인덱스를 걸면 무조건 빨라진다고 생각하기 쉬운데요.
실제 면접에서는 “인덱스를 걸었습니다”에서 끝나는 게 아니라,
왜 빨라졌는지 / 왜 안 빨라질 수도 있는지까지 자주 물어봅니다.

이 질문은 단순히 인덱스 개념을 아는지보다,
쿼리 실행 계획과 데이터 특성을 같이 보고 판단할 수 있는지를 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- 조건 컬럼의 카디널리티가 낮지는 않은가?
- 인덱스를 타더라도 결국 많은 row를 읽는 구조는 아닌가?
- WHERE, ORDER BY, JOIN 조건과 인덱스 순서가 맞는가?
- 인덱스 컬럼에 함수나 형변환을 적용해서 인덱스를 못 타는 상황은 아닌가?
- 옵티마이저가 인덱스보다 Full Scan이 낫다고 판단한 이유는 무엇인가?

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- 인덱스가 있는데도 Full Scan이 발생하는 경우는 언제인가요?
- 카디널리티가 낮은 컬럼에 인덱스를 걸면 왜 효과가 작을 수 있나요?
- 단일 인덱스 여러 개와 복합 인덱스 하나는 어떻게 다르게 동작하나요?
- 실행 계획에서 어떤 지표를 보고 병목을 판단하시나요?
- 인덱스를 많이 만들면 어떤 단점이 있나요?

---

## 답변

> **한 줄 핵심**: 인덱스는 공짜 지름길이 아니라 "B-tree 탐색 + 랜덤 I/O 테이블 접근"이라는 비용을 가진 경로다 — 이 비용이 Full Scan(순차 읽기)보다 쌀 때만 빠르다.

### 1문 1답

**Q. 조건 컬럼의 카디널리티가 낮지는 않은가? (낮은 컬럼에 인덱스를 걸면 왜 효과가 작을 수 있나요?)**
→ 카디널리티가 낮은 컬럼은 값의 종류가 적어서(상태·성별처럼) 조건을 걸어도 대상 row를 거의 걸러내지 못한다. 예를 들어 `status='ACTIVE'`가 전체의 30%를 차지하면 인덱스로 걸러지는 양이 거의 없다(후보가 여전히 전체의 30%나 남는다). 게다가 인덱스로 걸린 그 많은 row를 테이블에서 하나씩 랜덤 I/O로 읽어야 하므로, 필터 효과는 작은데 접근 비용은 큰 이중 손해가 발생한다. 인덱스의 존재 이유가 "읽을 후보를 크게 줄이는 것"인데, 값 종류가 적으면 이 전제 자체가 무너진다. 그래서 옵티마이저가 아예 Full Scan을 선택하기도 한다.

**Q. 인덱스를 타더라도 결국 많은 row를 읽는 구조는 아닌가?**
→ 인덱스 경로의 비용은 대략 'B-tree 탐색 + 읽을 row 수 × (랜덤 I/O + 테이블 룩업)'으로 볼 수 있다. 반면 Full Scan은 '전체 페이지 × 순차 I/O'인데, 랜덤 I/O는 순차 I/O보다 훨씬 비싸다. 따라서 조건의 선택도가 낮아 읽을 row가 많아지면 랜덤 I/O가 누적되어 인덱스 경로가 오히려 손해가 된다. "전체의 15~25% 이상을 읽으면 Full Scan이 낫다"는 경험칙이 자주 인용되지만, 실제 임계점은 DBMS 비용 모델·스토리지(SSD/HDD)·캐시 적중률에 따라 달라 절대 기준은 아니다. 결국 인덱스를 '탔는지'가 아니라 '몇 row를 어떤 방식으로 읽는지'가 성능을 가른다.

**Q. WHERE, ORDER BY, JOIN 조건과 인덱스 순서가 맞는가?**
→ 복합 인덱스 `(a,b)`는 먼저 a로 정렬하고 그 안에서 다시 b로 정렬하는 구조다. 그래서 leftmost prefix 규칙상 선행 컬럼 a 없이 `WHERE b=?`만 주면 탐색 시작점을 잡을 수 없어 인덱스를 제대로 못 쓴다. 또 ORDER BY가 인덱스 정렬 순서와 어긋나면 정렬을 위해 filesort가 추가로 붙어 비용이 커진다. 설계 원칙은 등호(=) 조건 컬럼을 앞에, 범위(>, <, BETWEEN) 조건 컬럼을 뒤에 두는 것인데, 범위 조건 뒤에 오는 컬럼은 이미 정렬이 흐트러져 탐색에 쓰이지 못하기 때문이다. 실행 계획에서 possible_keys에는 있는데 key로 선택되지 않으면 이 순서 불일치를 의심한다.

**Q. 인덱스 컬럼에 함수나 형변환을 적용해서 인덱스를 못 타는 상황은 아닌가?**
→ B-tree 인덱스는 컬럼의 원본 값을 기준으로 정렬해 두기 때문에, 값을 가공하는 조건은 그 정렬을 활용할 수 없다. 예컨대 `DATE(created_at)=?`, 문자열↔숫자 형변환, 앞 와일드카드 `LIKE '%x'` 같은 조건은 정렬된 원본 기준의 시작점을 잡지 못한다. 이런 조건을 non-sargable이라 부르며, 인덱스를 아예 타지 못하고 `type=ALL`의 풀스캔으로 떨어진다. 실행 계획에서 key가 선택되지 않는 것으로 확인할 수 있다. 해결하려면 조건 쪽을 가공하지 말고(예: 범위 조건으로 바꾸기) 필요 시 함수 기반/생성 컬럼 인덱스를 고려한다.

**Q. 옵티마이저가 인덱스보다 Full Scan이 낫다고 판단한 이유는 무엇인가?**
→ 옵티마이저는 통계(추정 row 수)를 근거로 인덱스 경로와 Full Scan의 비용을 각각 계산해 더 싼 쪽을 고른다. 읽을 row가 많아 랜덤 I/O 비용이 순차 스캔보다 크다고 계산되면 Full Scan을 선택하는데, 이건 잘못이 아니라 "안 타는 게 이득"이라는 합리적 판단인 경우가 많다. 문제는 통계가 낡았을 때인데, 추정 row 수가 실제와 어긋나면 비용 계산 자체가 틀어져 잘못된 계획이 나온다. 그래서 `ANALYZE TABLE`로 통계를 갱신하거나, MySQL 8.0의 `EXPLAIN ANALYZE`로 예상 row와 실측 row의 괴리를 확인하는 것이 중요하다. 값 분포가 편중돼 있으면 히스토그램(8.0)으로 분포 정보를 보완하기도 한다.

**Q. 인덱스가 있는데도 Full Scan이 발생하는 경우는 언제인가요?**
→ 크게 세 갈래로 나뉜다. 첫째는 아예 인덱스를 "못 타는" 경우로, 함수·형변환·앞 와일드카드 같은 non-sargable 조건이 여기 해당한다. 둘째는 인덱스를 "안 타는 게 이득"인 경우로, 읽을 row가 많아 랜덤 I/O 비용이 순차 스캔보다 커서 옵티마이저가 스스로 Full Scan을 고르는 상황이다. 셋째는 통계가 낡아 옵티마이저가 오판하는 경우다. 앞의 둘은 정상적인 동작이고 마지막 하나만 손봐야 할 문제라는 구분이 실무에서 중요하다.

**Q. 단일 인덱스 여러 개와 복합 인덱스 하나는 어떻게 다르게 동작하나요?**
→ MySQL은 기본적으로 하나의 테이블 접근에 인덱스를 하나만 선택한다(index merge는 제한적 예외다). 그래서 단일 인덱스가 여러 개 있어도 `WHERE a AND b`에서는 보통 그중 하나만 쓰고 나머지 조건은 인덱스 레벨에서 걸러내지 못한 채 테이블에서 필터링한다. 반면 복합 인덱스 `(a,b)`는 두 조건을 인덱스 레벨에서 함께 걸러 후보를 크게 줄이므로 압도적으로 유리하다. 설계할 때는 등호 조건 컬럼을 앞, 범위 조건 컬럼을 뒤에 두는 원칙을 따른다. 즉 "여러 조건이 AND로 함께 쓰이는 쿼리"에는 단일 인덱스 여러 개보다 잘 설계된 복합 인덱스 하나가 낫다.

**Q. 실행 계획에서 어떤 지표를 보고 병목을 판단하시나요?**
→ 먼저 `type`을 보는데 ALL→index→range→ref→eq_ref 순으로 접근 방식이 개선되는 것이고, ALL이면 풀스캔이라는 뜻이다. `rows`는 스캔 추정 건수, `filtered`는 그중 조건으로 남는 비율을 보여준다. `Extra`도 핵심인데 `Using filesort`·`Using temporary`는 정렬·임시테이블이 생겼다는 경고 신호이고, `Using index`는 테이블 접근 없이 인덱스만으로 끝난 커버링 인덱스라는 좋은 신호다. 마지막으로 MySQL 8.0의 `EXPLAIN ANALYZE`는 실제 시간과 실제 row를 보여주므로, 예상치와 크게 벌어지는 지점이 곧 통계 오판이자 병목 지점이다.

**Q. 인덱스를 많이 만들면 어떤 단점이 있나요?**
→ 쓰기 작업(INSERT·UPDATE·DELETE)마다 관련된 모든 인덱스의 B-tree를 함께 갱신해야 하므로 쓰기 성능이 떨어진다. 또 인덱스는 스토리지 공간과 버퍼풀 메모리를 잠식한다. 게다가 후보 인덱스가 많아질수록 옵티마이저가 엉뚱한 인덱스를 고를 확률도 올라간다. 결국 인덱스는 읽기 성능을 쓰기 성능·공간과 맞바꾸는 것이므로, 실제 쿼리 패턴에 근거해 꼭 필요한 만큼만 최소로 유지하는 것이 원칙이다.

### 면접 답변 (구술용)

인덱스가 있어도 느린 경우는 크게 다섯 가지입니다. 첫째, 인덱스 컬럼에 함수나 형변환을 적용하면 인덱스를 아예 못 탑니다. 둘째, 카디널리티가 낮은 컬럼처럼 조건의 선택도가 낮으면 인덱스를 타도 결국 많은 row를 랜덤 I/O로 읽어야 해서 Full Scan보다 느려질 수 있고, 옵티마이저가 아예 Full Scan을 선택하기도 합니다. 셋째, 복합 인덱스의 컬럼 순서가 WHERE·ORDER BY와 안 맞으면 인덱스를 못 쓰거나 filesort가 추가됩니다. 넷째, SELECT 컬럼이 인덱스에 없으면 row마다 테이블 룩업이 발생합니다. 다섯째, 통계가 낡으면 옵티마이저가 잘못된 계획을 세웁니다. 그래서 "인덱스를 걸었다"가 아니라 실행 계획에서 실제로 몇 row를 어떤 방식으로 읽는지 확인하는 것이 중요합니다.

### 원리 이해 (왜 그런가)

**비용 모델이 모든 판단의 근거다:**

```
인덱스 경로 비용 = B-tree 탐색 + (읽을 row 수 × 랜덤 I/O + 테이블 룩업)
Full Scan 비용  = 전체 페이지 × 순차 I/O
※ 랜덤 I/O ≫ 순차 I/O  →  읽을 row가 많으면 인덱스가 오히려 손해
```

**느린 원인 5가지 정리:**

| # | 원인 | 메커니즘 (왜) | 확인 방법 |
|---|------|--------------|-----------|
| 1 | 인덱스를 못 탐 | B-tree는 **원본 값** 순서로 정렬 → `DATE(created_at)=?`, 형변환, `LIKE '%x'`는 탐색 시작점을 못 잡음 (non-sargable) | `type=ALL`, key 미선택 |
| 2 | 낮은 선택도 | `status='ACTIVE'`가 30%면 거의 안 걸러짐 + 그 row들을 랜덤 I/O로 읽음 → 이중 손해 | `rows` 큼, 옵티마이저가 Full Scan 선택 |
| 3 | 복합 인덱스 순서 불일치 | `(a,b)`는 a로 정렬 후 b로 정렬 → `WHERE b=?`만으론 시작점 없음 (**leftmost prefix**) | possible_keys에 있는데 key 미사용 |
| 4 | 커버링 실패 | SELECT 컬럼이 인덱스에 없음 → row마다 테이블 접근(랜덤 I/O) | Extra에 `Using index` 없음 |
| 5 | 통계 낡음 | 옵티마이저는 통계(추정 row 수)로 비용 계산 → 통계가 틀리면 계획도 틀림 | `EXPLAIN ANALYZE`로 예상 vs 실측 괴리, `ANALYZE TABLE`로 갱신 |

### 꼬리질문 Q&A

**Q. 인덱스가 있는데도 Full Scan이 발생하는 경우는?**
→ **못 타는 경우(함수·형변환·앞 와일드카드) + 안 타는 게 이득인 경우(읽을 row가 많아 랜덤 I/O 비용 > 순차 스캔 비용) + 통계 오판.**
- "전체의 15~25% 이상 읽으면 Full Scan이 낫다"는 경험칙이 자주 인용되지만, 실제 임계점은 DBMS 비용 모델·스토리지(SSD/HDD)·캐시 적중률에 따라 다르다(절대 기준 아님).

**Q. 카디널리티가 낮은 컬럼의 인덱스는 왜 효과가 작나?**
→ **거의 안 걸러지는데(필터 효과 ↓) 접근은 랜덤 I/O(비용 ↑)라 이중 손해.**
- 인덱스의 존재 이유 = "읽을 후보를 크게 줄이는 것". 값 종류가 적으면(성별, 상태) 이 전제가 무너진다.

**Q. 단일 인덱스 여러 개 vs 복합 인덱스 하나?**
→ **MySQL은 기본적으로 테이블 접근에 인덱스 하나만 선택(index merge는 제한적 예외) — `WHERE a AND b`는 복합 `(a,b)`가 인덱스 레벨에서 둘 다 걸러 압도적으로 유리.**
- 복합 인덱스 설계 원칙: 등호 조건 컬럼 앞, 범위 조건 컬럼 뒤 (범위 뒤 컬럼은 탐색에 못 쓰임).

**Q. 실행 계획에서 어떤 지표를 보나?**
→ **`type`(ALL→index→range→ref→eq_ref 순으로 개선), `rows`, `filtered`, `Extra`(Using filesort/temporary = 경고, Using index = 커버링).**
- MySQL 8.0 `EXPLAIN ANALYZE`는 실제 시간·실제 row를 보여줌 → 예상치와 괴리가 큰 지점 = 통계/오판 지점.

**Q. 인덱스를 많이 만들면 단점은?**
→ **쓰기마다 모든 인덱스 B-tree 갱신(쓰기 성능 저하) + 스토리지·버퍼풀 잠식 + 옵티마이저 오판 확률 증가.**
- 인덱스 = 읽기 성능을 쓰기 성능·공간과 교환하는 것. 쿼리 패턴 기반 최소 유지가 원칙.

### 🌱 심화 키워드
- **클러스터링 인덱스 / 세컨더리 인덱스** — InnoDB 세컨더리 인덱스는 PK를 품고 있어 룩업이 "인덱스→PK→테이블" 2단계가 되는 구조
- **커버링 인덱스(covering index)** — 테이블 접근 자체를 없애는 최적화, Extra의 `Using index`
- **Index Condition Pushdown(ICP)** — 조건 필터링을 스토리지 엔진 레벨로 내려 룩업 수를 줄이는 기법
- **히스토그램(MySQL 8.0)** — 카디널리티만으로 부족한 값 분포 정보를 옵티마이저에 제공
- **선택도(selectivity)** — "조건이 row를 얼마나 걸러내는가", 인덱스 효용의 척도

### ❓ 더 파볼 질문
- **B-tree(정확히는 B+tree)가 디스크 기반 인덱스의 표준이 된 이유는?**
  ↳ 디스크는 페이지(보통 16KB) 단위로 읽으므로, 노드 하나를 페이지 하나에 맞춰 자식을 수백 개씩 두면(큰 팬아웃) 수억 건도 트리 높이 3~4로 커버된다 — 조회당 디스크 I/O가 서너 번으로 고정된다는 뜻이다. 게다가 B+tree는 리프들이 연결 리스트로 이어져 있어 범위 스캔이 순차 I/O가 된다. "페이지 단위 I/O에 최적화된 구조"라는 게 핵심이다.
- **InnoDB에서 PK가 크면 왜 모든 세컨더리 인덱스가 함께 커지는가?**
  ↳ InnoDB의 세컨더리 인덱스는 리프에 row 주소가 아니라 **PK 값**을 저장하고, 조회 시 그 PK로 클러스터링 인덱스를 다시 탐색한다. 그래서 PK 크기가 모든 세컨더리 인덱스에 복제된다 — PK를 작게(가능하면 단조 증가 정수) 유지하라는 권고의 근거다.
- **`ORDER BY x LIMIT 10`이 옵티마이저의 인덱스 선택을 왜곡하는 케이스는?**
  ↳ 옵티마이저가 "x 정렬 인덱스를 따라가다 조건에 맞는 row 10개만 채우면 일찍 끝난다"고 추정해 정렬 인덱스를 고르는데, WHERE 조건에 맞는 row가 희소하면 10개를 채우기 위해 사실상 인덱스 전체를 훑게 된다 — 추정(균등 분포 가정)과 실제(편중 분포)의 괴리가 만드는 대표적 성능 함정이다.
