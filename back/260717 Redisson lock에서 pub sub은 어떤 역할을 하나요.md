# 260717 : Redisson lock에서 pub/sub은 어떤 역할을 하나요?

Redisson lock에서 pub/sub은 어떤 역할을 하나요?

이 질문은 Redisson 분산락이 단순히 Redis에 key를 저장하는 수준을 넘어, 락 대기와 해제를 어떻게 효율적으로 처리하는지 이해하고 있는지 확인하기 좋습니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- Redisson에서 어떤 스레드가 lock 획득에 실패하면 무작정 Redis에 계속 요청을 보내며 확인하지 않습니다.
- 대신 lock이 해제될 때 알림을 받을 수 있도록 Redis pub/sub 채널을 구독합니다.
- Lock을 가진 스레드가 unlock하면 Redisson은 대기 중인 클라이언트에게 pub/sub으로 알림을 보냅니다.
- 대기 중인 클라이언트는 그 알림을 받은 뒤 다시 lock 획득을 시도합니다.
- 이 방식은 spin lock처럼 계속 polling하는 방식보다 Redis 부하를 줄이고, 락 해제 시점에 더 효율적으로 재시도할 수 있게 합니다.
- 다만 pub/sub 알림 자체가 비즈니스 정합성을 보장하는 것은 아니므로, timeout, lease time, transaction, 멱등성 설계는 별도로 필요합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- Spin lock 방식과 pub/sub 방식의 차이는 무엇인가요?
- Pub/sub 알림을 놓치면 어떻게 되나요?
- Redisson lock에서 wait time은 pub/sub 대기와 어떤 관계가 있나요?
- Pub/sub 방식이 Redis 부하를 줄이는 이유는 무엇인가요?
- Redisson을 써도 lock 획득 실패나 timeout은 어떻게 처리해야 하나요?

---

## 답변

> **한 줄 핵심**: pub/sub은 락 **대기**를 "폴링" 대신 "해제 알림 구독"으로 바꾸는 장치다 — 부하는 낮추고 인계는 빠르게. 단 알림은 "풀렸다"는 신호일 뿐 획득 보장이 아니며(깨어난 대기자들은 다시 경쟁), 알림 유실에 대비한 타임아웃 재시도가 정확성을 받친다.

### 1문 1답

**Q. Redisson lock에서 pub/sub은 어떤 역할을 하나요?**

**A.** pub/sub은 락 대기를 "폴링" 대신 "해제 알림 구독"으로 바꾸는 장치다. 락 획득에 실패한 클라이언트는 무작정 재시도하는 대신 그 락 전용 채널을 구독하고 대기하며, 홀더가 unlock하면 Redisson의 해제 Lua 스크립트가 락 삭제와 함께 채널에 해제 메시지를 publish해서 — 해제와 알림이 한 스크립트라 "해제됐는데 알림이 안 나가는" 틈이 없다 — 대기자들이 깨어나 재획득을 시도하고, 성공하거나 남은 wait time 동안 반복한다. 효과는 두 가지다 — 스핀 방식은 대기자 N명이 각자 주기적으로 재시도해 Redis를 계속 두드리지만 pub/sub은 유휴 대기 중 요청이 없어 부하가 낮고, 해제 즉시 알림이 가 인계가 빠르다. 다만 알림은 락이 풀렸다는 신호일 뿐 획득 보장이 아니라 깨어난 대기자들은 다시 경쟁하고, 공정 락이 아니면 순서도 보장되지 않는다. 그리고 pub/sub은 대기 효율 장치일 뿐 락의 정합성 한계(TTL 경계, 트랜잭션과의 관계, 멱등성 필요)를 바꾸지는 않는다.

**Q. Redisson은 lock 획득에 실패한 클라이언트를 어떻게 대기시키나요?**

**A.** 락 획득에 실패한 클라이언트는 무작정 재시도 요청을 보내는 대신, 그 락 전용 pub/sub 채널을 구독하고 대기 상태(내부적으로 세마포어 대기, 시간 상한 있음)에 들어갑니다. 락 홀더가 unlock하면 Redisson의 해제 Lua 스크립트가 락 삭제와 함께 채널에 해제 메시지를 publish합니다. 해제와 알림이 한 스크립트라서 "해제됐는데 알림이 안 나가는" 틈이 없습니다. 알림을 받은 대기 클라이언트들은 깨어나 재획득을 시도하고, 승자 한 명을 제외한 나머지는 다시 대기합니다. 이 과정을 성공하거나 남은 wait time이 소진될 때까지 반복하며, 예산이 다하면 tryLock이 false를 반환하고 구독을 해제합니다.

**Q. Spin lock 방식과 pub/sub 방식의 차이는 무엇인가요?**

**A.** 가장 큰 차이는 주도권입니다 — spin은 대기자가 능동적으로 주기적으로 직접 확인(폴링)하고, pub/sub은 수동적으로 해제 알림을 기다리는 이벤트 기반입니다. 그래서 spin은 부하와 인계 지연이 "폴링 간격" 하나에 묶이는데, 간격을 줄이면 부하가 늘고 늘리면 지연이 커지는 트레이드오프가 있습니다. 반면 pub/sub은 유휴 대기 중 부하가 없고 해제 즉시 반응하지만, 구독 채널·커넥션 같은 관리 인프라가 필요합니다. 이는 OS 동시성의 spin lock vs 조건변수(wait/notify) 구도가 분산 환경에 재현된 것입니다. 경합이 심할수록 spin은 최악이 되고 pub/sub은 완만해서, "부하가 몰리는 순간에 더 안정적"이라는 것이 실질 가치입니다.

**Q. pub/sub만으로 보장되지 않아 별도로 설계해야 하는 것은 무엇인가요?**

**A.** 알림은 락이 "풀렸다"는 신호일 뿐 획득을 보장하지 않으므로, 깨어난 대기자들은 다시 경쟁하고 공정 락이 아니면 순서도 보장되지 않습니다. 또한 pub/sub은 어디까지나 대기 효율을 높이는 성능 최적화 계층일 뿐입니다. 락의 정합성 한계 — TTL 만료 경계, 트랜잭션과의 관계, 멱등성 필요 — 는 pub/sub이 바꾸지 못합니다. 그래서 timeout, lease time, transaction, 멱등성은 별도로 설계해야 합니다. 즉 pub/sub은 "빠르게 대기하고 재시도하는 법"을 최적화할 뿐, 정확성을 보장하는 장치가 아닙니다.

**Q. Pub/sub 알림을 놓치면 어떻게 되나요?**

**A.** 알림을 놓쳐도 교착되지는 않습니다. Redis pub/sub은 fire-and-forget이라 전달 보장이 없는데(260626), Redisson은 이 점을 전제로 대기에 시간 상한(락 TTL·남은 wait 기반)을 걸어 둡니다. 그래서 알림을 못 받더라도 타임아웃 후 깨어나 재시도하게 됩니다. 구조로 보면 빠른 길은 알림이고 안전망은 타임아웃이며, pub/sub은 성능 최적화 계층이고 정확성(교착 없음)은 시간 상한 재시도가 받칩니다. 결국 알림을 놓쳤을 때의 비용은 실패가 아니라 "타임아웃만큼의 지연"일 뿐입니다.

**Q. Redisson lock에서 wait time은 pub/sub 대기와 어떤 관계가 있나요?**

**A.** wait time은 전체 대기 예산이고, 구독·알림 대기·재시도가 전부 그 예산 안에서 이루어집니다. 흐름으로 보면 첫 시도 실패 → 채널 구독 → 남은 예산 한도로 대기 → 알림 또는 타임아웃에 깨어나 재시도 → … → 예산 소진 시 false 반환과 구독 해제 순입니다. 즉 pub/sub 대기는 wait time과 별개로 흐르는 시간이 아니라, 주어진 wait time을 효율적으로 소비하는 방법입니다. spin이 같은 예산을 폴링으로 소비한다면, pub/sub은 같은 예산을 알림 기반으로 소비하는 차이입니다. 그래서 wait time을 넘겨 대기하는 일은 없습니다.

**Q. Pub/sub 방식이 Redis 부하를 줄이는 이유는 무엇인가요?**

**A.** 요청이 발생하는 구조 자체가 다르기 때문입니다. spin은 부하가 "대기자 수 × 폴링 빈도"에 비례해 상시 발생하지만, pub/sub은 "해제 이벤트 횟수"에만 비례합니다. 예를 들어 대기자 100명이 100ms 간격으로 폴링하면 락이 풀릴 때까지 초당 1,000회의 요청이 계속됩니다. 반면 pub/sub이면 유휴 중에는 요청이 0이고, 해제 순간에만 publish 1회와 재시도 100회가 발생합니다. 그래서 경합이 심할수록(대기자가 많을수록) 두 방식의 격차가 크게 벌어집니다.

**Q. Redisson을 써도 lock 획득 실패나 timeout은 어떻게 처리해야 하나요?**

**A.** 이것은 라이브러리가 대신 정해 줄 수 없는 비즈니스 결정이므로, tryLock의 false 분기를 "정상 흐름"으로 설계해야 합니다. 선택지는 즉시 실패 응답("이미 처리 중입니다"), 백오프 재시도, 큐잉, 폴백이고, 기준은 260713에서 정리한 락의 목적입니다 — 중복 억제면 즉시 실패, 순차화면 재시도입니다. 코드 수준에서는 성공 경로의 finally에서 unlock을 보장하되, isHeldByCurrentThread로 자기 락일 때만 풀어야 합니다. 또 InterruptedException을 처리하고, 실패율 메트릭을 남겨 경합 추이를 관측하는 것까지가 완성입니다. 즉 실패를 예외가 아닌 설계된 경로로 다루는 것이 핵심입니다.

### 면접 답변 (구술용)

pub/sub은 Redisson에서 락 대기를 폴링 없이 처리하는 알림 채널입니다. 흐름으로 설명하면 — 락 획득에 실패한 클라이언트는 무작정 재시도 요청을 보내는 대신 그 락 전용 채널을 구독하고 대기 상태에 들어갑니다. 락 홀더가 unlock하면 Redisson의 해제 Lua 스크립트가 락 삭제와 함께 채널에 해제 메시지를 publish하고 — 해제와 알림이 한 스크립트라 "해제됐는데 알림이 안 나가는" 틈이 없습니다 — 대기 중이던 클라이언트들이 깨어나 재획득을 시도합니다. 성공하거나, 남은 wait time 동안 이 과정을 반복합니다. 효과는 두 가지입니다. 첫째 부하 절감 — 스핀 방식은 대기자 N명이 각자 주기적으로 재시도해서 락이 풀릴 때까지 N 곱하기 폴링 빈도만큼 Redis를 계속 두드리지만, pub/sub은 유휴 대기 중 요청이 없고 해제 순간의 publish 한 번과 재시도 N번으로 끝납니다. 둘째 빠른 인계 — 스핀은 해제 시점부터 다음 폴링까지의 공백만큼 락이 놀지만, pub/sub은 해제 즉시 알림이 갑니다. 다만 선을 그어야 할 게 있습니다 — 알림은 락이 풀렸다는 신호일 뿐 획득 보장이 아니라서 깨어난 대기자들은 다시 경쟁하고, 공정 락이 아니면 순서도 보장되지 않습니다. 그리고 pub/sub은 대기 효율 장치일 뿐, 락의 정합성 한계 — TTL 경계, 트랜잭션과의 관계, 멱등성 필요 — 를 바꾸지는 않습니다.

### 원리 이해 (왜 그런가)

**대기 흐름 전체:**

```
tryLock 실패
  → 락 전용 채널 구독 + 대기 (내부적으로 세마포어 대기, 시간 상한 있음)
  → [홀더] unlock → 해제 Lua: 락 삭제 + 해제 메시지 publish (한 스크립트 = 원자적)
  → [대기자들] 깨어남 → 재획득 경쟁 → 승자 1명, 나머지는 다시 대기
  → 남은 wait time 소진 시 tryLock = false 반환 + 구독 해제
```

**spin vs pub/sub — OS 동시성과의 대응:**

| | spin (폴링) | pub/sub (알림) |
|---|---|---|
| 대기 방식 | 능동 — 주기적으로 직접 확인 | 수동 — 해제 알림을 기다림 |
| Redis 부하 | 대기자 수 × 폴링 빈도 (경합 심할수록 악화) | 유휴 중 0, 해제 시 publish 1회 |
| 인계 지연 | 해제 ~ 다음 폴링 사이 공백 | 해제 즉시 |
| 트레이드오프 | 간격 줄이면 부하↑, 늘리면 지연↑ | 구독 관리 인프라 필요 |
| OS 유비 | spin lock | 조건변수 (wait/notify) |

경합이 심해질수록 spin은 최악이 되고 pub/sub은 완만하다는 점 — "부하가 몰리는 순간에 더 안정적"이라는 것이 운영 관점의 실질 가치입니다.

**알림 유실 대비 — 이중 구조**: Redis pub/sub은 fire-and-forget이라 전달 보장이 없습니다(260626). Redisson은 이를 전제로 대기에 시간 상한(락 TTL·남은 wait 기반)을 걸어 두므로, 알림을 못 받아도 타임아웃 후 깨어나 재시도합니다(거의 확실 — 타임아웃 폴백 구조). 즉 **빠른 길은 알림, 안전망은 타임아웃** — pub/sub은 성능 최적화 계층이고 정확성(교착 없음)은 시간 상한 재시도가 받칩니다.

### 꼬리질문 Q&A

**Q. Spin lock 방식과 pub/sub 방식의 차이는 무엇인가요?**

**A.** **주도권이 다르다 — spin은 대기자가 능동적으로 주기 확인(폴링), pub/sub은 수동적으로 알림 대기(이벤트 기반).**
spin은 구현이 단순하지만 부하와 지연이 "폴링 간격" 하나에 묶인 트레이드오프이고(줄이면 부하, 늘리면 지연), pub/sub은 부하가 낮고 반응이 빠른 대신 구독 관리가 필요합니다. OS 동시성의 spin lock vs 조건변수(wait/notify) 구도가 분산 환경에 재현된 것이라고 연결하면 CS 이해도가 잘 드러납니다.

**Q. Pub/sub 알림을 놓치면 어떻게 되나요?**

**A.** **교착되지 않는다 — Redisson은 대기에 시간 상한을 걸어 두므로 알림이 안 와도 타임아웃 후 깨어나 재시도한다.**
Redis pub/sub이 fire-and-forget(전달 보장 없음)이라는 사실을 전제로 설계된 이중 구조입니다. 알림은 "빨리 깨어나는 최적화"이고, 놓쳤을 때의 비용은 실패가 아니라 "타임아웃만큼의 지연"입니다(거의 확실 — 타임아웃 폴백 동작).

**Q. Redisson lock에서 wait time은 pub/sub 대기와 어떤 관계가 있나요?**

**A.** **wait time이 전체 대기 예산이고, 구독·알림 대기·재시도가 전부 그 예산 안에서 이루어진다.**
첫 시도 실패 → 구독 → (남은 예산 한도로) 대기 → 알림 또는 타임아웃에 깨어나 재시도 → … → 예산 소진 시 false 반환과 구독 해제. 즉 pub/sub 대기는 wait time과 별개의 시간이 아니라, wait time을 효율적으로 소비하는 방법입니다.

**Q. Pub/sub 방식이 Redis 부하를 줄이는 이유는 무엇인가요?**

**A.** **요청 발생 구조가 다르기 때문 — spin은 부하가 "대기자 수 × 폴링 빈도"에 비례해 상시 발생하고, pub/sub은 "해제 이벤트 횟수"에만 비례한다.**
대기자 100명이 100ms 간격으로 폴링하면 초당 1,000회의 요청이 락이 풀릴 때까지 계속되지만, pub/sub이면 유휴 중 요청이 0이고 해제 순간에만 publish 1회 + 재시도 100회가 발생합니다. 경합이 심할수록(대기자가 많을수록) 격차가 벌어집니다.

**Q. Redisson을 써도 lock 획득 실패나 timeout은 어떻게 처리해야 하나요?**

**A.** **라이브러리가 대신 정해줄 수 없는 비즈니스 결정이므로, tryLock의 false 분기를 "정상 흐름"으로 설계해야 한다.**
선택지는 즉시 실패 응답("이미 처리 중입니다"), 백오프 재시도, 큐잉, 폴백이고 — 기준은 260713에서 정리한 락의 목적입니다(중복 억제면 즉시 실패, 순차화면 재시도). 코드 수준에서는 성공 경로의 finally에서 unlock 보장(단 isHeldByCurrentThread로 자기 락일 때만), InterruptedException 처리, 그리고 실패율 메트릭을 남겨 경합 추이를 관측하는 것까지가 완성입니다.

### 🌱 심화 키워드
- **조건변수(wait/notify)** — pub/sub 대기의 OS 동시성 원형
- **thundering herd** — 해제 알림에 대기자 전원이 동시에 깨어나 경쟁하는 현상
- **fair lock** — 인계 순서를 보장하는 변형과 그 비용 (→ 260715)
- **fire-and-forget** — Redis pub/sub의 전달 보장 수준 (→ 260626)
- **isHeldByCurrentThread** — finally unlock에서 자기 락 여부를 확인하는 API

### ❓ 더 파볼 질문
- **해제 알림에 대기자 100명이 동시에 깨어나면(thundering herd) 무슨 일이 생기고 어떻게 완화하나?**
  ↳ 100명이 동시에 재획득을 시도해 순간 경합 스파이크가 생기고 99명은 다시 실패한다. 완화는 fair lock으로 한 명씩 순차 인계하거나(비용 있음), 재시도에 백오프+지터를 섞어 깨어나는 시점을 분산하는 것 — 260713의 지터 원리가 그대로 적용된다.
- **Redis Cluster 환경에서 pub/sub은 어떤 비용이 있나?**
  ↳ 클러스터의 일반 pub/sub은 메시지를 전 노드에 브로드캐스트해서 노드가 늘수록 비용이 커진다. Redis 7의 sharded pub/sub(SSUBSCRIBE)은 채널을 특정 샤드에 귀속시켜 이 문제를 개선했다 — 대량 락 채널을 클러스터에서 쓸 때 고려 지점이다.
- **락 대기 알림을 pub/sub이 아니라 blocking 명령(BLPOP류)으로 구현하면 어떻게 다른가?**
  ↳ 대기자마다 리스트에 blocking pop을 걸어두고 해제 시 토큰을 push하는 방식도 가능하다 — 이러면 "한 명만 깨우기"(thundering herd 회피)가 자연스럽다. 대신 대기자 수만큼 blocking 커넥션이 잡히는 비용이 생긴다. "알림 브로드캐스트 vs 토큰 전달"의 구조 차이를 비교해보면 대기 설계의 감이 잡힌다.
