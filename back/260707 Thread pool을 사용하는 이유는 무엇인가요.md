# 260707 Thread pool을 사용하는 이유는 무엇인가요?

**오늘의 백엔드 질문**

Thread pool을 사용하는 이유는 무엇인가요?

이 질문은 단순히 “스레드를 재사용한다”는 답변에서 끝나면 조금 아쉽습니다. 면접에서는 보통 서버 자원 관리, 요청 처리량, 응답 지연, 장애 전파까지 함께 보고 싶어 합니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- 요청마다 스레드를 새로 만들면 생성/소멸 비용이 계속 발생합니다.
- 스레드가 무제한으로 늘어나면 CPU context switching, 메모리 사용량 증가, OOM 같은 문제가 생길 수 있습니다.
- Thread pool은 미리 정해진 수의 스레드를 재사용해서 작업 처리 비용을 줄이고, 동시에 실행되는 작업 수를 제한합니다.
- 작업량이 많아질 경우 queue에 대기시키거나, rejection policy를 통해 더 이상 받을 수 없는 작업을 제어할 수 있습니다.
- 결국 thread pool은 성능 최적화뿐 아니라 서버가 감당 가능한 범위 안에서 안정적으로 동작하게 만드는 장치입니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- Thread pool size는 어떤 기준으로 정하나요?
- CPU-bound 작업과 I/O-bound 작업의 pool size는 어떻게 달라질까요?
- Queue가 가득 차면 어떤 일이 발생하나요?
- Core pool size와 max pool size의 차이는 무엇인가요?
- 공용 thread pool을 여러 기능에서 같이 쓰면 어떤 문제가 생길 수 있나요?

---

## 답변

> **한 줄 핵심**: thread pool은 ① 스레드 생성/소멸 비용 제거(재사용)이자, 더 중요하게는 ② 동시 실행량에 상한을 둬 서버를 과부하에서 지키는 admission control 장치다 — 초과분은 큐와 거부 정책으로 명시적으로 제어한다.

### 면접 답변 (구술용)

두 축으로 답할 수 있는데, 비용 절감과 자원 보호이고 실무에서 더 중요한 건 후자입니다. 먼저 비용 — 스레드 생성은 OS 커널 자원 할당과 스택 메모리 확보가 필요한 비싼 작업이라, 요청마다 만들고 버리면 그 고정비를 계속 지불합니다. 풀은 만들어 둔 스레드를 재사용해 이 비용을 초기 1회로 끝냅니다. 다음으로 자원 보호 — 스레드가 요청량에 비례해 무한히 늘어나면 두 가지로 죽습니다. 실행 가능한 스레드가 코어 수를 크게 넘으면 CPU가 실제 일 대신 컨텍스트 스위칭에 시간을 쓰게 되고, 스레드당 스택 메모리가 쌓이다 결국 "unable to create new native thread" OOM으로 프로세스가 죽습니다. 풀은 동시 실행 스레드 수에 상한을 둬서 트래픽이 아무리 몰려도 서버가 감당 가능한 만큼만 동시에 실행하게 강제합니다. 그리고 상한을 넘는 작업은 큐에 대기시키고, 큐도 차면 거부 정책으로 어떻게 거절할지 선택합니다 — 제한이 없으면 과부하 때 전체가 함께 죽지만, 제한이 있으면 일부만 거절하고 나머지는 정상 처리하는 우아한 성능 저하가 가능해집니다. 그래서 thread pool은 성능 최적화 도구이면서 동시에 수용량 관리 장치입니다.

### 원리 이해 (왜 그런가)

**ThreadPoolExecutor의 동작 순서 — 직관과 다른 게 함정:**

```
작업 제출 → ① 스레드 수 < core          → 새 스레드 생성해 실행
          → ② core 다 참               → 큐에 넣음 (max로 안 늘어남!)
          → ③ 큐가 가득 참              → max까지 스레드 추가 생성
          → ④ max 도달 + 큐 가득        → RejectedExecutionHandler 호출
```

흔한 오해가 "바쁘면 바로 max까지 늘겠지"인데, 실제로는 **큐가 가득 차야** max로 늘기 시작합니다. 그래서 무한 큐(기본 생성자의 LinkedBlockingQueue) + max 설정 조합은 큐가 절대 차지 않으므로 max가 영원히 무의미합니다. (참고: Tomcat의 executor는 자체 TaskQueue로 이 순서를 뒤집어 큐잉 전에 max까지 먼저 늘립니다 — 웹 요청은 대기보다 즉시 처리가 우선이라는 설계 판단입니다.)

**거부 정책(RejectedExecutionHandler) 비교:**

| 정책 | 동작 | 언제 |
|------|------|------|
| AbortPolicy (기본) | RejectedExecutionException 던짐 | 호출자가 실패를 인지하고 처리해야 할 때 |
| CallerRunsPolicy | 제출한 스레드가 직접 실행 | 제출자가 느려지며 유입이 저절로 줄어드는 backpressure가 필요할 때 |
| Discard / DiscardOldest | 조용히 버림 | 유실 허용 작업만 (로그 샘플링 등) |

**풀 크기 산정 — 이론과 현실:**
- 출발점은 Java Concurrency in Practice의 공식: CPU-bound ≈ 코어 수(+1), I/O-bound ≈ 코어 수 × (1 + 대기시간/연산시간). 근거는 "스레드가 대기하는 비율만큼 여분이 있어야 CPU가 놀지 않는다"입니다.
- 하지만 실제 상한은 하위 자원이 결정합니다 — DB 커넥션이 10개면 DB 작업 스레드 100개 중 90개는 커넥션 대기만 합니다. 부하 테스트로 병목 지점에 맞춰 조정하는 것이 마무리입니다.

### 꼬리질문 Q&A

**Q. Thread pool size는 어떤 기준으로 정하나요?**
→ **작업 성격(CPU/I/O 비율)으로 이론값을 잡고, 하위 자원 한계와 부하 테스트로 보정한다.**
JCiP 공식이 출발점이지만, 스레드를 아무리 늘려도 유효 동시성은 가장 좁은 자원(DB 풀, 외부 API 쿼터)에 묶인다는 점이 실무의 핵심입니다.

**Q. CPU-bound와 I/O-bound 작업의 pool size는 어떻게 달라지나요?**
→ **CPU-bound는 코어 수 안팎, I/O-bound는 대기 비율만큼 곱해서 크게.**
CPU 작업은 스레드가 늘 CPU를 쓰므로 코어보다 많으면 스위칭 낭비만 늘어납니다. I/O 작업은 스레드 시간 대부분이 대기라 CPU가 비어 있으므로, 예를 들어 대기 90ms/연산 10ms면 코어당 약 10개가 있어야 코어가 활용됩니다.

**Q. Queue가 가득 차면 어떤 일이 발생하나요?**
→ **max까지 스레드를 추가로 만들고, 그것도 차면 거부 정책이 실행된다.**
기본 AbortPolicy는 예외를 던져 호출자에게 실패를 알리고, CallerRunsPolicy는 제출 스레드가 직접 실행해 자연스러운 backpressure를 만듭니다. 주의할 것은 무한 큐입니다 — 거부가 영원히 안 일어나는 대신 대기 시간과 메모리가 무한히 자라므로, "거부가 없다"는 게 장점이 아니라 장애를 뒤로 미루는 것일 수 있습니다. 큐 용량 제한이 중요한 이유입니다.

**Q. Core pool size와 max pool size의 차이는?**
→ **core는 평상시 유지 스레드 수, max는 한계치 — 단 max는 "큐가 가득 찼을 때만" 동작한다.**
위 동작 순서(core → 큐 → 큐 가득 시 max → 거부)가 이 질문의 진짜 답입니다. 무한 큐면 max가 무의미해진다는 함정, 그리고 Tomcat은 순서를 바꿨다는 예외까지 말하면 이해 깊이가 드러납니다.

**Q. 공용 thread pool을 여러 기능에서 같이 쓰면 어떤 문제가 생기나요?**
→ **한 기능의 느린 작업이 풀을 독점하면 무관한 기능까지 함께 멈춘다 — 장애 전파.**
예를 들어 외부 API가 무응답이 되면 그 호출 작업들이 풀 스레드를 전부 점유하고, 같은 풀을 쓰던 내부 알림 발송까지 마비됩니다. 대응이 bulkhead 패턴 — 기능/의존성별로 풀을 분리해 한 칸이 침수돼도 배 전체가 가라앉지 않게 격리하는 것입니다. 실무 함정으로 CompletableFuture 기본 실행기(ForkJoinPool.commonPool)를 여러 기능이 무심코 공유하다 병목이 되는 사례가 흔합니다 — 용도별 전용 Executor 지정이 안전합니다.

### 🌱 심화 키워드
- **RejectedExecutionHandler** — 과부하 시의 정책을 코드로 표현하는 지점
- **bulkhead** — 선박 격벽에서 온 자원 격리 패턴 (Resilience4j 등에 구현체)
- **ForkJoinPool / work-stealing** — 분할정복·비동기 조합용의 또 다른 풀 모델
- **JCiP 스레드 수 공식** — N_threads = N_cpu × U_cpu × (1 + W/C)
- **admission control** — "받을 수 있는 만큼만 받는다"는 시스템 설계 일반 원칙

### ❓ 더 파볼 질문
- **Tomcat은 어떻게 "큐잉 전에 max까지 증설"로 순서를 바꿨나?**
  ↳ 자체 TaskQueue가 offer()에서 "현재 스레드 수 < max면 false를 반환"해 executor가 큐 삽입에 실패한 것으로 착각하고 스레드를 먼저 늘리게 만든다. 표준 컴포넌트의 동작을 큐 구현으로 우회한 재미있는 사례다.
- **keepAliveTime과 allowCoreThreadTimeOut은 어떤 역할인가?**
  ↳ core를 초과해 만들어진 스레드는 keepAliveTime 동안 유휴하면 회수된다. allowCoreThreadTimeOut을 켜면 core 스레드까지 회수 대상이 된다 — 트래픽 파도가 큰 서비스에서 유휴 자원을 줄이는 손잡이다.
- **Virtual Thread 시대에도 풀링이 필요한 자원은 뭔가?**
  ↳ 스레드는 싸져서 풀링할 이유가 사라지지만(VT는 풀링하지 말라는 게 공식 가이드), DB 커넥션·파일 핸들·외부 API 쿼터는 여전히 유한하다. "풀링의 대상이 스레드에서 진짜 희소 자원으로 이동한다"가 핵심 통찰이다.
