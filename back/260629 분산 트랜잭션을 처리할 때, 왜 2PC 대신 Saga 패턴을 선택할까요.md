# 260629 분산 트랜잭션을 처리할 때, 왜 2PC 대신 Saga 패턴을 선택할까요?

분산 트랜잭션을 처리할 때, 왜 2PC 대신 Saga 패턴을 선택할까요?

요즘 이력서에 MSA, Kafka, Outbox, Saga, 이벤트 기반 구조를 적는 분들이 많아졌는데요.
면접에서는 “써봤다”보다 “왜 그 방식을 선택했는지”를 더 깊게 물어보는 경우가 많습니다.

이 질문은 단순히 2PC와 Saga의 정의를 외웠는지보다,
분산 환경에서 정합성, 가용성, 장애 복구, 서비스 간 결합도를 어떻게 바라보는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- 2PC는 여러 시스템을 하나의 트랜잭션처럼 묶어 강한 정합성을 보장하려는 방식
- 2PC는 Coordinator 장애, 참여자 락 점유, 응답 지연 시 전체 트랜잭션 대기 문제가 생길 수 있음
- Saga는 각 서비스의 로컬 트랜잭션을 순차적으로 실행하고 실패 시 보상 트랜잭션으로 복구하는 방식
- Saga는 강한 즉시 정합성보다 최종 정합성을 선택하는 구조
- 결제, 주문, 재고처럼 실패 복구 흐름이 중요한 도메인에서는 보상 로직 설계가 핵심

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- 2PC가 가용성 측면에서 불리한 이유는 무엇인가요?
- Saga 패턴에서 보상 트랜잭션은 어떻게 설계하나요?
- Choreography Saga와 Orchestration Saga의 차이는 무엇인가요?
- Saga 중간 단계에서 메시지 발행이 실패하면 어떻게 처리하나요?
- Saga를 쓰면 데이터 정합성이 깨지는 순간이 생기는데, 사용자는 그 상태를 어떻게 보게 되나요?

## 답변

> **한 줄 핵심**: 2PC는 강한 정합성의 대가로 가용성·확장성을 내놓는데(블로킹 프로토콜 + 전원 생존 전제 + 지원 시스템 제한), MSA에서는 그 대가가 치명적이라 — 잠깐의 중간 상태를 허용하는 대신 각 서비스의 자율성을 지키는 Saga(로컬 트랜잭션 + 보상)를 선택한다.

### 1문 1답

**Q. 분산 트랜잭션을 처리할 때, 왜 2PC 대신 Saga 패턴을 선택할까요?**

**A.** 2PC는 강한 정합성을 주는 대신 가용성·확장성을 내주는데, MSA에서는 그 대가가 치명적이라 Saga를 택한다. 2PC가 어려운 이유는 세 가지다 — 블로킹 프로토콜이라 prepare 단계에서 참여자들이 락을 쥔 채 coordinator를 기다리다 coordinator가 죽으면 스스로 결정 못 하고 락을 쥔 채 대기해 그 데이터를 쓰는 다른 트랜잭션까지 막히고, 전체가 가장 느린 참여자에 종속돼 서비스가 늘수록 전원 생존 확률이 곱셈으로 떨어지며, 현실적으로 Kafka·많은 NoSQL·외부 API는 XA에 참여하지 않아 선택지가 안 된다. Saga는 전역 트랜잭션을 포기하고 각 서비스의 로컬 트랜잭션을 순차 실행하되 중간 실패 시 이미 커밋된 단계를 보상 트랜잭션으로 되돌린다. 락을 서비스 경계 너머로 잡지 않아 각 서비스가 자율적으로 동작하고, 잠깐의 불일치를 허용하는 대신 최종 정합성을 보장한다. 본질은 "즉시 정합성 vs 가용성·자율성"의 트레이드오프이고, 대부분의 도메인은 '결제 확인 중' 같은 짧은 중간 상태를 허용할 수 있어 Saga를 택하되 복잡성이 보상 로직 설계로 이동한다는 점까지 말해야 완성된 답이다.

**Q. 2PC는 어떻게 여러 시스템을 하나의 트랜잭션처럼 묶어 강한 정합성을 보장하나요?**

**A.** 2PC는 이름 그대로 두 단계로 진행된다. 먼저 prepare 단계에서 coordinator가 모든 참여자에게 "커밋할 준비가 됐는가"를 묻고, 각 참여자는 필요한 락을 쥔 채 준비 완료를 응답한다. 모든 참여자가 준비됐다고 답하면 commit 단계에서 일괄 커밋하고, 하나라도 준비되지 못하면 전원 롤백한다. 이렇게 전원의 합의 위에서만 커밋하므로 여러 시스템이 마치 하나의 트랜잭션인 것처럼 강한 즉시 정합성을 보장한다. 다만 이 강한 정합성은 참여자 전원이 살아있어야 하고 결정을 기다리는 동안 락을 쥐고 있어야 한다는 대가 위에 성립한다.

**Q. 2PC는 왜 Coordinator 장애·참여자 락 점유·응답 지연 시 문제가 생기나요?**

**A.** 2PC가 블로킹 프로토콜이기 때문이다. prepare 단계에서 참여자들은 락을 쥔 채 coordinator의 최종 결정을 기다리는데, 이때 coordinator가 죽으면 참여자들은 커밋도 롤백도 스스로 결정하지 못한다 — 혼자 판단했다가 다른 참여자와 어긋날 수 있어 기다릴 수밖에 없기 때문이다. 그동안 그 데이터를 쓰려는 다른 트랜잭션까지 전부 막혀, 장애가 "한 컴포넌트 문제"에서 "락 병목을 통한 전체 정지"로 증폭되고, coordinator는 그 자체로 단일 장애점이 된다. 게다가 트랜잭션 성립에 참여자 전원의 생존이 필요하므로, 참여 서비스가 늘수록 전원이 살아있고 빠를 확률이 곱셈으로 떨어진다(99.9%×5개≈99.5%). 이는 서비스를 잘게 쪼개 늘려 가는 MSA 방향과 정면으로 충돌한다.

**Q. Saga는 각 서비스의 로컬 트랜잭션을 어떻게 실행하고 실패 시 어떻게 복구하나요?**

**A.** Saga는 전역 트랜잭션을 아예 포기하고, 각 서비스가 자기 로컬 트랜잭션만 순차적으로 실행하는 방식이다. 예를 들어 주문 생성 → 결제 승인 → 재고 차감을 각각 개별 커밋으로 진행한다. 그러다 중간 단계(가령 재고 차감)가 실패하면, 이미 커밋된 앞 단계들을 보상 트랜잭션으로 역순으로 되돌린다 — 결제 취소, 주문 실패 처리 순으로 말이다. 여기서 보상은 DB 롤백이 아니라 "의미적 역연산"으로, 승인을 취소로, 차감을 복원으로 되돌리는 별도의 비즈니스 로직이다. 락을 서비스 경계 너머로 잡지 않으니 각 서비스가 자율적으로 동작하고, 그 대가로 잠깐의 불일치를 허용하는 최종 정합성을 갖는다.

**Q. Saga가 강한 즉시 정합성 대신 최종 정합성을 선택한다는 것은 무슨 의미인가요?**

**A.** 트랜잭션이 진행되는 동안 '결제 확인 중' 같은 중간 상태가 잠깐 외부에 노출될 수 있다는 뜻이다. 2PC가 락으로 그 중간 상태를 아예 감추는 대신 가용성을 내주는 반면, Saga는 그 중간 상태를 허용하는 대신 서비스의 가용성과 자율성을 지킨다. 본질은 "즉시 정합성 vs 가용성·자율성"의 트레이드오프다. 그런데 대부분의 도메인은 짧은 중간 상태를 감내할 수 있기 때문에 Saga를 택하는 것이 합리적이다. 대신 그 대가로 복잡성이 인프라(프로토콜)에서 애플리케이션의 보상 로직 설계로 옮겨 간다는 점을 함께 인식해야 한다.

**Q. 결제·주문·재고처럼 실패 복구가 중요한 도메인에서 보상 트랜잭션은 어떻게 설계하나요?**

**A.** 각 단계마다 비즈니스 수준의 "의미적 역연산"을 정의하는 것이 핵심이다. 재고 차감의 보상은 재고 복원, 결제 승인의 보상은 환불·취소이며, 이는 기술적 DB 롤백이 아니라 이미 커밋된 결과를 비즈니스적으로 되돌리는 별도 로직이다. 이미 커밋된 것을 다루므로 보상 자체도 하나의 정식 비즈니스 로직으로 취급해, 멱등하게 만들고 재시도 큐에 태우며 실패 시 알람이 오도록 설계해야 한다. 또 되돌릴 수 없는 작업(외부 통보, 이메일 발송)은 가능한 한 뒤쪽, 즉 pivot 이후에 배치해 애초에 보상할 일을 줄인다. 보상 진행 중에도 조회가 들어오므로 PENDING·COMPENSATING·FAILED 같은 상태를 명시적으로 기록해 두어야 한다.

**Q. Choreography Saga와 Orchestration Saga의 차이는 무엇인가요?**

**A.** Choreography는 중앙 조정자 없이 각 서비스가 서로의 이벤트에 반응해 다음 동작을 이어 가는 방식이다. 결합도가 낮고 단순한 흐름에 좋지만, 단계가 많아지면 전체 흐름이 어느 한곳에 적혀 있지 않아 추적과 수정이 어렵고 순환 구독 같은 함정이 생긴다. 반대로 Orchestration은 중앙 orchestrator가 각 단계에 명령을 내리고 상태를 관리하는 방식이라, 전체 흐름이 한곳에 명시되어 추적과 보상 관리가 쉽지만 로직이 orchestrator에 집중된다. 실무 감각으로는 단계가 3개 이상이거나 보상이 복잡해지면 orchestration이 유리하다는 것이 일반적 기준이다.

**Q. Saga 중간 단계에서 메시지 발행이 실패하면 어떻게 처리하나요?**

**A.** 이는 "로컬 트랜잭션은 커밋됐는데 다음 단계로 넘길 이벤트가 발행되지 못하는" 상황으로, 정확히 Outbox 패턴이 푸는 문제다. 각 단계에서 상태 변경과 이벤트 기록을 같은 로컬 트랜잭션으로 묶어 두면, 커밋과 동시에 이벤트가 DB에 안전하게 남는다. 그 뒤 별도의 릴레이가 그 기록을 읽어 발행을 보장하므로, 이벤트가 중간에 증발해 Saga 체인이 끊기는 일이 없어진다. 서버가 죽어도 릴레이가 재기동 후 이어서 발행하기 때문이다. Saga와 Outbox가 세트로 쓰이는 이유가 바로 이것이다.

**Q. Saga를 쓰면 데이터 정합성이 깨지는 순간이 생기는데, 사용자는 그 상태를 어떻게 보게 되나요?**

**A.** 중간 상태를 숨기지 말고 도메인 상태로 승격시키는 것이 핵심이다. 예컨대 "주문 접수됨 → 결제 확인 중 → 확정/실패"처럼 진행 단계를 명시적인 상태로 모델링한다. 사용자에게는 지금 진행 중이라는 것을 보여주고, 보상이 완료되면 사유와 함께 최종 상태로 전환해 준다. 이때 UI뿐 아니라 CS와 정산까지 같은 상태 모델을 공유해야, "돈은 나갔는데 주문이 없다" 같은 문의에도 시스템 전체가 일관된 답을 줄 수 있다. 결국 최종 정합성 시스템의 UX는 일시적 불일치를 사용자의 언어로 번역해 주는 일이다.

### 면접 답변 (구술용)

2PC가 MSA에서 어려운 이유는 세 가지입니다. 첫째, 블로킹 프로토콜입니다 — prepare 단계에서 모든 참여자가 락을 쥔 채 coordinator의 결정을 기다리는데, coordinator가 죽으면 참여자들은 커밋도 롤백도 스스로 결정하지 못하고 락을 쥔 채 대기하고, 그동안 그 데이터를 쓰는 다른 트랜잭션까지 전부 막힙니다. 둘째, 전체가 가장 느린 참여자에 종속됩니다 — 참여 서비스가 늘수록 전원이 살아있고 빠를 확률은 곱셈으로 떨어져서, 서비스를 늘리는 MSA 방향과 정면충돌합니다. 셋째, 현실적으로 Kafka나 많은 NoSQL, 외부 API는 XA에 참여하지 않아서 애초에 선택지가 안 되는 경우가 많습니다. Saga는 전역 트랜잭션을 포기하고 각 서비스의 로컬 트랜잭션을 순차 실행하되, 중간 실패 시 이미 커밋된 단계들을 보상 트랜잭션으로 되돌립니다. 락을 서비스 경계 너머로 잡지 않으니 각 서비스가 자율적으로 동작하고, 잠깐의 불일치를 허용하는 대신 최종 정합성을 보장합니다. 본질은 "즉시 정합성 vs 가용성·자율성"의 트레이드오프이고, 대부분의 도메인은 '결제 확인 중' 같은 짧은 중간 상태를 허용할 수 있어서 Saga를 택합니다. 대신 복잡성이 보상 로직 설계로 이동한다는 것까지 말해야 완성된 답입니다.

### 원리 이해 (왜 그런가)

**2PC vs Saga 비교표:**

| | 2PC | Saga |
|---|---|---|
| 정합성 | 강한 즉시 정합성 | 최종 정합성 (중간 상태 존재) |
| 락 | prepare~commit 동안 참여자 전원이 락 유지 | 각 로컬 트랜잭션 동안만 (서비스 경계 넘는 락 없음) |
| 장애 시 | coordinator 죽으면 전원 블로킹 | 실패 단계부터 보상 실행, 나머지 서비스는 계속 동작 |
| 가용성 | 전원 생존 필요 → 곱셈으로 하락 (99.9% × 5개 ≈ 99.5%) | 서비스별 독립 |
| 지원 | XA 구현 시스템만 (Kafka·다수 NoSQL 불가) | 로컬 트랜잭션만 있으면 됨 |
| 복잡성의 위치 | 프로토콜(인프라)에 숨음 | **보상 로직 설계(애플리케이션)로 이동** |

**Saga 실행 흐름 (주문 예시):**

```
주문 생성(커밋) → 결제 승인(커밋) → 재고 차감(실패!)
                                      ↓ 보상 체인 역순 실행
             ← 결제 취소(보상)  ← 주문 실패 처리(보상)
※ 보상 = DB 롤백이 아니라 "의미적 역연산" (승인↔취소, 차감↔복원)
```

**보상 설계 원칙 3가지:**
1. 보상도 실패할 수 있다 → 멱등하게 만들고 재시도 큐에 태운다.
2. 되돌릴 수 없는 작업(외부 통보, 이메일)은 가능한 한 뒤쪽(pivot 이후)에 배치해 보상할 일 자체를 줄인다.
3. 보상 중에도 조회가 들어오므로 상태(PENDING/COMPENSATING/FAILED)를 명시적으로 기록한다.

### 꼬리질문 Q&A

**Q. 2PC가 가용성 측면에서 불리한 이유는?**

**A.** **coordinator가 단일 장애점이고, prepare 이후 coordinator 장애 시 참여자들이 락을 쥔 채 블로킹되기 때문.**
참여자는 혼자 판단해서 커밋/롤백하면 다른 참여자와 어긋날 수 있어서 기다릴 수밖에 없습니다 — 장애가 "한 컴포넌트 문제"에서 "락 병목을 통한 전체 정지"로 증폭되는 구조입니다. 또 트랜잭션 성립에 참여자 전원의 생존이 필요하므로 시스템이 늘수록 전체 가용성이 곱으로 떨어집니다.

**Q. 보상 트랜잭션은 어떻게 설계하나요?**

**A.** **각 단계마다 비즈니스 수준의 "의미적 역연산"을 정의한다 — DB 롤백이 아니다.**
재고 차감의 보상은 재고 복원, 결제 승인의 보상은 환불/취소입니다. 이미 커밋된 것을 되돌리는 것이므로 기술적 롤백이 불가능하고, 그래서 보상 자체도 하나의 정식 비즈니스 로직으로 — 멱등하게, 재시도 가능하게, 실패 시 알람이 오게 — 설계해야 합니다.

**Q. Choreography와 Orchestration의 차이는?**

**A.** **Choreography는 중앙 조정자 없이 각 서비스가 이벤트에 반응하고, Orchestration은 중앙 orchestrator가 명령과 상태를 관리한다.**
Choreography는 결합도가 낮고 단순한 흐름에 좋지만, 단계가 많아지면 전체 흐름이 어디에도 적혀 있지 않아 추적·수정이 어렵고 순환 구독 같은 함정이 생깁니다. Orchestration은 흐름이 한곳에 명시되어 추적과 보상 관리가 쉽지만 orchestrator에 로직이 집중됩니다. 실무 감각으로는 단계가 3개 이상이거나 보상이 복잡하면 orchestration이 유리하다는 게 일반적 기준입니다.

**Q. Saga 중간 단계에서 메시지 발행이 실패하면?**

**A.** **"로컬 트랜잭션은 커밋됐는데 다음 단계 이벤트가 안 나가는" 상황 — 정확히 Outbox 패턴(260618)이 푸는 문제다.**
각 단계에서 상태 변경과 이벤트 기록을 같은 로컬 트랜잭션으로 묶고 릴레이가 발행을 보장하면, Saga 체인이 중간에 증발하는 일이 없습니다. Saga와 Outbox가 세트로 쓰이는 이유입니다.

**Q. 정합성이 깨진 중간 상태를 사용자는 어떻게 보게 되나요?**

**A.** **숨기지 말고 도메인 상태로 승격시킨다 — "주문 접수됨 → 결제 확인 중 → 확정/실패".**
사용자에게는 진행 중임을 보여주고, 보상이 완료되면 사유와 함께 최종 상태로 전환합니다. UI·CS·정산까지 이 상태 모델을 공유해야 "돈은 나갔는데 주문이 없다"는 문의에도 시스템이 일관된 답을 줄 수 있습니다. 최종 정합성 시스템의 UX는 일시적 불일치를 사용자의 언어로 번역하는 일입니다.

### 🌱 심화 키워드
- **보상 트랜잭션(compensating transaction) / pivot** — Saga 단계 분류: 보상 가능 → pivot(분기점) → 재시도 가능
- **semantic lock** — Saga의 격리성 부재를 보완하는 대책(상태 필드로 "처리 중" 표시)
- **TCC(Try-Confirm-Cancel)** — 자원 예약 기반의 또 다른 분산 트랜잭션 패턴
- **Saga orchestrator / process manager** — 상태 기계로 Saga를 구동하는 컴포넌트
- **XA / JTA** — 2PC의 표준 인터페이스 (무엇이 지원 안 되는지 알아야 선택 근거가 됨)

### ❓ 더 파볼 질문
- **Saga는 격리성(Isolation)이 없어서 생기는 이상 현상이 있다는데, 어떤 것이고 어떻게 막나?**
  ↳ 커밋된 중간 상태를 다른 트랜잭션이 읽거나(dirty read 유사) 덮어쓰는 문제가 있다. 대책이 semantic lock(상태 필드를 PENDING으로 마킹해 다른 로직이 건드리지 않게), 교환 가능한(commutative) 업데이트 설계, 값 재검증 등이며 Richardson 책에서 countermeasures로 정리돼 있다.
- **TCC는 Saga와 뭐가 다른가?**
  ↳ TCC는 Try 단계에서 자원을 "예약"(재고 가예약, 한도 홀드)해 두고 Confirm/Cancel로 확정·해제한다. 예약 개념 덕분에 격리성이 Saga보다 좋지만, 모든 참여 서비스가 Try/Confirm/Cancel 3개 API를 구현해야 하는 부담이 있다. 예약이 자연스러운 도메인(좌석, 한도)에서 유리하다.
- **보상이 불가능한 작업(이메일 발송, 외부 통보)은 어떻게 다루나?**
  ↳ Saga의 마지막(pivot 이후 재시도 가능 구간)에 배치해 "앞 단계가 전부 확정된 뒤에만 실행"되게 한다. 즉 되돌릴 수 없는 일은 되돌릴 일이 없는 위치로 옮기는 것이 설계 원칙이다.
