# 260708 JPA 영속성 컨텍스트는 무엇이고 어떤 역할을 하나요?

**오늘의 백엔드 질문**

JPA 영속성 컨텍스트는 무엇이고 어떤 역할을 하나요?

이 질문은 JPA를 “그냥 ORM”으로만 쓰는지, 아니면 내부 동작을 이해하고 쓰는지를 확인하기 좋은 질문입니다. 특히 1차 캐시, 변경 감지, flush, transaction과 자연스럽게 연결됩니다.

좋은 답변에는 이런 흐름이 들어가면 좋습니다.

- 영속성 컨텍스트는 JPA가 Entity를 관리하는 논리적인 저장 공간입니다.
- EntityManager를 통해 조회하거나 저장한 Entity는 영속성 컨텍스트에 올라가고, JPA는 이 Entity의 상태를 추적합니다.
- 같은 transaction 안에서 동일한 Entity를 다시 조회하면 DB를 다시 조회하지 않고 1차 캐시에서 반환할 수 있습니다.
- Entity의 값이 변경되면 JPA가 변경 내용을 감지하고, flush 시점에 update SQL을 생성합니다.
- 즉, 영속성 컨텍스트는 1차 캐시, 동일성 보장, 변경 감지, 쓰기 지연 같은 기능을 통해 DB 접근을 효율적으로 관리합니다.

꼬리 질문으로는 이런 것들이 나올 수 있습니다.

- 영속성 컨텍스트의 1차 캐시는 어떤 key로 Entity를 관리하나요?
- flush와 commit은 어떤 관계인가요?
- Dirty Checking은 어떤 방식으로 동작하나요?
- 준영속 상태의 Entity는 변경 감지가 될까요?
- 같은 Entity를 두 번 조회했을 때 `==` 비교가 true가 될 수 있는 이유는 무엇인가요?

---

## 답변

> **한 줄 핵심**: 영속성 컨텍스트는 애플리케이션과 DB 사이의 버퍼·캐시 계층으로 — "트랜잭션 동안의 작업을 메모리에 모아뒀다가 flush 시점에 한 번에 동기화"하기 위해 존재하며, 1차 캐시·동일성 보장·변경 감지·쓰기 지연이 그 위에서 동작한다.

### 면접 답변 (구술용)

영속성 컨텍스트는 EntityManager가 엔티티를 관리하는 논리적 저장 공간으로, 애플리케이션과 DB 사이의 버퍼이자 캐시 계층입니다. 존재 이유부터 말하면 — 객체 변경을 즉시 SQL로 바꾸면 DB 왕복이 잦고 같은 데이터를 반복 조회하면 낭비라서, 트랜잭션 동안의 작업을 메모리에 모아뒀다가 한 번에 동기화하기 위한 중간층입니다. 역할은 네 가지입니다. 첫째, 1차 캐시 — 엔티티 타입과 @Id 값을 key로 보관해서 같은 컨텍스트에서 같은 PK 조회는 DB에 가지 않습니다. 둘째, 동일성 보장 — 같은 PK는 항상 같은 객체 참조를 돌려주므로 == 비교가 성립하고, 애플리케이션 레벨에서 반복 가능한 읽기와 유사한 효과를 줍니다. 셋째, 변경 감지 — 영속화 시점에 스냅샷을 떠 두고 flush 때 현재 값과 비교해 달라진 엔티티의 UPDATE를 자동 생성합니다. update() 메서드를 호출하지 않는 이유입니다. 넷째, 쓰기 지연 — persist해도 SQL을 즉시 보내지 않고 모아뒀다가 flush에 일괄 전송하므로 JDBC 배치로 묶을 수 있습니다. 스프링에서는 보통 트랜잭션과 생명주기를 같이하는데, Spring Boot는 OSIV가 기본 켜져 있어 요청 끝까지 유지된다는 점도 함께 알아두면 좋습니다. 주의점은 대량 배치 처리 시 1차 캐시가 계속 자라므로 주기적으로 flush와 clear로 비워줘야 한다는 것입니다.

### 원리 이해 (왜 그런가)

**구조와 4대 기능:**

```
애플리케이션 ──▶ [영속성 컨텍스트]                        ──▶ DB
                  ├ 1차 캐시: Map<(타입, @Id), 엔티티>       (조회 절약)
                  ├ 스냅샷: 영속화 시점의 필드 값 복사본       (변경 감지 기준)
                  └ 쓰기 지연 SQL 저장소: INSERT/UPDATE 대기열 (flush에 일괄 전송)
```

| 기능 | 동작 | 효과·근거 |
|------|------|-----------|
| 1차 캐시 | (타입 + @Id)로 보관, 같은 PK 조회 시 캐시 반환 | 반복 조회 비용 제거 — 단, **같은 컨텍스트 안에서만** |
| 동일성 보장 | 같은 PK → 같은 인스턴스 | `==` 성립, 애플리케이션 레벨 REPEATABLE READ 유사 효과 |
| 변경 감지 | flush 때 스냅샷과 현재 값을 필드 단위 비교 → UPDATE 생성 | "객체를 바꾸면 DB가 따라온다" — **영속 상태일 때만** 동작 |
| 쓰기 지연 | SQL을 모아 flush에 전송 | JDBC 배치 가능 + 락 보유 시간 단축(SQL 실행을 커밋 직전으로 미룸) |

**flush의 자동 발생 3시점과 각각의 이유:**
1. **트랜잭션 커밋 직전** — 커밋 전에 변경이 DB에 반영돼 있어야 하므로.
2. **JPQL 등 쿼리 실행 직전**(FlushModeType.AUTO) — 방금 persist한 데이터가 쿼리 결과에 안 잡히는 모순을 막기 위해.
3. **명시적 `em.flush()`**.

flush ≠ commit: flush는 SQL을 "전송"한 것이지 "확정"이 아니라서, flush 후에도 롤백할 수 있습니다.

### 꼬리질문 Q&A

**Q. 1차 캐시는 어떤 key로 Entity를 관리하나요?**
→ **엔티티 타입 + 식별자(@Id 값) 조합.**
그래서 캐시 적중은 PK 조회(em.find) 이야기입니다. JPQL은 일단 DB에 쿼리를 날리고, 가져온 row의 PK가 이미 1차 캐시에 있으면 캐시의 인스턴스를 반환해 동일성은 유지합니다 — "쿼리는 나가되 객체는 캐시 우선"이라는 미묘한 동작까지 알면 좋습니다.

**Q. flush와 commit은 어떤 관계인가요?**
→ **flush = 변경분을 SQL로 만들어 DB에 전송(동기화), commit = 그 트랜잭션을 확정. flush했어도 커밋 전이면 롤백 가능하다.**
commit 시점에 flush가 자동 호출되고, 그 외에 JPQL 실행 직전과 명시적 호출 시에도 발생합니다. "flush가 곧 저장 확정"이라고 답하면 감점 포인트입니다.

**Q. Dirty Checking은 어떤 방식으로 동작하나요?**
→ **영속화 시점의 스냅샷을 보관해 두고, flush 때 현재 값과 필드 단위로 비교해서 달라진 엔티티의 UPDATE SQL을 만든다.**
비용 관점의 근거도 중요합니다 — 관리 엔티티 수만큼 비교가 일어나므로 대량 로딩 시 flush가 느려질 수 있고, 조회 전용이면 `@Transactional(readOnly = true)`로 스냅샷 비교와 flush를 생략시키는 것이 최적화가 됩니다. 참고로 Hibernate의 UPDATE는 기본이 모든 컬럼을 포함한 정적 SQL(구문 재사용 목적)이고, 변경 컬럼만 쓰려면 `@DynamicUpdate`를 붙입니다.

**Q. 준영속 상태의 Entity는 변경 감지가 될까요?**
→ **안 된다 — 변경 감지는 엔티티의 능력이 아니라 컨텍스트의 기능인데, 준영속은 컨텍스트가 더 이상 관리하지 않는 상태다.**
스냅샷 비교 대상 자체가 아닙니다. 다시 반영하려면 merge()를 쓰는데, merge는 넘긴 객체를 영속화하는 게 아니라 식별자로 조회한 영속 엔티티에 값을 복사하고 **그 새 영속 객체를 반환**합니다. 반환값을 써야 하고, 인자로 넘긴 객체는 여전히 준영속이라는 게 흔한 함정입니다.

**Q. 같은 Entity를 두 번 조회했을 때 `==`가 true일 수 있는 이유는?**
→ **1차 캐시의 동일성 보장 — 두 번째 조회가 같은 key(타입+PK)로 캐시에서 같은 인스턴스를 돌려주기 때문.**
성립 조건이 곧 한계입니다: 같은 영속성 컨텍스트 안에서만 참이고, 트랜잭션(컨텍스트)이 다르면 각자 다른 인스턴스라 false입니다. 그래서 엔티티의 논리적 동등성은 ==가 아니라 비즈니스 키 기반 equals/hashCode로 따로 설계해야 합니다.

### 🌱 심화 키워드
- **엔티티 생명주기** — 비영속(new) → 영속(managed) → 준영속(detached) / 삭제(removed) 상태 전이
- **FlushModeType (AUTO/COMMIT)** — flush 타이밍 제어
- **@DynamicUpdate** — 변경 컬럼만 UPDATE (트레이드오프: 구문 캐시 재사용 포기)
- **OSIV(Open Session In View)** — 컨텍스트 생존 범위를 요청 끝까지 늘리는 설정과 그 대가 (→ 260617)

### ❓ 더 파볼 질문
- **IDENTITY 전략을 쓰면 왜 쓰기 지연이 무력화되나?**
  ↳ IDENTITY는 DB가 INSERT를 실행해야 PK가 생기는데, 영속 상태가 되려면 1차 캐시 key인 PK가 필요하다. 그래서 persist 즉시 INSERT가 나가고, 결과적으로 INSERT 배치가 불가능해진다. 대량 삽입 성능이 중요하면 SEQUENCE 전략 + allocationSize를 검토하는 이유다.
- **2차 캐시는 1차 캐시와 뭐가 다르고 왜 잘 안 쓰나?**
  ↳ 1차 캐시는 트랜잭션 범위의 사적 캐시라 동시성 문제가 없지만, 2차 캐시는 SessionFactory 수준의 공유 캐시라 무효화·동시성·클러스터 일관성 문제를 함께 떠안는다. 실무에서는 그 복잡성 때문에 Redis 같은 외부 캐시를 명시적으로 쓰는 쪽이 흔하다.
- **readOnly = true는 내부적으로 무엇을 바꾸나?**
  ↳ Hibernate의 flush mode를 MANUAL로 바꿔 자동 flush와 스냅샷 비교(dirty checking)를 생략하게 한다 — 메모리와 CPU를 아끼는 실질 효과가 있다. 추가로 라우팅 데이터소스 구성에서는 리플리카(읽기 전용 DB)로 보내는 분기 근거로도 쓰인다.
