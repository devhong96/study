## 답변

> **한 줄 핵심**: DB와 브로커라는 두 시스템에 대한 쓰기(dual write)는 원자적으로 묶을 수 없으므로, **이벤트 발행을 "같은 DB 트랜잭션 안의 outbox INSERT"로 바꿔** 두 시스템 문제를 한 시스템 문제로 환원하는 패턴이다.

### 면접 답변 (구술용)

DB 변경과 메시지 발행의 원자성 문제, 이른바 dual write problem을 해결하는 패턴입니다. DB와 Kafka는 서로 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없어서, "커밋 후 발행"이면 발행 직전에 죽었을 때 이벤트가 유실되고, "발행 후 커밋"이면 롤백 시 유령 이벤트가 남습니다. Outbox는 이벤트 발행을 DB 쓰기로 바꿔서 풉니다 — 비즈니스 데이터 변경과 outbox 테이블 INSERT를 같은 로컬 트랜잭션에서 수행하면 DB 원자성이 "둘은 함께 성공하거나 함께 실패한다"를 보장하고, 별도 릴레이가 outbox를 읽어 Kafka로 발행합니다. 릴레이가 죽어도 이벤트는 DB에 남아 있으니 유실이 구조적으로 막힙니다. 대가는 at-least-once가 된다는 것이라, 중복은 Consumer 멱등성으로 흡수합니다. "유실은 구조로 막고, 중복은 멱등성으로 막는다"는 분업입니다.

### 원리 이해 (왜 그런가)

**dual write가 왜 안 되는가 — 순서를 바꿔도 깨진다:**

```
[커밋 → 발행]  커밋 ✓ … (crash) … 발행 ✗  → 데이터는 바뀌었는데 이벤트 없음 (유실)
[발행 → 커밋]  발행 ✓ … 커밋 실패(롤백)    → 없던 일에 대한 이벤트 (유령 이벤트)
※ 재시도로 해결 불가 — 재시도 주체도 함께 죽을 수 있음
```

**Outbox 동작 흐름:**

```
① [하나의 DB 트랜잭션] 비즈니스 테이블 변경 + outbox INSERT  ← 원자성은 DB가 보장
② [릴레이] outbox에서 미발행 이벤트 읽기 → Kafka 발행 → 발행 완료 표시
   (릴레이가 죽으면? 이벤트는 DB에 그대로 → 재시작 후 이어서 발행 = 유실 불가)
③ [Consumer] 이벤트 ID 기반 멱등 처리  ← 재발행(중복)은 여기서 흡수
```

**릴레이 구현 2방식:**

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| Polling | 주기적으로 미발행 row SELECT | 구현 단순, 인프라 추가 없음 | 폴링 주기만큼 지연, DB 부하 |
| CDC (Debezium) | DB 변경 로그(binlog/WAL)에서 outbox INSERT 감지 | 낮은 지연, 폴링 부하 없음 | Kafka Connect/Debezium 운영 복잡도 |

### 꼬리질문 Q&A

**Q. Outbox Pattern은 어떤 문제를 해결하나요?**
→ **dual write problem — 두 시스템에 대한 쓰기의 원자성 부재.** 커밋과 발행 사이 모든 장애 시나리오에서 정합성이 유지된다.

**Q. Outbox 테이블에는 어떤 컬럼이 필요할까요?**
→ **id(이벤트 고유 ID = 멱등성 키), aggregate_type/aggregate_id(파티션 키), event_type, payload(JSON), created_at, 발행 상태.**
- aggregate_id를 Kafka 파티션 키로 쓰는 근거: Kafka는 같은 파티션 안에서만 순서를 보장하므로, 같은 엔티티의 이벤트를 같은 파티션에 모아 순서를 지킨다.

**Q. Polling인가요, CDC인가요?**
→ **둘 다 가능 — 지연·부하가 중요하면 CDC, 단순함이 중요하면 Polling.** 트래픽이 작으면 Polling으로 시작해도 충분하다.

**Q. 중복 발행 시 Consumer 멱등성은?**
→ **이벤트 ID를 처리 이력 테이블(unique 제약)에 기록하며 처리하고, 가능하면 이력 기록과 비즈니스 반영을 같은 DB 트랜잭션으로 묶는다.**
- 릴레이가 "발행 성공 후 완료 표시 전에 죽는" 구간이 존재하는 한 재발행은 필연 — 그래서 at-least-once.

**Q. Outbox 테이블이 커지면?**
→ **발행 완료분은 삭제/아카이빙, 대량이면 날짜 파티셔닝 후 오래된 파티션 DROP.**
- DROP PARTITION이 DELETE보다 싼 근거: DELETE는 row 단위 undo·인덱스 갱신, DROP은 메타데이터 연산.

**Q. Saga와 Outbox는 각각 어떤 문제를 해결하나요?**
→ **Outbox = 한 서비스 안에서 "DB 변경 + 이벤트 발행"의 원자성. Saga(260629) = 여러 서비스에 걸친 비즈니스 트랜잭션의 정합성(보상 포함).**
- 실제로는 Saga의 각 단계가 Outbox로 이벤트를 안전하게 발행하는 조합이 일반적.

### ❓ 더 파볼 질문
- **Outbox 없이 "Kafka에 먼저 쓰고 Kafka에서 DB로" 구조로 뒤집으면 문제가 어떻게 달라지나?**
  ↳ Kafka가 진실의 원천(source of truth)이 되고 DB는 이벤트를 소비해 만든 프로젝션이 된다 — dual write는 사라지지만, "방금 쓴 것이 조회에 바로 안 보이는" 읽기 일관성 문제와 프로젝션 재구축·스키마 진화 관리가 새 과제로 등장한다. 즉 문제가 없어지는 게 아니라 "쓰기 원자성 문제"가 "읽기 일관성 문제"로 자리를 바꾸는 것이다(이벤트 소싱/CQRS의 세계).
- **CDC 릴레이의 지연(lag)은 어떻게 모니터링하고, 밀렸을 때 순서 보장은 유지되나?**
  ↳ Debezium/Kafka Connect가 노출하는 메트릭(처리 중인 binlog 위치와 최신 위치의 차이, 이벤트 타임스탬프 지연)으로 lag를 관측하고 알람을 건다. 밀려도 binlog는 순서대로 읽히므로 같은 파티션 키 안에서의 순서는 유지된다 — 지연은 "늦게 도착"이지 "뒤섞임"이 아니라는 점이 소비자 설계의 전제가 된다.
- **발행 완료 표시 자체가 실패하면(발행 ✓, 완료 UPDATE ✗) 어떤 일이 생기고 왜 허용 가능한가?**
  ↳ 릴레이가 재시작 후 그 이벤트를 미발행으로 보고 다시 발행한다 — 즉 중복만 생기고 유실은 없다. Outbox의 설계 철학이 바로 이것이다: 애매한 실패는 전부 "재발행(중복)" 쪽으로 쓰러지게 만들고, 중복은 Consumer 멱등성이 흡수한다. 유실은 복구 불가지만 중복은 흡수 가능하다는 비대칭이 이 선택의 근거다.
