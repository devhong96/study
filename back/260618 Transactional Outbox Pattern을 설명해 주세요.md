## 답변

> **한 줄 핵심**: DB와 브로커라는 두 시스템에 대한 쓰기(dual write)는 원자적으로 묶을 수 없으므로, **이벤트 발행을 "같은 DB 트랜잭션 안의 outbox INSERT"로 바꿔** 두 시스템 문제를 한 시스템 문제로 환원하는 패턴이다.

### 1문 1답

**Q. Transactional Outbox Pattern을 설명해 주세요.**

**A.** DB 변경과 메시지 발행의 원자성 문제, 이른바 dual write problem을 해결하는 패턴이다. DB와 Kafka는 서로 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없어서, "커밋 후 발행"이면 발행 직전에 죽었을 때 이벤트가 유실되고 "발행 후 커밋"이면 롤백 시 유령 이벤트가 남는다. Outbox는 이벤트 발행을 DB 쓰기로 바꿔 푸는데, 비즈니스 데이터 변경과 outbox 테이블 INSERT를 같은 로컬 트랜잭션에서 수행하면 DB 원자성이 "둘은 함께 성공하거나 함께 실패한다"를 보장하고, 별도 릴레이가 outbox를 읽어 Kafka로 발행한다. 릴레이가 죽어도 이벤트는 DB에 남아 있어 유실이 구조적으로 막힌다. 대가는 at-least-once가 된다는 것이라, 중복은 Consumer 멱등성으로 흡수한다 — "유실은 구조로 막고, 중복은 멱등성으로 막는다"는 분업이다.

**Q. DB 저장과 메시지 발행은 왜 원자적으로 묶기 어려운가?**

**A.** DB와 Kafka(브로커)는 서로 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없기 때문이다. 이것이 dual write problem인데, 순서를 어떻게 바꿔도 깨진다. "커밋 후 발행"이면 커밋은 됐는데 발행 직전에 프로세스가 죽었을 때 데이터는 바뀌었지만 이벤트가 없어 유실이 생긴다. 반대로 "발행 후 커밋"이면 발행은 나갔는데 뒤이어 커밋이 롤백되면 없던 일에 대한 유령 이벤트가 남는다. 단순 재시도로도 못 푸는데, 재시도를 담당하는 주체마저 함께 죽을 수 있기 때문이다.

**Q. Outbox 테이블에는 어떤 컬럼(데이터)을 저장해야 하나요?**

**A.** 이벤트 고유 ID(멱등성 키), aggregate_type과 aggregate_id(파티션 키), event_type, payload(JSON), created_at, 그리고 발행 상태 정도가 필요하다. 여기서 이벤트 ID는 Consumer가 중복을 걸러내는 멱등성 키로 쓰인다. aggregate_id를 Kafka 파티션 키로 삼는 이유가 중요한데, Kafka는 같은 파티션 안에서만 순서를 보장하므로 같은 엔티티(예: 같은 주문)의 이벤트들을 같은 파티션에 모아야 순서가 지켜지기 때문이다. payload에는 이벤트 소비에 필요한 데이터를 담고, 발행 상태로 미발행/발행완료를 구분해 릴레이가 무엇을 보낼지 판단한다.

**Q. 이벤트 발행 실패 시 어떻게 재시도할 것인가?**

**A.** 핵심은 이벤트가 이미 DB의 outbox 테이블에 안전하게 남아 있다는 점이다. 별도 릴레이가 outbox에서 미발행 이벤트를 읽어 Kafka로 발행하고 발행 완료로 표시하는데, 이 릴레이가 도중에 죽더라도 이벤트는 DB에 그대로 있으므로 재시작 후 이어서 발행하면 된다. 그래서 유실이 구조적으로 불가능해진다. 대신 "발행은 성공했는데 완료 표시 직전에 죽는" 구간이 존재하는 한 같은 이벤트의 재발행은 필연이라, 전달 보장 수준은 at-least-once가 된다. 이때 생기는 중복은 유실과 달리 흡수 가능하므로 Consumer 멱등성으로 처리한다.

**Q. 같은 이벤트가 여러 번 발행되면 Consumer는 어떻게 중복 처리(멱등성 보장)하나요?**

**A.** 이벤트 ID를 unique 제약이 걸린 처리 이력(inbox) 테이블에 기록하면서 처리하고, 가능하면 이력 기록과 비즈니스 반영을 같은 DB 트랜잭션으로 묶어 둘이 함께 확정되도록 한다. 그러면 같은 ID를 가진 이벤트가 다시 들어와도 unique 제약 위반으로 걸러져 두 번 반영되지 않는다. Outbox 설계는 애매한 실패를 전부 "재발행(중복)" 쪽으로 쓰러지게 만들어 유실을 없애는 대신, 그 중복을 Consumer가 흡수하도록 역할을 나눈다. 즉 "유실은 구조로 막고, 중복은 멱등성으로 막는다"는 분업이다. 유실은 복구가 불가능하지만 중복은 흡수 가능하다는 비대칭이 이 선택의 근거다.

**Q. Outbox 테이블이 계속 커질 때 성능 저하는 어떻게 방지하나요?**

**A.** 발행이 끝난 row는 그대로 쌓아두지 말고 삭제하거나 아카이빙한다. 양이 많다면 날짜 기준으로 파티셔닝해 두고 오래된 파티션을 통째로 DROP하는 방식이 효율적이다. DROP PARTITION이 대량 DELETE보다 싼 이유는, DELETE는 row 단위로 undo 로그와 인덱스 갱신이 드는 반면 DROP은 메타데이터 연산으로 끝나기 때문이다. 이렇게 오래된 데이터를 값싸게 정리하면 테이블이 무한정 커져 조회·발행이 느려지는 것을 막을 수 있다.

**Q. Outbox Pattern은 어떤 문제를 해결하나요?**

**A.** 두 시스템에 대한 쓰기의 원자성이 없다는 dual write problem을 해결한다. 방법은 이벤트 발행을 "같은 DB 트랜잭션 안의 outbox INSERT"로 바꿔, 서로 다른 두 시스템 문제를 한 시스템(DB) 문제로 환원하는 것이다. 비즈니스 데이터 변경과 outbox INSERT가 같은 로컬 트랜잭션에서 일어나므로 DB의 원자성이 "둘은 함께 성공하거나 함께 실패한다"를 보장한다. 그 결과 커밋과 발행 사이의 모든 장애 시나리오에서 정합성이 유지된다. 핵심 철학은 "유실은 구조로 막고, 중복은 멱등성으로 막는다"는 분업이다.

**Q. Outbox 이벤트를 Kafka로 발행하는 방식은 Polling인가요, CDC인가요?**

**A.** 둘 다 가능하며 트레이드오프가 다르다. Polling 방식은 미발행 row를 주기적으로 SELECT해서 발행하는데, 구현이 단순하고 추가 인프라가 없다는 장점이 있지만 폴링 주기만큼 지연이 생기고 반복 조회로 DB 부하가 있다. CDC 방식(Debezium)은 DB의 변경 로그(binlog/WAL)에서 outbox INSERT를 감지해 발행하므로 지연이 낮고 폴링 부하가 없는 대신, Kafka Connect·Debezium을 운영하는 복잡도가 붙는다. 그래서 지연·부하가 중요하면 CDC, 단순함이 중요하거나 트래픽이 작으면 Polling으로 시작하는 식으로 고른다.

**Q. Saga Pattern과 Outbox Pattern은 각각 어떤 문제를 해결하나요?**

**A.** Outbox는 한 서비스 안에서 "DB 변경 + 이벤트 발행"의 원자성을 보장하는 패턴이다. 반면 Saga(260629)는 여러 서비스에 걸친 비즈니스 트랜잭션의 정합성을, 실패 시 보상 트랜잭션까지 포함해 관리하는 패턴이다. 즉 해결 범위가 로컬(한 서비스 내부)이냐 분산(여러 서비스)이냐로 갈린다. 실제로는 둘이 대립하는 게 아니라, Saga의 각 단계가 자신의 이벤트를 Outbox로 안전하게 발행하는 식으로 함께 쓰이는 경우가 일반적이다.

### 면접 답변 (구술용)

DB 변경과 메시지 발행의 원자성 문제, 이른바 dual write problem을 해결하는 패턴입니다. DB와 Kafka는 서로 다른 시스템이라 하나의 트랜잭션으로 묶을 수 없어서, "커밋 후 발행"이면 발행 직전에 죽었을 때 이벤트가 유실되고, "발행 후 커밋"이면 롤백 시 유령 이벤트가 남습니다. Outbox는 이벤트 발행을 DB 쓰기로 바꿔서 풉니다 — 비즈니스 데이터 변경과 outbox 테이블 INSERT를 같은 로컬 트랜잭션에서 수행하면 DB 원자성이 "둘은 함께 성공하거나 함께 실패한다"를 보장하고, 별도 릴레이가 outbox를 읽어 Kafka로 발행합니다. 릴레이가 죽어도 이벤트는 DB에 남아 있으니 유실이 구조적으로 막힙니다. 대가는 at-least-once가 된다는 것이라, 중복은 Consumer 멱등성으로 흡수합니다. "유실은 구조로 막고, 중복은 멱등성으로 막는다"는 분업입니다.

### 원리 이해 (왜 그런가)

**dual write가 왜 안 되는가 — 순서를 바꿔도 깨진다:**

```
[커밋 → 발행]  커밋 ✓ … (crash) … 발행 ✗  → 데이터는 바뀌었는데 이벤트 없음 (유실)
[발행 → 커밋]  발행 ✓ … 커밋 실패(롤백)    → 없던 일에 대한 이벤트 (유령 이벤트)
※ 재시도로 해결 불가 — 재시도 주체도 함께 죽을 수 있음
```

**Outbox 동작 흐름:**

```
① [하나의 DB 트랜잭션] 비즈니스 테이블 변경 + outbox INSERT  ← 원자성은 DB가 보장
② [릴레이] outbox에서 미발행 이벤트 읽기 → Kafka 발행 → 발행 완료 표시
   (릴레이가 죽으면? 이벤트는 DB에 그대로 → 재시작 후 이어서 발행 = 유실 불가)
③ [Consumer] 이벤트 ID 기반 멱등 처리  ← 재발행(중복)은 여기서 흡수
```

**릴레이 구현 2방식:**

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| Polling | 주기적으로 미발행 row SELECT | 구현 단순, 인프라 추가 없음 | 폴링 주기만큼 지연, DB 부하 |
| CDC (Debezium) | DB 변경 로그(binlog/WAL)에서 outbox INSERT 감지 | 낮은 지연, 폴링 부하 없음 | Kafka Connect/Debezium 운영 복잡도 |

### 꼬리질문 Q&A

**Q. Outbox Pattern은 어떤 문제를 해결하나요?**

**A.** **dual write problem — 두 시스템에 대한 쓰기의 원자성 부재.** 커밋과 발행 사이 모든 장애 시나리오에서 정합성이 유지된다.

**Q. Outbox 테이블에는 어떤 컬럼이 필요할까요?**

**A.** **id(이벤트 고유 ID = 멱등성 키), aggregate_type/aggregate_id(파티션 키), event_type, payload(JSON), created_at, 발행 상태.**
- aggregate_id를 Kafka 파티션 키로 쓰는 근거: Kafka는 같은 파티션 안에서만 순서를 보장하므로, 같은 엔티티의 이벤트를 같은 파티션에 모아 순서를 지킨다.

**Q. Polling인가요, CDC인가요?**

**A.** **둘 다 가능 — 지연·부하가 중요하면 CDC, 단순함이 중요하면 Polling.** 트래픽이 작으면 Polling으로 시작해도 충분하다.

**Q. 중복 발행 시 Consumer 멱등성은?**

**A.** **이벤트 ID를 처리 이력 테이블(unique 제약)에 기록하며 처리하고, 가능하면 이력 기록과 비즈니스 반영을 같은 DB 트랜잭션으로 묶는다.**
- 릴레이가 "발행 성공 후 완료 표시 전에 죽는" 구간이 존재하는 한 재발행은 필연 — 그래서 at-least-once.

**Q. Outbox 테이블이 커지면?**

**A.** **발행 완료분은 삭제/아카이빙, 대량이면 날짜 파티셔닝 후 오래된 파티션 DROP.**
- DROP PARTITION이 DELETE보다 싼 근거: DELETE는 row 단위 undo·인덱스 갱신, DROP은 메타데이터 연산.

**Q. Saga와 Outbox는 각각 어떤 문제를 해결하나요?**

**A.** **Outbox = 한 서비스 안에서 "DB 변경 + 이벤트 발행"의 원자성. Saga(260629) = 여러 서비스에 걸친 비즈니스 트랜잭션의 정합성(보상 포함).**
- 실제로는 Saga의 각 단계가 Outbox로 이벤트를 안전하게 발행하는 조합이 일반적.

### ❓ 더 파볼 질문

**Q. Outbox 없이 "Kafka에 먼저 쓰고 Kafka에서 DB로" 구조로 뒤집으면 문제가 어떻게 달라지나?**

**A.** Kafka가 진실의 원천(source of truth)이 되고 DB는 이벤트를 소비해 만든 프로젝션이 된다 — dual write는 사라지지만, "방금 쓴 것이 조회에 바로 안 보이는" 읽기 일관성 문제와 프로젝션 재구축·스키마 진화 관리가 새 과제로 등장한다. 즉 문제가 없어지는 게 아니라 "쓰기 원자성 문제"가 "읽기 일관성 문제"로 자리를 바꾸는 것이다(이벤트 소싱/CQRS의 세계).

**Q. CDC 릴레이의 지연(lag)은 어떻게 모니터링하고, 밀렸을 때 순서 보장은 유지되나?**

**A.** Debezium/Kafka Connect가 노출하는 메트릭(처리 중인 binlog 위치와 최신 위치의 차이, 이벤트 타임스탬프 지연)으로 lag를 관측하고 알람을 건다. 밀려도 binlog는 순서대로 읽히므로 같은 파티션 키 안에서의 순서는 유지된다 — 지연은 "늦게 도착"이지 "뒤섞임"이 아니라는 점이 소비자 설계의 전제가 된다.

**Q. 발행 완료 표시 자체가 실패하면(발행 ✓, 완료 UPDATE ✗) 어떤 일이 생기고 왜 허용 가능한가?**

**A.** 릴레이가 재시작 후 그 이벤트를 미발행으로 보고 다시 발행한다 — 즉 중복만 생기고 유실은 없다. Outbox의 설계 철학이 바로 이것이다: 애매한 실패는 전부 "재발행(중복)" 쪽으로 쓰러지게 만들고, 중복은 Consumer 멱등성이 흡수한다. 유실은 복구 불가지만 중복은 흡수 가능하다는 비대칭이 이 선택의 근거다.
