# 260706 Redis 분산락은 실제로 무엇을 막아 주는 걸까요?

Redis 분산락은 실제로 무엇을 막아 주는 걸까요?

면접에서 Redis 분산락 이야기가 나오면 많은 분들이
“동시성 문제를 해결하기 위해 사용합니다”라고 답변을 시작합니다.
그런데 면접관이 더 궁금해하는 건
Redis 분산락이 모든 정합성 문제를 해결해 주는지,
아니면 특정 구간에 동시에 진입하는 것을 제한하는 도구인지를 구분하고 있는지입니다.
이 질문은 단순히 Redis 락 구현 방법을 외웠는지보다,
락이 보장하는 범위와 보장하지 못하는 영역을 이해하고 있는지를 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- Redis 분산락은 특정 critical section에 동시에 들어오는 요청을 제한한다는 점
- 락을 잡았다고 DB 트랜잭션 정합성이 자동으로 보장되는 것은 아니라는 점
- 락 만료 시간, 작업 시간, 네트워크 지연이 맞물리면 락이 먼저 풀릴 수 있다는 점
- 락 해제 시 본인이 획득한 락인지 확인해야 한다는 점
- 결제, 재고, 쿠폰 같은 도메인에서는 락뿐 아니라 멱등성, DB 제약조건, 트랜잭션 설계가 함께 필요하다는 점

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- Redis 분산락과 DB 비관적 락은 어떤 차이가 있나요?
- Redis 락의 TTL은 어떻게 정해야 하나요?
- 락을 획득한 서버가 작업 중 죽으면 어떻게 되나요?
- 락이 만료된 뒤 기존 요청이 뒤늦게 DB를 업데이트하면 어떤 문제가 생기나요?
- 분산락을 썼는데도 멱등성 키가 필요한 이유는 무엇인가요?

---

## 답변

> **한 줄 핵심**: Redis 분산락이 보장하는 것은 "여러 인스턴스에 걸친 임계 구역 동시 진입 제한(상호배제)" 하나뿐이다 — 트랜잭션 정합성·롤백·시간차 중복은 못 막으므로, 최종 방어선은 "락이 뚫려도 데이터가 안 깨지는 층"(unique 제약·조건부 UPDATE·멱등성)에 둬야 한다.

### 1문 1답

**Q. Redis 분산락은 실제로 무엇을 막아 주는 걸까요?**

**A.** Redis 분산락이 보장하는 것은 정확히 하나 — 여러 인스턴스에 걸쳐 특정 임계 구역에 동시에 진입하는 것을 제한하는 상호배제다. 그 이상을 기대하면 설계가 어긋나는데, 락을 잡아도 정합성이 깨지는 시나리오가 세 가지 있다. 첫째, TTL 만료 경계 — 작업이 GC pause나 네트워크 지연으로 TTL보다 길어지면 락이 먼저 풀려 다른 요청이 진입하고, 원래 요청이 자기 락이 풀린 줄 모른 채 뒤늦게 DB를 쓰면 동시에 쓴 것과 같아진다. 둘째, 락 해제와 커밋의 순서 — 커밋 전에 락을 풀면 다음 요청이 커밋 전 데이터를 읽고 판단하므로, 락 안에서 커밋까지 끝내고 해제해야 한다. 셋째, 남의 락 해제 — 내 TTL이 만료되고 남이 잡은 상태에서 무조건 DEL하면 남의 락을 지우므로, 소유자 값을 넣고 Lua로 비교와 삭제를 원자적으로 해야 한다. 그래서 결제·재고·쿠폰 도메인에선 정합성의 최종 방어선을 DB unique 제약·조건부 UPDATE·낙관적 락·멱등성 키에 두고, 락은 "없어도 안 깨지는데 있으면 더 효율적인" 장치로 쓰는 게 올바른 구조다.

**Q. Redis 분산락은 정확히 무엇을 보장하나요?**

**A.** Redis 분산락이 보장하는 것은 정확히 하나, 여러 인스턴스에 걸쳐 특정 임계 구역(critical section)에 동시에 진입하는 것을 제한하는 상호배제입니다. 즉 같은 순간에 여러 요청이 같은 구간을 "실행"하는 것을 억제하는 성능 최적화이자 부수효과 보호 장치입니다. 이것이 락이 해줄 수 있는 전부이고, 그 이상을 기대하는 순간 설계가 어긋납니다. 예컨대 트랜잭션 정합성, 롤백, 시간차로 발생하는 중복 실행 같은 것은 락이 막아주지 못합니다. 게다가 뒤에서 볼 TTL 만료 경계 같은 상황에서는 이 상호배제조차 뚫릴 수 있으므로, 락은 "완벽한 보장"이 아니라 "대부분의 동시 진입을 줄여주는 장치"로 이해해야 합니다.

**Q. 락을 잡으면 DB 트랜잭션 정합성도 자동으로 보장되나요?**

**A.** 아니오입니다. 락은 임계 구역 진입만 제한할 뿐, DB 트랜잭션의 정합성을 대신 보장해 주지 않습니다. 대표적인 함정이 락 해제와 커밋의 순서입니다. 만약 커밋이 끝나기 전에 락을 먼저 풀면, 다음 요청이 아직 커밋되지 않은 옛 데이터를 읽고 판단해 버리므로 정합성이 깨집니다. 특히 @Transactional 메서드 안에서 락을 잡고 풀면 실제 커밋은 메서드가 끝난 뒤 일어나기 때문에 이 순서가 어긋나기 쉽습니다. 그래서 락 안에서 커밋까지 마친 뒤 해제해야 하고, 정합성의 최종 방어선은 DB unique 제약·조건부 UPDATE·낙관적 락에 따로 두어야 합니다.

**Q. 락 해제 시 왜 본인이 획득한 락인지 확인해야 하나요?**

**A.** 내 작업이 GC pause 등으로 길어져 TTL이 먼저 만료되면, 그 사이 다른 요청이 정상적으로 새 락을 잡을 수 있습니다. 이 상태에서 내가 뒤늦게 깨어나 무조건 DEL로 락을 지우면, 내 것이 아니라 남이 잡은 락을 삭제하게 됩니다. 그러면 상호배제가 깨져 또 다른 요청까지 임계 구역에 진입하는 연쇄 문제가 생깁니다. 그래서 락의 value에 소유자 UUID를 넣어 두고, 해제할 때 "그 값이 내 것일 때만 삭제"하도록 해야 합니다. 이 비교와 삭제는 반드시 Lua로 원자적으로 실행해야, 비교와 삭제 사이에 다른 요청이 끼어드는 틈을 없앨 수 있습니다.

**Q. 결제·재고·쿠폰 도메인에서는 락 외에 무엇이 더 필요한가요?**

**A.** 결제·재고·쿠폰처럼 정합성이 돈과 직결되는 도메인에서는 락 하나에만 의존해서는 안 됩니다. 락은 "불필요한 동시 작업을 미리 줄여주는 장치"로만 쓰고, 정합성의 최종 방어선은 DB unique 제약, 조건부 UPDATE, 낙관적 락, 멱등성 키에 둡니다. 그 이유는 락은 TTL 경계나 늦은 쓰기 같은 상황에서 뚫릴 수 있는 반면, 이 층들은 커밋 시점에 DB가 직렬화하거나 중복 요청을 걸러내 실제로 데이터를 지켜주기 때문입니다. 즉 락은 성능·효율을 위한 앞단이고, 데이터가 깨지지 않게 하는 진짜 방어는 DB와 멱등성 층이 담당하는 역할 분담입니다. 그래서 올바른 사고방식은 "락이 있으니 안전하다"가 아니라 "락이 없어도 데이터가 안 깨지는데, 락으로 더 효율적으로 만든다"입니다.

**Q. Redis 분산락과 DB 비관적 락은 어떤 차이가 있나요?**

**A.** 둘은 락의 수명과 보장 방식이 다릅니다. DB 비관적 락(SELECT FOR UPDATE)은 트랜잭션과 수명이 정확히 일치해 커밋이나 롤백 시 자동으로 해제되므로, "락이 풀리는 시점"과 "데이터가 확정되는 시점"이 원자적으로 일치합니다. 덕분에 해제와 커밋의 순서가 어긋나는 문제가 원천적으로 발생하지 않습니다. 대신 락 대기와 부하가 DB에 집중되고, 보호할 수 있는 대상도 DB 안의 자원으로 한정됩니다. 반면 Redis 락은 DB 밖의 자원(외부 API 구간)이나 여러 저장소에 걸친 구간도 보호할 수 있고 DB 앞단에서 경합을 미리 줄여주지만, TTL 기반이라 보장이 약합니다. 그래서 단일 DB 안의 정합성 문제라면 DB 락·제약이 우선 검토 대상입니다.

**Q. Redis 락의 TTL은 어떻게 정해야 하나요?**

**A.** TTL은 두 방향의 요구를 동시에 만족해야 합니다. 한편으로는 정상 작업 시간의 최대치, 즉 p99에 GC 정지와 네트워크 지연 여유까지 더한 값보다 길어야 작업 도중에 락이 풀리지 않습니다. 다른 한편으로는 홀더가 죽었을 때 다른 요청이 오래 막히지 않도록, 장애 시 허용 가능한 잠금 시간보다는 짧아야 합니다. 이 두 조건이 서로 충돌하면 고정 TTL로는 답이 없습니다. 그럴 때는 Redisson watchdog처럼 "짧은 TTL을 걸고 작업이 진행되는 동안 주기적으로 갱신"하는 방식이 실용적인데, 정상 작업 중에는 계속 갱신되어 안 풀리고 홀더가 죽으면 갱신이 멈춰 빠르게 풀리기 때문입니다.

**Q. 락을 획득한 서버가 작업 중 죽으면 어떻게 되나요?**

**A.** 락을 획득한 서버가 작업 중에 죽어도, TTL이 만료되면 락이 자동으로 해제되어 다음 요청이 진행할 수 있습니다. 만약 TTL이 없다면 그 락은 아무도 풀어주지 못해 영구 데드락이 되므로, TTL은 분산락의 필수 요소입니다. 다만 여기서 주의할 점은 "락 해제"가 곧 "데이터 복구"는 아니라는 것입니다. 죽기 직전까지 진행하던 작업의 잔재는, 아직 커밋 전이라면 트랜잭션 롤백이 정리해 주고 이미 커밋된 뒤라면 멱등 재시도나 보상 트랜잭션으로 따로 정리해야 합니다. 즉 TTL은 락의 생존성만 회복시킬 뿐, 데이터 정합성은 별도 장치가 책임집니다.

**Q. 락이 만료된 뒤 기존 요청이 뒤늦게 DB를 업데이트하면 어떤 문제가 생기나요?**

**A.** 이것은 분산락의 가장 유명한 취약 지점입니다. 예를 들어 A가 TTL 10초로 락을 잡았는데 GC pause로 12초를 멈추면, 10초 시점에 락이 만료되어 B가 락을 잡고 작업을 시작합니다. 그런데 A가 깨어나 자기가 아직 락 홀더인 줄 알고 뒤늦게 DB에 쓰면, 결국 A와 B가 동시에 쓴 것과 같은 충돌 상태가 됩니다. 근본 해법은 쓰기 시점에 검증하는 것으로, 이론적 정답은 fencing token(락을 얻을 때마다 단조 증가하는 토큰을 받고, 저장소가 더 낮은 토큰의 쓰기를 거부하는 방식)입니다. 다만 이는 저장소의 협조가 필요하므로, 실무에서는 낙관적 락(version 비교)이나 조건부 UPDATE(`WHERE stock >= n`)로 "내가 읽었던 상태 그대로일 때만 쓰기"를 강제해 같은 효과를 냅니다.

**Q. 분산락을 썼는데도 멱등성 키가 필요한 이유는 무엇인가요?**

**A.** 락과 멱등성 키는 애초에 막는 대상이 다릅니다. 락은 "동시(같은 시각)"의 진입을 막고, 멱등성 키는 "순차적 중복(시간차를 두고 다시 실행되는 것)"을 막습니다. 더블클릭, 네트워크 재시도, MQ의 재전달 같은 상황은 각각 순서대로 락을 잡고 각각 "정상적으로" 두 번 실행됩니다. 락 입장에서는 동시 진입이 아니므로 아무 문제가 없는 흐름이고, 그래서 락만으로는 이 중복을 막을 수 없습니다. 같은 논리 작업이 두 번 반영되는 것을 막는 일은 요청 단위로 부여하는 멱등성 키의 몫이며, 결국 락과 멱등성은 경쟁 관계가 아니라 서로 다른 축을 담당하는 보완 관계입니다.

### 면접 답변 (구술용)

Redis 분산락이 보장하는 것은 정확히 하나 — 여러 인스턴스에 걸쳐 특정 임계 구역에 동시에 진입하는 것을 제한하는 상호배제입니다. 그 이상을 기대하는 순간 설계가 어긋납니다. 락을 잡아도 정합성이 깨지는 시나리오가 세 가지 있습니다. 첫째, TTL 만료 경계입니다 — 작업이 GC pause나 네트워크 지연으로 TTL보다 길어지면 락이 먼저 풀려 다른 요청이 진입하고, 원래 요청이 자기 락이 풀린 줄 모른 채 뒤늦게 DB를 쓰면 결국 동시에 쓴 것과 같아집니다. 둘째, 락 해제와 커밋의 순서입니다 — 커밋이 끝나기 전에 락을 풀면 다음 요청이 커밋 전 데이터를 읽고 판단하므로, 락 안에서 커밋까지 끝내고 해제해야 합니다. 셋째, 남의 락 해제입니다 — 내 TTL이 만료되고 남이 락을 잡은 상태에서 무조건 DEL하면 남의 락을 지우므로, 소유자 값을 넣고 Lua로 비교와 삭제를 원자적으로 해야 합니다. 그래서 결제·재고·쿠폰 도메인에서는 락을 "불필요한 동시 작업을 줄이는 장치"로 쓰고, 정합성의 최종 방어선은 DB unique 제약, 조건부 UPDATE, 낙관적 락, 멱등성 키에 둡니다. "락이 있으니 안전하다"가 아니라 "락이 없어도 안 깨지는데, 락으로 더 효율적이다"가 올바른 구조입니다.

### 원리 이해 (왜 그런가)

**락이 뚫리는 3대 시나리오:**

```
[1] TTL 만료 경계의 늦은 쓰기
A 락 획득(TTL 10s) → A가 GC pause로 12s 멈춤 → 10s에 락 만료 → B 획득·작업
→ A가 깨어나 "자기가 아직 락 홀더인 줄 알고" DB 쓰기 → A·B의 쓰기 충돌
※ 락은 이 경계 상황을 혼자 못 막는다 (Kleppmann의 Redlock 비판 지점, 해법 = fencing token)

[2] 락 해제 ↔ 커밋 순서 역전
락 안에서: 재고 읽기 → 차감 → (락 해제) → 커밋
→ 해제~커밋 사이에 다음 요청이 "옛 재고"를 읽고 또 차감
※ @Transactional 메서드 안에서 락을 잡고 풀면 커밋은 메서드 밖 — 순서가 깨지기 쉽다

[3] 남의 락 해제
내 TTL 만료 → B가 락 획득 → 내가 무조건 DEL → B의 락 삭제 → C까지 진입
※ 해법: value에 소유자 UUID + Lua로 compare-and-delete 원자 실행 (→ 260714)
```

**역할 분담 구조 (결론):**

| 층 | 담당 | 성격 |
|----|------|------|
| Redis 분산락 | 동시 "실행" 진입 억제 | 성능 최적화·부수효과 보호 — 경계 상황에서 뚫릴 수 있음 |
| DB 제약/조건부 UPDATE/낙관적 락 | 잘못된 "데이터" 차단 | 최종 방어선 — 커밋 시점에 DB가 직렬화, 뚫리지 않음 |
| 멱등성 키 | 시간차 "중복 실행" 차단 | 재시도·더블클릭·재전달 방어 (→ 260710) |

### 꼬리질문 Q&A

**Q. Redis 분산락과 DB 비관적 락(SELECT FOR UPDATE)의 차이는?**

**A.** **수명과 보장 방식이 다르다 — DB 락은 트랜잭션과 수명이 일치해 커밋/롤백 시 자동 해제되므로 "락 풀림과 데이터 확정"이 원자적으로 일치한다.**
그래서 위 시나리오 2가 원천적으로 없습니다. 대신 락 대기와 부하가 DB에 집중되고 DB 안의 자원만 보호합니다. Redis 락은 DB 밖 자원(외부 API 구간)이나 여러 저장소에 걸친 구간도 보호할 수 있고 DB 앞단에서 경합을 줄여주지만, TTL 기반이라 보장이 약합니다. 단일 DB 안의 정합성 문제라면 DB 락/제약이 우선 검토 대상입니다.

**Q. TTL은 어떻게 정해야 하나요?**

**A.** **정상 작업 시간의 최대치(p99 + GC/네트워크 여유)보다 길게, 장애 시 허용 가능한 잠금 시간보다 짧게.**
두 조건이 충돌하면 고정 TTL 대신 Redisson watchdog처럼 "짧은 TTL + 작업 중 주기 갱신" 방식이 실용적입니다 — 정상이면 안 풀리고, 죽으면 빨리 풀립니다(→ 260716).

**Q. 락을 획득한 서버가 작업 중 죽으면?**

**A.** **TTL 만료로 자동 해제되어 다음 요청이 진행한다 — TTL이 없으면 영구 데드락이므로 TTL은 필수다.**
단 "락 해제"가 "데이터 복구"는 아닙니다. 죽기 직전까지의 작업 잔재는 커밋 전이면 트랜잭션 롤백이, 커밋 후면 멱등 재시도나 보상이 정리합니다.

**Q. 락 만료 후 기존 요청이 뒤늦게 DB를 업데이트하면?**

**A.** **위 시나리오 1 — 두 요청이 동시에 쓴 것과 같은 상태가 된다.**
근본 해법은 쓰기 시점 검증입니다. 이론적 정답은 fencing token(락 획득마다 단조 증가 토큰을 받고, 저장소가 더 낮은 토큰의 쓰기를 거부)이지만 저장소 협조가 필요하므로, 실무에서는 낙관적 락(version 비교)이나 조건부 UPDATE(`WHERE stock >= n`)로 "내가 읽었던 상태 그대로일 때만 쓰기"를 강제하는 방식이 같은 효과를 냅니다.

**Q. 분산락을 썼는데도 멱등성 키가 필요한 이유는?**

**A.** **락과 멱등성은 막는 대상이 다르다 — 락은 "동시(같은 시각)" 진입, 멱등성은 "순차적 중복(시간차 재실행)".**
더블클릭, 네트워크 재시도, MQ 재전달은 각각 순서대로 락을 잡고 각각 "정상적으로" 두 번 실행됩니다 — 락 입장에서는 아무 문제 없는 흐름입니다. 같은 논리 작업의 이중 반영을 막는 것은 요청 단위 멱등성 키(→ 260710)의 몫입니다.

### ❓ 더 파볼 질문

**Q. ZooKeeper나 etcd 기반 분산락은 Redis 락과 뭐가 다른가?**

**A.** 합의(quorum) 기반 저장소라 failover에도 락 상태가 유실되지 않고, 세션·임시 노드(ephemeral node)로 홀더의 생사를 감지해 TTL 추정 없이 락을 회수할 수 있다. 대신 쓰기 지연이 크고 운영이 무겁다 — "정확성이 최우선이면 합의 기반, 성능·간편함이 우선이면 Redis"라는 구도다.

**Q. DB 락만으로 충분해서 Redis 락이 아예 불필요한 경우는 언제인가?**

**A.** 보호 대상이 단일 DB 안에 있고 트랜잭션이 짧으면 SELECT FOR UPDATE가 더 단순하고 강하다. Redis 락의 적정 용도는 "DB 앞에서 경합을 미리 줄이고 싶거나, 보호 대상이 DB 밖(외부 API·파일)일 때"다.

**Q. 락 없이 설계할 수 있는 경우는?**

**A.** 조건부 UPDATE(`UPDATE stock SET qty = qty - 1 WHERE id = ? AND qty >= 1`)와 unique 제약만으로 동시성이 제어되는 경우가 생각보다 많다 — DB의 row lock이 이미 직렬화를 해주기 때문이다. "락을 추가하기 전에, DB가 이미 주는 직렬화로 충분한지"를 먼저 묻는 것이 좋은 설계 순서다.
