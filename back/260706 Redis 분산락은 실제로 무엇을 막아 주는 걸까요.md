# 260704 Redis 분산락은 실제로 무엇을 막아 주는 걸까요?

[오늘의 백엔드 질문]
Redis 분산락은 실제로 무엇을 막아 주는 걸까요?

면접에서 Redis 분산락 이야기가 나오면 많은 분들이
“동시성 문제를 해결하기 위해 사용합니다”라고 답변을 시작합니다.
그런데 면접관이 더 궁금해하는 건
Redis 분산락이 모든 정합성 문제를 해결해 주는지,
아니면 특정 구간에 동시에 진입하는 것을 제한하는 도구인지를 구분하고 있는지입니다.
이 질문은 단순히 Redis 락 구현 방법을 외웠는지보다,
락이 보장하는 범위와 보장하지 못하는 영역을 이해하고 있는지를 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- Redis 분산락은 특정 critical section에 동시에 들어오는 요청을 제한한다는 점
- 락을 잡았다고 DB 트랜잭션 정합성이 자동으로 보장되는 것은 아니라는 점
- 락 만료 시간, 작업 시간, 네트워크 지연이 맞물리면 락이 먼저 풀릴 수 있다는 점
- 락 해제 시 본인이 획득한 락인지 확인해야 한다는 점
- 결제, 재고, 쿠폰 같은 도메인에서는 락뿐 아니라 멱등성, DB 제약조건, 트랜잭션 설계가 함께 필요하다는 점

꼬리질문으로는 이런 질문이 이어질 수 있습니다.
ㄴ> Redis 분산락과 DB 비관적 락은 어떤 차이가 있나요?
ㄴ> Redis 락의 TTL은 어떻게 정해야 하나요?
ㄴ> 락을 획득한 서버가 작업 중 죽으면 어떻게 되나요?
ㄴ> 락이 만료된 뒤 기존 요청이 뒤늦게 DB를 업데이트하면 어떤 문제가 생기나요?
ㄴ> 분산락을 썼는데도 멱등성 키가 필요한 이유는 무엇인가요?

---

## 답변

> **한 줄 핵심**: Redis 분산락이 보장하는 것은 "여러 인스턴스에 걸친 임계 구역 동시 진입 제한(상호배제)" 하나뿐이다 — 트랜잭션 정합성·롤백·시간차 중복은 못 막으므로, 최종 방어선은 "락이 뚫려도 데이터가 안 깨지는 층"(unique 제약·조건부 UPDATE·멱등성)에 둬야 한다.

### 면접 답변 (구술용)

Redis 분산락이 보장하는 것은 정확히 하나 — 여러 인스턴스에 걸쳐 특정 임계 구역에 동시에 진입하는 것을 제한하는 상호배제입니다. 그 이상을 기대하는 순간 설계가 어긋납니다. 락을 잡아도 정합성이 깨지는 시나리오가 세 가지 있습니다. 첫째, TTL 만료 경계입니다 — 작업이 GC pause나 네트워크 지연으로 TTL보다 길어지면 락이 먼저 풀려 다른 요청이 진입하고, 원래 요청이 자기 락이 풀린 줄 모른 채 뒤늦게 DB를 쓰면 결국 동시에 쓴 것과 같아집니다. 둘째, 락 해제와 커밋의 순서입니다 — 커밋이 끝나기 전에 락을 풀면 다음 요청이 커밋 전 데이터를 읽고 판단하므로, 락 안에서 커밋까지 끝내고 해제해야 합니다. 셋째, 남의 락 해제입니다 — 내 TTL이 만료되고 남이 락을 잡은 상태에서 무조건 DEL하면 남의 락을 지우므로, 소유자 값을 넣고 Lua로 비교와 삭제를 원자적으로 해야 합니다. 그래서 결제·재고·쿠폰 도메인에서는 락을 "불필요한 동시 작업을 줄이는 장치"로 쓰고, 정합성의 최종 방어선은 DB unique 제약, 조건부 UPDATE, 낙관적 락, 멱등성 키에 둡니다. "락이 있으니 안전하다"가 아니라 "락이 없어도 안 깨지는데, 락으로 더 효율적이다"가 올바른 구조입니다.

### 원리 이해 (왜 그런가)

**락이 뚫리는 3대 시나리오:**

```
[1] TTL 만료 경계의 늦은 쓰기
A 락 획득(TTL 10s) → A가 GC pause로 12s 멈춤 → 10s에 락 만료 → B 획득·작업
→ A가 깨어나 "자기가 아직 락 홀더인 줄 알고" DB 쓰기 → A·B의 쓰기 충돌
※ 락은 이 경계 상황을 혼자 못 막는다 (Kleppmann의 Redlock 비판 지점, 해법 = fencing token)

[2] 락 해제 ↔ 커밋 순서 역전
락 안에서: 재고 읽기 → 차감 → (락 해제) → 커밋
→ 해제~커밋 사이에 다음 요청이 "옛 재고"를 읽고 또 차감
※ @Transactional 메서드 안에서 락을 잡고 풀면 커밋은 메서드 밖 — 순서가 깨지기 쉽다

[3] 남의 락 해제
내 TTL 만료 → B가 락 획득 → 내가 무조건 DEL → B의 락 삭제 → C까지 진입
※ 해법: value에 소유자 UUID + Lua로 compare-and-delete 원자 실행 (→ 260714)
```

**역할 분담 구조 (결론):**

| 층 | 담당 | 성격 |
|----|------|------|
| Redis 분산락 | 동시 "실행" 진입 억제 | 성능 최적화·부수효과 보호 — 경계 상황에서 뚫릴 수 있음 |
| DB 제약/조건부 UPDATE/낙관적 락 | 잘못된 "데이터" 차단 | 최종 방어선 — 커밋 시점에 DB가 직렬화, 뚫리지 않음 |
| 멱등성 키 | 시간차 "중복 실행" 차단 | 재시도·더블클릭·재전달 방어 (→ 260710) |

### 꼬리질문 Q&A

**Q. Redis 분산락과 DB 비관적 락(SELECT FOR UPDATE)의 차이는?**
→ **수명과 보장 방식이 다르다 — DB 락은 트랜잭션과 수명이 일치해 커밋/롤백 시 자동 해제되므로 "락 풀림과 데이터 확정"이 원자적으로 일치한다.**
그래서 위 시나리오 2가 원천적으로 없습니다. 대신 락 대기와 부하가 DB에 집중되고 DB 안의 자원만 보호합니다. Redis 락은 DB 밖 자원(외부 API 구간)이나 여러 저장소에 걸친 구간도 보호할 수 있고 DB 앞단에서 경합을 줄여주지만, TTL 기반이라 보장이 약합니다. 단일 DB 안의 정합성 문제라면 DB 락/제약이 우선 검토 대상입니다.

**Q. TTL은 어떻게 정해야 하나요?**
→ **정상 작업 시간의 최대치(p99 + GC/네트워크 여유)보다 길게, 장애 시 허용 가능한 잠금 시간보다 짧게.**
두 조건이 충돌하면 고정 TTL 대신 Redisson watchdog처럼 "짧은 TTL + 작업 중 주기 갱신" 방식이 실용적입니다 — 정상이면 안 풀리고, 죽으면 빨리 풀립니다(→ 260716).

**Q. 락을 획득한 서버가 작업 중 죽으면?**
→ **TTL 만료로 자동 해제되어 다음 요청이 진행한다 — TTL이 없으면 영구 데드락이므로 TTL은 필수다.**
단 "락 해제"가 "데이터 복구"는 아닙니다. 죽기 직전까지의 작업 잔재는 커밋 전이면 트랜잭션 롤백이, 커밋 후면 멱등 재시도나 보상이 정리합니다.

**Q. 락 만료 후 기존 요청이 뒤늦게 DB를 업데이트하면?**
→ **위 시나리오 1 — 두 요청이 동시에 쓴 것과 같은 상태가 된다.**
근본 해법은 쓰기 시점 검증입니다. 이론적 정답은 fencing token(락 획득마다 단조 증가 토큰을 받고, 저장소가 더 낮은 토큰의 쓰기를 거부)이지만 저장소 협조가 필요하므로, 실무에서는 낙관적 락(version 비교)이나 조건부 UPDATE(`WHERE stock >= n`)로 "내가 읽었던 상태 그대로일 때만 쓰기"를 강제하는 방식이 같은 효과를 냅니다.

**Q. 분산락을 썼는데도 멱등성 키가 필요한 이유는?**
→ **락과 멱등성은 막는 대상이 다르다 — 락은 "동시(같은 시각)" 진입, 멱등성은 "순차적 중복(시간차 재실행)".**
더블클릭, 네트워크 재시도, MQ 재전달은 각각 순서대로 락을 잡고 각각 "정상적으로" 두 번 실행됩니다 — 락 입장에서는 아무 문제 없는 흐름입니다. 같은 논리 작업의 이중 반영을 막는 것은 요청 단위 멱등성 키(→ 260710)의 몫입니다.

### ❓ 더 파볼 질문
- **ZooKeeper나 etcd 기반 분산락은 Redis 락과 뭐가 다른가?**
  ↳ 합의(quorum) 기반 저장소라 failover에도 락 상태가 유실되지 않고, 세션·임시 노드(ephemeral node)로 홀더의 생사를 감지해 TTL 추정 없이 락을 회수할 수 있다. 대신 쓰기 지연이 크고 운영이 무겁다 — "정확성이 최우선이면 합의 기반, 성능·간편함이 우선이면 Redis"라는 구도다.
- **DB 락만으로 충분해서 Redis 락이 아예 불필요한 경우는 언제인가?**
  ↳ 보호 대상이 단일 DB 안에 있고 트랜잭션이 짧으면 SELECT FOR UPDATE가 더 단순하고 강하다. Redis 락의 적정 용도는 "DB 앞에서 경합을 미리 줄이고 싶거나, 보호 대상이 DB 밖(외부 API·파일)일 때"다.
- **락 없이 설계할 수 있는 경우는?**
  ↳ 조건부 UPDATE(`UPDATE stock SET qty = qty - 1 WHERE id = ? AND qty >= 1`)와 unique 제약만으로 동시성이 제어되는 경우가 생각보다 많다 — DB의 row lock이 이미 직렬화를 해주기 때문이다. "락을 추가하기 전에, DB가 이미 주는 직렬화로 충분한지"를 먼저 묻는 것이 좋은 설계 순서다.
