## 답변

> **한 줄 핵심**: 살아있는 객체의 **이동(compaction)을 G1은 STW 안에서, ZGC는 애플리케이션 실행과 동시에** 한다 — 그래서 G1의 pause는 힙·라이브셋에 비례해 늘 수 있고, ZGC의 pause는 밀리초 미만으로 힙 크기와 무관하다.

### 면접 답변 (구술용)

GC는 크게 마킹, 객체 이동, 참조 갱신 단계로 이루어지는데, 마킹은 두 GC 모두 대부분 동시적으로 합니다. 근본 차이는 이동을 어디서 하느냐입니다. G1은 힙을 수천 개 region으로 나누고 가비지가 많은 region부터 수거해서 pause 목표(기본 200ms)를 맞추려 하지만, 살아있는 객체를 다른 region으로 복사하는 evacuation은 STW 중에 합니다. 그래서 힙이 크고 살아있는 객체가 많을수록 pause가 길어질 수 있습니다. ZGC는 colored pointer와 load barrier라는 장치로 객체 이동을 애플리케이션과 동시에 수행합니다 — 이동 중인 객체를 애플리케이션이 읽으면 barrier가 그 자리에서 새 주소로 교정해 줍니다. 그 결과 STW가 밀리초 미만이고 힙 크기와 무관합니다. 대신 ZGC는 모든 참조 읽기에 barrier 비용을 내고 동시 GC가 CPU를 나눠 쓰므로 처리량은 다소 손해입니다. 그래서 수십 GB 힙에서 p99 지연이 중요한 서비스는 ZGC, 일반 웹 서비스는 G1이 기본 선택입니다.

### 원리 이해 (왜 그런가)

**비교표 (구조 → 결과):**

| | G1GC | ZGC |
|---|---|---|
| 설계 목표 | pause를 **예측 가능**하게 (`-XX:MaxGCPauseMillis` 기본 200ms) | pause를 **힙 크기와 무관한 상수**로 (sub-ms) |
| 힙 구조 | region 분할, Young/Old 역할 동적 배정 | region 유사 구조 + colored pointer |
| 마킹 | 대부분 동시(concurrent) | 동시 |
| **객체 이동** | **STW evacuation** (복사 중 접근 차단 필요) | **동시 relocation** (barrier가 접근을 교정) |
| pause 결정 요인 | 이동할 라이브셋 크기 | 힙 크기 무관, 상수 수준 |
| 비용 | 상대적으로 낮은 상시 오버헤드 | 모든 참조 읽기에 load barrier + 동시 GC의 CPU 사용 |
| 적합 | 일반 웹 서비스(JDK 9+ 기본 GC) | 대형 힙 + 엄격한 지연 SLA |

**ZGC의 두 장치 (동시 이동이 가능한 이유):**
- **Colored pointer** — 64비트 포인터의 여분 비트에 GC 메타데이터(마킹/재배치 상태)를 심는다. "포인터 자체가 상태를 안다."
- **Load barrier** — JIT가 참조 읽기 지점마다 삽입하는 짧은 검사 코드. 나쁜 색(이동됐거나 미처리) 참조를 읽으면 그 자리에서 새 주소로 교정(self-healing) 후 반환. **큰 멈춤 하나를 모든 읽기에 잘게 나눠 내는 발상.**
- JDK 21의 Generational ZGC(JEP 439)가 세대 구분을 도입해 처리량 약점을 상당히 개선.

### 꼬리질문 Q&A

**Q. G1의 Region 구조는 기존 Young/Old 구조와 어떻게 다른가?**
→ **고정·연속 영역이 아니라 region 단위로 역할을 동적 배정 — "전체 Old를 한 번에" 대신 "이득 큰 region 몇 개만" 선택 수거가 가능해진다.**
- 이 선택적 수거가 pause 목표 관리의 전제 조건 (Garbage-First라는 이름의 유래).

**Q. Load Barrier란?**
→ **힙 참조를 읽을 때마다 실행되는 삽입 코드 — colored pointer의 비트를 보고 필요하면 교정 후 반환.**

**Q. STW가 길어지면 서비스 영향은?**
→ **모든 응답 일괄 지연(p99 스파이크) → 업스트림 타임아웃·재시도 폭주 → 헬스체크 실패 → LB 제외 → 남은 인스턴스로 부하 집중, 증폭 연쇄.**

**Q. GC 튜닝 시 먼저 볼 지표는?**
→ **문제 정의 먼저: 지연 문제면 pause 분포(p99/최대 — 평균은 스파이크를 숨김), 처리량 문제면 GC CPU 시간 비율.**
- 이어서 GC 빈도, GC 후 생존 힙(라이브셋 — 우상향이면 누수 의심), promotion 비율, Full GC 유무(있으면 원인 제거가 최우선).

**Q. 메모리를 늘리면 GC 문제가 항상 해결되나?**
→ **아니오 — 빈도는 줄지만 한 번의 수거량이 늘어 pause가 길어질 수 있다(특히 STW 복사를 하는 G1). "빈도 ↔ 길이" 트레이드오프지 총비용 제거가 아니다.**
- 누수면 시점만 미루고, 라이브셋이 큰 워크로드면 GC 교체(ZGC)나 할당 패턴 개선이 맞는 처방.

### 🌱 심화 키워드
- **evacuation / relocation** — "객체 이동"의 G1/ZGC 용어, 이 질문의 축
- **SATB(Snapshot-At-The-Beginning)** — G1 동시 마킹의 일관성 유지 기법
- **remembered set / card table** — region 간 참조 추적 (부분 수거를 가능케 하는 자료구조)
- **humongous object** — region 크기 절반을 넘는 객체가 G1에서 일으키는 문제
- **Generational ZGC(JEP 439)** — 약한 세대 가설을 ZGC에 적용해 처리량 개선

### ❓ 더 파볼 질문
- **Shenandoah는 같은 목표를 브룩스 포인터로 푸는데, ZGC의 colored pointer와 무엇이 다른가?**
  ↳ Shenandoah(전통 방식)는 객체마다 forwarding pointer 공간을 둬서 이동 시 새 주소를 가리키게 하는 간접 참조 방식이었고, ZGC는 포인터 자체의 여분 비트에 상태를 심어 barrier가 판단한다. 메모리 오버헤드의 위치(객체별 포인터 vs 주소 비트)와 barrier 설계가 다르며, 둘 다 "동시 이동"이라는 같은 목표의 다른 구현이라는 점이 비교 포인트다(Shenandoah도 이후 버전에서 방식을 개선했다 — 세부는 버전 의존).
- **G1에서 humongous 할당이 많으면 왜 Full GC로 이어지기 쉬운가?**
  ↳ region 절반을 넘는 객체는 연속된 region들을 통째로 점유하는 humongous 객체로 처리되는데, 연속 공간을 요구하므로 힙에 빈 region이 흩어져 있으면(단편화) 할당에 실패한다. 이 실패가 반복되면 G1은 전체를 압축하는 Full GC로 후퇴한다 — "큰 배열·큰 컬렉션을 반복 할당하는 코드"가 G1에서 특히 해로운 이유다.
- **"GC 로그에서 pause는 짧은데 응답 지연이 크다"면 무엇을 의심해야 하나?**
  ↳ GC pause에 잡히지 않는 멈춤 요인들 — 대표적으로 safepoint 도달 대기(TTSP: 모든 스레드가 멈추길 기다리는 시간, 긴 루프·JNI가 늘림), 그 외 스왑/페이지 폴트, JIT 디옵티마이제이션, 커널·노이지 네이버 문제다. "STW 시간 = GC 로그의 pause"가 아니라 "safepoint 대기 + pause"라는 걸 알면 진단 폭이 넓어진다.
