# 260623 @Transactional과 @Async를 함께 사용할 때 주의할 점은 무엇일까요?

@Transactional과 @Async를 함께 사용할 때 주의할 점은 무엇일까요?

취준생/주니어 이력서를 보다 보면
“비동기 처리로 성능 개선”, “트랜잭션 분리”, “이벤트 기반 처리” 같은 표현이 자주 등장합니다.

그런데 면접에서 막상 물어보면
@Transactional과 @Async가 각각 어떻게 동작하는지,
둘을 같이 썼을 때 트랜잭션이 어디까지 유지되는지 헷갈려 하는 경우가 꽤 많습니다.

이 질문은 단순히 어노테이션 사용법을 외웠는지보다,
Spring AOP, ThreadLocal, 트랜잭션 컨텍스트, 비동기 실행 흐름을 이해하고 있는지 확인하는 질문에 가깝습니다.

같이 체크해보면 좋은 포인트는 아래와 같습니다.

- @Transactional이 Spring AOP 프록시 기반으로 동작한다는 점
- 트랜잭션 컨텍스트가 보통 현재 스레드에 묶인다는 점
- @Async는 별도 스레드에서 메서드를 실행한다는 점
- 내부 호출/self-invocation 시 AOP가 적용되지 않을 수 있다는 점
- 비동기 작업 실패 시 롤백, 예외 처리, 재처리를 어떻게 설계할지

꼬리질문으로는 이런 질문이 이어질 수 있습니다.

- @Transactional은 내부적으로 어떻게 동작하나요?
- @Async는 어떤 Executor를 사용하나요?
- 트랜잭션 안에서 비동기 메서드를 호출하면 같은 트랜잭션이 유지되나요?
- 비동기 작업에서 예외가 발생하면 기존 트랜잭션은 롤백되나요?
- 비동기 처리가 꼭 필요하다면 Outbox Pattern이나 Message Queue를 고려할 수 있을까요?

---

## 답변

> **한 줄 핵심**: 트랜잭션 컨텍스트는 ThreadLocal(스레드)에 묶여 있는데 @Async는 다른 스레드에서 실행된다 — 모든 주의점(트랜잭션 미전파, 롤백 안 됨, 커밋 전 데이터 안 보임)은 이 한 문장에서 파생된다.

### 1문 1답

**Q. @Transactional과 @Async를 함께 사용할 때 주의할 점은 무엇일까요?**

**A.** 핵심은 트랜잭션 컨텍스트가 ThreadLocal에 저장되는데 @Async는 메서드를 다른 스레드에서 실행한다는 점이고, 주의점 대부분이 여기서 파생된다. 첫째, 트랜잭션이 전파되지 않아 비동기 메서드는 호출자의 트랜잭션에 참여하지 않고 자체 @Transactional이 있으면 완전히 새 트랜잭션을 연다. 둘째, 비동기 작업의 예외가 다른 스레드에서 터지므로 호출자를 롤백시키지 못하고, void 반환이면 예외가 조용히 사라질 수도 있다. 셋째, 호출자가 커밋하기 전에 비동기 작업이 같은 데이터를 조회하면 변경 전 데이터를 읽어서, 커밋 이후 실행이 필요하면 AFTER_COMMIT 이벤트를 써야 한다. 넷째, 둘 다 프록시 기반이라 같은 클래스 안에서 this로 호출하면 어노테이션이 무시된다. 마지막으로 비동기 작업은 성패가 분리된 독립 작업이라 실패 시 재시도·이력·보상을 명시적으로 설계해야 하고, 유실이 허용되지 않으면 @Async 대신 Outbox와 메시지 큐로 가는 것이 맞다.

**Q. @Transactional이 Spring AOP 프록시 기반으로 동작한다는 것은 무슨 뜻인가요?**

**A.** @Transactional은 어노테이션 자체가 아니라 Spring이 만들어 주는 AOP 프록시로 동작한다. 빈 등록 시점에 프록시가 생성되고, 이 프록시가 메서드 호출을 가로채 PlatformTransactionManager로 커넥션을 얻은 뒤 autocommit을 끄고 그 트랜잭션 상태를 ThreadLocal에 바인딩한 다음 실제 메서드를 실행한다. 정상적으로 끝나면 커밋하고, 롤백 규칙에 해당하는 예외가 올라오면 롤백한다. 즉 "프록시가 호출을 가로채야 트랜잭션이 시작된다"는 것이 핵심이고, 이 성질이 곧 self-invocation이 무효인 이유와 "예외가 프록시 경계를 지나가야 롤백된다"는 성질의 공통 근거가 된다.

**Q. 트랜잭션 컨텍스트가 현재 스레드에 묶인다는 것은 무슨 의미인가요?**

**A.** Spring은 현재 트랜잭션의 커넥션과 동기화 상태를 TransactionSynchronizationManager의 ThreadLocal에 바인딩한다. ThreadLocal은 이름 그대로 스레드별 저장소이므로, 지금 이 스레드에서 시작한 트랜잭션은 다른 스레드에서는 보이지 않는다. 그래서 "트랜잭션 안에서 호출했으니 당연히 같은 트랜잭션이겠지"라는 직관이 스레드가 바뀌는 순간 구조적으로 깨진다. @Async와 함께 쓸 때 생기는 문제들 — 트랜잭션 미전파, 롤백 안 됨, 커밋 전 데이터 미가시 — 이 모두 이 "트랜잭션이 스레드에 묶여 있다"는 한 문장에서 파생된다. 결국 트랜잭션의 경계는 코드의 호출 관계가 아니라 스레드를 따라간다.

**Q. @Async가 별도 스레드에서 메서드를 실행한다는 점은 어떤 영향을 주나요?**

**A.** @Async는 메서드를 호출자와 다른 스레드에서 실행하는데, 트랜잭션 컨텍스트는 호출자 스레드의 ThreadLocal에만 있으므로 새 스레드에는 그 바인딩이 없다. 그래서 첫째로 트랜잭션이 전파되지 않아 비동기 메서드는 호출자의 트랜잭션에 참여하지 못하고, 자체 @Transactional이 있으면 완전히 새로운 트랜잭션을 시작한다. 둘째로 비동기 작업에서 난 예외는 호출자의 콜스택을 지나가지 않아 호출자를 롤백시키지 못한다. 셋째로 호출자가 아직 커밋하기 전이라면 비동기 작업이 같은 데이터를 조회해도 변경 전 값을 읽는다. 이 세 가지가 모두 "실행 스레드가 다르다"는 한 가지 사실에서 나온다.

**Q. 내부 호출(self-invocation) 시 AOP가 적용되지 않을 수 있는 이유는 무엇인가요?**

**A.** @Transactional과 @Async는 둘 다 프록시가 호출을 가로채야 동작하는데, 같은 클래스 안에서 this.method()로 부르면 프록시를 거치지 않고 실제 객체를 직접 호출하게 된다. 프록시를 우회했으니 트랜잭션 시작이나 비동기 전환 같은 부가 로직이 끼어들 틈이 없어, 어노테이션이 조용히 무시된다. 문제는 컴파일 에러도 예외도 나지 않아서 "붙였는데 왜 안 되지"로 헤매기 쉽다는 점이다. 해결은 호출 대상을 별도 빈으로 분리해 외부에서 프록시를 거쳐 호출되게 하거나 구조를 재설계하는 것이다.

**Q. 비동기 작업 실패 시 롤백·예외 처리·재처리는 어떻게 설계해야 하나요?**

**A.** 비동기 작업은 호출자와 성패가 분리된 독립 작업이라, 실패해도 호출자가 알아서 되돌려 주지 않으므로 재시도·이력·보상을 명시적으로 설계해야 한다. 특히 void를 반환하는 @Async 메서드에서 예외가 나면 AsyncUncaughtExceptionHandler로 흘러가 기본적으로 로그만 남고 사라져, "실패했는데 아무도 모르는" 상태가 되기 쉽다. 그래서 결과가 중요하면 Future나 CompletableFuture로 결과를 받아 실패 이력을 저장하는 편이 안전하다. 더 근본적으로 @Async의 작업 큐는 JVM 메모리에 있어 서버가 재시작되면 큐에 있던 작업이 그냥 사라진다는 한계가 있다. 따라서 유실이 허용되지 않는 작업이라면 @Async 대신 Outbox와 메시지 큐로 가는 것이 맞다.

**Q. @Async는 어떤 Executor를 사용하나요?**

**A.** 지정하지 않으면 Spring Boot가 자동 구성한 ThreadPoolTaskExecutor(빈 이름 applicationTaskExecutor)를 사용한다. 문제는 이 기본 설정이 core 스레드 8에 큐 용량이 사실상 무제한이라는 점이다. 부하가 몰리면 큐가 무한히 자라며 메모리를 압박하고, 큐가 가득 차야 만들어지는 max 스레드는 사실상 영원히 생성되지 않는다. 그래서 운영에서는 풀 크기·큐 용량·거부 정책을 명시한 전용 Executor를 만들어 @Async("빈이름")으로 지정하는 것이 안전하다. 기본값에 의존하면 평소엔 멀쩡하다가 트래픽이 몰릴 때 문제가 드러나기 때문이다.

**Q. 트랜잭션 안에서 비동기 메서드를 호출하면 같은 트랜잭션이 유지되나요?**

**A.** 아니오, 유지될 수 없다. 트랜잭션은 ThreadLocal에 묶여 있는데 @Async가 배정한 실행 스레드는 호출자와 다른 스레드이기 때문이다. 비동기 메서드에 @Transactional이 붙어 있다면 그건 호출자 트랜잭션에 참여하는 것이 아니라 완전히 새로운 트랜잭션을 여는 것이다. 그 결과 두 트랜잭션은 서로의 커밋과 롤백을 전혀 알지 못하고 독립적으로 성패가 갈린다. "같은 메서드 흐름 안에서 불렀다"는 코드상의 관계는 트랜잭션 경계와 무관하다.

**Q. 비동기 작업에서 예외가 발생하면 기존 트랜잭션은 롤백되나요?**

**A.** 아니오. 롤백은 예외가 프록시 경계를 지나갈 때 트리거되는데, 비동기 작업의 예외는 다른 스레드에서 터지므로 호출자 스레드의 콜스택을 지나가지 않기 때문이다. 예외적으로 호출자가 Future.get()으로 결과를 기다리다 ExecutionException을 받아 다시 던지면 롤백되긴 한다. 하지만 그렇게 블로킹해서 결과를 기다릴 거라면 애초에 비동기로 만든 의미가 사라지는 모순이 생긴다. 그래서 "비동기 작업의 실패가 본 작업을 취소해야 한다"는 요구가 있다면 @Async는 그 요구에 맞는 도구가 아니다.

**Q. 비동기 처리가 꼭 필요하다면 Outbox Pattern이나 Message Queue를 고려할 수 있을까요?**

**A.** 예, 다만 기준은 "그 작업이 유실돼도 되는가"다. 캐시 워밍처럼 유실돼도 그만인 작업이면 @Async로 충분하다. 그러나 알림 발송이나 정산 연동처럼 유실이 허용되지 않는 작업이면 Outbox + MQ가 맞는데, MQ는 메시지를 브로커에 영속화해 소비 실패 시 재시도와 DLQ가 가능한 반면 @Async의 인메모리 큐는 서버가 죽으면 그대로 소멸하기 때문이다. 여기에 트랜잭션 정합성까지, 즉 "커밋된 것만 발행"을 보장해야 한다면 Outbox 패턴으로 상태 변경과 이벤트 기록을 같은 트랜잭션에 묶는다. 결국 유실 허용 여부가 @Async와 MQ를 가르는 판단 기준이 된다.

### 면접 답변 (구술용)

핵심은 트랜잭션 컨텍스트가 ThreadLocal에 저장되는데 @Async는 메서드를 다른 스레드에서 실행한다는 점입니다. 그래서 첫째, 트랜잭션이 전파되지 않습니다 — 비동기 메서드는 호출자의 트랜잭션에 참여하지 않고, 자체 @Transactional이 있으면 완전히 새로운 트랜잭션을 시작합니다. 둘째, 비동기 작업의 예외가 호출자를 롤백시키지 못합니다 — 예외가 다른 스레드에서 터지므로 호출자의 트랜잭션 경계를 지나가지 않고, void 반환이면 예외가 조용히 사라질 수도 있습니다. 셋째, 호출자가 커밋하기 전에 비동기 작업이 같은 데이터를 조회하면 변경 전 데이터를 읽습니다 — 커밋 이후 실행이 필요하면 AFTER_COMMIT 이벤트를 써야 합니다. 넷째, 둘 다 프록시 기반이라 같은 클래스 안에서 this로 호출하면 어노테이션이 조용히 무시됩니다. 마지막으로 비동기 작업은 호출자와 성패가 분리된 독립 작업이므로 실패 시 재시도·이력·보상을 명시적으로 설계해야 하고, 유실이 허용되지 않는 작업이라면 @Async 대신 Outbox와 메시지 큐로 가는 것이 맞습니다.

### 원리 이해 (왜 그런가)

**문제의 뿌리 — 트랜잭션과 스레드의 결합:**

Spring은 현재 트랜잭션(커넥션, 동기화 상태)을 `TransactionSynchronizationManager`의 ThreadLocal에 바인딩합니다. ThreadLocal은 이름 그대로 스레드별 저장소이므로, @Async가 배정한 새 스레드에는 그 바인딩이 존재하지 않습니다. "트랜잭션 안에서 호출했으니 당연히 같은 트랜잭션이겠지"라는 직관이 구조적으로 성립하지 않는 이유입니다.

**파생되는 문제 4가지 정리:**

| # | 문제 | 왜 생기나 | 대응 |
|---|------|-----------|------|
| 1 | 트랜잭션 미전파 | 새 스레드에는 ThreadLocal 바인딩이 없음 | 비동기 메서드는 독립 트랜잭션으로 설계 |
| 2 | 예외가 롤백을 못 일으킴 | 롤백은 예외가 프록시 경계를 "지나갈 때" 발생 — 다른 스레드의 예외는 호출자 콜스택을 안 지나감. void면 `AsyncUncaughtExceptionHandler`(기본: 로그만)로 흘러가 "실패했는데 아무도 모름" | Future/CompletableFuture로 결과 수신, 실패 이력 저장 |
| 3 | 커밋 전 데이터 미가시 | 비동기 작업은 별도 커넥션 — READ_COMMITTED에서 커밋 전 변경은 남에게 안 보임 | `@TransactionalEventListener(phase = AFTER_COMMIT)`으로 커밋 후 실행 |
| 4 | self-invocation 무효 | 프록시가 호출을 가로채야 동작하는데 `this.method()`는 프록시를 우회 | 별도 빈으로 분리, 또는 구조 재설계 |

**@Async의 근본 한계 — 유실 가능성:** @Async의 작업 큐는 JVM 메모리에 있습니다. 서버가 재시작되면 큐에 있던 작업이 그냥 사라지고, 실패 재시도도 직접 만들어야 합니다. 그래서 "유실돼도 되는가?"가 @Async vs MQ의 판단 기준이 됩니다.

### 꼬리질문 Q&A

**Q. @Transactional은 내부적으로 어떻게 동작하나요?**

**A.** **빈 등록 시 프록시가 생성되고, 프록시가 메서드 호출을 가로채 트랜잭션을 시작·커밋·롤백한다.**
구체적으로는 호출을 가로챈 프록시가 `PlatformTransactionManager`를 통해 커넥션을 얻고 autocommit을 끄고 ThreadLocal에 바인딩한 뒤 실제 메서드를 실행하고, 정상 종료면 커밋, 롤백 규칙에 해당하는 예외면 롤백합니다. "프록시가 가로챈다"는 이 메커니즘이 self-invocation 무효와 "예외가 경계를 지나가야 롤백된다"는 성질의 공통 근거입니다.

**Q. @Async는 어떤 Executor를 사용하나요?**

**A.** **지정하지 않으면 Spring Boot가 자동 구성한 ThreadPoolTaskExecutor(빈 이름 applicationTaskExecutor)를 쓴다.**
문제는 Boot 기본 설정이 core 8에 큐 용량 무제한이라는 점입니다(spring.task.execution.* 기본값 기준 — 거의 확실). 부하가 몰리면 큐가 무한히 자라며 메모리를 압박하고, 큐가 차야 동작하는 max 스레드는 사실상 영원히 안 만들어집니다. 그래서 운영에서는 풀 크기·큐 용량·거부 정책을 명시한 전용 Executor를 만들어 `@Async("빈이름")`으로 지정하는 것이 안전합니다.

**Q. 트랜잭션 안에서 비동기 메서드를 호출하면 같은 트랜잭션이 유지되나요?**

**A.** **아니오. 트랜잭션은 ThreadLocal에 묶여 있고 실행 스레드가 다르므로 유지될 수 없다.**
비동기 메서드에 @Transactional이 붙어 있다면 그건 참여가 아니라 완전히 새로운 트랜잭션입니다. 두 트랜잭션은 서로의 커밋/롤백을 모릅니다.

**Q. 비동기 작업에서 예외가 발생하면 기존 트랜잭션은 롤백되나요?**

**A.** **아니오. 예외가 호출자 스레드의 콜스택을 지나가지 않기 때문이다.**
예외적으로 호출자가 Future.get()으로 결과를 기다리다가 ExecutionException을 받아서 다시 던지면 롤백되긴 하는데, 그렇게 블로킹할 거라면 애초에 비동기로 만든 의미가 없다는 모순이 생깁니다. 즉 "비동기 실패가 본 작업을 취소해야 한다"는 요구라면 @Async가 맞는 도구가 아닙니다.

**Q. 비동기 처리가 꼭 필요하다면 Outbox나 MQ를 고려할 수 있을까요?**

**A.** **예 — 기준은 "그 작업이 유실돼도 되는가"다.**
캐시 워밍처럼 유실 허용이면 @Async로 충분합니다. 알림 발송이나 정산 연동처럼 유실 불가면 Outbox + MQ가 맞습니다 — MQ는 브로커에 영속화되어 소비 실패 시 재시도와 DLQ가 가능하지만, @Async의 인메모리 큐는 서버가 죽으면 소멸하기 때문입니다. 트랜잭션 정합성까지 필요하면 Outbox(260618)로 "커밋된 것만 발행"을 보장합니다.

### 🌱 심화 키워드
- **TransactionSynchronizationManager** — 트랜잭션-스레드 바인딩의 실체, 이 주제의 뿌리
- **AsyncUncaughtExceptionHandler** — void 비동기 메서드의 예외가 흘러가는 곳
- **@TransactionalEventListener(AFTER_COMMIT)** — "커밋된 경우에만 후속 작업"의 표준 수단
- **TaskDecorator** — 비동기 스레드로 MDC/컨텍스트를 복사하는 표준 지점 (→ 260624)
- **CompletableFuture 예외 모델** — exceptionally/handle, join 시 CompletionException 래핑

### ❓ 더 파볼 질문
**Q. AFTER_COMMIT 리스너 안에서 DB를 변경하려면 왜 REQUIRES_NEW가 필요한가?**

**A.** AFTER_COMMIT 시점에는 원래 트랜잭션이 이미 커밋을 마친 상태라, 리스너가 기본 전파(REQUIRED)로 그 트랜잭션 리소스에 참여해도 추가 변경이 커밋될 기회가 없다. 그래서 리스너에서 DB 쓰기를 하려면 REQUIRES_NEW로 새 트랜잭션을 열어야 확실히 반영된다(메커니즘은 거의 확실 — 출처 귀속은 확실치 않음).

**Q. CompletableFuture에서 예외는 어떤 경로로 전파되나?**

**A.** 예외는 future 객체 안에 저장되고, get()은 ExecutionException으로, join()은 CompletionException으로 감싸 던진다. 아무도 결과를 조회하지 않으면 예외는 어디에도 나타나지 않으므로, exceptionally()/handle()로 처리 경로를 명시하는 것이 안전하다.

**Q. AFTER_COMMIT + @Async 조합도 서버가 죽으면 유실되는데, 유실 불가 요건은 어떻게 푸나?**

**A.** 이벤트를 메모리로 넘기지 말고 Outbox 테이블에 남기는 것으로 바꾼다. 커밋과 함께 DB에 기록이 남으므로 서버가 죽어도 릴레이가 재기동 후 이어서 발행한다 — "커밋 후 실행"과 "유실 불가"를 동시에 만족하는 조합은 사실상 Outbox뿐이다.
