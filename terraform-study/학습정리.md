# Terraform 학습 정리

> 로컬 Docker 실습 + AWS 예제로 "문법 + 전체 흐름 + 베스트 프랙티스"를 잡은 정리 노트.

---

# Part 1. 핵심 개념 (큰 그림)

## 한 문장
> **나는 "원하는 최종 상태"를 코드(`.tf`)에 적는다 → 테라폼이 "지금 상태"와 비교해 → 차이(diff)만 실제 인프라에 적용한다.**

이게 **선언형(declarative)**. "어떻게"가 아니라 "무엇"만 적는다.
명령형 스크립트(`docker run` 3번 = 컨테이너 3개)와 정반대.

## 삼각 비교 — 테라폼이 평생 하는 일
```
   ① 코드(.tf)        ② 장부(tfstate)        ③ 실제 인프라
  "원하는 상태"   ↔   "내가 아는 상태"    ↔    "진짜 상태"
```
이 셋을 일치시키는 게 전부.

## 핵심 사이클
| 명령 | 하는 일 | 비유 |
|------|---------|------|
| `init` | provider(플러그인) 다운로드. 폴더당 최초 1회 | `npm install` |
| `plan` | 코드 vs 현재 비교 → 무엇이 바뀔지 미리보기 (변경 X) | `git diff` |
| `apply` | diff를 실제 적용 + tfstate 갱신 | `git commit & push` |
| `destroy` | tfstate에 적힌 것 전부 삭제 | 정리/롤백 |

## 기억할 3가지
1. **plan은 미리보기, apply가 실행.** apply 전 항상 plan으로 확인.
2. **tfstate(장부)가 심장.** 테라폼은 이 파일로 "현재"를 안다. 없으면 아무것도 모름.
3. **선언형이라 멱등(idempotent).** 같은 코드 100번 apply해도 결과 동일. 차이 없으면 아무것도 안 함.

# Part 2. 흐름 상세

## 코드 안의 블록 구조
```
terraform { }   ← 어떤 provider를 쓸지 선언      (설정)
provider  { }   ← provider에 어떻게 접속할지       (연결)
variable  { }   ← 입력값                          (IN)
data      { }   ← 기존 리소스를 "조회" (생성 아님) (READ)
resource  { }   ← "이런 리소스가 존재하게 해줘"     (본체) ★핵심
output    { }   ← apply 후 보여줄 결과값           (OUT)
```

## 의존성은 자동
리소스끼리 `<타입>.<이름>.<속성>`으로 참조하면, 테라폼이 순서를 **스스로 계산**한다.
```
docker_image.nginx ──(image_id 참조)──▶ docker_container.web
       먼저 생성                            그 다음 생성
```
→ 순서를 내가 안 적는다. (명령형과의 결정적 차이)

# Part 3. 진입점 / 폴더 / 실행 규칙 (Q&A로 정리)

## main() 같은 시작 함수가 있나?
**없다.** 진입점은 "함수"가 아니라 **`terraform`을 실행한 폴더 = 루트 모듈(root module)**.
- `main.tf`는 그냥 관례. 특별하지 않음. 파일명 바꿔도 동작 동일.
- 폴더 안의 **모든 `.tf`를 통째로 합쳐서** 읽음. import도, 파일 순서도 없음.

| | Spring | Terraform |
|--|--------|-----------|
| 시작점 | `main()` 함수 | `terraform` 친 **폴더**(루트 모듈) |
| 트리거 | `java -jar` | `terraform apply` |
| 파일 순서/import | 있음 | 없음 (폴더 내 `.tf` 자동 병합) |
| 실행 순서 | 내가 지정 | 참조 기반 자동 계산 |

## 어디서 apply하든 같은 결과인가?
- **다른 폴더 = 다른 결과.** 폴더마다 `tfstate`가 따로라 서로의 존재를 모름. 각 폴더가 독립된 세계.
- **같은 폴더 = 같은 결과 (멱등).** 단 전제: state·변수·대상 계정이 같아야 함.
- 같은 폴더라도 결과가 달라지는 경우: state 변경, 변수 변경, 계정/리전 변경, data 소스 실시간 조회 결과 변화, 외부에서 인프라 변경(drift).

> 결과를 결정하는 건 "어느 위치냐"가 아니라 **"어느 폴더 + 그 폴더의 state·변수·계정"**.

## 빈 폴더에서 apply하면?
- "테라폼이 있는 폴더"가 아니라 **".tf 파일이 있는 폴더"**가 정확한 표현. (테라폼 도구는 전역 설치됨)
- `.tf` 없는 빈 폴더 → `Error: No configuration files` (무시가 아니라 **에러로 거부** = 안전장치).
- `git`과 동일: `git`은 어디서든 실행되지만 `.git` 없으면 거부.

## 폴더 안의 폴더를 한 번에 실행하려면? (모듈)
- 하위 폴더 `.tf`는 **자동 포함 안 됨.** `module` 블록으로 **직접 지목**해야 함.
```hcl
module "web_a" {
  source        = "./modules/webserver"  # ← 이 폴더를 써라 (자동 탐색 아님)
  external_port = 9200
}
```
- 루트에서 `apply` 1회 → 지목된 모든 모듈이 **하나의 그래프 + 하나의 tfstate**로 실행됨.
- 모듈 = "입력(variable) 받아 리소스 만들고 결과(output) 돌려주는 함수". 그냥 폴더.

```
infra/  (루트, 여기서 apply)
├── main.tf            module "network"/"database"/"server" 호출
└── modules/
    ├── network/   (VPC .tf)
    ├── database/  (RDS .tf)
    └── server/    (EC2 .tf)
```


# Part 5. AWS 베스트 프랙티스 — 아키텍처 설계 흐름

> `05-aws-ec2`를 "EC2 한 대 만든다"가 아니라 **"인프라를 어떤 순서로 설계하는가"** 관점으로 정리.
> 실무에서 새 인프라를 짤 때 머릿속으로 밟는 순서다.

## 설계 0단계 — 무엇을, 어디에 (요구사항 → 경계 정하기)
"웹서버 1대를 서울 리전에 띄운다"를 먼저 말로 정의.
이게 `variable`로 추출할 후보를 결정한다:
- 변하는 것(리전·사양·접속 IP) → **변수로**
- 안 변하는 구조(보안그룹 규칙 형태·인스턴스 정의) → **코드 본문으로**

```hcl
# variables.tf — "환경마다 다를 수 있는 것"을 입력으로 뽑아냄
variable "region"        { default = "ap-northeast-2" }
variable "instance_type" { default = "t3.micro" }       # 프리티어
variable "my_ip_cidr"    { default = "0.0.0.0/0" }       # SSH 허용 대역
```
**BP:** 하드코딩하지 말고 "바뀔 만한 건 변수로". 환경 분리(dev/prod)의 출발점.

## 설계 1단계 — 어디에 어떻게 접속하나 (Provider & 인증 경계)
```hcl
# provider.tf
terraform {
  required_version = ">= 1.5"
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }  # 버전 고정
  }
}
provider "aws" { region = var.region }
```
**BP:**
- **버전 고정**(`~> 5.0`): provider 업데이트로 갑자기 깨지는 것 방지.
- **자격증명을 코드에 절대 안 적음**: `~/.aws/credentials`·환경변수로 분리. git 유출 = 사고.

## 설계 2단계 — 이미 있는 것을 조회 (Data 소스, 의존성의 시작점)
새로 만들기 전에 "기존 환경에 의존하는 값"을 먼저 조회한다.
```hcl
data "aws_ami" "amazon_linux" { ... }   # 최신 AMI ID (리전·시점마다 다름)
data "aws_vpc" "default"      { default = true }  # 기본 VPC
```
**BP:** AMI ID 하드코딩 금지. 리전마다 다르고 수시로 바뀌므로 **동적 조회**.
→ 이 data들이 아래 리소스가 참조할 "뿌리"가 된다.

## 설계 3단계 — 보안 경계부터 (Security Group)
서버보다 **방화벽(보안그룹)을 먼저** 설계한다. 무엇이 들어오고 나갈 수 있나?
```hcl
resource "aws_security_group" "web" {
  vpc_id = data.aws_vpc.default.id          # ← 2단계 data 참조

  ingress { from_port=22 to_port=22 cidr_blocks=[var.my_ip_cidr] }  # SSH: 내 IP만
  ingress { from_port=80 to_port=80 cidr_blocks=["0.0.0.0/0"] }     # HTTP: 전체
  egress  { from_port=0 to_port=0 protocol="-1" cidr_blocks=["0.0.0.0/0"] }
}
```
**BP:** SSH(22)를 `0.0.0.0/0`로 열지 말 것 → 변수로 본인 IP만. **최소 권한**.

## 설계 4단계 — 본체 (Compute)
이제 서버. 앞 단계 결과물(AMI·보안그룹)을 **참조**해서 조립한다.
```hcl
resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux.id   # 2단계 참조
  instance_type          = var.instance_type
  vpc_security_group_ids = [aws_security_group.web.id]    # 3단계 참조 ★
  user_data              = <<-EOF ... nginx 설치 ... EOF  # 부팅 자동화
  tags = { Name = "${var.project_name}-web", ManagedBy = "terraform" }
}
```
**참조가 곧 설계 순서다:** 인스턴스가 보안그룹을 참조 → 테라폼이 "보안그룹 먼저, 인스턴스 나중"을 자동 결정.
```
data.aws_vpc ─▶ security_group ─▶ instance ◀─ data.aws_ami
```
**BP:**
- **모든 리소스에 태그**: 비용 추적·소유자 식별·일괄 정리. `ManagedBy=terraform`로 "수동 손대지 마" 표시.
- **이름에 prefix 변수**(`${var.project_name}-`): 환경/프로젝트 충돌 방지.
- `user_data`로 부팅 설정 자동화: 서버 안에서 손으로 안 만짐 (불변 인프라 지향).

## 설계 5단계 — 결과 노출 (Outputs)
"이 인프라를 쓰려면 알아야 할 값"만 밖으로 내보낸다.
```hcl
output "public_ip"   { value = aws_instance.web.public_ip }
output "web_url"     { value = "http://${aws_instance.web.public_ip}" }
```
**BP:** 다음 단계(다른 모듈·사람)가 쓸 값만 output. 모듈 간 연결의 인터페이스가 됨.

## 설계 6단계 — 검증 → 적용 (실행 흐름)
```bash
terraform fmt        # 정렬 (저장 습관)
terraform validate   # 문법/참조 검증 (인증 불필요)  ← 05-aws-ec2는 여기까지 통과
terraform plan       # 무엇이 바뀔지 리뷰 (PR에 첨부해 팀 확인)
terraform apply      # 승인 후 적용 (실무는 CI/CD가 실행, 사람이 콘솔 안 만짐)
```

## 설계 ∞단계 — 팀/실무로 확장 (이 예제의 다음 스텝)
| 추가할 것 | 왜 |
|----------|-----|
| **원격 State** (S3 backend + DynamoDB 락) | 팀 공유·동시 apply 방지·유실 방지. 혼자 학습 vs 팀 실무를 가르는 결정적 차이 |
| **module로 추출** | 반복 구성 재사용. network/db/server 분리 |
| **dev/prod tfvars 분리** | 같은 코드 + 다른 값 → 환경 복제 |
| **plan을 PR 리뷰** | "뭐가 바뀌는지" 코드리뷰처럼 검증 |
| **CI/CD 연동** | 사람이 직접 apply 안 함. 승인 기반 자동화 |

## 설계 흐름 한 장 요약
```
0. 요구사항 → 변수로 뽑을 것 결정
1. Provider + 인증 경계  (어디에, 버전고정, 키 분리)
2. Data 조회            (기존 의존성: AMI, VPC)
3. 보안 경계            (Security Group — 서버보다 먼저)
4. 본체                 (EC2 — 앞 결과를 참조해 조립)
5. Outputs              (다음이 쓸 값만 노출)
6. fmt → validate → plan → apply
∞. 원격state·module·환경분리·CI/CD 로 팀 확장
```
**관통하는 원칙:** 바깥(설정·인증) → 의존성 조회 → 보안 → 본체 → 노출.
참조 관계가 곧 설계 순서이고, 그 순서를 테라폼이 자동으로 지켜준다.
