## 1. 왜 배포 전략이 필요한가

새 버전을 올릴 때 가장 단순한 방법은 **"기존 걸 끄고 새 걸 켜는 것"**이다.
하지만 이러면:

- 끄고 켜는 사이 서비스가 멈춘다 (**다운타임**)
- 새 버전에 버그가 있으면 **모든 사용자**가 즉시 피해를 본다

그래서 **다운타임 없이**, **문제 시 빠르게 되돌릴 수 있게(rollback)** 배포하는 방법이 필요하다.
→ 블루-그린 / 카나리 / 롤링.

## 2. 블루-그린 (Blue-Green)

**핵심 아이디어: 똑같은 환경을 두 개 만들어 두고, 통째로 스위치를 넘긴다.**

```
[ 사용자 ] → [ 라우터 ] ──→ 🔵 Blue  (현재 운영 중, v1)
                            🟢 Green (새 버전 v2 배포 + 테스트)

                    ↓ 스위치 전환

[ 사용자 ] → [ 라우터 ] ──→ 🟢 Green (이제 운영 중, v2)
                            🔵 Blue  (대기, 문제 시 복귀용)
```

- Blue = 현재 트래픽을 받는 환경, Green = 새 버전을 올리는 환경 (동일 규모)
- Green에 새 버전을 배포하고 충분히 테스트
- 문제없으면 라우터를 **한 번에** Green으로 전환
- Blue는 그대로 남겨둠 → 문제 시 **라우터만 다시 Blue로** 돌리면 즉시 롤백

| 장점 | 단점 |
|---|---|
| 롤백이 즉각적이고 간단 | 인프라가 2배로 필요 (비용 큼) |
| 전환 전 새 환경을 완전히 테스트 가능 | "전부 아니면 전무" 전환 → 문제 시 전체 사용자 영향 |
| | DB를 공유하면 복잡해짐 |

### 어원

- 용어는 Jez Humble & David Farley의 책 **《Continuous Delivery》(2010)**에서 소개되어 퍼졌다.
- 핵심은 **"색깔 자체에 의미가 없다"**는 점. 두 환경이 완전히 대등하고, "운영 중"이라는 역할이 계속 번갈아 바뀌기 때문에 **우열·순서를 암시하지 않는 중립적인 두 색**이 필요했다.
  - "운영/스테이징", "주/부", "1번/2번" 등은 서열·순서를 암시 → 부적합
- **빨강-초록을 피한 이유**: 빨강/초록은 신호등·테스트 결과(실패/통과)처럼 "나쁨/좋음" 의미가 이미 박혀 있고, 적록색맹이 구분하기 어렵다.

## 3. 카나리 (Canary)

**핵심 아이디어: 새 버전을 소수에게만 먼저 노출하고, 괜찮으면 점점 비율을 늘린다.**

> **이름 유래**: 옛 광부들이 유독가스를 감지하려고 카나리아 새를 데리고 들어갔다. 새가 먼저 위험을 알려주듯, 소수 사용자가 "먼저 위험을 감지"하는 역할.

```
1단계:  v1 95%  |  v2 5%   ← 모니터링
2단계:  v1 75%  |  v2 25%  ← 지표 정상이면 확대
3단계:  v1 50%  |  v2 50%
4단계:  v1 0%   |  v2 100% ← 완료
```

- 새 버전을 일부(예: 5%) 사용자에게만 노출
- 에러율·응답속도 등 지표를 모니터링
- 정상이면 25% → 50% → 100%로 단계적 확대
- 문제 발견 시 그 단계에서 멈추고 롤백 → **영향받은 사용자가 소수에 그침**

| 장점 | 단점 |
|---|---|
| 피해 범위(blast radius) 최소화 | 모니터링·관측 체계가 잘 갖춰져야 함 |
| 실제 운영 트래픽으로 검증 | 배포가 느림 |
| | 두 버전이 오래 동시에 돌아감 |

## 4. 롤링 (Rolling)

**핵심 아이디어: 서버를 몇 대씩 묶어서 순차적으로 새 버전으로 교체한다.**

```
서버 10대를 2대씩 교체:

[v1][v1][v1][v1][v1][v1][v1][v1][v1][v1]
[v2][v2][v1][v1][v1][v1][v1][v1][v1][v1]  ← 2대 교체
[v2][v2][v2][v2][v1][v1][v1][v1][v1][v1]  ← 2대 더
 ...
[v2][v2][v2][v2][v2][v2][v2][v2][v2][v2]  ← 완료
```

- 전체 서버 중 일부만 내려서 새 버전으로 교체, 끝나면 다음 묶음
- 교체 중에도 나머지 서버가 트래픽을 받으므로 다운타임 없음
- 추가 인프라 불필요 (기존 서버를 그대로 돌려씀)

| 장점 | 단점 |
|---|---|
| 추가 비용 없음 | 롤백도 순차적이라 느림 |
| 쿠버네티스 등에서 기본 제공, 구현이 쉬움 | 배포 중 두 버전이 섞여서 돌아감 |
| | "비율 제어"는 불가 |

## 5. 세 전략 한눈에 비교

| 항목 | 블루-그린 | 카나리 | 롤링 |
|---|---|---|---|
| 추가 인프라 비용 | 높음 (2배) | 약간 | 없음 |
| 롤백 속도 | 매우 빠름 | 빠름 | 느림 |
| 피해 범위 제어 | 안 됨 (전부 전환) | 우수 (비율 조절) | 안 됨 |
| 두 버전 공존 | 거의 없음 | 있음 (길다) | 있음 |
| 모니터링 의존도 | 보통 | 높음 | 보통 |

## 6. 롤링 vs 카나리 — 무엇이 다른가

둘 다 점진적이라 헷갈리지만, **목적이 다르다.**

> **롤링은 "교체 작업", 카나리는 "검증 작업"이다.**

- **롤링**: 목적은 다운타임 없이 버전을 다 갈아끼우는 것
- **카나리**: 목적은 새 버전이 진짜 괜찮은지 확인하는 것

비유:
- 롤링 = **울타리 페인트칠** — 판자 한 장씩 칠하고, 목표는 끝까지 다 칠하는 것
- 카나리 = **새 메뉴 시식** — 한 테이블에 먼저 맛보게 하고 반응을 본 뒤 전체 메뉴에 올림

### 진행을 멈추고 판단하느냐 (가장 큰 차이)

```
롤링:   v2 20% → 40% → 60% → 80% → 100%
        (헬스체크만 통과하면 자동으로 끝까지 진행)
        판단 기준: "새 서버가 떴는가?"

카나리: v2 5% → ⏸️관찰 → 25% → ⏸️관찰 → 50% → ⏸️관찰 → 100%
        (각 단계에서 멈춰 지표 보고 go/no-go 결정)
        판단 기준: "에러율·응답속도·매출 지표가 정상인가?"
```

### 비교표

| | 롤링 | 카나리 |
|---|---|---|
| 목적 | 버전 교체 | 버전 검증 |
| 진행 방식 | 헬스체크 통과 시 자동으로 끝까지 | 단계마다 멈춰 지표 보고 결정 |
| 판단 기준 | 서버가 떴는가 | 품질 지표가 정상인가 |
| 대상 지정 | 불가 (무작위) | 가능 (내부 직원/특정 지역/베타 신청자 등) |
| 문제 시 피해 | 진행된 만큼 | 소수로 한정 |

> **둘은 배타적이지 않다.** 실무에선 "카나리로 5%만 먼저 검증 → 통과하면 나머지는 롤링으로 쭉 교체"처럼 섞어 쓰는 경우가 많다.

## 7. DB 스키마 변경의 영향

### 왜 공통적으로 문제가 되나

세 전략 모두 배포 도중에는 **구버전(v1)·신버전(v2) 앱이 동시에 돌아가는데, DB는 보통 하나를 공유**한다.

```
   🔵 v1 앱  ──┐
              ├──→  [ 같은 DB 하나 ]
   🟢 v2 앱  ──┘
```

→ 스키마는 **v1·v2 양쪽 모두와 호환**돼야 한다.

**핵심 비대칭**:

> 앱 롤백은 싸고 안전하지만, **DB 롤백은 위험하다.**
> 앱은 이전 이미지로 다시 띄우면 끝이지만, `DROP COLUMN` 후엔 사라진 데이터를 복구할 수 없다.

### 전략별 영향

- **블루-그린**: 가장 뼈아프다. 파괴적 스키마 변경을 하는 순간, 라우터를 Blue로 되돌려도 Blue(v1)가 사라진 컬럼을 찾다가 깨진다 → 간판 기능인 "즉시 롤백"이 거짓말이 됨.
- **카나리**: v1·v2가 가장 오래 공존 → 그 기간 내내 호환 필요. 호환 요구가 제일 길다.
- **롤링**: 카나리와 같은 문제, 공존 기간이 짧을 뿐.

### 해결책: Expand/Contract 패턴 (확장-수축)

핵심 원칙: **스키마 변경을 앱 배포에서 분리하고, 여러 단계로 쪼갠다.**

예시 — `name` 컬럼을 `full_name`으로 **이름 변경**:

❌ **나쁜 방법** (한 방에):
```sql
ALTER TABLE users RENAME COLUMN name TO full_name;
-- 실행 순간 name을 조회하던 v1 앱이 전부 깨짐
```

✅ **좋은 방법** (Expand → Migrate → Contract):
```
1. Expand(확장):  full_name 컬럼을 새로 추가 (기존 name은 그대로 둠)
                  → 추가만 했으니 v1도 v2도 안 깨짐
2. Migrate(이행): v2 앱 배포 — 읽기는 full_name,
                  쓰기는 name·full_name 양쪽 모두에
                  → 기존 데이터는 백필(backfill)로 복사
3. (대기)         모든 v1 인스턴스가 사라지고 v2가 안정적인지 확인
4. Contract(수축): name 컬럼을 DROP
                  → 이제 아무도 name을 안 쓰니 안전
```

→ 어느 단계에서 멈춰도, 어느 단계로 롤백해도 깨지지 않는다.

### 기억할 원칙 3가지

1. **DB 마이그레이션과 앱 배포를 분리**한다 — "컬럼 삭제 + 앱 배포"를 한 릴리스에 묶지 말 것
2. **추가는 자유, 삭제·변경은 단계적으로** — `ADD COLUMN`은 안전, `DROP`/`RENAME`은 expand/contract로 쪼갠다
3. **스키마는 항상 앞 단계 호환(backward compatible)** — 구버전 앱이 한 대라도 살아있으면 그 앱이 쓰는 스키마를 건드리지 않는다

## 8. 버전 이격(Version Skew) 문제

> **버전 이격은 배포 전략이 "해결"해 주는 게 아니다.**
> 무중단 배포를 하는 한 **반드시 생기는 부산물**이며, 전략은 이격이 얼마나 오래·크게 생기는지 그 **"모양"만** 결정한다.
> 이격은 전략의 문제가 아니라 **애플리케이션 설계의 문제**다.

### 전략은 이격의 "모양"만 바꾼다

| | 이격 지속 시간 | 특징 |
|---|---|---|
| 블루-그린 | 짧음 (전환은 순간) | 단, 롤백 가능 기간 동안은 v1을 살려둬야 함 |
| 카나리 | 가장 김 (수 시간~수일) | 의도적으로 오래 공존 → 호환 요구 최강 |
| 롤링 | 중간 | 교체 진행 내내 v1·v2 무작위로 섞임 |

### 이격은 DB만의 문제가 아니다

v1·v2가 닿는 **모든 경계면**에서 생긴다:

- **DB 스키마**
- **서비스 간 API** — 배포 중인 서비스를 다른 서비스가 호출
- **프론트 ↔ 백엔드** — 브라우저에 캐싱된 구버전 JS가 새 백엔드를 호출 (큰 함정)
- **비동기 메시지/큐** — v1이 넣은 메시지를 v2가 꺼내 읽음 (또는 반대)
- **캐시·세션** — Redis 등에 저장된 데이터 포맷이 버전마다 다름

### 진짜 해결책 — 호환성 규율 (전략과 무관)

핵심 원칙: **모든 경계면을 양방향 호환되게 설계한다.**

| # | 기법 | 설명 |
|---|---|---|
| ① | Expand/Contract | 추가만 하고, 삭제·변경은 단계적으로 (모든 경계면에 적용) |
| ② | Tolerant Reader | 받는 쪽은 모르는 필드가 와도 무시, 없는 필드는 기본값 처리 |
| ③ | N-1 호환 규칙 | 모든 버전은 바로 직전 버전과 반드시 호환되게 설계 |
| ④ | 피처 플래그 | 코드 배포와 기능 활성화를 분리 |
| ⑤ | 백엔드의 구 프론트 지원 | 캐싱된 구버전 프론트 요청도 한동안 받아줌 (API 버저닝 등) |
| ⑥ | 세션 일관성 | 스티키 세션으로 한 사용자를 한 버전에 고정, 또는 무상태(stateless) 설계 |

**④ 피처 플래그 — 코드 배포와 기능 활성화 분리:**
```
1. v2 코드를 배포 (새 기능은 플래그 OFF) → 동작은 v1과 동일, 이격 없음
2. 모든 인스턴스가 v2 코드를 갖게 됨
3. 그 다음 플래그를 ON → 모두가 동시에 새 동작으로 전환
```
"코드가 퍼지는 순간"과 "동작이 바뀌는 순간"을 떼어놓으면, 이격 구간엔 동작이 동일해서 안전하다.

## 9. 핵심 요약

- **블루-그린**: 똑같은 환경 두 개를 두고 통째로 스위치 전환. 즉시 롤백이 강점, 인프라 2배가 약점.
- **카나리**: 새 버전을 소수에게만 노출 → 지표 보고 단계적 확대. 피해 범위 최소화가 강점, 모니터링 의존이 약점. **목적은 "검증"**.
- **롤링**: 서버를 몇 대씩 순차 교체. 추가 비용 없음이 강점, 느린 롤백이 약점. **목적은 "교체"**.
- **DB 스키마 변경**: 어느 전략을 쓰든 따라오는 숙제. 답은 전략 변경이 아니라 **Expand/Contract로 변경 자체를 안전하게 쪼개는 것**.
- **버전 이격**: 무중단 배포의 필연적 부산물. 전략은 이격의 지속 시간만 정할 뿐, 실제 해결은 **모든 경계면을 N-1 호환으로 설계하고, 위험한 변경을 단계적으로 쪼개고, 코드 배포와 기능 활성화를 피처 플래그로 분리하는 애플리케이션 규율**이다.
