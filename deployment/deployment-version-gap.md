# 배포 전략과 버전 이격(Version Skew) 문제

## 개요

배포 전략 세 가지(Rolling / Blue-Green / Canary) 중 무엇을 쓰든, **구/신 버전이 동시에 떠 있는 순간은 무조건 존재**한다. Blue-Green조차 스위칭 직전엔 공존한다. 차이는 공존 기간의 길이뿐이며, 그 사이에 발생하는 **이격(skew) 문제**를 어떻게 다루느냐가 핵심이다.

---

## 1. 이격이 발생하는 지점

| 영역 | 이격 사례 |
|------|----------|
| **API 스키마** | 신버전이 응답 필드명을 변경 → 구버전 클라이언트/내부 서비스가 못 읽음 |
| **DB 스키마** | 신버전이 NOT NULL 컬럼 추가 → 구버전 인스턴스의 INSERT 실패 |
| **메시지 큐 페이로드** | 구버전이 발행한 메시지를 신버전이 못 읽거나 반대 케이스 |
| **세션/캐시 직렬화** | Redis 등에 저장된 객체 포맷 변경 → 역직렬화 실패 |
| **상태 일관성** | 같은 사용자가 요청마다 다른 버전에 라우팅 → 화면 깜빡임, 동작 차이 |

---

## 2. 해결 원칙

### 2.1 N-1 하위 호환성 (가장 중요)

> 신버전은 항상 구버전과 호환되게 배포한다.

- API 필드는 **추가만**, 삭제는 다음 릴리스로 미룸
- 응답 포맷 변경 시 새 필드 추가 + 구 필드 한동안 유지
- 클라이언트는 모르는 필드를 무시 (예: `@JsonIgnoreProperties(ignoreUnknown=true)`)

### 2.2 Expand-Contract 패턴 (DB 스키마 변경 필수)

한 배포에 끝내려 하면 거의 100% 터진다. 보통 3~4 릴리스로 쪼갠다.

```
릴리스 1: 새 컬럼 추가 (nullable)              → 구버전 영향 없음
릴리스 2: 신버전이 양쪽(구 + 신 컬럼)에 쓰기
릴리스 3: 백필 수행 + 신버전이 신 컬럼만 읽기
릴리스 4: 구 컬럼 DROP
```

### 2.3 Feature Flag

- 코드는 미리 배포, 기능 활성화는 플래그로 분리
- **배포(Deploy)와 릴리스(Release)를 분리**하는 개념
- 이격 발생 시 플래그를 꺼서 즉시 롤백 가능

### 2.4 API 버저닝

- 변경 폭이 크면 `/api/v1`, `/api/v2`를 병행 운영
- 구버전 클라이언트가 모두 마이그레이션된 후 v1 제거

### 2.5 Sticky Session (보조 수단)

- 같은 사용자는 같은 버전 인스턴스로 라우팅 → 단기 화면 깜빡임 방지
- **서버 간 통신/DB 이격은 막지 못함** → 어디까지나 보조

---

## 3. 배포 전략별 비교

| 전략 | 공존 기간 | 이격 리스크 | 주의점 |
|------|----------|------------|--------|
| **Blue-Green** | 매우 짧음 (스위칭 순간) | 낮음 | DB는 공유되므로 스키마 이격은 여전히 발생 |
| **Rolling** | 중간 (수 분~수십 분) | 중간 | 하위 호환성 필수 |
| **Canary** | 길다 (수 시간~수 일) | 가장 높음 | 하위 호환성 + 정밀 모니터링 필수 |

---

## 4. 결론

- 배포 전략은 **공존 시간을 조절할 뿐**, 이격 자체를 없애지 않는다.
- 핵심 원칙: **"신버전 코드 = 구버전과 호환되는 코드"** 를 코드/DB 양쪽에 강제할 것.
- DB 스키마 변경은 항상 Expand-Contract로 단계화.
- Feature Flag로 배포와 릴리스를 분리하면 운영 안정성이 크게 올라간다.

## ❓ 남은 질문

1. Expand-Contract에서 롤백이 필요해지면, 이미 진행한 단계는 어떻게 되돌리나?

   → **답:** 각 단계가 하위 호환이라 코드는 이전 버전으로 되돌리면 되지만, 컬럼 DROP처럼 파괴적인 스키마 변경(Contract)은 되돌릴 수 없으므로 마지막 단계로 미루고 앞 단계 안정화를 충분히 확인한 뒤 진행한다.

2. Blue-Green도 DB를 공유한다면, 스위칭 후 문제가 생겨 Green→Blue로 되돌릴 때 무엇이 위험한가?

   → **답:** Green이 실행한 스키마 변경이나 데이터 쓰기가 이미 공유 DB에 남아 있어, 구버전 Blue가 그 데이터를 못 읽거나 잘못 처리할 수 있다. 그래서 DB 변경도 앞뒤 버전 모두와 호환되게(하위 호환) 만들어야 트래픽 롤백이 안전하다.

3. Canary에서 신버전 오류율만으로 판단하기 어려운 이격 문제는 어떻게 조기 발견하나?

   → **답:** 오류율뿐 아니라 지연시간, 비즈니스 지표(주문 성공률 등)를 구/신 버전별로 분리 측정하고, 역직렬화 실패나 큐 소비 지연 같은 이격 특유의 신호를 함께 모니터링한다.
